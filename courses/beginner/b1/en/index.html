B1 · What is an AI agent

Not autocomplete and not a chatbot — a worker that does things itself and shows you what it did.

Why this is the first chapter

Hey folks, welcome 👋

If you stepped into AI in 2026 and keep hearing the word “agent” — it’s normal that your head is a mush. Chatbot, assistant, agent, copilot, autocomplete — they all do something to code, and from the outside it looks like there’s no difference.

There is a difference, and it’s huge. If you don’t catch it, your first month with these tools will go in “it did something, and I have no idea what” mode. Personally that drove me up the wall in my own start, so let’s begin with the basics.

By the end of this chapter you can tell three things apart that beginners constantly mix up:

  1. A search engine — finds. You read.
  2. A chatbot — answers. You read and decide.
  3. An agent — does. You look at what it did and accept or roll back.

That third thing is the topic of the entire course.

Thirty seconds on the difference

Imagine your sink is broken.

  • Search engine — that’s a YouTube query “how to fix a sink.” You get links, the rest is on you.
  • Chatbot — that’s a neighbour plumber explaining over the phone what to unscrew, in what order. You hold the wrench, he talks.
  • Agent — that’s the neighbour plumber who showed up with the wrench, unscrewed everything, swapped the gasket, screwed it back, and says “done, check it doesn’t leak.”

Feel the difference? In the first two cases you do the work. In the third — he does. Your job in the third case is to check that he actually fixed it, didn’t make it worse.

With an AI agent it’s exactly the same. It opens files itself, writes code itself, runs commands itself, commits itself. Your job is to set a task, draw the boundaries, and then verify the result.

Minimum diagram: plan → act → evaluate

Any decent agent always has three phases:

flowchart LR
    A([Task from you]) --> P[PLAN<br/>agent describes<br/>what it will do]
    P -->|you approve| D[ACT<br/>agent changes files,<br/>runs commands]
    D --> C[EVALUATE<br/>match result<br/>against the plan]
    C -->|ok| OK([Accept])
    C -->|nope| R([Roll back and clarify])
    R -.-> P

In one line: plan → act → evaluate.

  1. Plan. The agent states what it’s about to do, before it changes anything.
  2. Act. The agent executes the plan — in a clearly bounded zone (folder, branch, sandbox).
  3. Evaluate. The agent (or you) compares the result with what was planned.

Sounds simple. In practice, most beginner mistakes are skipping phase 1 or phase 3:

  • Skipped the plan — the agent ran off to do things, five minutes later you have 40 changed files and no idea what happened.
  • Skipped the evaluation — the agent said “done,” you pushed to the repo, an hour later you discover it deleted half the tests.

In this course we learn from day one to never skip any of the three phases. It’s not paranoia — it’s taste and discipline, and as we’ll say many times, those matter more than any tool.

🥷 Vibe Samurai asks: “If the agent is so smart, why babysit it?”

Answer: Smart ≠ context-aware. The agent will cheerfully refactor a file that was holding all the logic. It’s not malicious — it just doesn’t know that file is special. Evaluating is how you stay the senior engineer of your own project, even if you don’t yet call yourself an engineer.

Why the agent is a “worker,” not a “manual”

If you’re used to Stack Overflow or to ChatGPT in a browser — there’s a familiar pattern: you asked, it answered, you copy-pasted.

With an agent it’s different. The agent doesn’t hand back text — it hands back changes. In files. In your folder. Right now.

Which means:

  • If you give it access to everything — it will work on everything.
  • If you give it prod credentials — it can take down prod.
  • If you ask it to “delete the extra stuff” — it will delete what it considers extra.

That’s not a scare story, that’s working reality. Think of the agent as a new employee on probation. You wouldn’t give that person prod creds on day one, right? Don’t give them to the agent either. First — sandbox, small task, verification, trust.

The full kit of a modern agent

When I say “modern agent” — I mean tools like Claude Code, Cursor, Codex CLI, Devin. They all have roughly the same kit:

  • Read files in your folder
  • Write and edit files
  • Run commands (tests, build, git)
  • Google things and read the web
  • Remember what happened in this session (and sometimes across sessions)

So it’s not a text chatter. It’s a full worker with hands (files, commands), eyes (your code, your logs), a voice (replies in the terminal), and memory (session context).

From which the main thing follows: the safety boundary is not the agent — it’s you. You decide which folder it works in, which commands it can run, which files it sees, what it absolutely doesn’t touch.

Your first agentic cycle

No theorising. Open a terminal and ask Claude Code (or Cursor, or Codex CLI) to do the safest possible thing — read your folder and tell you what it sees:

claude "Read every file in this folder and write a one-paragraph summary. Do not change anything."

That command is your “look, don’t touch” mode. No changes, no commands, just reading.

Read what the agent wrote. If it understood your project — great, you can move on. If it understood wrong — also great, now you see where its blind spots are, and you can give it context next time.

This habit — opening a session with “look, don’t touch” — will stay with you the whole course. Not perfect, but actually works.

Recap (what to take away)

  • Agent ≠ chatbot. It doesn’t hand back text, it makes changes in your files.
  • Plan → Act → Evaluate — three phases you can’t skip.
  • The safety boundary is you, not the tool itself. Sandbox, small task, verification.
  • Start every session with “look, don’t touch” — that’s your training wheels.
