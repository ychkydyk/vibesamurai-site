B6 · Reading the code the agent wrote

Basic orientation in the code the agent created. Not 'learn to code' — stop being blind in your own project.

Hook

WITHOUT orientation:             WITH orientation:

"Something in JS,         →     index.html — page structure
 I don't get it..."              style.css  — how it looks
                                 app.js     — what happens on click
                                                ↑
                                       roughly clear

Hey folks, welcome 👋

This chapter is not about learning to program. And it’s not “read Cormen and you’ll get your project.” It’s about basic orientation: you open a file, see code, and roughly understand what this thing does. Not line by line, not perfectly — but you understand where the logic lives, where the styling lives, and where to look if something broke.

Twenty to thirty minutes. After it you won’t be afraid to open files the agent wrote.

Concept: orientation ≠ programming

Imagine you moved into a new flat. To live there you don’t need to know how the wiring inside the walls works. You need to know: that’s the switch, that’s the socket, that’s the meter. If a fuse blew — you know where to look.

Same with code. To work with an agent you don’t need every language construct. You need to know:

┌─────────────────────────────────────────────────────┐
│  1. WHICH FILE IS RESPONSIBLE FOR WHAT              │
│  2. WHERE IN THE FILE IS LOGIC, WHERE IS STYLING    │
│  3. HOW TO LOCATE A SPECIFIC FEATURE IN THE CODE    │
└─────────────────────────────────────────────────────┘

That’s enough to:

  • Tell the agent “fix this specific chunk,” not “fix it somewhere”
  • Notice if the agent touches things it shouldn’t
  • Roughly understand a console error

Minimum orientation by file types

Most learning projects consist of a few typical file types. Memorise on “this is about what”:

ExtensionWhat’s insideWhy
.htmlPage structureWhat’s on the page — heading, buttons, form, list
.cssStylingColours, fonts, spacing, sizes
.jsLogicWhat happens on actions — click, input, reload
.jsonData/configDependency list, configs, saved state
.mdText for humansREADME, documentation, notes

That’s the bare minimum. If a project has .html, .css, .js — you already know which to look in for what.

A couple more important files

  • package.json — list of dependencies (what the project uses) and run scripts. If it doesn’t start, this is usually checked first.
  • README.md — project instructions. Decent agents write it. If present — read it; “how to run” and “what is this” are there.
  • .gitignore — list of what not to store in git. Usually node_modules/, .env, temp files.

Example: looking inside the todo app from B5

Open the my-todo/ folder in a text editor. E.g. VS Code — free, on every OS, the agent loves it.

You see:

my-todo/
├── index.html
├── style.css
├── app.js
└── README.md

From the names it’s already roughly clear — structure, style, logic, description.

Round 1. Peek at index.html

Opened — see something like:

<!DOCTYPE html>
<html lang="en">
<head>
  <title>My todo list</title>
  <link rel="stylesheet" href="style.css">
</head>
<body>
  <h1>My todo list</h1>
  <input id="task-input" placeholder="New task">
  <button id="add-btn">Add</button>
  <ul id="task-list"></ul>
  <script src="app.js"></script>
</body>
</html>

Don’t try to understand every line. Just glance and pick out the familiar parts:

  • <title>My todo list</title> — browser tab title
  • <h1>My todo list</h1> — big heading on the page
  • <input ...> — input field where you type a task
  • <button ...>Add</button> — that button
  • <ul id="task-list"> — empty list where tasks get added
  • <link rel="stylesheet" href="style.css"> — “pull styling from style.css”
  • <script src="app.js"></script> — “pull logic from app.js”

See? You already can point and say — “here’s the heading, here’s the button.” That’s the full package for our task.

Round 2. Peek at style.css

body {
  font-family: sans-serif;
  max-width: 600px;
  margin: 40px auto;
}

button {
  background: #4CAF50;
  color: white;
  border: none;
  padding: 8px 16px;
}

.done {
  text-decoration: line-through;
  color: gray;
}

Same logic — recognise the familiar:

  • font-family: sans-serif — what font
  • background: #4CAF50 — colour (green)
  • text-decoration: line-through — strikethrough text
  • .done — about a completed task

Again, without details — you know this is styling, and if you want to change the button colour — it’s here.

Round 3. Peek at app.js

Scariest by looks, same logic. Look at comments and names:

const input = document.getElementById('task-input');
const addBtn = document.getElementById('add-btn');
const list = document.getElementById('task-list');

addBtn.addEventListener('click', () => {
  // what happens when "Add" is clicked
  const text = input.value;
  if (!text) return;
  addTask(text);
  input.value = '';
  saveTasks();
});

function addTask(text) {
  // create a new list element
  ...
}

function saveTasks() {
  // save to localStorage
  ...
}

What we understand without knowing JS:

  • addBtn.addEventListener('click', ...) — “when addBtn is clicked, do this”
  • addTask(text) — function that adds a task
  • saveTasks() — function that saves
  • Names speak for themselves — addTask, saveTasks, taskInput

So later when you want to change something in saving — you know to dig into saveTasks in app.js.

Trick #1: ask the agent for an explainer

The most underrated move for a newcomer. When a file doesn’t make sense — ask the agent to explain:

“Explain in three sentences what app.js does. No code, plain words, as for someone who’s never seen JavaScript.”

The agent says something like:

“This file implements the todo app’s logic. When you click ‘Add,’ it takes the text from the input, creates a new list element, and saves all tasks in browser memory (localStorage). Reloading the page doesn’t lose tasks — they’re loaded back from that memory.”

That’s it, you have a file map. Not perfect, but enough.

Trick #2: “show me where X”

When something needs changing and you don’t know where it lives, don’t go hunting — ask:

“Which file and which line has the code that saves tasks to localStorage? Show me the chunk itself.”

The agent shows the exact spot. Then you either ask it to change there, or look with your own eyes to confirm that’s what’s changing.

Trick #3: track what the agent touches

In Claude Code after every change you see which files were modified. That’s critical to avoid missing “accidental” edits.

If you asked “change the button colour” and the agent touches app.js — stop, something’s off. Why the logic? Ask:

“Why did you change app.js? I only asked for the button colour. Revert that change.”

It’s normal — to re-ask. It’s not “distrust of the agent,” it’s basic hygiene.

What to do when “nothing makes sense at all”

Case 1. Opened a file, 500 lines, head spinning

Don’t panic. Nobody reads 500 lines in a row. Do this:

  1. Ask the agent: “give me a table of contents for this file — what functions, what each does, one line each”
  2. Find the one for what you care about
  3. Ask to show just that one

Case 2. You see a function name — don’t know what it means

Ask the agent: “what does setInterval mean here, in plain words?” Not a stupid question. The best way to learn — ask in context on a live example.

Case 3. Console error — heap of red text

Copy the first error (usually the most important) and ask:

“This error appeared in the browser console: <paste>. What does it mean and where to fix it?”

The agent shows file, line, and usually proposes the fix.

Practice (20–30 minutes)

Task 1 (15 minutes) — explainers

Open your my-todo from B5 (or any folder with agent code). For each file ask the agent:

“Explain in three sentences what this file does. No code, plain words.”

Rewrite the answers in your own words into a notebook or README. Not for show — the process itself cements understanding.

Task 2 (15 minutes) — find the feature

Open app.js (or another logic file). Find the function for one concrete feature — e.g. “delete a task” or “save to localStorage.” Do two things:

  1. Say out loud (or write) — what this function does step by step, in your words
  2. Cross-check with the agent: “function <name> — do I understand correctly that it works like this?”

If yes — great, you just read real code. If not — the agent corrects, you extend your understanding.

Recap

  • Orientation ≠ programming. Knowing “which file does what” is already 80 % of the benefit.
  • Extensions speak for themselves: .html — structure, .css — styling, .js — logic, .json — config, .md — text.
  • Top trick — ask the agent for explainers, don’t “learn the language.” In context, on a live example, bit by bit.

Vibe Samurai

🥷 Vibe Samurai asks: “What if I just don’t want to read code at all? I’m not a programmer — I thought the agent does it all.”

Answer: Nobody’s asking you to read code line by line. But if you never open your files, something interesting happens: you become fully dependent on the agent, which sometimes doesn’t know what it did either. Minimum orientation is your control over your own project. Like driving a car without knowing where the hood is: you can drive, but if something rattles — you’re helpless. Not perfect, but actually works — even minimal code reading turns you from passenger into driver.

📚 Further reading

Reading other people’s code is a skill trained separately from writing. Below — about that skill and references nice to use when you “didn’t get a word.”

Articles on code reading

  • “How to Read Code Written by Someone Else” (Towards Data Science) — short walkthrough of techniques: trace from the output, find tests, use “Go to definition”: towardsdatascience.com/the-most-efficient-way-to-read-code-written-by-someone-else.
  • “How to read someone else’s code!” (MIT Media Lab, classic short note by Henry Lieberman): web.media.mit.edu/~lieber/Teaching/IISDW/Read-Code.html. Old but honest — teaches “don’t try to understand everything at once.”

References

  • MDN — JavaScript Guide (if the project is JS — the only reference worth keeping in a tab): developer.mozilla.org/en-US/docs/Web/JavaScript/Guide.
  • MDN — Window.localStorage (a typical thing in learning projects: “how the agent saves state in the browser”): developer.mozilla.org/en-US/docs/Web/API/Window/localStorage.
  • VS Code — Getting Started (how to use “Go to definition,” symbol search, and why they matter when reading code): code.visualstudio.com/docs/getstarted/getting-started.

Tip: habit — see an unfamiliar word in code, immediately look it up on MDN or ask the agent to explain “in plain words.” In a month a vocabulary builds itself.
