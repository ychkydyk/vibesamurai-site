B6 · Чтение кода, который написал агент

Базовая ориентация в коде, который агент создал. Не «научиться кодить», а перестать быть слепым в своём проекте.

Черновик B6 в голосе DmitriyM (см. scripts/tg_voice_analysis/_summary.md).

Hook

БЕЗ ориентации:                  С ориентацией:

«Тут что-то на JS,        →     index.html — структура страницы
 я не понимаю...»                style.css  — как это выглядит
                                 app.js     — что происходит при клике
                                                ↑
                                       примерно понятно

Друзья, всем привет! 👋

Эта глава — не про то, чтобы научиться программировать. И это не «прочитайте Кормена и поймёте свой проект». Это про базовую ориентацию: открываете файл, видите код, и примерно понимаете, что эта штука делает. Не построчно, не идеально — но понимаете, в каком месте лежит логика, в каком — оформление, и куда смотреть, если что-то сломалось.

Двадцать-тридцать минут. После главы вы не будете бояться открывать файлы, которые написал агент.

Концепция: ориентация ≠ программирование

Представьте, что вы пришли в новую квартиру. Чтобы там жить, не обязательно знать, как устроена электропроводка в стенах. Достаточно понимать: вот выключатель, вот розетка, вот счётчик. Если перегорело — знаете, куда смотреть.

С кодом то же самое. Чтобы работать с агентом, не обязательно знать каждую конструкцию языка. Достаточно понимать:

┌─────────────────────────────────────────────────────┐
│  1. КАКОЙ ФАЙЛ ЗА ЧТО ОТВЕЧАЕТ                      │
│  2. ГДЕ В ФАЙЛЕ ЛОГИКА, ГДЕ ОФОРМЛЕНИЕ              │
│  3. КАК НАЙТИ КОНКРЕТНУЮ ФИЧУ В КОДЕ                │
└─────────────────────────────────────────────────────┘

Этого хватит, чтобы:

  • Скзать агенту «вот в этом куске поправь», а не «поправь где-нибудь»
  • Заметить, если агент трогает то, что не должен
  • Понять, что значит ошибка из консоли — хотя бы примерно

Минимальная ориентация по типам файлов

Большинство учебных проектов состоит из нескольких типичных типов файлов. Запомните на уровне «это про что вообще»:

РасширениеЧто внутриЗачем
.htmlСтруктура страницыЧто есть на странице — заголовок, кнопки, форма, список
.cssОформлениеЦвета, шрифты, отступы, размеры
.jsЛогикаЧто происходит при действиях — клик, ввод, перезагрузка
.jsonДанные/настройкиСписок зависимостей, конфиги, сохранённое состояние
.mdТекст для людейREADME, документация, заметки

Это минимум миниморум. Если в проекте есть .html, .css, .js — вы уже знаете, в каком из них что искать.

Ещё пара важных файлов

  • package.json — список зависимостей (что проект использует) и скриптов запуска. Если что-то не запускается, обычно смотрят сюда первым.
  • README.md — инструкция по проекту. Хорошие агенты её пишут. Если есть — прочитайте, там есть «как запускать» и «что это вообще такое».
  • .gitignore — список того, что не сохранять в git. Обычно туда попадают node_modules/, .env, временные файлы.

Пример: смотрим в todo-приложение из B5

Открываем папку my-todo/ в текстовом редакторе. Например, VS Code — бесплатный, есть на всех OS, агент его любит.

Видим:

my-todo/
├── index.html
├── style.css
├── app.js
└── README.md

Уже из имён всё примерно понятно — структура, стиль, логика, описание.

Раунд 1. Заглянем в index.html

Открыли — увидели что-то вроде:

<!DOCTYPE html>
<html lang="ru">
<head>
  <title>Мой todo-лист</title>
  <link rel="stylesheet" href="style.css">
</head>
<body>
  <h1>Мой todo-лист</h1>
  <input id="task-input" placeholder="Новая задача">
  <button id="add-btn">Добавить</button>
  <ul id="task-list"></ul>
  <script src="app.js"></script>
</body>
</html>

Не пытайтесь понять каждую строку. Просто посмотрите глазами и узнайте знакомые куски:

  • <title>Мой todo-лист</title> — заголовок в браузере, видели сверху вкладки
  • <h1>Мой todo-лист</h1> — большой заголовок на странице
  • <input ...> — поле ввода, куда пишете задачу
  • <button ...>Добавить</button> — та самая кнопка
  • <ul id="task-list"> — пустой список, куда добавляются задачи
  • <link rel="stylesheet" href="style.css"> — «подключи оформление из style.css»
  • <script src="app.js"></script> — «подключи логику из app.js»

Видите? Вы уже можете показать пальцем — «вот тут заголовок, вот тут кнопка». Это полный фарш для нашей задачи.

Раунд 2. Заглянем в style.css

body {
  font-family: sans-serif;
  max-width: 600px;
  margin: 40px auto;
}

button {
  background: #4CAF50;
  color: white;
  border: none;
  padding: 8px 16px;
}

.done {
  text-decoration: line-through;
  color: gray;
}

Логика та же — узнавайте знакомое:

  • font-family: sans-serif — какой шрифт
  • background: #4CAF50 — цвет (зелёный)
  • text-decoration: line-through — зачёркнутый текст
  • .done — это про задачу, которая выполнена

Опять, не вдаваясь в детали — вы понимаете, что тут оформление, и если хочется поменять цвет кнопки — это сюда.

Раунд 3. Заглянем в app.js

Самое страшное по виду, но логика та же. Смотрим на уровне комментариев и имён:

const input = document.getElementById('task-input');
const addBtn = document.getElementById('add-btn');
const list = document.getElementById('task-list');

addBtn.addEventListener('click', () => {
  // что происходит при клике на «Добавить»
  const text = input.value;
  if (!text) return;
  addTask(text);
  input.value = '';
  saveTasks();
});

function addTask(text) {
  // создаём новый элемент списка
  ...
}

function saveTasks() {
  // сохраняем в localStorage
  ...
}

Что мы здесь понимаем без знания JS:

  • addBtn.addEventListener('click', ...) — «когда кликают на addBtn, делать это»
  • addTask(text) — функция, которая добавляет задачу
  • saveTasks() — функция, которая сохраняет
  • Имена говорят сами за себя — addTask, saveTasks, taskInput

И если потом захочется поменять что-то в сохранении — вы знаете, что лезть надо в saveTasks в app.js.

Лайфхак №1: попросите агента сделать explainer

Самый недооценённый приём для новичка. Когда не понимаете файл — попросите агента объяснить:

«Объясни мне в трёх предложениях, что делает файл app.js. Без кода, простыми словами, как для человека, который раньше никогда не видел JavaScript.»

Агент выдаст что-то вроде:

«Этот файл реализует логику todo-приложения. Когда вы нажимаете «Добавить», он берёт текст из поля ввода, создаёт новый элемент списка и сохраняет все задачи в браузерной памяти (localStorage). Перезагрузка страницы не теряет задачи — они подгружаются обратно из этой памяти.»

Всё, теперь у вас есть карта файла. Не идеальная, но достаточная.

Лайфхак №2: «покажи мне, где X»

Когда что-то надо поменять, и вы не знаете, где это лежит, не лезьте искать — спросите:

«В каком файле и в какой строке находится код, который сохраняет задачи в localStorage? Покажи мне сам кусок.»

Агент покажет конкретное место. Дальше вы либо просите его поменять там, либо смотрите своими глазами, чтобы убедиться, что меняется именно это.

Лайфхак №3: следите за тем, что агент трогает

В Claude Code после каждого изменения видно — какие файлы были изменены. Это критично, чтобы не пропустить «случайные» правки.

Если попросили «измени цвет кнопки», а агент трогает app.js — стоп, что-то не так. Зачем ему логика? Спросите:

«Зачем ты изменил app.js? Я просил только цвет кнопки. Откати это изменение.»

Это нормально — переспрашивать. Это не «недоверие к агенту», это базовая гигиена.

Что делать, когда «совсем ничего не понятно»

Случай 1. Открыли файл, и там 500 строк, голова кругом

Не паникуйте. Никто не читает 500 строк подряд. Делайте так:

  1. Спросите агента: «дай мне оглавление этого файла — какие функции, что каждая делает, в одной строке»
  2. Найдите ту, которая делает то, что вас интересует
  3. Просите показать только её

Случай 2. Видите название функции — не понимаете, что значит

Спросите агента: «что значит setInterval в этом коде, простыми словами». Это не глупый вопрос. Это лучший способ научиться — спрашивать в контексте, на живом примере.

Случай 3. Ошибка в консоли — куча красного текста

Скопируйте первую ошибку (обычно она самая важная) и спросите:

«Эта ошибка появилась в консоли браузера: <вставить>. Что это значит и где это исправить?»

Агент покажет файл, строку, и обычно сразу предложит фикс.

Practice (20-30 минут)

Задание 1 (15 минут) — explainer’ы

Откройте свой my-todo из B5 (или любую папку с агентским кодом). Для каждого файла попросите агента:

«Объясни в трёх предложениях, что делает этот файл. Без кода, простыми словами.»

Перепишите ответы себе своими словами в блокнот или README. Это нужно не для красоты — этот процесс сам по себе закрепляет понимание.

Задание 2 (15 минут) — найдите фичу

Откройте app.js (или другой логический файл). Найдите функцию, которая отвечает за одну конкретную фичу — например, «удаление задачи» или «сохранение в localStorage». Сделайте две вещи:

  1. Расскажите вслух (или напишите) — что эта функция делает по шагам, своими словами
  2. Сверьтесь с агентом: «функция <имя> — правильно ли я понимаю, что она работает так-то?»

Если правильно — отлично, вы только что прочитали реальный код. Если нет — агент поправит, вы дополните своё понимание.

Recap

  • Ориентация ≠ программирование. Знать «какой файл за что отвечает» — уже 80% пользы.
  • Расширения говорят сами за себя: .html — структура, .css — оформление, .js — логика, .json — настройки, .md — текст.
  • Главный лайфхак — просить explainer’ы у агента, а не «учить язык». В контексте, на живом примере, по чуть-чуть.

Вайб Самурай

🥷 Вайб Самурай спрашивает: «А если я в принципе не хочу читать код? Я же не программист, я думал, агент сам всё сделает.»

Ответ: Никто не заставляет вас читать код построчно. Но если вы вообще никогда не открываете свои файлы, то происходит интересное: вы становитесь полностью зависимы от агента, который, бывает, и сам не знает, что натворил. Минимальная ориентация — это ваш контроль над своим же проектом. Это как ездить на автомобиле, не зная, где капот: ехать можно, но если что-то стучит — вы беспомощны. Не идеально, но реально работает — даже минимальное чтение кода превращает вас из пассажира в водителя.

Дальше

В B7 — Безопасность для начинающего разберём, что агенту нельзя давать и как защитить себя на этапе обучения. Без паранойи, но и без наивности — есть несколько простых правил, которые избавят вас от 90% типовых факапов: про API-ключи, про секреты, про папки, в которые лучше не пускать.

Если зашло — поделитесь главой с тем, кто работает с агентом, но в код не смотрит вообще. Если у вас есть файл, в котором ничего не понятно — закиньте кусок в комменты, разберём вместе. За лайк и комментарий — отдельная благодарность, ваша обратная связь реально помогает шлифовать материал.

📚 Дальше: посмотреть и почитать

Чтение чужого кода — это навык, который качается отдельно от написания. Подборка ниже — про этот навык и про справочники, в которые приятно лазить, когда «не понял слово».

Статьи про чтение кода

  • «How to Read Code Written by Someone Else» (Towards Data Science) — короткий разбор приёмов: трассировать от выхода, искать тесты, использовать «Go to definition»: towardsdatascience.com/the-most-efficient-way-to-read-code-written-by-someone-else.
  • «How to read someone else’s code!» (MIT Media Lab, классическая короткая заметка Henry Lieberman): web.media.mit.edu/~lieber/Teaching/IISDW/Read-Code.html. Старая, но честная — учит «не понимать всё сразу».

Справочники

  • MDN — JavaScript Guide (если в проекте JS — это единственный справочник, который стоит держать открытым во вкладке): developer.mozilla.org/en-US/docs/Web/JavaScript/Guide.
  • MDN — Window.localStorage (типовая штука в учебных проектах: «как агент сохраняет состояние в браузере»): developer.mozilla.org/en-US/docs/Web/API/Window/localStorage.
  • VS Code — Getting Started (как пользоваться «Go to definition», поиском по символам, и зачем оно при чтении кода): code.visualstudio.com/docs/getstarted/getting-started.

Совет: заведите привычку — увидели незнакомое слово в коде, сразу ищите его на MDN или просите агента объяснить «своими словами». Через месяц словарь сам соберётся.
