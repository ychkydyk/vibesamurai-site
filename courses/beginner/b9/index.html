B9 · Деплой первого проекта

От локальной папки до публичной ссылки. GitHub Pages, Cloudflare Pages — бесплатно, 15 минут, без серверов.

Черновик B9 в голосе DmitriyM (см. scripts/tg_voice_analysis/_summary.md).

Hook

ДО:                              ПОСЛЕ:

«У меня есть проект,    →       «Вот ссылка:
 но он только у меня             https://ваше-имя.github.io/my-todo
 на компе»                       — открой, потыкай»

Друзья, всем привет! 👋

Эта глава — про тот самый момент, когда ваш todo-приложение из B5 перестаёт быть «локальной папкой на компе» и становится публичной ссылкой, которой можно поделиться. Это момент, когда вы реально что-то сделали — потому что есть URL, и его можно отправить.

Пятнадцать минут. Без серверов, без админки, без денег. Через GitHub Pages.

Концепция: статика — это просто файлы

Ваш todo (из B5) — это статический сайт. Три файла:

my-todo/
├── index.html
├── style.css
└── app.js

Никакой базы данных, никакого бэкенда — всё работает в браузере пользователя. Это означает, что разместить такой сайт — это скопировать файлы на любой сервер, который умеет их отдавать.

Бесплатных сервисов, которые умеют это из коробки, навалом:

┌─────────────────────────────────────────────────────┐
│  GitHub Pages       — самый простой, бесплатно      │
│  Cloudflare Pages   — быстрее, чуть сложнее         │
│  Vercel             — для следующего шага           │
│  Netlify            — для следующего шага           │
└─────────────────────────────────────────────────────┘

В этой главе берём GitHub Pages, потому что у вас уже есть git (B4) и репозиторий, и второй шаг — просто включить настройку.

Пример: деплоим todo на GitHub Pages

Шаг 1. Создаём аккаунт GitHub (если нет)

Идите на github.com → Sign up. Имя выберите осознанно — оно будет в URL вашего сайта (https://ВАШЕ-ИМЯ.github.io/проект).

Шаг 2. Создаём репозиторий на GitHub

В правом верхнем углу GitHub: + → New repository.

  • Repository name: my-todo
  • Description: «Мой первый todo-лист»
  • Public — обязательно. GitHub Pages для приватных репо доступен только на платных тарифах.
  • НЕ ставьте «Add README», «Add .gitignore», «Add license» — у нас уже есть свои файлы.
  • Жмём Create repository.

GitHub покажет инструкцию «как залить существующий проект». Берём оттуда команды.

Шаг 3. Заливаем локальный репозиторий на GitHub

В терминале, в папке my-todo:

git remote add origin https://github.com/ВАШЕ-ИМЯ/my-todo.git
git branch -M main
git push -u origin main

При первом пуше GitHub попросит авторизоваться. Дальше работают personal access tokens (PAT) — гайд по их получению есть в самой GitHub-документации, или попросите вашего агента провести через шаги.

После пуша зайдите в репозиторий на GitHub в браузере — увидите свои файлы.

Шаг 4. Включаем GitHub Pages

В репозитории на сайте GitHub:

  1. Settings (вверху справа)
  2. Слева — Pages
  3. Source: Deploy from a branch
  4. Branch: main, папка / (root)
  5. Save

Через 30-60 секунд GitHub соберёт сайт. Сверху появится зелёная плашка с URL — обычно https://ВАШЕ-ИМЯ.github.io/my-todo/.

Шаг 5. Открываем ссылку

Откройте URL в браузере. Видите тот же todo, что у себя на компе. Теперь он в интернете.

Поделитесь ссылкой с кем-нибудь. Это полный фарш — ваш первый публичный продукт.

Если что-то не работает

Случай 1. По ссылке 404

  • Проверьте в Settings → Pages, что статус «Your site is live at …»
  • Подождите минуту-две — иногда первая сборка медленная
  • Убедитесь, что index.html лежит в корне репозитория (не в подпапке)

Случай 2. Сайт открывается, но нет стилей/без логики

Это значит — пути к файлам сломаны. В index.html пути должны быть относительными, без / в начале:

<!-- ХОРОШО -->
<link rel="stylesheet" href="style.css">
<script src="app.js"></script>

<!-- ПЛОХО — на GitHub Pages подкаталога это сломает -->
<link rel="stylesheet" href="/style.css">
<script src="/app.js"></script>

Если у вас плохие пути — попросите агента: «Поправь пути к ассетам в index.html, чтобы работало и локально, и на GitHub Pages в подкаталоге.»

Случай 3. Изменил файлы локально — на сайте по-прежнему старое

GitHub Pages обновляется только при push. После изменений:

git add .
git commit -m "правки"
git push

Через 30 секунд сайт обновится.

Случай 4. «Personal access token» — что это вообще

Это пароль для git, который работает вместо обычного пароля. Создаётся в GitHub: Settings → Developer settings → Personal access tokens → Tokens (classic) → Generate new token.

Дайте ему доступ только к repo, поставьте срок жизни 90 дней. Сохраните в надёжное место (или используйте менеджер паролей).

Не коммитьте PAT в код. Помните из B7 — секреты не в репозиторий. Если случайно закоммитили — сразу отзовите (revoke), потом разбирайтесь.

Cloudflare Pages — альтернатива на следующий шаг

Когда освоитесь с GitHub Pages, можно попробовать Cloudflare Pages (pages.dev). Плюсы:

  • Быстрее в большинстве регионов (CDN-сеть)
  • Не требует публичного репо — приватный тоже работает на бесплатном тарифе
  • Кастомные домены ставятся в один клик

Минусы — на старте чуть больше настройки. Если вам уже хватает GitHub Pages — не меняйте, не плодите сущности. На опыте — переезжать туда стоит, когда хочется свой домен или есть конкретная скорость-проблема.

Чек-лист «продукт готов поделиться»

Прежде чем кидать ссылку другу, пройдитесь:

  • Открывается с телефона (не только с десктопа) — откройте ссылку с мобилы, проверьте
  • Работают все фичи — не только «добавить», но и «удалить», «сохранить»
  • Нет ошибок в консоли — F12 → Console, должно быть пусто или близко к этому
  • Понятно, что это — есть заголовок, понятно, как пользоваться
  • Не утекли секреты — посмотрите файлы в репозитории на GitHub, нет ли там .env, ключей, паролей

Если все галочки — отправляйте. Не идеально, но реально работает.

Practice (30 минут)

Задание 1 (20 минут) — задеплойте свой todo

Пройдите все 5 шагов выше. Получите URL. Откройте его с телефона — это важный финальный тест.

Если на каком-то шаге застряли — спросите агента в Claude Code:

«Я делаю деплой на GitHub Pages, у меня шаг N не получается, ошибка: <вставить>. Что не так?»

Агент знает GitHub Pages хорошо, проведёт через любой ступор.

Задание 2 (10 минут) — поделитесь ссылкой

Отправьте URL минимум одному человеку, у которого спросите обратную связь:

«Открой со своего телефона, попробуй добавить пару задач, скажи, что не так.»

Это первое реальное использование вашего проекта другим человеком. Запишите фидбек — это точка роста на следующие итерации.

Recap

  • Статика = просто файлы. Любой статический сайт можно задеплоить за минуты, бесплатно.
  • GitHub Pages — самый простой первый деплой. Создал репо → залил → включил Pages → готово.
  • Чек-лист перед шерингом: мобила, фичи, консоль, понятность, утечки секретов.

Скептик Самурай

🥷 Скептик Самурай спрашивает: «А зачем мне деплой? Я же только учусь, кому нужен мой кривой todo в интернете?»

Ответ: Дело не в этом конкретном todo. Дело в петле: вы что-то сделали → вышло в мир → кто-то посмотрел → вы получили реакцию → улучшили. Если ваши проекты годами лежат в локальной папке — вы не получаете обратной связи и не учитесь на реальном использовании. У всех, кого я знаю в индустрии — даже у нулевых новичков — есть публичный портфель. Не потому что «надо для работы», а потому что видимый результат мотивирует. Один URL — это полный фарш мотивации на следующий проект. Не идеально, но реально работает.

Дальше

В B10 — Как оценивать, что агент сделал хорошо разберём, как понять, что результат агента — это «хорошо», а не «выглядит хорошо». Чек-листы, скептические проверки, типичные «дыры», в которые провалится новичок. Без этой главы вы будете принимать первое, что красиво выглядит — а это половина агентских факапов.

Если зашло — поделитесь главой с тем, у кого «всё на компе и никто не видит». Если задеплоили — киньте ссылку в комменты, реально хочется посмотреть, что у вас получилось. За лайк и комментарий — отдельная благодарность, ваша обратная связь реально помогает шлифовать материал.
