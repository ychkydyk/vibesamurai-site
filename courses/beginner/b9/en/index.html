B9 · Deploying the first project

From a local folder to a public link. GitHub Pages, Cloudflare Pages — free, 15 minutes, no servers.

Hook

BEFORE:                          AFTER:

"I have a project,        →     "Here's the link:
 but it's only on my             https://your-name.github.io/my-todo
 own machine"                    — open it, tap around"

Hey folks, welcome 👋

This chapter is about that moment when your todo app from B5 stops being “a local folder on my laptop” and becomes a public link you can share. It’s the moment you actually built something — because there’s a URL, and it can be sent.

Fifteen minutes. No servers, no admin, no money. Via GitHub Pages.

Concept: static = just files

Your todo (from B5) is a static site. Three files:

my-todo/
├── index.html
├── style.css
└── app.js

No database, no backend — everything runs in the user’s browser. So “hosting” such a site is just copying files to any server that knows how to serve them.

Free services that do this out of the box are plenty:

┌─────────────────────────────────────────────────────┐
│  GitHub Pages       — simplest, free                │
│  Cloudflare Pages   — faster, a bit more setup      │
│  Vercel             — for the next step             │
│  Netlify            — for the next step             │
└─────────────────────────────────────────────────────┘

We take GitHub Pages in this chapter because you already have git (B4) and a repo, and the second step is just flipping a setting.

Example: deploying todo to GitHub Pages

Step 1. Create a GitHub account (if you don’t have one)

Go to github.com → Sign up. Pick the username deliberately — it ends up in the URL of your site (https://YOUR-NAME.github.io/project).

Step 2. Create a repository on GitHub

Top-right on GitHub: + → New repository.

  • Repository name: my-todo
  • Description: “My first todo list”
  • Public — required. GitHub Pages for private repos is only on paid plans.
  • Don’t tick “Add README,” “Add .gitignore,” “Add license” — we already have our own files.
  • Click Create repository.

GitHub will show “how to push an existing project.” We take those commands.

Step 3. Push the local repo to GitHub

In the terminal, in the my-todo folder:

git remote add origin https://github.com/YOUR-NAME/my-todo.git
git branch -M main
git push -u origin main

On first push GitHub asks for auth. From there it’s personal access tokens (PATs) — guidance is in GitHub’s own docs, or ask your agent to walk you through the steps.

After push, open the repo on GitHub in the browser — you’ll see your files.

Step 4. Enable GitHub Pages

In the repo on GitHub:

  1. Settings (top right)
  2. Left side — Pages
  3. Source: Deploy from a branch
  4. Branch: main, folder / (root)
  5. Save

In 30–60 seconds GitHub builds the site. A green banner appears at the top with the URL — usually https://YOUR-NAME.github.io/my-todo/.

Step 5. Open the link

Open the URL in a browser. You see the same todo as on your laptop. Now it’s on the internet.

Share the link with someone. That’s the full package — your first public product.

If something doesn’t work

Case 1. The URL shows 404

  • Verify in Settings → Pages that the status is “Your site is live at …”
  • Wait a minute or two — first build is sometimes slow
  • Make sure index.html sits at the root of the repo (not in a subfolder)

Case 2. Site opens but no styles / no logic

That means file paths are broken. In index.html paths must be relative, no leading /:

<!-- GOOD -->
<link rel="stylesheet" href="style.css">
<script src="app.js"></script>

<!-- BAD — breaks on GitHub Pages subpaths -->
<link rel="stylesheet" href="/style.css">
<script src="/app.js"></script>

If your paths are bad — ask the agent: “Fix asset paths in index.html so it works both locally and on a GitHub Pages subpath.”

Case 3. Edited files locally — site still shows old content

GitHub Pages only updates on push. After changes:

git add .
git commit -m "edits"
git push

In 30 seconds the site updates.

Case 4. “Personal access token” — what’s that

It’s a password for git that replaces your regular password. Created in GitHub: Settings → Developer settings → Personal access tokens → Tokens (classic) → Generate new token.

Give it repo access only, set a 90-day lifetime. Save it somewhere secure (or a password manager).

Don’t commit a PAT into code. From B7 — secrets don’t go in the repo. If you accidentally committed — revoke immediately, then deal with git.

Cloudflare Pages — alternative for the next step

Once you’re comfortable with GitHub Pages, try Cloudflare Pages (pages.dev). Pros:

  • Faster in most regions (CDN network)
  • Doesn’t require a public repo — private works on the free tier too
  • Custom domains in one click

Cons — slightly more setup up front. If GitHub Pages is enough for you — don’t switch, don’t multiply entities. From experience — move when you want a custom domain or have a concrete speed problem.

Checklist “product ready to share”

Before sending the link to a friend, walk through:

  • Opens on mobile (not only desktop) — open the link from your phone, verify
  • All features work — not only “add” but also “delete,” “save”
  • No console errors — F12 → Console, should be empty or near-empty
  • Clear what it is — there’s a heading, clear how to use it
  • No leaked secrets — look at files in the GitHub repo, no .env, keys, passwords

If everything ticked — send it. Not perfect, but actually works.

Practice (30 minutes)

Task 1 (20 minutes) — deploy your todo

Walk through all 5 steps above. Get the URL. Open it on your phone — important final test.

If you got stuck at some step — ask the agent in Claude Code:

“I’m deploying to GitHub Pages, step N fails with error: . What’s wrong?”

The agent knows GitHub Pages well, walks you out of any block.

Task 2 (10 minutes) — share the link

Send the URL to at least one person, ask for feedback:

“Open it on your phone, try adding a couple of tasks, tell me what’s wrong.”

That’s the first real use of your project by another human. Note the feedback — that’s the growth edge for the next iteration.

Recap

  • Static = just files. Any static site can be deployed in minutes for free.
  • GitHub Pages — the simplest first deploy. Create repo → push → enable Pages → done.
  • Pre-share checklist: mobile, features, console, clarity, secret leaks.

Vibe Samurai

🥷 Vibe Samurai asks: “Why do I need to deploy? I’m just learning, who cares about my janky todo on the internet?”

Answer: It’s not about this specific todo. It’s about the loop: you built something → it went into the world → someone looked → you got feedback → you improved. If your projects sit in a local folder for years — you don’t get feedback and don’t learn from real use. Everyone I know in the industry — even absolute beginners — has a public portfolio. Not because “it’s needed for work,” but because visible results motivate. One URL is the full package of motivation for the next project. Not perfect, but actually works.

📚 Further reading

There are many static hosts and each has its quirks. A short list — official docs for the three I personally use (GitHub Pages for learning, Cloudflare Pages for prod with a domain, Vercel — when a frontend framework joins).

Official docs

  • GitHub Pages — Creating a site (full guide: from creating a repo to publishing, with the user/organisation sites section): docs.github.com/en/pages/getting-started-with-github-pages/creating-a-github-pages-site.
  • Cloudflare Pages — Get started (the “deploy static as-is” section, no build): developers.cloudflare.com/pages/get-started.
  • Vercel — Deployments (for later: pushing static via CLI or git, no servers): vercel.com/docs/deployments.

Token security

  • GitHub — Managing your personal access tokens (how to create a PAT, which scopes, how to revoke): docs.github.com/en/authentication/keeping-your-account-and-data-secure/managing-your-personal-access-tokens. Read the fine-grained tokens section — that’s the 2026 recommendation, not classic PATs.

Tip: don’t rush to move from GitHub Pages to Cloudflare/Vercel “because cooler.” A move makes sense when there’s a concrete reason — custom domain, speed, dynamic features. Otherwise don’t multiply entities.
