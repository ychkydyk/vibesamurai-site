B11 · Capstone: build your own tool

The Beginner track's final project. From 'learning task' to a personal tool you'll use yourself.

Hook

B5: todo for everyone      B11: something
                           you personally need

"learning project"   →     "I use it
                            every day"

Hey folks, welcome 👋

This is the final chapter of the Beginner track. No new concepts — we gather everything from B1–B10 into one real project. The key difference from the B5 todo: this is not a learning task. It’s a thing you personally need, and you’ll actually use it yourself.

This is the most important thing I can hand you in this track. Because another todo is an exercise. Building your own tool is the point of no return: you realised you can build the programs you need.

An hour and a half to two hours. Make a nice cup of tea.

Concept: “everyday programs” as a build target

It used to be that “program” = “product for millions of users.” Now in the agent era everyone has a personal right to a program:

┌─────────────────────────────────────────────────────┐
│  Before:                                            │
│  "I want X" → buy off-the-shelf or live without     │
│                                                     │
│  Now:                                               │
│  "I want X" → describe to the agent → get it        │
└─────────────────────────────────────────────────────┘

Not theory. Already reality for anyone who finished a track like this one.

How to pick a capstone idea

Rule 1. It must solve your own pain. Not “I thought people might want this.” But “I will personally use it.” The more often — the better.

Rule 2. Realistic scope — 15–30 minutes of agent work, not “a month-long project.”

Rule 3. Local or static. No complex servers at the start.

Examples (for inspiration)

  • Personal habit tracker — table “days / habits,” click = ✓, persists in the browser
  • Unit converter for your hobby (photographer — aperture/shutter/ISO; barista — recipes)
  • Random picker — “where to eat today” from your list
  • Calorie calculator over your standard meals (not for everyone, your menu)
  • “Before flying” checklist — recurring, ticked, resets after a trip
  • Pomodoro timer with your intervals (the standard 25/5 isn’t for everyone)
  • Note generator — you type 5 buzzwords, a structured mini-note comes out
  • Spend tracker — three buttons “coffee / food / transport,” persists, end of month shows the total
  • “What I know about X” — a personal mini-wiki for a topic you keep returning to
  • Meds / supplements schedule — reminder at your times

These aren’t “genius” ideas. They’re useful to you specifically. That’s the point.

Capstone steps

In order, no fluff. Each step references the chapter where we covered the detail.

Step 1. Idea → formula (10 minutes)

Take the idea and write it by the B3 formula:

WHAT:
WHERE:
HOW WE KNOW IT'S DONE:

Example (habit tracker):

  • WHAT: build a web-page habit tracker
  • WHERE: new folder ~/projects/habits, no build step, plain HTML/CSS/JS
  • HOW WE KNOW IT’S DONE:
    • Current date in the header
    • List of 5 habits (Water, Workout, Reading, Meditation, No sugar)
    • Under each weekday — checkboxes for the 5 habits
    • Clicking a checkbox persists
    • Reload doesn’t lose state
    • “New week” button — resets checkboxes, shifts the date

That’s your brief.

Step 2. Setup (5 minutes)

mkdir habits
cd habits
git init
echo ".env" > .gitignore
git add .gitignore
git commit -m "init"
claude

See B4 (git) and B7 (.gitignore immediately).

Step 3. Plan (10 minutes)

Send the agent your brief from Step 1. Say:

“Show the plan first — which files, what structure, no code.”

Read the plan. If you agree — “go.” If something’s off — fix now.

See B5.

Step 4. First iteration (15–25 minutes)

The agent writes the files. Run → check. Use the checklist from B10:

  • Match the criteria
  • Edge cases
  • Console
  • git diff
  • Reload

After acceptance — commit (see B4).

Step 5. 2–3 edits (20–30 minutes)

Something will need fixing. Phrase by the formula, give to the agent. After every accepted edit:

git add .
git commit -m "<short what>"

See B3 (how to phrase), B10 (checklist), B4 (commit).

Step 6. Deploy (15 minutes)

If your project is static (HTML/CSS/JS, no backend):

# Create the GitHub repo (see B9)
git remote add origin https://github.com/YOUR-NAME/habits.git
git branch -M main
git push -u origin main

# Enable GitHub Pages in Settings → Pages

Get a URL. Open on a phone.

See B9.

Step 7. Use it (open-ended time)

The main point. Today, tomorrow, the day after — try using your own program. Bookmark it on your phone, tick checkboxes, try “new week.”

After a week of use you’ll have 2–3 edits that are obvious from the inside:

  • “Font too small, can’t see on the go”
  • “Want a sixth habit”
  • “Want to see last week”

Each edit is a new iteration: formula → agent → checklist → commit → deploy. The full menu of your own product lifecycle.

What NOT to do for the capstone

  • Don’t pick a “mega-idea.” “Social network for dog owners” — not today. Today — one small useful thing.
  • Don’t try to add login, a database, a complex backend. Local/static, all in the browser.
  • Don’t get stuck on design. Design is easiest to improve later. Function first.
  • Don’t sit silently for 4 hours without git status. After every edit — commit. Per B4.

Example: habit tracker, real story

One of the folks in an early course cohort built this tracker for themselves. Two weeks later — sent a screenshot: 14 days straight, all 5 boxes ticked. Said the free apps in the store kept “pushing ads” and they dropped them. Their own — no ads, for them, simple — they didn’t drop it.

That’s the point of the capstone. Not “make another thing,” but get a tool you chose yourself, built yourself, and use yourself.

Practice — that’s the capstone

Main task (1.5–2 hours) — build and deploy

One project. From formula to URL on GitHub Pages. All 7 steps above.

Extra task (a day or a week) — use it

Drop the link in your bookmarks. Use it. Make 2–3 edits based on actual use. Commit, redeploy.

After a week of use you have:

  • Your tool in prod
  • 5–10 commits in history
  • A working loop “idea → edit → deploy”
  • Real understanding that you can build your own tools

Recap

  • Capstone = your own tool. Not learning, not “for the portfolio,” but what you use yourself.
  • Every step is from earlier chapters. Formula (B3), git (B4), end-to-end flow (B5), security (B7), deploy (B9), evaluation (B10).
  • The main thing — use it. Building is half. Using for a week and improving is the other half and the big shift.

Vibe Samurai

🥷 Vibe Samurai asks: “What if my idea isn’t original — it already exists in the store? Why build my own?”

Answer: Because the store version isn’t yours. Store apps are universal, with ads, with every conceivable feature you don’t need. Yours is tuned to your specific habit, no extras, in your language, on your domain. Analogy: every morning you could buy coffee at a café, but at home — faster, cheaper, your way. Same with tools. And — even if the store “has it,” building your own trains the build muscle. In a month or two that becomes a superpower — not depending on the store at all. Not perfect, but actually works.

📚 Further reading

The capstone is your first personal program. Two source types come in handy next: where to draw ideas for the following ones and where to see what others build on the same topic.

Ideas and galleries

  • Indie Hackers — Ideas Database (live base of personal ideas from people who build rather than “genius-plan”): indiehackers.com/ideas. Filter for “not SaaS,” look at simple ones.
  • awesome-claude-code (hesreallyhim) — curated repo with real CLAUDE.md files, slash-commands, small tools for Claude Code: github.com/hesreallyhim/awesome-claude-code. Good source to see how others shape their project for the agent.

Workflows and patterns

  • Simon Willison — Agentic Engineering Patterns (live blog of one of the best writers on agent practice): simonwillison.net/tags/agentic-engineering. Not a textbook — practitioner notes, useful to see how grown-ups work with the same tools.

Tip: don’t copy ideas wholesale. Take the skeleton (e.g. “local browser tracker”) and rewrite to your pain. Your capstone = you use it yourself, not “looks like the other guy’s.”
