B10 · How to evaluate the agent's output

Checklist and sceptical checks. Telling 'actually works' from 'looks working'. Half of agent screw-ups live here.

Hook

"LOOKS GOOD"                  "ACTUALLY WORKS"

Opened — pretty   →           Opened — pretty
Accepted                       Checked every feature
                               Checked edge cases
                               Checked console
                               Read the diff
                               → accepted

Hey folks, welcome 👋

This chapter is about that moment when the agent finished, everything “looks” fine, and you want to hit accept and move on. That’s the main trap for a newcomer. Half of agent screw-ups don’t show up at once — they show up two days later when “looked working” code actually breaks on the second click.

Twenty to thirty minutes. After the chapter you’ll have a verification checklist that takes 3–5 minutes but saves hours later.

Concept: “works” and “looks working” are different things

LOOKS POTENTIALLY              WORKS
✓ Opened without error         ✓ Opened without error
✓ Picture in place             ✓ All 5 "how we know it's done" items
                               ✓ Edge cases checked
                               ✓ Console clean
                               ✓ Only the right files changed
                               ✓ Reload doesn't break

Looks good is a surface test. The eye saw the familiar, the brain ticked “ok.”

Works is a functional test. Every promised feature checked by hand.

The difference is 5–7 minutes of checks. And how many of your nerves survive next week.

Acceptance checklist (5 minutes)

Memorise and walk it after every edit by the agent. Base hygiene, like git status after changes.

The “accept or not” decision drawn as a flowchart:

flowchart TB
    A([Agent says<br/>"done"]) --> B{"How we know it's done"<br/>criteria verified<br/>by hand?}
    B -->|no| F([Roll back<br/>git restore .])
    B -->|yes| C{Edge cases<br/>passed?}
    C -->|no| U([Clarify<br/>and redo])
    C -->|yes| D{Console and<br/>git diff clean?}
    D -->|no| U
    D -->|yes| E{Reload<br/>didn't break it?}
    E -->|no| U
    E -->|yes| OK([Accept<br/>+ commit])
    U -.-> A
    F -.-> A

Bottom-up: while any check isn’t green — clarify or roll back. No “I’ll figure it out later.”

1. Match the request’s criteria

Recall the B3 formula: WHAT + WHERE + HOW WE KNOW IT’S DONE. Take your “how we know” and verify each item by hand.

If the request said “added a task → it appeared in the list” — add a task and look. Not “should appear” — did it appear.

2. Edge cases

“Edge case” = boundary scenarios: empty input, very long text, special characters, rapid clicks.

Minimum list:

  • Empty input (click “add” without typing)
  • Very long input (300 chars)
  • Special chars in text (<script>, quotes, emoji)
  • Rapid repeated clicks
  • Deleting the last item

90 % of bugs hide in edge cases.

3. Browser console

F12 → Console. What’s there?

  • Empty — great
  • Yellow warnings — usually ok, but read what they mean
  • Red errors — stop. Show the agent: “Console error <…>. What’s that?“

4. Git diff

git diff

Shows exactly which lines changed. Eyeball-scroll:

  • Did only the expected files change?
  • No extra changes (e.g. you had one todo and the agent rewrote three files)?
  • No suspicious lines — eval(), localStorage.setItem("token", ...), new <script src="http://...">?

If something’s off — git restore . (rollback) and reformulate.

5. Page reload (where applicable)

For web projects: after changes hit F5 in the browser. Still works? State persisted if it should have?

Extended checklist (5–10 minutes, before deploy/share)

When you want to share the result — extras:

6. Mobile device

Open the URL (if you deployed in B9) on a phone. Often desktop is fine and mobile:

  • Buttons too small
  • Text runs off-screen
  • Swipe/tap doesn’t work

7. Another browser

Tested in Chrome? Open in Firefox/Safari. Sometimes things work only in Chrome and quietly break elsewhere.

8. Clean session

Open in “incognito” / “private window.” That clears localStorage and cookies. If your todo doesn’t start “from scratch” there — you have a first-run bug on your machine masked by old data.

9. What should NOT work — really doesn’t?

Suppose your todo doesn’t sync to the cloud (it’s local). Verify — does anything go to the cloud? Open DevTools → Network and see where your site reaches in the background. There shouldn’t be requests to unknown domains.

Principle: if network requests appear you didn’t ask for — that’s a serious signal. Ask the agent: “where did the request to <domain> come from, I didn’t ask for it.”

Anti-patterns newcomers fall into

Anti-pattern 1. “Looks = works”

Opened, saw — “oh, button, form, all there” → accepted. No feature checks. Two days later half doesn’t work.

Cure: the checklist above. Always.

Anti-pattern 2. “Agent wrote it = it’s right”

The agent confidently reports “done, added function X.” Actually it added a similar function that’s 70 % right and 30 % off.

Cure: check by hand the key scenarios. Trust the agent — yes, but with verification.

Anti-pattern 3. “Accept now, figure out later”

“I’ll just accept, check later.” Never checks. A full menu of hidden bugs piles up, in a week the project won’t start, and nobody remembers what’s what.

Cure: commit after every accepted edit (B4). If a bug surfaces — git log shows where it appeared, and git restore rolls back painlessly.

Anti-pattern 4. “Green = good”

Tests passed → it works. A common mistake of experienced programmers too. Tests check only what’s written in them. Did you write the right test?

Cure: by hand, with your eyes. Tests are a supplement, not a replacement.

Anti-pattern 5. “Works for me = works for everyone”

“Opens on my machine — ok.” For the user:

  • Different browser
  • Different OS
  • Different screen size
  • Older version

Cure: mobile, clean session, another browser — points 6–8 of the extended checklist.

A short aside ⚡

The most common comment from folks: “yesterday everything worked, today it’s broken — the agent broke something.”

In 9 out of 10 cases — nobody broke anything. Yesterday you didn’t check properly, today you spotted a bug that was there then too. Basic cognitive bias: “works for me” = “works in principle.” It doesn’t.

The checklist is insurance against yourself.

Example: applying the checklist to todo

Say you asked the agent: “Add the ability to delete a task with a ❌ button.”

Agent said “done.”

Checklist (3 minutes):

  1. Match: opened — is there a ❌ next to each task? Yes. Click deletes? Yes.
  2. Edge:
    • Delete the last task — list empty? Not broken?
    • Delete rapidly in a row — all deleted?
    • And if I completed a task and then delete — works?
  3. Console: F12 → Console. Empty.
  4. Diff: git diff → only app.js and style.css changed (for the button) — expected.
  5. Reload: F5 after delete — tasks really gone, not “restored from localStorage.”

All good? Commit. Not all? Clarify with the agent or roll back.

Practice (30 minutes)

Task 1 (15 minutes) — checklist your todo

Open your todo from B5 (or any current project). Walk the basic checklist (items 1–5). Find at least one bug you didn’t notice before.

If you found zero — great, tidy project. Or you didn’t check well.

If you found something — ask the agent to fix and walk the checklist again. Habit reinforced.

Task 2 (15 minutes) — checklist → text file

In the project root create CHECKLIST.md:

# Before commit
- [ ] All "how we know it's done" criteria verified by hand
- [ ] Edge: empty input, long text, special chars
- [ ] Console F12 — no red
- [ ] git diff — only the expected changed
- [ ] F5 — state persists as it should

# Before sharing
- [ ] Mobile
- [ ] Another browser
- [ ] Incognito (clean session)
- [ ] Network: no extra requests

Make a habit of opening this file before every accepted edit. In 2 weeks it’s in your head; delete the file.

Recap

  • “Looks good” ≠ “works.” 5 minutes of checklist saves hours later.
  • Base checklist: criteria → edge → console → diff → reload. After every edit.
  • Anti-patterns: “accept now, check later,” “green = good,” “works for me.” All three are known traps.

Vibe Samurai

🥷 Vibe Samurai asks: “Spending half an hour on checks! I want fast. And the agent wrote it — didn’t it check itself?”

Answer: The agent checked against what you asked. If you didn’t ask — it did the minimum sufficient to look working. Verification is your responsibility, not the agent’s. Sounds unfair, but it’s like with a contractor: you can hand over a job, but accepting the work is on you. Otherwise every “unchecked” bug stays yours — and the agent moved on. Five minutes of checks vs hours of redo — the math is simple. Not perfect, but actually works.
