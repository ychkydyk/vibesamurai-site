B7 · Security for beginners

What not to give the agent. Secrets, API keys, dangerous commands. No paranoia, no naivety.

Hook

WITHOUT rules:                    WITH rules:

API key       → in code      your key → .env, in .gitignore
password      → in chat      agent doesn't see the password
"yes" to all  → agent        agent asks before rm,
                  rm -rf /                    git push, sudo

Hey folks, welcome 👋

This chapter is not a scare story. I’m not going to scare you with “hackers” and “leaks.” But there are a few simple rules that save you from 90 % of typical trouble: from an accidentally published API key to a wiped photo folder. Twenty minutes — and you have base hygiene.

The main thing to land: the agent isn’t malicious, but it’s extremely diligent. Give it rights to do something dangerous — it will do it without thinking. Your job — don’t give unnecessary rights.

Concept: three trust levels

Imagine the agent is a capable intern you gave a laptop and credentials. The intern isn’t a vandal, but they act literally. Say “delete extra files” — they may delete your diploma because “the name looks similar.”

So trust levels:

┌──────────────────────────────────────────────────┐
│  L0 — Read-only       "look, don't touch"        │
│  L1 — Local edits     "change files in this folder"│
│  L2 — Run commands    "you can run tests"        │
│  L3 — External actions "push, deploy, payments"  │
└──────────────────────────────────────────────────┘

Rule: start every session at L0. Raise only when the agent’s behaviour is clear. Never grant L3 “for life.”

If you draw the trust boundary on a napkin — who sees what and where they can’t look — it looks like:

flowchart LR
    subgraph WORLD[Outside world]
        GH[(GitHub /<br/>others' services)]
        AI[(LLM provider<br/>chat history)]
    end
    subgraph MACHINE[Your machine]
        HOME[~/ home, photos,<br/>documents]
        ENV[.env<br/>secrets]
        subgraph PROJ[Project folder]
            CODE[code + .gitignore]
            AGENT[Agent<br/>sees only this]
        end
    end
    AGENT -.doesn't see.-> HOME
    AGENT -.doesn't see values.-> ENV
    ENV -- read as an<br/>env variable --> CODE
    AGENT --> CODE
    CODE --> GH
    AGENT -- prompt + context --> AI

Key from the diagram: the agent sees the folder it was launched in and the values of env variables through your code, not the secret string itself. Anything above the project folder or in .env is “dark” unless you yourself paste it into the chat.

What you MUST NOT give the agent

1. Secrets in chat

If you wrote to the agent in chat:

“Here’s my OpenAI API key: sk-abc123..., use it”

— that’s already an incident. Because:

  • That key now lives in the chat history the provider stores
  • If you later screenshot the chat and post it — the key went public
  • The agent can in some tool call accidentally forward this key to an external service

Right way: put the key into an environment variable (.env file or export VAR=... in shell). Tell the agent:

“Use the key from OPENAI_API_KEY in the environment.”

The agent reads the value from the system, but doesn’t see the actual string in its memory.

2. Passwords, tokens, keys — directly in code

The most common newcomer mistake:

const apiKey = "sk-abc123def456...";  // ⚠️ ⚠️ ⚠️

Then — git commit, git push, and your key is public on GitHub. Bots scan public repos every few minutes — your key will be used within minutes.

Right way — always:

const apiKey = process.env.OPENAI_API_KEY;

And the key — in a .env file that is in .gitignore:

# .gitignore
.env
.env.local
node_modules/

If you accidentally committed a key — don’t “edit history” or remove it from the last commit first. First revoke the key at the provider (seconds), then deal with git. Revoke is primary, history cleanup is secondary.

3. Access to folders you’re not working in

By default the agent works in the folder you launched it from. That’s good.

cd my-project
claude       # the agent sees my-project and subfolders

Don’t run the agent from your home directory or the drive root — otherwise it has access to everything. Including your photos, documents, correspondence.

Right way — a dedicated project folder. The agent looks there, and only there.

4. Commands without confirmation

In Claude Code (and in Cursor, and in Codex CLI) by default the agent asks before running commands. That’s built-in protection. Don’t disable it until you know what you’re doing.

If you ever see a flag like --dangerously-skip-permissions or “yolo mode” — stop and think. That’s for the experienced who know the risks. A newcomer doesn’t need it.

A short aside ⚡

Heard a story recently — a guy set up “safe tutorial mode” with --yolo (agent does everything without asking), said “build me a site,” went for coffee. Came back — the agent decided the Downloads/ folder was “cluttered” and tidied it. Eight years of accumulated files. Out of a sense of order.

The agent isn’t bad. The human gave it rights without thinking. Brewed it himself.

Hygiene rules (memorise like letters)

Rule 1. .env + .gitignore — always

In any project that has at least one key/token:

# Create .env (for secrets)
echo "OPENAI_API_KEY=sk-..." > .env

# Immediately add to .gitignore
echo ".env" >> .gitignore

Check that .env doesn’t land in the commit:

git status

If .env isn’t in the list — great. If it is — add to .gitignore and don’t commit.

Rule 2. “Look, don’t touch” at the start

First message in a new session:

“Read the files in this folder. Don’t change anything, don’t run anything. Tell me what’s here.”

That gives a project map and prevents the agent from doing something out of habit. When you’ve mapped the structure — raise the trust level.

Rule 3. Commit before every dangerous action

Before:

  • Letting the agent run commands
  • Installing a new library (npm install ...)
  • Rewriting something big

Commit. From B4 — that’s a rollback point. If anything goes wrong, git restore . brings it all back.

Rule 4. Don’t trust “safe” commands

Commands that sound safe but can be dangerous:

CommandWhat’s dangerous
rm <file>Deletion is irreversible in most cases
npm install <pkg>Can pull malicious code via dependencies
git push --forceOverwrites server history, breaks colleagues
sudo <anything>System changes, easy to break the OS
chmod 777 ...Open permissions = anyone can edit the file
> file.txtRedirect “wipes” the file entirely

If the agent itself wants to run one of these — read what exactly and why. Don’t hit “Yes” reflexively.

Rule 5. Suspicious sources

If the agent suggests:

  • Downloading from a little-known site (curl http://random.site/install.sh | sh)
  • Installing a package you didn’t ask for
  • Opening a “test link” in the browser

Stop. Ask:

“What’s this package/site, why is it needed, who’s the author? Give me a source link.”

A good agent gives a link and explains. A bad one will hand-wave, and here your scepticism is your first defence. This move pays off regularly — better to ask one extra time and avoid trouble than later wrestle with the full package of consequences.

Example: correct initial project setup

# 1. Make a folder, init git
mkdir my-safe-project
cd my-safe-project
git init

# 2. Create .gitignore IMMEDIATELY, before the first commit
cat > .gitignore <<EOF
.env
.env.local
node_modules/
*.log
.DS_Store
EOF

# 3. Make .env for secrets (if you'll have any)
touch .env

# 4. First commit — empty project with the right protection
git add .gitignore
git commit -m "init: safety baseline"

# 5. Now launch the agent
claude

First message:

“This is a new project. Only .gitignore is in the folder. Tell me what file structure you propose for . Plan only, no code.”

That’s a fully safe start. The agent can’t break anything because there’s nothing yet.

Practice (20–30 minutes)

Task 1 (10 minutes) — .gitignore for all your projects

Open every folder where you built projects in B5 and earlier. Check:

ls -la | grep .gitignore

If no .gitignore — create one. If there is — verify .env, node_modules/, and any heavy/secret folders are included.

Task 2 (15 minutes) — audit of “what the agent touches”

In the next session with the agent, after every change, check:

git status

What exactly changed? Does it match what you asked? If there are “extra” changes — that’s a signal: re-ask the agent or roll back.

Make it a habit: after every accepted edit — git status + a quick look. That’s base hygiene, same as washing hands.

Recap

  • Three trust levels: read-only → edits → run → external actions. Always start at the minimum.
  • Secrets in .env, and .env in .gitignore. Never write keys into chat or directly into code.
  • Launch the agent in a dedicated project folder. Not from home, not from the drive root.

Vibe Samurai

🥷 Vibe Samurai asks: “I have a learning project, no real value — what’s there to protect? I just want to try quickly.”

Answer: A learning project is a habit trainer. If on a learning project you immediately always make .gitignore and don’t paste secrets into chat — on a real project it becomes reflex, and you won’t lose money/data/nerves. If on a learning project “I can be sloppy” — same on a real one, and one day it bites. Cost of the right habit at the start — a minute per project. Cost of not having it can be a bill for thousands of dollars for a stolen API key. Not perfect, but actually works — ten minutes of hygiene at the start save hours (and sometimes dollars) later.

📚 Further reading

Security is a topic where primary sources beat summaries. A short, verified list I’d give any beginner.

Official documentation

  • Claude Code — Sandboxing — code.claude.com/docs/en/sandboxing. How Claude Code itself separates “allowed” from “forbidden” at the OS level. Useful to know what the tool protects and what’s your job.
  • Anthropic — Claude Code auto mode: a safer way to skip permissions — anthropic.com/engineering/claude-code-auto-mode. Honest breakdown by Anthropic of what happens when you turn off confirmations and where that’s acceptable.

Attacks and threats — to know what to fear

  • OWASP Top 10 for Agentic Applications (2026) — genai.owasp.org/resource/owasp-top-10-for-agentic-applications-for-2026. Map of typical agent risks. Not a scare story — a structured list.
  • Simon Willison — prompt injection (series) — simonwillison.net/tags/prompt-injection. The person who coined “prompt injection.” Best place to spend an hour and understand how an agent can be “talked into” something via slipped text.

Secrets and git

  • GitGuardian — Secure Your Secrets With .env — blog.gitguardian.com/secure-your-secrets-with-env. Detailed walkthrough of .env + .gitignore and what to do if a key did leak.

Tip: if reading time is short — start with OWASP Top 10 for Agentic Applications. It’s the cheapest way to stop stepping on the same rakes everyone else does.
