B7 · Безопасность для начинающего

Что агенту нельзя давать. Секреты, API-ключи, опасные команды. Без паранойи, но и без наивности.

Черновик B7 в голосе DmitriyM (см. scripts/tg_voice_analysis/_summary.md).

Hook

БЕЗ правил:                      С правилами:

API-ключ      → в коде      ваш ключ → .env, в .gitignore
пароль        → в чате      агент не видит пароля
"да" на всё   → агент        агент спрашивает на rm,
                  rm -rf /                    git push, sudo

Друзья, всем привет! 👋

Эта глава — не страшилка. Я не буду пугать вас «хакерами» и «утечками». Но есть несколько простых правил, которые избавят вас от 90% типовых неприятностей: от случайно опубликованного API-ключа до затёртой папки с фотографиями. Двадцать минут — и у вас есть базовая гигиена.

Главное, что хочу донести: агент не злой, но он очень исполнительный. Если вы дадите ему права делать что-то опасное — он сделает, не задумавшись. Ваша задача — не давать ненужных прав.

Концепция: три уровня доверия

Представьте, что агент — это толковый стажёр, которому вы дали ноут и доступы. Стажёр не вредитель, но он действует буквально. Если сказали «удали лишние файлы» — может удалить ваш диплом, потому что «название похожее».

Поэтому уровни доверия:

┌──────────────────────────────────────────────────┐
│  L0 — Read-only      «только посмотреть»         │
│  L1 — Локальные правки  «меняй файлы в этой папке»│
│  L2 — Запуск команд  «можешь запускать тесты»    │
│  L3 — Внешние действия  «push, deploy, оплата»   │
└──────────────────────────────────────────────────┘

Правило: начинаете каждую сессию с L0. Повышаете, только когда понятно, что делает агент. Никогда не выдаёте L3 «на всю жизнь».

Что НЕЛЬЗЯ давать агенту

1. Секреты в чат

Если вы написали агенту в чате:

«Вот мой API-ключ для OpenAI: sk-abc123..., используй его»

— это уже инцидент. Потому что:

  • Этот ключ теперь в истории чата, которая хранится у провайдера
  • Если вы потом скриншотите чат и где-то постите — ключ ушёл в публику
  • Агент может в каком-то tool call случайно переслать этот ключ во внешний сервис

Правильно: положите ключ в переменную окружения (.env файл или export VAR=... в шелл). Скажите агенту:

«Используй ключ из OPENAI_API_KEY в окружении.»

Агент прочитает значение из системы, но не увидит саму строку в своей памяти.

2. Пароли, токены, ключи — в код напрямую

Самая частая ошибка новичка:

const apiKey = "sk-abc123def456...";  // ⚠️ ⚠️ ⚠️

И потом — git commit, git push, и ваш ключ в открытом доступе на GitHub. Боты сканируют публичные репозитории каждые несколько минут — ваш ключ начнут использовать за минуты.

Правильно — всегда:

const apiKey = process.env.OPENAI_API_KEY;

И ключ — в .env файле, который в .gitignore:

# .gitignore
.env
.env.local
node_modules/

Если вы случайно закоммитили ключ — не удаляйте его из последнего коммита и не «правьте историю». Сначала отзовите ключ у провайдера (это секунды), потом разбирайтесь с git. Отзыв первичен, чистка истории — вторична.

3. Доступ к папкам, в которых не работаете

Агент по умолчанию работает в той папке, из которой вы его запустили. Это хорошо.

cd my-project
claude       # агент видит my-project и подпапки

Не запускайте агента из домашней директории или из корня диска — иначе ему доступно всё. Включая ваши фотографии, документы, переписку.

Правильно — отдельная папка под проект. Агент туда смотрит, и только туда.

4. Команды без подтверждения

В Claude Code (и в Cursor, и в Codex CLI) по умолчанию агент спрашивает разрешения перед запуском команд. Это встроенная защита. Не отключайте её, пока не понимаете, что делаете.

Если когда-нибудь увидите флаг типа --dangerously-skip-permissions или «yolo mode» — остановитесь и подумайте. Это для опытных, кто знает, чем рискует. Новичку не нужно.

Короткая вставка ⚡

Слышал недавно историю — человек поставил «безопасный туториальный режим» с --yolo (агент делает всё без вопросов), сказал «сделай мне сайт», ушёл за кофе. Вернулся — агент решил, что папка Downloads/ «захламлена», и подчистил. Восемь лет накопленных файлов. Из чувства порядка.

Это не агент плохой. Это человек дал ему права, не подумав. Заварил эту кашу сам.

Правила гигиены (запомните как буквы)

Правило 1. .env + .gitignore — всегда

В любом проекте, где есть хоть один ключ/токен:

# Создайте .env (для секретов)
echo "OPENAI_API_KEY=sk-..." > .env

# Сразу добавьте в .gitignore
echo ".env" >> .gitignore

Проверьте, что .env не попадает в коммит:

git status

Если в списке нет .env — отлично. Если есть — добавьте в .gitignore и не коммитьте.

Правило 2. «Только посмотреть» в начале

Первое сообщение в новой сессии:

«Прочитай файлы в этой папке. Ничего не меняй, ничего не запускай. Расскажи мне, что тут лежит.»

Это даёт карту проекта и не даёт агенту натворить с непривычки. Когда поняли структуру — поднимаете уровень доверия.

Правило 3. Коммит до каждого опасного действия

Перед тем, как:

  • Дать агенту запускать команды
  • Установить новую библиотеку (npm install ...)
  • Переписать что-то большое

Коммитнитесь. Из B4 помните — это точка отката. Если что-то пойдёт не так, git restore . вернёт всё назад.

Правило 4. Не доверяйте «безопасным» командам

Команды, которые звучат безопасно, но могут быть опасны:

КомандаЧем опасна
rm <файл>Удаление необратимо в большинстве случаев
npm install <пакет>Может тянуть зловредный код через зависимости
git push --forceЗатирает историю на сервере, у коллег ломается
sudo <что-угодно>Системные изменения, легко сломать ОС
chmod 777 ...Открытые права = любой может изменить файл
> file.txtПеренаправление «затирает» файл целиком

Если агент сам хочет запустить что-то из этого — прочитайте, что именно и зачем. Не нажимайте «Yes» рефлексом.

Правило 5. Подозрительные источники

Если агент предлагает:

  • Скачать что-то с малоизвестного сайта (curl http://random.site/install.sh | sh)
  • Установить пакет, которого вы не просили
  • Открыть в браузере «ссылку для проверки»

Стоп. Спросите:

«Что это за пакет/сайт, зачем он нужен и кто его автор? Дай ссылку на источник.»

Хороший агент даст ссылку и объяснит. Плохой — наплетёт, и тут ваш скепсис первый защитник. Этот ход выручает регулярно — лучше спросить лишний раз и не нарваться, чем потом возиться с полным фаршем последствий.

Пример: правильная начальная настройка проекта

# 1. Создаём папку, инициализируем git
mkdir my-safe-project
cd my-safe-project
git init

# 2. Создаём .gitignore СРАЗУ, до первого коммита
cat > .gitignore <<EOF
.env
.env.local
node_modules/
*.log
.DS_Store
EOF

# 3. Создаём .env для секретов (если будут)
touch .env

# 4. Первый коммит — пустой проект с правильной защитой
git add .gitignore
git commit -m "init: safety baseline"

# 5. Теперь запускаем агента
claude

Первое сообщение:

«Это новый проект. В папке только .gitignore. Расскажи, какую структуру файлов ты предлагаешь для <моей задачи>. Только план, без кода.»

Это полностью безопасный старт. Агент не может ничего сломать, потому что ничего ещё и нет.

Practice (20-30 минут)

Задание 1 (10 минут) — .gitignore для всех ваших проектов

Откройте все папки, в которых вы делали проекты в B5 и предыдущих. Проверьте:

ls -la | grep .gitignore

Если .gitignore нет — создайте. Если есть — проверьте, что туда включены .env, node_modules/, и любые папки с тяжёлыми/секретными файлами.

Задание 2 (15 минут) — аудит «что трогает агент»

В следующей сессии с агентом, после каждого изменения, проверяйте:

git status

Что точно изменилось? Совпадает ли это с тем, что вы просили? Если есть «лишние» изменения — это сигнал: либо переспросить агента, либо откатиться.

Заведите привычку: после каждой принятой правки — git status + быстрый взгляд. Это базовая гигиена, такая же, как мыть руки.

Recap

  • Три уровня доверия: read-only → правки → запуск → внешние действия. Начинайте всегда с минимума.
  • Секреты — в .env, и .env — в .gitignore. Никогда не пишите ключи в чат или в код напрямую.
  • Запускайте агента в отдельной папке проекта. Не из домашней директории, не из корня диска.

Скептик Самурай

🥷 Скептик Самурай спрашивает: «У меня учебный проект, никакой ценности — что тут защищать? Я просто хочу быстро попробовать.»

Ответ: Учебный проект — это тренажёр привычек. Если на учебном проекте вы научились сразу делать .gitignore и не пускать секреты в чат — на боевом проекте это будет рефлексом, и вы не потеряете деньги/данные/нервы. Если на учебном «можно небрежно» — на боевом то же самое, и однажды это вылезет. Цена правильной привычки на старте — минута на проект. Цена её отсутствия может быть счёт на тысячи долларов за украденный API-ключ. Не идеально, но реально работает — десять минут на гигиену в начале экономят часы (и иногда деньги) потом.
