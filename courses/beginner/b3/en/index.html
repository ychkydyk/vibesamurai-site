B3 · First conversation

How to ask the agent and what to ask. Anatomy of a good request. When to accept, when to say 'no, redo it'.

Hook

YOUR REQUEST:                 AGENT UNDERSTANDS:
"make it pretty"      →       ??? (something, you won't guess)
"change the Login    →        a specific edit
 button colour to blue"

Hey folks, welcome 👋

In this chapter — the most important skill of the whole course: how to ask the agent so it gets you on the first try. No “magic prompts,” no “ChatGPT secret hacks from info-hawkers.” A simple structure that always works. Twenty minutes, and you’ll save yourself tens of hours over the coming months.

Concept: a request = a small contract

Think of each request to the agent as a mini-contract. Three things have to be in it:

┌─────────────────────────────────────────────────────┐
│  1. WHAT TO DO       — concrete action              │
│  2. WHERE            — file / folder / function     │
│  3. HOW WE KNOW IT'S DONE — acceptance criterion    │
└─────────────────────────────────────────────────────┘

That’s the base formula. If your request misses any of the three — the agent will guess. Sometimes right, sometimes not. Lottery.

Bad vs good

Bad:

“Help with authorization”

What’s wrong: unclear what to help with, where the code is, what counts as done. The agent will invent its own story — usually not yours.

Good:

“In the file src/login.js there’s a sign-in form. Add a check that the password is at least 8 characters. Done when: 1) for passwords shorter than 8 a message ‘Password too short’ appears, 2) for long enough — the form submits as before.”

What’s good: clear what (validation), where (src/login.js), how we know (two concrete behaviours).

Anatomy of a good request

Doesn’t have to be bulky. The minimum — one sentence by the formula:

” in so that <HOW WE KNOW IT’S DONE>.”

Extensions — as needed:

  • “First show me the plan” — if the task is bigger than one move
  • “Don’t touch X” — if there are areas not to go into
  • “Use Y” — if you have preferences on tools/libraries
  • “Ask if unclear” — gives the right to re-ask rather than guess

Template to copy

What: ...
Where: ...
How we know it's done: ...
Don't touch: ...
Show me the plan first.

Keep this template handy — open in a text editor or as a bookmark. The first weeks you’ll consult it often, later — reflexively.

Typical beginner mistakes

1. “Help me”

The most common. What “help” means is undefined. The agent does something to its taste. Cure: always swap “help” for a concrete verb.

2. “Make it pretty”

What does “pretty” mean? Bigger font? Different colour? Animation? You won’t guess. Cure: show an example (“like this screenshot,” “like site X”), or give concrete parameters.

3. “Fix the bug”

Which bug? Where? How to reproduce? What should be there instead? Cure: describe the repro first, then the expectation.

4. A wall of text

Also bad: 5 paragraphs with every possible detail. The agent burns context figuring out what’s important. Cure: short by formula + optional “don’t touch.”

5. “Do what you think is best”

Sounds polite, actually — abdicating responsibility. The agent will do “what it thinks,” which rarely matches what you wanted. Cure: state preferences yourself, even uncertain ones.

Example: a bad request travelling to a good one

Round 1. “Make me a business-card site.”

— What goes on the site? Dark or light? Name, role, contacts? One file or a folder?

Round 2. “HTML page, dark theme, my name ‘Ivan Ivanov,’ role ‘AI engineer,’ three contacts: email, telegram, github.”

— OK. Font preferences? Images/photos needed?

Round 3. “Sans-serif, no images. File index.html in the current folder. Done when: opening the file in a browser shows the name, role, and three clickable contacts on a dark background. Show the plan before starting.”

That’s a request you don’t have to clarify. The agent knows what to do, where, what counts as done.

Can you write that on the first try? Yes. After 2–3 weeks of practice it becomes reflex.

When to accept, when to say “no, redo”

If you draw one full “move” of a conversation with the agent on a napkin — it looks like:

flowchart LR
    U([Your request<br/>by formula]) --> P[Agent thinks<br/>and writes a plan]
    P --> A[Agent acts<br/>files / commands]
    A --> R[You inspect<br/>the result]
    R -->|ok| OK([Accept<br/>+ commit])
    R -->|almost| F([Clarify<br/>and redo])
    R -->|not at all| X([Roll back<br/>start over])
    F -.-> U
    X -.-> U

After the agent did something, you have three buttons:

✅ “Accept” — when

  • Result fully matches the “how we know it’s done” from the request
  • No extra “bonus” changes you didn’t ask for
  • Only the expected files changed

🔁 “Clarify and redo” — when

  • Most of it is right, but something is off
  • You need small tweaks (text, colour, spacing)
  • A nudge “OK, but use Geist instead of Inter” — normal workflow

❌ “Roll back, start over” — when

  • The agent did “not what was asked,” wandered off
  • Files were touched that shouldn’t have been
  • Clearly cheaper to rephrase the task from scratch than to fix it

Rolling back is normal. It’s not “a failure” or “the agent is bad.” It’s a normal working motion. In git the rollback is git restore (or hitting “Reject” in Cursor). In Claude Code — say “revert the latest changes.”

Practice (30 minutes)

Task 1 (15 minutes) — rewrite a bad request

Take your real request you might give an agent. E.g. “build me a portfolio page,” “help with my CV,” “write a Telegram weather bot.” Rewrite by the formula WHAT / WHERE / HOW WE KNOW.

Compare both versions. Which one is clearer to you? That’s the rule: if you can answer questions about it, the agent definitely can.

Task 2 (15 minutes) — fire a good request

In an empty folder open claude and send your rewritten request. Look at the result. Check it against “how we know it’s done” — match or not?

If yes — great, formula works. If not — what was missing in the formula? Note it for yourself. Apply next time.

Recap

  • Request = mini-contract of three parts: WHAT + WHERE + HOW WE KNOW.
  • Optional: “don’t touch,” “show me the plan first,” “use X.”
  • Three buttons after the result: accept / clarify-redo / roll-back-start-over. Rollback is normal.

Vibe Samurai

🥷 Vibe Samurai asks: “That’s wasted time on phrasing! I want fast, not contract-writing.”

Answer: Ten seconds on the formula vs 10 minutes of “it did the wrong thing, I’m redoing.” Feels faster to “be short” — actually slower by multiples. Same as code review: “no need for tests, I write it right anyway” — everyone thought that, then caught bugs in prod. The formula is your mini-test before the request. Not perfect, but actually works.
