B4 · What's a repo and why commit

Git in 20 minutes — only what's needed. PR as a way to say 'stop, look at what I did'.

Hook

WITHOUT git:                   WITH git:
Agent broke something.    →    Agent broke something.
Something forever. 😱          git restore .  ←  everything rolled back 🙂

Hey folks, welcome 👋

This chapter is about the main safety wheel of working with an agent. Before this you could only watch the agent and worry “what if it breaks something” — after it you have a physical roll-back button for any agent mistake. That changes confidence a lot.

We take git exactly as much as needed for safety and the basic flow. Not “git mastery.” Twenty minutes.

Concept: a repository = a folder with memory

A repository (or just “repo”) is your project folder plus the memory of every change in it.

my-project/        ← regular folder
├── index.html
└── style.css

my-project-with-git/    ← repository
├── .git/               ← this is what turns the folder into a repo
├── index.html
└── style.css

The hidden .git/ folder is the “memory.” Inside it sits the history of every change in the project. Deleted a file — it’s in history. Changed a line — old version in history. At any moment you can return to “as it was.”

Why you care, working with an agent

When the agent writes code in a regular folder — every change is irreversible. If it deleted a needed file, rewrote an important function, added a bug — you don’t have a “time machine.”

When the agent writes code in a repository — every change can be rolled back with one command. That’s not theory, that’s just how it works.

The minimum git commands that cover 90 % of tasks

CommandWhat it doesWhen
git initTurn a folder into a repoOnce, at the start of a new project
git statusShow which files changedEach time, as “what do I have now”
git add .Stage all changes for savingBefore a commit
git commit -m "..."Save changes into historyWhen something working landed
git restore .Roll back uncommitted changesWhen you broke something and want it back
git log --onelineShow commit historyWhen you want to know “what have I been doing”

Remember these six. The rest — later.

The “changed → saved” loop in one paragraph

  1. Change files (yourself or via the agent)
  2. git status — check what changed
  3. git add . — stage everything
  4. git commit -m "what I did" — save into history
  5. Back to step 1

If at step 1 you broke something and want to go back to the last commit — git restore .. Everything after the commit disappears. What’s in commits stays.

Example: create a repo and make a first commit

In the terminal:

mkdir my-first-repo
cd my-first-repo
git init

Done, you have a repo. Make a simple file:

echo "# My first repo" > README.md

Look at the state:

git status

Shows “Untracked files: README.md” — meaning “there’s a file, but not under memory yet.” Add to memory:

git add .
git commit -m "first file"

Done. Now break something — say, deleted the file by accident:

rm README.md

Want it back:

git restore README.md

File is back. That’s the time machine in action.

Why commit often

Each commit is a rollback point. The more often you commit, the more points, the safer.

A good rhythm with an agent:

1. Describe the task to the agent by the B3 formula
2. Agent did it → checked → works
3. git add . + git commit -m "short what we did"
4. Next task

So every accepted edit = a commit. Ten tasks a day — ten commits. Normal.

If at some point the agent did something weird and the previous commit was an hour ago — git restore . rolls back to an hour ago. Lost an hour, oh well. If the previous commit was a week ago — a week is lost. Feel the difference?

Principle: commits are free save points. Don’t be stingy. I even commit on “I’m sure it’ll work” first — never know.

GitHub, GitLab, and the rest — just storage

git is a program on your computer. GitHub is a service on the internet where you can store a copy of the repo. And — crucially — where you can make a pull request (PR).

A PR (or MR — Merge Request on GitLab) is a way to propose changes and get a review before they merge into the main branch.

Why you care, working with an agent:

  • When the agent does something serious — it can do it in a separate branch, open a PR, and you review the changes in a nice UI before accepting.
  • In a team a PR is the standard way to say “colleague, look at what I did.”

We won’t dive deep into PR flow at the Beginner stage. Enough to know: git is local, GitHub is “where others can see and discuss.”

Practice (30 minutes)

Task 1 (15 minutes) — your first repository

Walk through the full loop by hand:

mkdir hello-git
cd hello-git
git init

# create a file — any way you like
echo "hello world" > hello.txt

git status         # check the state
git add .
git commit -m "first file"

# change the file
echo "hello git" > hello.txt

git status         # check again
git restore hello.txt    # rolled back

# verify — should be as it was after the commit
cat hello.txt      # prints "hello world"

If the last cat hello.txt printed “hello world,” not “hello git” — congrats, the time machine works.

Task 2 (15 minutes) — connect with the agent

In the same folder (or any new one with git init):

claude

Ask the agent:

“Create a file notes.md with three bullet points about something useful.”

After the agent did it — you yourself in the terminal:

git add .
git commit -m "agent added notes"

Done, you saved the state after the agent’s edit. Next time the agent breaks something — git restore . brings you back exactly here.

Recap

  • Repo = folder + change memory (.git/).
  • Six commands cover 90 % of tasks: init, status, add, commit, restore, log.
  • Commit often — every accepted agent edit = a commit. Each commit = a rollback point.

Vibe Samurai

🥷 Vibe Samurai asks: “I thought git was for serious programmers with teams. I’m solo, why do I need it?”

Answer: Git isn’t about “team work,” it’s about reversibility. A solo developer with git is a developer with a time machine. A solo developer without git only has the present moment. When you work with an agent writing code into your files — a time machine is mandatory, otherwise in two weeks you’ll be afraid to hit “apply.” Not perfect, but actually works — even my personal projects have git, otherwise hours of work disappear on nothing.

📚 Further reading

The six commands from this chapter will last a long time. But sooner or later you’ll wonder “how is git built inside” — for that moment, a short list of official sources.

Books and docs

  • Pro Git (Scott Chacon & Ben Straub, free online book, updated for Git 2.x) — git-scm.com/book/en/v2. The canon. Chapters 1–2 are exactly what I’d cover in a “git deep dive” session.
  • GitHub Docs — About Git (a short explainer of “what’s a VCS” by GitHub themselves): docs.github.com/en/get-started/using-git/about-git.
  • GitHub Docs — Git basics (a minimal tutorial series: init, add, commit, push): docs.github.com/en/get-started/git-basics.

Tip: don’t try to read Pro Git “all in a weekend.” Open the table of contents, read chapters 1–2, the rest — on demand. The book is a reference, not a lecture.
