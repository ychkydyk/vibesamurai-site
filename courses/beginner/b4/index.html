B4 · Что такое репозиторий и зачем коммитить

Git за 20 минут — только то, что нужно. PR как способ сказать «стоп, посмотри что я сделал».

Черновик B4 в голосе DmitriyM (см. scripts/tg_voice_analysis/_summary.md).

Hook

БЕЗ git:                       С git:
Агент что-то сломал.    →     Агент что-то сломал.
Что-то навсегда. 😱           git restore .  ←  всё откатилось 🙂

Друзья, всем привет! 👋

Эта глава — про главное страховое колесо работы с агентом. Если до неё вы могли только смотреть, как агент работает, и переживать «вдруг что-то сломает» — то после неё у вас будет физическая кнопка отката на любую агентскую ошибку. Это сильно меняет уверенность в работе.

Git мы возьмём ровно настолько, насколько нужен для безопасности и базовой работы. Не «git mastery». Двадцать минут.

Концепция: репозиторий = папка с памятью

Репозиторий (или просто «репо») — это папка вашего проекта плюс память всех изменений в ней.

my-project/        ← обычная папка
├── index.html
└── style.css

my-project-with-git/    ← репозиторий
├── .git/               ← вот это превращает папку в репо
├── index.html
└── style.css

Скрытая папка .git/ — это и есть «память». Внутри неё лежит история каждого изменения в проекте. Удалили файл — он есть в истории. Поменяли строку — старая версия в истории. Можете в любой момент вернуть как было.

Зачем это вам с агентом

Когда агент пишет код в обычной папке — каждое изменение необратимо. Если он удалил нужный файл, переписал важную функцию, добавил баг — у вас нет «машины времени».

Когда агент пишет код в репозитории — каждое изменение можно откатить одной командой. Это не теория, это просто рабочий механизм.

Минимум команд git, которого хватит на 90% задач

КомандаЧто делаетКогда
git initПревратить папку в репозиторийОдин раз, в начале нового проекта
git statusПоказать какие файлы изменилисьКаждый раз, как «что у меня сейчас»
git add .Подготовить все изменения к сохранениюПеред коммитом
git commit -m "..."Сохранить изменения в историюКогда что-то рабочее получилось
git restore .Откатить несохранённые измененияКогда что-то сломали и хотите вернуть
git log --onelineПосмотреть историю коммитовКогда хочется понять «что я делал»

Запомните эти шесть. Остальное — потом.

Цикл «изменил → сохранил» одним абзацем

  1. Меняете файлы (сами или агентом)
  2. git status — проверили что изменилось
  3. git add . — приготовили все изменения
  4. git commit -m "что я сделал" — сохранили в историю
  5. Снова к шагу 1

Если на шаге 1 что-то сломалось и хочется вернуть до последнего коммита — git restore .. Всё, что было после коммита, исчезает. То что в коммитах — остаётся.

Пример: создаём репозиторий и делаем первый коммит

В терминале:

mkdir my-first-repo
cd my-first-repo
git init

Готово, у вас репозиторий. Создадим простой файл:

echo "# Мой первый репо" > README.md

Посмотрим состояние:

git status

Покажет «Untracked files: README.md» — то есть «есть файл, но он пока не под памятью». Добавим в память:

git add .
git commit -m "первый файл"

Готово. Теперь поломаем — допустим, удалили файл случайно:

rm README.md

Хотим вернуть:

git restore README.md

Файл на месте. Это и есть машина времени в действии.

Зачем коммитить часто

Каждый коммит — это точка отката. Чем чаще коммитите, тем больше точек, тем безопаснее.

Хороший ритм работы с агентом:

1. Описать задачу агенту по формуле из B3
2. Агент сделал → проверили → работает
3. git add . + git commit -m "коротко что сделали"
4. Следующая задача

То есть каждая принятая правка = коммит. Десять задач в день — десять коммитов. Это нормально.

Если в какой-то момент агент сделал что-то странное и прошлый коммит был час назад — git restore . откатит на час назад. Час работы потеряли, ну ладно. Если же прошлый коммит был неделю назад — неделя работы потеряна. Чувствуете разницу?

Принципиально: коммиты — это бесплатные точки сохранения. Жадничать на них нельзя. Я даже на «I am sure it’ll work» сначала коммичу — мало ли.

GitHub, GitLab и другие — это просто хранилища

git — программа на вашем компьютере. GitHub — это сервис в интернете, где можно хранить копию репозитория. И — критично — где можно делать pull request (PR).

PR (или MR — Merge Request на GitLab) — это способ предложить изменения и получить ревью до того, как их применят в основную ветку.

Зачем это вам с агентом:

  • Когда агент делает что-то серьёзное — он может сделать это в отдельной ветке, открыть PR, и вы смотрите изменения в удобном интерфейсе до того, как принять.
  • В команде PR — это стандартный способ сказать «коллега, посмотри что я сделал».

В этом курсе мы не будем глубоко погружаться в PR-flow на этапе Beginner. Достаточно знать: git — это локально, GitHub — это «там, где другие могут увидеть и обсудить».

Practice (30 минут)

Задание 1 (15 минут) — свой первый репозиторий

Пройдите весь цикл руками:

mkdir hello-git
cd hello-git
git init

# создайте файл — любым способом
echo "hello world" > hello.txt

git status         # посмотрите состояние
git add .
git commit -m "first file"

# измените файл
echo "hello git" > hello.txt

git status         # снова посмотрите
git restore hello.txt    # откатили

# проверьте — должно быть как было после коммита
cat hello.txt      # выведет "hello world"

Если последний cat hello.txt показал «hello world», а не «hello git» — поздравляю, машина времени работает.

Задание 2 (15 минут) — соединить с агентом

В той же папке (или любой новой с git init):

claude

Скажите агенту:

«Создай файл notes.md с тремя bullet-пунктами про что-нибудь полезное.»

После того как агент сделал — сами в терминале:

git add .
git commit -m "agent added notes"

Готово, вы сохранили состояние после агентского изменения. Если следующий раз агент что-то сломает — git restore . вернёт ровно сюда.

Recap

  • Репозиторий = папка + память изменений (.git/).
  • Шесть команд покрывают 90% задач: init, status, add, commit, restore, log.
  • Коммитьте часто — каждая принятая агентская правка = коммит. Каждый коммит = точка отката.

Скептик Самурай

🥷 Скептик Самурай спрашивает: «Я думал git — это для серьёзных программистов с командой. Я один, мне-то зачем?»

Ответ: Git — это не про «командную работу», это про обратимость. Один разработчик с git — это разработчик, у которого есть машина времени. Один разработчик без git — это разработчик, у которого есть только текущий момент. Когда вы работаете с агентом, который пишет код в ваши файлы — машина времени обязательна, иначе через две недели вы будете бояться нажимать «применить». Не идеально, но реально работает — даже у меня в личных проектах есть git, потому что иначе теряются часы работы на ровном месте.
