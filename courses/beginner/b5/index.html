Друзья, всем привет! 👋

Сорок-шестьдесят минут, и у вас в браузере реальное работающее todo-приложение, которое вы сделали с агентом, в репозитории, с коммитами. Это первый момент в курсе, когда абстрактные понятия из B1-B4 становятся полным фаршем — конкретным проектом, который работает.

И главное — мы делаем это не отступая от правил: режим «только посмотреть» в начале, формула запроса из B3, коммит после каждой принятой правки. Не идеально, но реально работает.

Концепция: end-to-end agent flow

Любой проект с агентом — независимо от размера — следует одному паттерну:

1. Сетап             — папка + git init
2. Постановка        — что хотим получить, по формуле B3
3. План (не код!)    — попросили агента план, прочитали
4. Первая итерация   — агент делает базовое
5. Запуск            — открыли результат, посмотрели
6. Правки            — что-то надо изменить, говорим агенту
7. Коммит            — после каждой принятой правки
8. Повтор 5-7        — пока не «готово»

В этой главе мы пройдём весь паттерн на простом, но реальном проекте: todo-приложение в браузере.

Пример: делаем todo с агентом

Шаг 1. Сетап

mkdir my-todo
cd my-todo
git init

Открываем агента:

claude

Шаг 2. Постановка (по формуле B3)

В чате Claude:

Что: Сделать простое todo-приложение в браузере Где: Текущая пустая папка Как поймём готовым:

  • Открыл index.html в браузере → вижу заголовок «Мой todo-лист», поле ввода, кнопку «Добавить», список задач
  • Добавил задачу → она появилась в списке
  • Нажал на задачу → она зачеркнулась как выполненная
  • Перезагрузил страницу → задачи сохранились (localStorage)

Технологии: чистый HTML/CSS/JS без библиотек, всё в одной папке, без сборки. Сначала покажи план перед тем как начнёшь писать код.

Это типичная постановка для начального проекта. Заметили — пять чётких критериев готовности. Нет «красиво», нет «удобно». Конкретно.

Шаг 3. План (не код!)

Агент покажет план — обычно три-четыре файла:

  • index.html — структура страницы
  • style.css — оформление
  • app.js — логика
  • README.md — описание

И список шагов: создать HTML, добавить стиль, реализовать логику, протестировать. Прочитайте план. Если согласны — «Хорошо, делай». Если хочется поправить — поправьте сейчас, а не потом.

Шаг 4. Первая итерация

Агент создаёт файлы. В терминале можно отслеживать:

ls           # увидеть появившиеся файлы
git status   # увидеть что нового

Шаг 5. Запуск

Откройте index.html в браузере. На macOS:

open index.html

На Windows:

start index.html

На Linux:

xdg-open index.html

Или просто двойной клик в файловом менеджере.

Появится страница. Попробуйте добавить пару задач, кликнуть, перезагрузить.

Шаг 6. Что-то не так? — правки

Допустим, оказалось, что:

  • Шрифт мелковат
  • Хочется удалять задачи кнопкой ❌
  • Кнопка «Добавить» по-английски, а хочется на русском

Скажите агенту:

Хорошо, работает. Три правки:

  1. Увеличь шрифт до 18px для списка задач
  2. Добавь кнопку ❌ возле каждой задачи — клик удаляет задачу
  3. Замени текст кнопки «Add» на «Добавить»

После правок я хочу видеть только эти три изменения, остальное не трогай.

Агент применит — посмотрите в браузере (нажмите F5).

Шаг 7. Коммит после каждой принятой правки

Хороший результат → сохранили в историю:

git add .
git commit -m "init todo: add, complete, persist"

После следующей пачки правок:

git add .
git commit -m "ui polish: font, delete button, ru text"

Через пару итераций у вас будет короткая, понятная история. Можно в любой момент откатить на любой из шагов.

Шаг 8. Повтор до «готово»

Что ещё хочется? Сортировка? Категории? Дедлайны? Лично меня обычно тянет залип в детали — на этом этапе остановитесь, как только todo решает вашу задачу. Слишком много фич = непочатый край и проект не доделается никогда.

Что делать когда не работает

Случай 1. Открыли index.html — пустая страница

  • Откройте в браузере Developer Tools (Cmd+Option+I / F12) → вкладка Console
  • Скопируйте ошибку
  • Скажите агенту: «В консоли браузера ошибка <…>. Исправь.»

Случай 2. Задачи добавляются, но при перезагрузке исчезают

Скажите агенту: «Задачи не сохраняются после перезагрузки. Проверь работу с localStorage в app.js, исправь.»

Случай 3. Агент сделал какую-то ересь, всё стало хуже

Не пытайтесь починить «починкой поверх починки». Откатывайтесь на последний рабочий коммит:

git restore .

И заново сформулируйте, что хотели, по формуле.

Случай 4. Совсем ничего не понятно

Скажите агенту: «Объясни мне в одном абзаце, что делает файл app.js. Без кода, простыми словами».

Это не сдача. Это нормальный шаг — попросить explainer, а не сразу «исправь».

Practice (40-60 минут)

Задание 1 (40 минут) — пройдите главу руками

От начала до конца. Свой my-todo, свои правки, свои коммиты. Не пропускайте git add . + git commit после каждой принятой правки — это самая важная привычка.

Задание 2 (20 минут) — добавьте свою фичу

Когда базовое todo работает — придумайте одну фичу, которая лично вам нужна. Что угодно: цветовые метки, поиск, экспорт в файл. Сформулируйте по формуле B3 и попросите агента. Закоммитьте.

Recap

  • End-to-end flow одинаков для любого проекта: сетап → постановка → план → итерация → запуск → правки → коммит.
  • Коммит после каждой принятой правки. Это базовая страховка от агентских косяков.
  • Не починка через починку. Если что-то стало хуже после правки — откат, потом переформулировать.

Скептик Самурай

🥷 Скептик Самурай спрашивает: «Я мог бы сам написать todo-лист за час. В чём смысл делать это через агента?»

Ответ: Смысл не в этом конкретном todo. Смысл — в том, что вы тренируете workflow, который потом применяете к проектам, которые сами не могли бы сделать. Через две недели по этой схеме вы сможете попросить агента сделать ваш личный helper для каких-то задач, которые до этого вы делали руками часами. Todo-приложение — это «велосипед без втулок» из автошколы. Никто на нём не ездит, на нём учатся держать руль. Не идеально, но реально работает.
