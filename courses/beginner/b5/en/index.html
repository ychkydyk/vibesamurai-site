B5 · Building the first project (todo app)

End-to-end: empty folder → agent writes → we run → we fix. What to do when it doesn't work.

Hey folks, welcome 👋

Forty to sixty minutes and you have a real working todo app in the browser that you built with an agent, in a repo, with commits. The first moment in the course where the abstractions of B1–B4 become the full package — a concrete project that works.

And — important — we do this without breaking the rules: “look, don’t touch” mode at the start, the B3 request formula, commit after each accepted edit. Not perfect, but actually works.

Concept: end-to-end agent flow

Any project with an agent — any size — follows one pattern:

1. Setup            — folder + git init
2. Brief            — what we want, by the B3 formula
3. Plan (not code!) — ask the agent for a plan, read it
4. First iteration  — agent builds the base
5. Run              — open the result, take a look
6. Edits            — say what to change
7. Commit           — after every accepted edit
8. Repeat 5-7       — until "done"

In this chapter we walk the whole pattern on a simple but real project: a browser todo app.

On a napkin the loop looks like:

flowchart LR
    S([Setup<br/>folder + git]) --> T[Brief<br/>WHAT / WHERE / HOW WE KNOW]
    T --> P[Agent writes the plan<br/>you read]
    P --> D[Agent acts<br/>file diff]
    D --> R[Run<br/>verify in browser]
    R -->|ok| C([Commit])
    R -->|edits| E[Small edit<br/>to the agent]
    E --> D
    C -.-> E

Inside one iteration — the three phases from B1 (plan → act → evaluate), and a commit turns “seems to work” into a rollback point you can always return to.

Example: building todo with the agent

Step 1. Setup

mkdir my-todo
cd my-todo
git init

Open the agent:

claude

Step 2. Brief (by the B3 formula)

In the Claude chat:

What: Build a simple browser todo app Where: Current empty folder How we know it’s done:

  • Open index.html in a browser → see the heading “My todo list,” an input field, an “Add” button, a list of tasks
  • Added a task → it appears in the list
  • Clicked a task → it strikes through as completed
  • Reloaded the page → tasks persisted (localStorage)

Tech: plain HTML/CSS/JS, no libraries, all in one folder, no build step. Show the plan first before writing any code.

That’s a typical brief for a starter project. Notice — five clear acceptance criteria. No “pretty,” no “convenient.” Concrete.

Step 3. Plan (not code!)

The agent shows the plan — usually three or four files:

  • index.html — page structure
  • style.css — styling
  • app.js — logic
  • README.md — description

Plus a step list: create HTML, add styles, implement logic, test. Read the plan. If you agree — “Good, go.” If you want to adjust — adjust now, not later.

Step 4. First iteration

The agent creates the files. In the terminal you can follow along:

ls           # see new files
git status   # see what's new

Step 5. Run

Open index.html in a browser. macOS:

open index.html

Windows:

start index.html

Linux:

xdg-open index.html

Or just a double-click in the file manager.

A page appears. Add a couple of tasks, click, reload.

Step 6. Something off? — edits

Say it turned out:

  • Font is small
  • You want a ❌ button to delete tasks
  • The “Add” button is in English, you want it in Russian

Tell the agent:

Good, it works. Three edits:

  1. Increase the font size to 18px for the task list
  2. Add a ❌ button next to every task — click deletes the task
  3. Replace the “Add” text on the button with “Добавить”

After the edits I want to see only these three changes, don’t touch anything else.

The agent applies — check the browser (F5).

Step 7. Commit after every accepted edit

A good result → save it to history:

git add .
git commit -m "init todo: add, complete, persist"

After the next batch of edits:

git add .
git commit -m "ui polish: font, delete button, ru text"

A couple of iterations later you have a short clear history. You can roll back to any step.

Step 8. Repeat to “done”

Want more? Sort? Categories? Deadlines? Personally I drift into details — at this stage stop as soon as the todo solves your problem. Too many features = endless tail and the project never ships.

What to do when it doesn’t work

Case 1. Opened index.html — blank page

  • Open Developer Tools in the browser (Cmd+Option+I / F12) → Console tab
  • Copy the error
  • Tell the agent: “Browser console error <…>. Fix it.”

Case 2. Tasks add but disappear on reload

Tell the agent: “Tasks don’t persist after reload. Check the localStorage logic in app.js, fix it.”

Case 3. The agent did something weird, things got worse

Don’t try to fix “fix on top of fix.” Roll back to the last working commit:

git restore .

And re-state what you wanted, by the formula.

Case 4. Nothing makes sense at all

Tell the agent: “Explain to me in one paragraph what app.js does. No code, plain words.”

That’s not surrender. It’s a normal step — ask for an explainer, not just “fix.”

Practice (40–60 minutes)

Task 1 (40 minutes) — walk the chapter by hand

Start to finish. Your my-todo, your edits, your commits. Don’t skip git add . + git commit after every accepted edit — that’s the most important habit.

Task 2 (20 minutes) — add your own feature

Once the base todo works — pick one feature you personally want. Anything: colour tags, search, export to file. Phrase it by the B3 formula and ask the agent. Commit.

Recap

  • End-to-end flow is the same for any project: setup → brief → plan → iteration → run → edits → commit.
  • Commit after every accepted edit. Base insurance against agent mistakes.
  • No fix-on-fix. If something got worse after an edit — roll back, then reformulate.

Vibe Samurai

🥷 Vibe Samurai asks: “I could write a todo list myself in an hour. What’s the point of doing it through an agent?”

Answer: The point isn’t this specific todo. The point is that you train the workflow which you then apply to projects you couldn’t build yourself. In two weeks of this rhythm you’ll ask the agent to build a personal helper for tasks you used to do by hand for hours. The todo app is the “no-handlebar bike” of driving school. Nobody rides it — they learn to hold the wheel. Not perfect, but actually works.
