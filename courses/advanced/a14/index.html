A14 · Локальные LLM на 8GB VRAM

llama.cpp b9050 + Qwen3.5-21B-Deckard IQ4_XS на 4070 8GB + 64GB RAM, ~5 t/s, OpenAI-совместимый API на 127.0.0.1:8080. Когда хватает, когда нет.

Что строим

Локальный LLM-узел на потребительском железе (8GB VRAM + 64GB RAM): запускаем 21B-модель в IQ4_XS квантизации, обслуживаем OpenAI-совместимый API на 127.0.0.1:8080, используем для рутинных задач (классификация, курация, фильтрация перед Sonnet).

Базовый стек

КомпонентВерсияЗачем
llama.cppb9050+Inference engine с offload в RAM
Qwen3.5-21B-DeckardIQ4_XS, 11.8GBМодель с разумным качеством на потреб. железе
OpenAI API endpoint127.0.0.1:8080Совместимость с любыми OpenAI SDK
Local diskE:\LLM\Хранилище gguf-файлов

Без llama.cpp 21B на 8GB VRAM не помещается — нужен offload в RAM, и для этого нужно много system RAM. 32GB маловато, 64GB — комфортно.

IQ4_XS: почему именно эта квантизация

Квантизация сжимает веса с 16-bit float до 4-bit (или меньше). IQ4_XS — i-quant на 4 бита (extra small), баланс между размером и качеством.

Q8_0   ────► 22 GB   качество ~100%   не помещается на 8GB+64GB
Q6_K   ────► 17 GB   качество ~98%    не помещается
Q5_K_M ────► 15 GB   качество ~96%    тяжело, медленно
IQ4_XS ────► 11.8 GB качество ~92%    нормально, 5 t/s    ← наш выбор
IQ3_M  ────►  9.4 GB качество ~85%    быстрее, но качество просаживается

Для классификации/фильтрации/курации 92% — больше, чем достаточно. Финальный текст — всегда уходит в Sonnet/Opus.

Запуск llama-server

& "E:\LLM\llama.cpp\build\bin\Release\llama-server.exe" `
    --model "E:\LLM\models\qwen3.5-21b-deckard-iq4_xs.gguf" `
    --host 127.0.0.1 --port 8080 `
    --ctx-size 16384 `
    --n-gpu-layers 28 `
    --threads 12 `
    --batch-size 512 `
    --parallel 1 `
    --no-mmap

Параметры:

  • --n-gpu-layers 28 — слои на GPU; остальное в RAM. 28 — эмпирический максимум для 4070 8GB при IQ4_XS;
  • --ctx-size 16384 — контекст 16k;
  • --no-mmap — на Windows иногда улучшает стабильность при offload;
  • --host 127.0.0.1 — обязательно (никогда 0.0.0.0, правило feedback_no_wildcard_http_server).

OpenAI-совместимый API

llama-server отдаёт /v1/chat/completions в формате OpenAI. Любой клиент работает без модификаций:

from openai import OpenAI

client = OpenAI(base_url="http://127.0.0.1:8080/v1", api_key="not-needed")

resp = client.chat.completions.create(
    model="qwen3.5-21b-deckard",
    messages=[
        {"role": "system", "content": "Ты курируешь контент. Отвечай YES/NO."},
        {"role": "user", "content": "Это пост про игры? «Сегодня вышла новая игра...»"},
    ],
    temperature=0.0,
    max_tokens=10,
)
print(resp.choices[0].message.content)  # → "YES"

Поэтому переключение с Claude API на локальную модель — это один URL.

Реальный кейс: курация и фильтрация

Из memory feedback_local_qwen_for_analysis: для курации/классификации/верификации использовать Qwen3.5-21B на 127.0.0.1:8080, не Claude API.

Сценарии, в которых локальный Qwen ставится по умолчанию:

  • классификация поста (категория, тон, наличие ссылок);
  • фильтрация перед RAG (см. A13: «хватает ли контекста?»);
  • верификация фактов (extract + cross-check внутри документа);
  • summarization до 500 слов;
  • генерация черновика для последующей доводки в Sonnet.

Финальный текст для клиента, важные решения, креатив для бренда — Sonnet/Opus.

Производительность

На 4070 8GB + 64GB RAM + Qwen3.5-21B IQ4_XS:

МетрикаЗначение
Tokens/sec (output)~5 t/s
Prompt eval (16k tokens)~30 секунд
Latency первого токена1–2 сек
Стабильность24/7 ok при правильном охлаждении

5 t/s — это медленно для интерактивных UI, но нормально для пакетной обработки: 1000 постов по 200 токенов ответа = ~40 000 токенов / 5 t/s = ~133 минуты на партию. Запускаем ночью / в фоне.

Когда локального не хватает

  • Финальный креативный текст — копирайт под бренд, КП, договор. Качество 92% IQ4_XS заметно.
  • Сложный reasoning — multi-step рассуждение, математика, code generation для прод-кода.
  • Запросы в реальном времени с высокой нагрузкой — 5 t/s не выдержит даже 10 одновременных юзеров.
  • Свежие факты — модель замороженная. Для freshness — RAG поверх (см. A13) или Sonnet с поиском.

Подводные камни

  • OOM при первом запуске — слишком много --n-gpu-layers. Снижайте, пока не помещается.
  • Контекст 16k реально занят процентов на 70% — модель работает корректно до ~10k токенов входа. Выше — деградация.
  • --no-mmap на не-Windows — наоборот, замедляет. Параметр платформо-зависимый.
  • Long-form generation — модель уходит в повторы или галлюцинации после ~1500 токенов output. Делите задачу.
  • Файл gguf скачан из ненадёжного источника — потенциальный supply chain. Качайте только с официальных HF репо.

Recap

  • llama.cpp + Qwen3.5-21B-Deckard IQ4_XS на 4070 8GB + 64GB RAM = ~5 t/s.
  • OpenAI-совместимый API на 127.0.0.1:8080 — один URL для совместимости со SDK.
  • Локальный узел — для курации, классификации, фильтрации, summarization.
  • Финал, креатив, сложный reasoning — в Sonnet/Opus.
  • 5 t/s годится для пакетной обработки, не для интерактивных UI.
