A14 · Local LLMs on 8GB VRAM

llama.cpp b9050 + Qwen3.5-21B-Deckard IQ4_XS on a 4070 8GB + 64GB RAM, ~5 t/s, OpenAI-compatible API on 127.0.0.1:8080. When it's enough, when it isn't.

What we’re building

A local LLM node on consumer hardware (8 GB VRAM + 64 GB RAM): run a 21B model in IQ4_XS quantisation, serve an OpenAI-compatible API on 127.0.0.1:8080, and use it for the routine tasks (classification, curation, filtering before Sonnet).

Base stack

ComponentVersionWhy
llama.cppb9050+Inference engine with RAM offload
Qwen3.5-21B-DeckardIQ4_XS, 11.8 GBModel with reasonable quality on consumer hardware
OpenAI API endpoint127.0.0.1:8080Compatibility with any OpenAI SDK
Local diskE:\LLM\gguf file storage

Without llama.cpp, 21B doesn’t fit on 8 GB VRAM — RAM offload is required, and that needs a lot of system RAM. 32 GB is tight, 64 GB is comfortable.

IQ4_XS: why this quantisation

Quantisation compresses weights from 16-bit float to 4-bit (or less). IQ4_XS is an i-quant at 4 bits (extra small), the balance point between size and quality.

Q8_0   ────► 22 GB   quality ~100%   doesn't fit on 8GB+64GB
Q6_K   ────► 17 GB   quality ~98%    doesn't fit
Q5_K_M ────► 15 GB   quality ~96%    tight, slow
IQ4_XS ────► 11.8 GB quality ~92%    fine, 5 t/s    ← our pick
IQ3_M  ────►  9.4 GB quality ~85%    faster but noticeable quality drop

For classification / filtering / curation 92 % is more than enough. Final text always goes to Sonnet / Opus.

Running llama-server

& "E:\LLM\llama.cpp\build\bin\Release\llama-server.exe" `
    --model "E:\LLM\models\qwen3.5-21b-deckard-iq4_xs.gguf" `
    --host 127.0.0.1 --port 8080 `
    --ctx-size 16384 `
    --n-gpu-layers 28 `
    --threads 12 `
    --batch-size 512 `
    --parallel 1 `
    --no-mmap

Parameters:

  • --n-gpu-layers 28 — layers on GPU; the rest in RAM. 28 is the empirical max for a 4070 8 GB on IQ4_XS;
  • --ctx-size 16384 — 16k context;
  • --no-mmap — on Windows sometimes improves stability under offload;
  • --host 127.0.0.1 — mandatory (never 0.0.0.0, rule feedback_no_wildcard_http_server).

OpenAI-compatible API

llama-server returns /v1/chat/completions in OpenAI format. Any client works without modification:

from openai import OpenAI

client = OpenAI(base_url="http://127.0.0.1:8080/v1", api_key="not-needed")

resp = client.chat.completions.create(
    model="qwen3.5-21b-deckard",
    messages=[
        {"role": "system", "content": "You curate content. Answer YES/NO."},
        {"role": "user", "content": "Is this a gaming post? 'A new game came out today...'"},
    ],
    temperature=0.0,
    max_tokens=10,
)
print(resp.choices[0].message.content)  # → "YES"

Which is why switching from Claude API to a local model is one URL.

Real case: curation and filtering

From memory feedback_local_qwen_for_analysis: for curation / classification / verification use Qwen3.5-21B on 127.0.0.1:8080, not Claude API.

Scenarios where the local Qwen is the default:

  • post classification (category, tone, presence of links);
  • pre-RAG filtering (see A13: “is the context enough?”);
  • fact verification (extract + cross-check inside the document);
  • summarisation up to 500 words;
  • a draft generation for later polishing in Sonnet.

Final text for the client, important decisions, brand creative — Sonnet / Opus.

Performance

On a 4070 8 GB + 64 GB RAM + Qwen3.5-21B IQ4_XS:

MetricValue
Tokens/sec (output)~5 t/s
Prompt eval (16k tokens)~30 seconds
First-token latency1–2 s
Stability24/7 with proper cooling

5 t/s is slow for interactive UIs but fine for batch processing: 1000 posts × 200 tokens per answer = ~40 000 tokens / 5 t/s = ~133 minutes per batch. Run overnight or in the background.

When local isn’t enough

  • Final creative text — brand copy, KP, contracts. The 92 % quality of IQ4_XS is noticeable here.
  • Hard reasoning — multi-step reasoning, mathematics, code generation for production.
  • Real-time requests under load — 5 t/s won’t survive even 10 concurrent users.
  • Fresh facts — the model is frozen. For freshness, layer RAG (see A13) or Sonnet with search.

Pitfalls

  • OOM on first start — too many --n-gpu-layers. Reduce until it fits.
  • 16k context is realistically ~70 % usable — the model behaves correctly up to ~10k input tokens. Above that — degradation.
  • --no-mmap on non-Windows — actually slows things down. The flag is platform-dependent.
  • Long-form generation — the model loops or hallucinates after ~1500 output tokens. Split the task.
  • gguf file from an untrusted source — potential supply chain risk. Download only from official HF repos.

Recap

  • llama.cpp + Qwen3.5-21B-Deckard IQ4_XS on a 4070 8 GB + 64 GB RAM = ~5 t/s.
  • OpenAI-compatible API on 127.0.0.1:8080 — one URL for SDK compatibility.
  • Local node — for curation, classification, filtering, summarisation.
  • Final, creative, hard reasoning — Sonnet / Opus.
  • 5 t/s fits batch processing, not interactive UIs.
