A9 · TouchDesigner + StreamDiffusion

Real-time generative video in TD: feeding audio/CV into StreamDiffusion, TdrDelay under Vulkan, the installation latency budget. Real-world traps when getting the patch up.

What we’re building

A real-time generative video installation: TouchDesigner takes audio / CV / MIDI input, feeds StreamDiffusion a condition image, and outputs a stylised video stream to a projector or screen with <200 ms latency.

Architecture

[audio/CV/MIDI] ─► TouchDesigner ─► tex_in ─► StreamDiffusion ─► tex_out ─► TD compositor ─► PROJ
                          │                       (LCM,           │
                          │                        SDXL Turbo,    │
                          ▼                        FLUX schnell)  │
                  control params ────────────────────────────────►│
                                                                  ▼
                                                              recording

The StreamDiffusion input condition is not “muted audio” — it’s a condition image: a TD scene render, a CV input from a camera, or a microspectrogram. Pass it via a shared GPU texture or a local named pipe / WS.

StreamDiffusion backends (May 2026)

BackendSpeedQualityWhen
LCM-LoRA on SDXL30–60 fps on a 4070mediumbaseline realtime
SDXL Turbo20–40 fpsgoodmost installations
FLUX schnell8–15 fps on a 4070very goodslow-art mode
TRT-StreamDiffusion60+ fps on a 4070goodbooths, heavy scenes

The installation latency budget is usually <200 ms total (input → output to projector); above that the audience catches the desync.

TdrDelay under Vulkan: a critical step

From production (memory feedback_tdr_for_trt): before running TRT-StreamDiffusion in TD you must set TdrDelay = 60 in the Windows registry and reboot the machine. Without this, Vulkan crashes with a fatal in load_config.

PowerShell snippet:

# Run as Administrator
New-ItemProperty -Path "HKLM:\SYSTEM\CurrentControlSet\Control\GraphicsDrivers" `
  -Name TdrDelay -Value 60 -PropertyType DWORD -Force
Restart-Computer

After the reboot, TRT load_config no longer crashes. This is a required step, not a “might be useful.”

Typical bug when bringing up TD + StreamDiffusion

One of the most common blockers when first wiring TD + StreamDiffusion on Windows: TouchDesigner’s Python doesn’t see the StreamDiffusion package installed in a separate venv. Symptom: importing inside TD fails with No module 'StreamDiffusion' despite a clean pip install. Fix — site.addsitedir() in the TD project’s init script.

The fix:

# init.py in the TD project
import site, sys
from pathlib import Path

VENV = Path(r"C:\path\to\streamdiffusion-venv")
SITE_PKG = VENV / "Lib" / "site-packages"
site.addsitedir(str(SITE_PKG))

import StreamDiffusion  # imports now

TD launches its own Python by default and doesn’t see your venv. site.addsitedir() adds the path to sys.path and registers .pth files. A plain sys.path.append is not enough.

After applying — Re-init the TD context. Without Re-init the old sys.path stays in memory.

Passing parameters from TD into SD

Parameters (prompt strength, denoise, cfg, controlnet weight) change in real time from TD parameters. Pass them through a shared dict or OSC:

# in the StreamDiffusion runner
params = td_bridge.get_params()  # {"prompt_strength": 0.8, "cfg": 1.5, ...}
sd_pipeline.update(**params)
frame = sd_pipeline(condition_tex)
td_bridge.push_frame(frame)

td_bridge is a thin named-pipe or Spout/Syphon channel — not HTTP. HTTP calls between TD and SD add 5–10 ms per request, which is unacceptable for realtime.

Latency budget

┌─────────────────────────────────────────────┐
│  INPUT (camera/audio/MIDI) →  TD            │   ~20 ms
│  TD compositor →  shared texture            │   ~15 ms
│  StreamDiffusion inference                  │   ~40-100 ms
│  Output texture →  TD                       │   ~15 ms
│  TD compositor →  projector                 │   ~30 ms
├─────────────────────────────────────────────┤
│  TOTAL                                      │   ~120-180 ms
└─────────────────────────────────────────────┘

If the budget overflows (you see “lag” in the projection vs the music), degrade: drop the condition texture size to 512×512, switch from FLUX schnell to SDXL Turbo, disable controlnet.

Pitfalls

  • “I have a 4090, it’ll just run” — on 8 GB VRAM (4070) FLUX schnell does ~10 fps. On a 4090 it’s better, but not “no budget.” Model size and step count call the tune.
  • TD main-thread hang — a known issue: a TD project hangs on the WebSocket port (typically :44444/:7878) even without an active WS client. Checklist: par_bindings / pack_json / frame_exec. Don’t diagnose by eye.
  • Vulkan on NVIDIA drivers <555 — unstable with StreamDiffusion. Upgrade to the latest stable driver before the installation.
  • Warm model in GPU cache — the first frame after start is 500+ ms, then normal. Warm up before the public launch.
  • Overheating on long installations — 4 hours of realtime inference on a 4070 sits at 70–80 °C. Plan cooling or throttle mode.

Recap

  • Stack: TD (input + compositor) + StreamDiffusion (inference) + shared texture/OSC (bridge).
  • TdrDelay = 60 + reboot — required for TRT-StreamDiffusion.
  • Pass parameters via named pipe / Spout / Syphon, not HTTP.
  • Latency budget ~120–180 ms; degradation path is declared up front.
  • Warm the model before the audience; control temperature.
