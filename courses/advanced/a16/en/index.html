A16 · Custom MCP servers

When to write your own MCP, when to take someone else's. Security audit of a third-party node. Allow-list, brand-tokens and media-archive as MCPs.

What we’re building

Your own MCP (Model Context Protocol) server that exposes a studio-specific resource as an agent tool: brand tokens, media archive, generation queue, telemetry. Plus a method for safely installing third-party MCPs.

Middle (M5) covered the MCP basics. Advanced is about applied use and safety stuff on a real stack.

Protocol state (May 2026)

The current MCP spec is 2025-11-25 (under the Linux Foundation Agentic AI). Since spec 2025-03-26 the recommended remote transport is Streamable HTTP (replaced SSE). Local servers still run over stdio.

Python SDK: pip install mcp. Primary framework — FastMCP with @tool / @resource / @prompt decorators.

When to write your own MCP

Custom MCP is justified when:

  1. The resource is unique to your studio — custom brand tokens, project layout, local indexes.
  2. An action is safer wrapped than left as a bash script — the agent shouldn’t see rm -rf, the agent should see delete_artifact(id).
  3. Audience is broader than one project — reused across all studio projects (brand MCP, archive MCP).
  4. The off-the-shelf MCP doesn’t cover an edge case — a thin wrapper is cheaper than forking someone else’s server.

Not justified if a shell command via bash-tool with permissions handles it.

Minimum MCP server (FastMCP)

# mcp_brand_tokens/server.py
from mcp.server.fastmcp import FastMCP
from pathlib import Path

mcp = FastMCP("brand-tokens")
TOKENS_DIR = Path(__file__).parent.parent / "brands"

@mcp.tool()
def list_brands() -> list[str]:
    """List available brands."""
    return [p.stem for p in TOKENS_DIR.glob("*.json")]

@mcp.tool()
def get_brand(brand: str) -> str:
    """Return JSON brand tokens for the given brand name."""
    path = TOKENS_DIR / f"{brand}.json"
    if not path.exists():
        return f"brand {brand} not found"
    return path.read_text()

if __name__ == "__main__":
    mcp.run(transport="stdio")

15 lines instead of hand-registering list_tools / call_tool. FastMCP builds the JSON schema from type hints and the docstring.

Registration in ~/.claude.json:

{
  "mcpServers": {
    "brand-tokens": {
      "command": "python",
      "args": ["-m", "mcp_brand_tokens.server"]
    }
  }
}

Rule: on any change to mcpServers in ~/.claude.json, synchronously update the MCP control panel (typical location — mcp-admin/mcps.json next to the project or in the studio’s tooling repo).

Security audit for a third-party MCP

The main rule: before installing any third-party MCP / custom node — a mandatory check:

  1. GitHub link to the source. Not “tried it, it worked,” but a concrete repo.
  2. Code audit:
    • network calls (what’s sent outside);
    • reads from .env / ~/.ssh / Documents;
    • eval/exec/subprocess on user input;
    • persistent processes (cron, scheduled tasks);
    • hooks/install scripts (what runs at install time);
    • transitive dependencies.
  3. Explicit “yes” from the user. Not “I see a green logo, fine.”

For skills additionally:

  • npx skills add always without -y, manual confirmation;
  • read SKILL.md in full — look for hidden directives, exfiltration, zero-width chars, transitive installers.

These rules save you from incidents that would otherwise pile up.

Allow-list

After installing an MCP — explicitly allow only the needed tools:

{
  "permissions": {
    "allow": [
      "mcp:brand-tokens:get_brand",
      "mcp:brand-tokens:list_brands"
    ],
    "deny": [
      "mcp:brand-tokens:write_*"
    ]
  }
}

Allow-list, not “everything allowed.” This is a Middle rule (M4 — Tool inventory & permissions), but studios slack on it for their own MCPs in particular. Don’t.

Real case: MCP admin panel

A typical configuration for one studio: 5 active MCPs (fs, playwright, github, comfyui, davinci) + a control panel (mcp-admin/) + a GUIDE.md describing each node.

Architecture:

  • mcps.json — single source of truth for installed MCPs;
  • ~/.claude.json — synced from mcps.json;
  • GUIDE.md — what for, how to enable/disable;
  • security_monitor (background PS monitor at C:\Claude\security_monitor\) — watches ~/.claude.json, alerts when something appears outside the panel.

The MCP admin is the “single console” for the agent’s tooling. Without it, three months later you have 15 MCP servers and no idea what they do.

MCP for a media archive

From archive production (see A13): MCP archive-search wraps the RAG stack and gives the agent a single tool search(query: str, k: int). The agent doesn’t know about BGE-M3, LanceDB, the reranker — those are implementation details.

@mcp.tool()
def search(query: str, k: int = 5) -> list[dict]:
    """Search the private archive (TG, files, audio).
    Returns top-k fragments with sources."""
    return rag_query(query, k=k)

Advantage over raw-RAG: the agent doesn’t leak extra data (only top-k reaches the context); the agent doesn’t get confused about embedder parameters.

Pitfalls

  • “Installed an MCP without looking” — the most common supply-chain channel. Apply the security audit strictly.
  • ~/.claude.json out of sync with the admin — you forgot to update mcps.json, a month later it’s unclear what’s wired.
  • MCP without allow-list — every one of its tools is reachable by the agent. At minimum, deny-list destructive operations.
  • Heavy MCP at every startup — if an MCP takes 3 s to start, the agent waits for it on every session. Long-running MCPs are separate daemons.
  • MCP logging on, secrets in logs — the default logger writes tool-call arguments. Mask sensitive fields.

Recap

  • Custom MCP — for unique studio resources and for wrapping dangerous commands.
  • Minimum MCP server with FastMCP — 15 lines of Python.
  • Security audit for a third-party MCP is mandatory (GitHub link → code audit → explicit yes).
  • Allow-list, not “everything allowed.”
  • MCP admin + security_monitor — a single control console.
  • MCP over RAG — hides implementation and limits leakage.
