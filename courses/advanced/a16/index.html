A16 · Кастомные MCP-серверы

Когда писать свой MCP, когда брать чужой. Security audit стороннего узла. Allow-list, инструменты бренд-токенов и медиа-архива как MCP.

Что строим

Собственный MCP (Model Context Protocol) сервер, который превращает специфический ресурс студии в инструмент агента: бренд-токены, медиа-архив, очередь генерации, telemetry. Плюс — методику безопасной установки чужих MCP.

Middle (M5) закрыл базу MCP. Advanced — про прикладное использование и safety stuff на боевом стеке.

Когда писать свой MCP

Свой MCP оправдан, когда:

  1. Ресурс уникален для вашей студии — кастомные бренд-токены, проектная структура, локальные индексы.
  2. Действие безопаснее обернуть, чем оставить bash-скрипт — agent не должен видеть rm -rf, agent должен видеть delete_artifact(id).
  3. Нужна аудитория шире чем один проект — переиспользуется во всех проектах студии (бренд-MCP, архив-MCP).
  4. Готовый MCP не покрывает edge case — написание тонкого враппера дешевле, чем форк чужого сервера.

Не оправдан, если можно решить shell-командой через bash-tool с разрешениями.

Состояние протокола (2026-05)

Текущая spec MCP — 2025-11-25 (Linux Foundation Agentic AI). С spec 2025-03-26 рекомендуемый remote-transport — Streamable HTTP (заменил SSE). Локальные серверы по-прежнему ездят через stdio.

Python SDK: pip install mcp. Основной фреймворк — FastMCP с decorators @tool / @resource / @prompt.

Минимальный MCP сервер (FastMCP)

# mcp_brand_tokens/server.py
from mcp.server.fastmcp import FastMCP
from pathlib import Path

mcp = FastMCP("brand-tokens")
TOKENS_DIR = Path(__file__).parent.parent / "brands"

@mcp.tool()
def list_brands() -> list[str]:
    """Список доступных брендов."""
    return [p.stem for p in TOKENS_DIR.glob("*.json")]

@mcp.tool()
def get_brand(brand: str) -> str:
    """Возвращает JSON с бренд-токенами по имени бренда."""
    path = TOKENS_DIR / f"{brand}.json"
    if not path.exists():
        return f"бренд {brand} не найден"
    return path.read_text()

if __name__ == "__main__":
    mcp.run(transport="stdio")

15 строк вместо ручной регистрации list_tools / call_tool. FastMCP сам строит JSON-схему из type hints и docstring.

Регистрация в ~/.claude.json:

{
  "mcpServers": {
    "brand-tokens": {
      "command": "python",
      "args": ["-m", "mcp_brand_tokens.server"]
    }
  }
}

Правило (memory feedback_mcp_panel_sync): при любом изменении mcpServers в ~/.claude.json синхронно править C:\Claude\mcp-admin\mcps.json — это панель управления.

Security audit стороннего MCP

Главное правило (memory feedback_comfyui_node_security): перед установкой любого стороннего MCP/custom node — обязательный чек:

  1. GitHub-ссылка на код. Не «попробовал, поработало» — а конкретный репо.
  2. Аудит кода:
    • сетевые вызовы (что отправляет наружу);
    • чтение .env/~/.ssh/Documents;
    • eval/exec/subprocess с user input;
    • постоянные процессы (cron, scheduled tasks);
    • hooks/install-scripts (что делается при установке);
    • транзитивные зависимости.
  3. Явное «да» от пользователя. Не «вижу зелёный логотип — норм».

Дополнительно для skill (feedback_skill_install_no_y, feedback_skill_install_audit):

  • npx skills add всегда без -y, ручное подтверждение;
  • читаем SKILL.md целиком — ищем скрытые директивы, exfiltration, zero-width chars, транзитивные установщики.

Эти правила экономят инциденты, которых иначе будет много.

Allow-list

После установки MCP — явно разрешаем только нужные инструменты:

{
  "permissions": {
    "allow": [
      "mcp:brand-tokens:get_brand",
      "mcp:brand-tokens:list_brands"
    ],
    "deny": [
      "mcp:brand-tokens:write_*"
    ]
  }
}

Allow-list, не «разрешено всё». Это правило Middle (M4 — Tool inventory & permissions), но именно на собственных MCP студия часто ленится — а зря.

Реальный кейс: MCP-админка

Из memory project_mcp_setup: 5 активных MCP (fs, playwright, github, comfyui, davinci) + панель C:\Claude\mcp-admin\ + GUIDE.md.

Архитектура:

  • mcps.json — единственная источник правды о подключённых MCP;
  • ~/.claude.json — синхронизируется из mcps.json;
  • GUIDE.md — что зачем, как включать/выключать;
  • security_monitor (фоновый PS-монитор C:\Claude\security_monitor\) — следит за изменениями ~/.claude.json, алертит, если появилось не из админки.

MCP-админка — это «единая консоль» для управления тулзингом агента. Без неё через 3 месяца у вас 15 MCP-серверов и неясно, что они делают.

MCP для медиа-архива

Из практики архива (см. A13): MCP archive-search оборачивает RAG-стек и отдаёт агенту единственный инструмент search(query: str, k: int). Агент не знает про BGE-M3, LanceDB, реранкер — это детали реализации.

@mcp.tool()
def search(query: str, k: int = 5) -> list[dict]:
    """Поиск по приватному архиву (TG, файлы, аудио).
    Возвращает top-k фрагментов с источниками."""
    return rag_query(query, k=k)

Преимущество перед raw-RAG: агент не утечёт лишние данные (только top-k уходит в контекст), агент не путается в параметрах эмбеддера.

Подводные камни

  • «Поставил MCP, не глянув» — самый частый канал supply-chain атак. Жёстко применяйте security audit.
  • ~/.claude.json без синка с админкой — забыли актуализировать mcps.json, через месяц непонятно, что подключено.
  • MCP без allow-list — все его инструменты доступны агенту. Минимум, что нужно — deny-list разрушительных операций.
  • Тяжёлый MCP в каждом запуске — если MCP стартует 3 секунды, агент ждёт его на каждом сеансе. Долгие MCP — отдельный демон.
  • Логирование MCP включено, секреты в логах — стандартный logger пишет аргументы tool calls. Маскируйте чувствительные поля.

Recap

  • Свой MCP — для уникальных ресурсов студии и оборачивания опасных команд.
  • Минимальный MCP-сервер — 30 строк Python.
  • Security audit стороннего MCP — обязателен (GitHub-ссылка → аудит кода → явное «да»).
  • Allow-list, не «разрешено всё».
  • MCP-админка + security_monitor — единая консоль контроля.
  • MCP над RAG — скрывает реализацию и ограничивает утечку.
