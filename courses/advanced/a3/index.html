A3 · Local vs Cloud матрица

Что держим локально, что — в облаке. Решение по cost, latency, privacy и амортизации железа. Чтобы не платить дважды за то, что можно посчитать на своём GPU.

Что строим

Decision matrix «локально vs облако» для каждого узла пайплайна студии: LLM, image, video, audio, RAG, CV. Решение принимаем не по идеологии («приватность важна» / «зачем своё железо»), а по четырём количественным осям.

Четыре оси решения

┌───────────────────────────────────────────────────────────┐
│  ОСЬ              ЛОКАЛЬНО                  ОБЛАКО        │
├───────────────────────────────────────────────────────────┤
│  Cost per unit    амортизация GPU           per-call $    │
│  Latency          1–10 c (тёплая модель)    100–500 ms    │
│  Privacy          данные не уходят          контракт+TLS  │
│  Freshness        замороженная модель       свежий weight │
└───────────────────────────────────────────────────────────┘

Решение per-узел: не «вся студия локально» и не «вся студия в облаке».

Когда локально выгоднее

Локальное обоснованно при выполнении хотя бы двух условий:

  1. Регулярная нагрузка — узел вызывается ≥1000 раз/месяц (иначе амортизация GPU не отбивается).
  2. Privacy-чувствительные данные — клиентские документы, приватный архив, лица.
  3. Качество модели достаточное — open-weight модель решает задачу не хуже cloud-альтернативы.
  4. Latency не критичен — терпите 5–10 c на запрос (или железо мощное).

Типичные «локально-выгодные» узлы:

  • LLM для курации/классификации — Qwen3.5-21B-Deckard, llama.cpp на 8GB VRAM + 64GB RAM. См. A14.
  • ASR — Whisper large-v3 локально, $0 за час аудио. Кейс tg_archive.
  • OCR — Tesseract / PaddleOCR, мгновенно и без сетевых задержек.
  • Embeddings — BGE-M3 локально, нулевая цена индексирования архива (см. A13).
  • TTS для стенда — Fish Speech на 127.0.0.1:7870, кешируется (см. A11).

Когда облако выгоднее

Cloud обоснован, когда:

  1. Разовая или нерегулярная задача — генерация финального ролика для клиента (одна штука в месяц).
  2. Нужна модель frontier-уровня — для финального копирайта/анализа никакой open-weight модели мая 2026 не хватит как Opus 4.7.
  3. Latency критичен и железа нет — масштабируемый inference через RunPod Flash / vLLM (см. A15).
  4. Задача редкая, но дорогая в инфре — большой image-генератор раз в неделю не оправдывает своё железо.

Типичные «облако-выгодные» узлы:

  • Финальный текст — Claude Opus 4.7 для финального копирайта/КП/договоров.
  • Креативный image для бренда — FLUX 2 через FAL, GPT Image 2 через OpenAI.
  • Video — Seedance 2.0 / Kling 3.0 / Runway. Локальное видео-поколение редко оправдано в 2026.
  • Identity training — Soul ID training, LoRA на cloud GPU (быстрее и дешевле, чем держать H100 локально).

Гибрид: cascade Local → Cloud

Самый ROI-сильный паттерн — каскад. Дешёвая локальная модель фильтрует/префильтрует, дорогое cloud — добивает финал.

вход ──► LOCAL Qwen3.5  ──► классификация / черновик
                            │
                            ▼
                     достаточно хорошо? ── да ──► OUTPUT
                            │
                            нет
                            ▼
              CLOUD Sonnet/Opus  ──► финал ── OUTPUT

Кейс: курация контента для канала. Локальный Qwen разбирает 10 000 постов в день, отбирает 50 кандидатов; Sonnet 4.6 ранжирует кандидатов; Opus 4.7 пишет финальный текст для топ-5. Из 10 000 LLM-вызовов в облако уходит ~55. Стоимость падает на два порядка.

Реальный кейс: как разнесён стек одной студии

Из практики проекта производства AI-контента:

  • Локально (4070 8GB + 64GB RAM): Qwen3.5-21B-Deckard (курация), BGE-M3 (RAG), Whisper (ASR), Fish Speech (TTS стенд), MediaPipe (CV), ComfyUI (превью).
  • Облако (paid API, по необходимости): Claude Opus 4.7 (финал), FLUX 2 / GPT Image 2 (бренд), Seedance 2.0 / Runway Unlimited (видео), Higgsfield Soul ID (идентичность).
  • Облако (own GPU pool): RunPod Flash для vLLM-инференса Qwen3.5 на масштабе (когда локальных 5 t/s не хватает).

Месячная экономика: ~70% LLM-вызовов закрываются локально, ~30% — облаком. Видео и финальная графика — целиком облако.

Подводные камни

  • «Локалка медленная — куплю A100» — посчитайте сначала real-time-нагрузку. На 1000 вызовов/месяц A100 за $1500/мес не окупается ни одной картинкой.
  • «Зачем локалка — у меня кредиты» — кредиты заканчиваются. Локальный узел в архитектуре = страховка от смены цен и блокировок.
  • «Локально приватнее всегда» — нет. Если ваш «локальный» сервер в датацентре провайдера с правом доступа сотрудников — это та же «облако», только хуже логированное.
  • Замороженная модель — open-weight Qwen3.5 в мае 2026 не знает события июня 2026. Для актуальных задач локалка не подходит.
  • Электричество — 4070 на 24/7 inference это +200–300 Вт постоянной нагрузки. На больших объёмах это реальные деньги.

Recap

  • Decision per-узел, не per-стек. Локально выгодно при ≥1000 вызовов/мес + privacy + достаточное качество.
  • Cascade Local → Cloud — самый ROI-сильный паттерн: дешёвая фильтрация локально, финал в облаке.
  • Локально: классификация, embeddings, ASR, RAG, CV, TTS-кеш. Облако: финал-текст, image/video gen, Soul ID training.
  • Не считать «локально приватнее» по умолчанию — приватность считается по конкретной угрозе.

Дальше: посмотреть и почитать

  • llama.cpp — github.com/ggerganov/llama.cpp. Эталонный локальный inference для quantized моделей.
  • vLLM — docs.vllm.ai. Production-grade inference; используется в RunPod Flash deployments.
  • Hugging Face — LLM leaderboards — huggingface.co/spaces/open-llm-leaderboard. Чтобы трезво оценить, на что способны open-weight модели на текущий месяц.
  • RunPod Flash — docs.runpod.io/serverless/flash. GA с 2026-04-30 — @Endpoint без Docker, FlashBoot, 500MB limit.
