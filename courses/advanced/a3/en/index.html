A3 · Local vs Cloud matrix

What stays on your hardware, what goes to the cloud. Decide by cost, latency, privacy, and amortisation — not ideology.

What we’re building

A per-node decision matrix for every component of the studio pipeline: LLM, image, video, audio, RAG, CV. The call is not made by ideology (“privacy matters” / “why bother with hardware”) — it’s made on four quantitative axes.

The four axes

┌───────────────────────────────────────────────────────────┐
│  AXIS             LOCAL                     CLOUD         │
├───────────────────────────────────────────────────────────┤
│  Cost per unit    GPU amortisation          per-call $    │
│  Latency          1–10 s (warm model)       100–500 ms    │
│  Privacy          data never leaves         contract+TLS  │
│  Freshness        frozen weights            current model │
└───────────────────────────────────────────────────────────┘

The decision is per-node, never “the whole studio runs locally” or “the whole studio in the cloud.”

When local wins

Local is justified when at least two of the following hold:

  1. Steady load — the node is invoked ≥1000×/month (otherwise GPU amortisation doesn’t pay back).
  2. Privacy-sensitive data — client documents, the private archive, faces.
  3. Open-weight quality is good enough — the open model solves the task no worse than the cloud alternative.
  4. Latency is non-critical — you can live with 5–10 s per call (or your hardware is fast enough).

Typical “local-wins” nodes:

  • LLM for curation / classification — Qwen3.5-21B-Deckard, llama.cpp on an 8 GB VRAM + 64 GB RAM box. See A14.
  • ASR — Whisper large-v3 locally, $0 per audio hour. Case: tg_archive.
  • OCR — Tesseract / PaddleOCR, instant and no round-trip latency.
  • Embeddings — BGE-M3 locally; zero cost to index an archive (see A13).
  • TTS for an installation — Fish Speech on 127.0.0.1:7870, cached (see A11).

When cloud wins

Cloud is justified when:

  1. One-off or irregular task — a single final clip for a client (one per month).
  2. You need a frontier-tier model — for final copy / analysis no open-weight model in May 2026 matches Opus 4.7.
  3. Latency is critical and you have no hardware — scalable inference via RunPod Flash / vLLM (see A15).
  4. The task is rare but infra-heavy — a large image generator once a week doesn’t earn back its own silicon.

Typical “cloud-wins” nodes:

  • Final text — Claude Opus 4.7 for final copy / proposals / contracts.
  • Brand creative imagery — FLUX 2 via FAL, GPT Image 2 via OpenAI.
  • Video — Seedance 2.0 / Kling 3.0 / Runway. Local video generation rarely pays in 2026.
  • Identity training — Soul ID, LoRA on cloud GPUs (faster and cheaper than keeping an H100 on premises).

Hybrid: Local → Cloud cascade

The strongest ROI pattern is a cascade. A cheap local model filters/pre-processes, an expensive cloud model finishes.

input ──► LOCAL Qwen3.5  ──► classification / draft
                            │
                            ▼
                    good enough? ── yes ──► OUTPUT
                            │
                            no
                            ▼
              CLOUD Sonnet/Opus  ──► final ── OUTPUT

Case: content curation for a channel. Local Qwen sweeps 10 000 posts a day and picks 50 candidates; Sonnet 4.6 ranks them; Opus 4.7 writes the final text for the top 5. Out of 10 000 LLM calls, ~55 go to the cloud. Cost drops two orders of magnitude.

Real case: how one studio split its stack

From a working AI content production project:

  • Local (4070 8 GB + 64 GB RAM): Qwen3.5-21B-Deckard (curation), BGE-M3 (RAG), Whisper (ASR), Fish Speech (TTS for installations), MediaPipe (CV), ComfyUI (previews).
  • Cloud (paid API, on demand): Claude Opus 4.7 (final copy), FLUX 2 / GPT Image 2 (brand creative), Seedance 2.0 / Runway Unlimited (video), Higgsfield Soul ID (identity).
  • Cloud (own GPU pool): RunPod Flash for vLLM inference of Qwen3.5 at scale (when local 5 t/s isn’t enough).

Monthly economics: ~70 % of LLM calls close locally, ~30 % via cloud. Video and final graphics — entirely cloud.

Pitfalls

  • “Local is slow — I’ll buy an A100” — count real-time load first. At 1000 calls/month an A100 at $1500/mo doesn’t earn back a single rendered image.
  • “Why local — I have credits” — credits run out. A local node is your insurance against price changes and account lockouts.
  • “Local is always more private” — not by default. If your “local” box sits in a provider’s datacentre where staff has access, it’s just cloud with worse logging.
  • Frozen weights — open-weight Qwen3.5 (May 2026) doesn’t know June 2026 events. For news-fresh tasks, local won’t do.
  • Electricity — a 4070 doing 24/7 inference adds 200–300 W of constant load. At scale that’s real money on the power bill.

Recap

  • Decisions are per-node, not per-stack. Local pays at ≥1000 calls/month + privacy + sufficient quality.
  • Local → Cloud cascade is the highest-ROI pattern: cheap filtering locally, final pass in the cloud.
  • Local: classification, embeddings, ASR, RAG, CV, TTS cache. Cloud: final copy, image/video gen, Soul ID training.
  • “Local is more private” is not free. Privacy is computed against a concrete threat, not assumed.
