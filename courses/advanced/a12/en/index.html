A12 · Telemetry → control loop

Closed loop sensor → event → LLM → visual command. WS protocol, event idempotency, backpressure. Diagnosing the TD main-thread hang.

What we’re building

The control loop for an installation: CV/sensor → event → LLM logic → command to the visual. A step beyond A11: there the booth reacted to a voice, here it reacts to the state of the space (people count, poses, activity, MIDI/OSC). Latency event → command ≤ 200 ms.

Loop architecture

SENSOR (CV/IMU/MIDI/OSC) ─► event_bus ─► dispatcher ─► LLM (optional) ─► action ─► visual
                                            │                                 │
                                            └──────► state store ◄────────────┘

Not every sensor sample goes to the LLM. Most events are simple (new face entered, someone raised a hand) and are processed by rules without LLM. The LLM kicks in when a “contextual” answer is needed.

WS event protocol

The base event format is a compact JSON with an idempotent id:

{
  "id": "evt_2026-05-29T14:22:11_a83f",
  "ts": 1748525531.221,
  "source": "cv:camera_main",
  "kind": "person.detected",
  "payload": { "track_id": 7, "pose": "standing", "confidence": 0.91 },
  "ttl_ms": 1500
}

Fields:

  • id — unique, dedup on the dispatcher side;
  • ts — client time, for ordering;
  • source — who sent it, for routing;
  • kind — event type; the dispatcher picks a rule/handler from this;
  • payload — event data;
  • ttl_ms — if the event isn’t processed inside the window it’s invalid (relevant for visual commands).

Idempotency

A sensor may send the same event twice (retry, duplicated stream). Without idempotency the visual fires a “double trigger.”

Simple dedup — a TTL cache on id:

recent_ids = TTLCache(maxsize=10000, ttl=10)

def handle(event: dict):
    if event["id"] in recent_ids:
        return  # duplicate
    recent_ids[event["id"]] = True
    route(event)

For commands to the visual — command_id is idempotent too. If the visual has already run the command, a repeat is ignored.

Backpressure

The sensor stream may be 30–60 events/sec. The visual handler is slower (especially if every event asks StreamDiffusion for a re-render). Without backpressure the queue swells, latency grows, and in the worst case the process OOM-crashes.

Strategies:

  1. Drop policy — inside an N ms window keep only the last event of a given kind (useful for continuous streams: position, volume).
  2. Sample — take every Kth sample.
  3. Aggregate — inside the window aggregate (mean / max / count) and send one summary command.
from collections import deque

class WindowAggregator:
    def __init__(self, window_ms=100):
        self.window_ms = window_ms
        self.buffer = deque()

    def push(self, event):
        self.buffer.append(event)
        cutoff = time.monotonic() * 1000 - self.window_ms
        while self.buffer and self.buffer[0]["ts"] * 1000 < cutoff:
            self.buffer.popleft()

    def aggregate(self):
        if not self.buffer:
            return None
        return {
            "count": len(self.buffer),
            "latest": self.buffer[-1],
            "mean_confidence": mean(e["payload"]["confidence"] for e in self.buffer),
        }

When to involve the LLM

An LLM in the loop is +200–800 ms per call. Plug it in only when rules aren’t enough:

  • No LLM: “person entered → fade-in,” “volume >0.7 → pulse,” “MIDI note 60 → scene 3.”
  • LLM: “several people of different age groups — which message to show,” “scene activity changed — rewrite the subtitle for the current video.”

A response cache keyed on event-key is mandatory — otherwise you pay for recurring situations.

Case: AI mirror for a business forum

A typical installation for a large forum (business, cultural, technology): an AI mirror at the pavilion entrance, a visitor approaches → CV classifies them by archetype → a personalised reply and visual composition is displayed.

Loop:

  • Sensor — camera + CV (e.g. 6 archetypes from a pre-prepared JSON catalogue).
  • Event — person.archetype_matched with archetype_id.
  • Rule — each archetype has matched text and visuals; the LLM is invoked only to generate the “personalised” portion of the reply.
  • Action — the mirror’s visual changes (background, typography, motion).

Spec for the CV contractor: an internal spec/{API,METRICS,WS_PROTOCOL}.md — an in-house WS protocol contract so the CV team can work independently of the ML side.

TD main-thread hang: diagnostics

Known issue: a TD project hangs on a WS port (typically :44444 / :7878) even without an active WS client.

Checklist:

  1. par_bindings — are all parameters resolved? A hanging bind on a missing operator kills the main thread.
  2. pack_json — is invalid JSON being parsed synchronously? Move it into td.run('...', delayFrames=1).
  3. frame_exec — is heavy work happening in frame_exec? Every frame_exec runs on the main thread.

Simple rule: the WS server and frame_exec in TD must live on different threads. Otherwise the handler blocks the frame, the frame blocks the handler — mutual deadlock.

Pitfalls

  • “Every sample is an event” — no. Sensor streams are aggregated. 30 fps × 60 s = 1800 events/min — that’s telemetry, not an event.
  • Without the id TTL cache — two sensors send one event, the visual fires twice. Dedup is mandatory.
  • LLM in every loop — money and latency. Only when rules aren’t enough.
  • OOM as the queue grows — monitor queue size; on overflow, drop old events instead of accumulating.
  • Logging on the main thread — slow writes block the loop. Logs go to a separate thread + async sink.

Recap

  • Loop = SENSOR → event_bus → dispatcher → (rule or LLM) → action → visual.
  • WS events: id (idempotency), ts, source, kind, payload, ttl_ms.
  • Backpressure: drop / sample / aggregate. Without it the loop falls apart.
  • Plug in the LLM only when rules aren’t enough; response cache is mandatory.
  • TD main-thread hang is a known pain. WS server and frame_exec on different threads.

Further reading

  • OSC (Open Sound Control) spec — opensoundcontrol.org/spec-1_0. Standard protocol for AV event exchange.
  • NATS / Redis Streams — nats.io, redis.io/docs/data-types/streams. Ready-made event buses if a project has more than two sensors.
  • MediaPipe Tasks API — developers.google.com/mediapipe/solutions/tasks. Ready CV tasks with event-style callbacks.
  • Twelve-Factor App § XII — Concurrency — 12factor.net/concurrency. Base worker/process model for loops.
