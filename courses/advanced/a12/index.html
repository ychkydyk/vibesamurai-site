Advanced Path · a12

A12 · Telemetry → control loop

Замкнутая петля сенсор → событие → LLM → команда визуалу. WS-протокол, идемпотентность событий, backpressure. Диагностика TD main-thread hang.

Что строим

Управляющую петлю инсталляции: CV/sensor — событие — LLM-логика — команда визуалу. Это шаг вперёд от A11: там стенд реагировал на голос, здесь — на состояние пространства (число людей, позы, активность, MIDI/OSC). Latency event→команда ≤ 200 ms.

Архитектура петли

SENSOR (CV/IMU/MIDI/OSC) ─► event_bus ─► dispatcher ─► LLM (опционально) ─► action ─► visual
                                            │                                 │
                                            └──────► state store ◄────────────┘

Не каждый сенсорный сэмпл идёт в LLM. Большинство событий — простые (новое лицо вошло, кто-то поднял руку) и обрабатываются без LLM по правилам. LLM включается, когда нужен «контекстный» ответ.

WS-протокол событий

Базовый формат события — компактный JSON с идемпотентным id:

{
  "id": "evt_2026-05-29T14:22:11_a83f",
  "ts": 1748525531.221,
  "source": "cv:camera_main",
  "kind": "person.detected",
  "payload": { "track_id": 7, "pose": "standing", "confidence": 0.91 },
  "ttl_ms": 1500
}

Поля:

  • id — уникален, дедупликация на стороне dispatcher;
  • ts — клиентское время, для упорядочивания;
  • source — кто прислал, для маршрутизации;
  • kind — тип события, по нему dispatcher выбирает rule/handler;
  • payload — данные события;
  • ttl_ms — если событие не обработано в окне, оно невалидно (актуально для visual-команд).

Идемпотентность

Сенсор может прислать одно и то же событие дважды (retry, дублированный поток). Без идемпотентности визуал выдаст «двойной триггер».

Простой дедуп — TTL-кэш на id:

recent_ids = TTLCache(maxsize=10000, ttl=10)

def handle(event: dict):
    if event["id"] in recent_ids:
        return  # дубль
    recent_ids[event["id"]] = True
    route(event)

Для команд визуалу — command_id тоже идемпотентен. Если визуал уже выполнил команду — повтор игнорируется.

Backpressure

Сенсорный поток может быть 30–60 событий/с. Visual handler — медленнее (особенно если каждое событие просит StreamDiffusion перерендер). Без backpressure очередь распухает, latency растёт, в худшем случае процесс OOM-падает.

Стратегии:

  1. Drop policy — за окном N мс берём только последнее событие данного kind (актуально для непрерывных потоков: позиция, громкость).
  2. Sample — берём каждый K-й сэмпл.
  3. Aggregate — внутри окна агрегируем (mean / max / count) и шлём одну сводную команду.
from collections import deque

class WindowAggregator:
    def __init__(self, window_ms=100):
        self.window_ms = window_ms
        self.buffer = deque()

    def push(self, event):
        self.buffer.append(event)
        cutoff = time.monotonic() * 1000 - self.window_ms
        while self.buffer and self.buffer[0]["ts"] * 1000 < cutoff:
            self.buffer.popleft()

    def aggregate(self):
        if not self.buffer:
            return None
        return {
            "count": len(self.buffer),
            "latest": self.buffer[-1],
            "mean_confidence": mean(e["payload"]["confidence"] for e in self.buffer),
        }

Когда подключать LLM

LLM в петле — это +200–800 ms на запрос. Подключаем только когда правила недостаточно:

  • Не нужен LLM: «человек вошёл — fade-in», «громкость >0.7 — pulse», «MIDI note 60 — scene 3».
  • Нужен LLM: «несколько людей разной возрастной группы — какой message показать», «активность сцены изменилась — переписать subtitle для текущего видео».

Кэш ответов LLM на event-key обязателен — иначе платите за повторяющиеся ситуации.

Реальный кейс: PMEF neuromirror

Из memory project_pmef_neuromirror: ИИ-зеркало для ПМЭФ. Галерея https://neuromirror-mockups.pages.dev, backend C:\Rendered\AK\ayna\ayna.

Петля:

  • Sensor — камера + CV (6 архетипов из content/archetypes.json).
  • Event — person.archetype_matched с archetype_id.
  • Rule — каждому архетипу соответствует подобранный текст и визуал; LLM подключается только для генерации «персонализированной» части реплики.
  • Action — визуал зеркала меняется (фон, типографика, motion).

Spec для CV-подрядчика: spec/{API,METRICS,WS_PROTOCOL}.md — собственный контракт WS-протокола, чтобы CV-команда работала независимо.

TD main-thread hang: диагностика

Из практики (memory project_td_audioreact_hang): Clode_React.toe залипает на :44444/:7878 даже без WS-клиента.

Чек-лист (повторяет диагностику из memory):

  1. par_bindings — все ли параметры разрешены? Висящий bind на отсутствующий operator валит main thread.
  2. pack_json — не парсится ли невалидный JSON синхронно? Перенесите в td.run('...', delayFrames=1).
  3. frame_exec — нет ли тяжёлой работы в frame_exec? Каждый frame_exec выполняется на main thread.

Простое правило: WS-сервер и frame_exec в TD должны быть в разных потоках. Иначе хэндлер блокирует кадр, кадр блокирует хэндлер — взаимная блокировка.

Подводные камни

  • «Каждый сэмпл — событие» — нет. Сенсорные потоки агрегируются. 30 fps × 60 c = 1800 событий/мин — это не «событие», это телеметрия.
  • Без TTL-кэша id — два сенсора шлют одно событие, визуал реагирует дважды. Дедуп обязателен.
  • LLM в каждой петле — деньги и latency. Только когда правила недостаточно.
  • OOM при росте очереди — мониторьте размер queue, при превышении — drop старых событий, не накапливайте.
  • Логирование на main thread — медленные writes блокируют петлю. Логи в отдельный поток + async sink.

Recap

  • Петля = SENSOR → event_bus → dispatcher → (rule or LLM) → action → visual.
  • WS-события: id (идемпотентность), ts, source, kind, payload, ttl_ms.
  • Backpressure: drop / sample / aggregate. Без неё petля разваливается.
  • LLM подключаем только когда правил недостаточно; кэш ответов обязателен.
  • TD main-thread hang — известная боль. WS-сервер и frame_exec — в разных потоках.
