A13 · Private RAG для архива

Приватный архив (мессенджер-каналы, файлы, аудио) → ASR/OCR/files/links → BGE-M3 + LanceDB → Sonnet 4.6. Идемпотентность повторного ingest. Кейс архива креативной студии.

Что строим

Приватная RAG-система над собственным архивом: разнородные источники (TG-каналы, файлы, аудио, скриншоты) собираются в индекс, retrieval идёт локально, генерация ответа — гибрид (Sonnet 4.6 для финала, локальный Qwen для рутины). Архив никуда не уходит — это критично для бизнеса и приватной переписки.

аудио              ──► Whisper STT        ──►  LanceDB                  │

ссылки             ──► fetch + dedup      ──►  link cache             rerank

                                                                         ▼

                                                                         │
                                                                         ▼

LanceDB — embedded, не требует Postgres/Milvus/Qdrant. Файл-на-диске, всё хранится в lance/-папке.

BGE-M3: почему этот эмбеддер

  • Multilingual — русский + английский + 100+ языков в одном индексе.
  • Long context — до 8192 токенов в одном чанке, не дробить мелко.
  • Локально — модель ~2GB на диске, работает на CPU терпимо, на GPU быстро.
  • Multi-vector — три типа представления (dense, sparse, multi-vector) в одной модели; повышает retrieval-качество при том же чанк-размере.
from FlagEmbedding import BGEM3FlagModel

model = BGEM3FlagModel("BAAI/bge-m3", use_fp16=True)

def embed(texts: list[str]) -> dict:
    out = model.encode(
        texts,
        batch_size=12,
        max_length=8192,
        return_dense=True,
        return_sparse=True,
    )
    return {"dense": out["dense_vecs"], "sparse": out["lexical_weights"]}

Для повседневного RAG-юзкейса используем только dense; sparse подключаем при просадке качества на коротких/специфичных запросах.

Идемпотентность повторного ingest

Главная боль RAG-систем: «прогнал ingest заново — индекс распух или дублировался». Решение: cache key на каждый чанк = sha256(source_id + chunk_position + content_hash).

def upsert_chunk(chunk: dict):
    key = hashlib.sha256(
        f"{chunk['source_id']}|{chunk['position']}|{chunk['content'][:512]}".encode()
    ).hexdigest()
    chunk["chunk_id"] = key
    db.merge_insert("documents") \
        .when_matched_update_all() \
        .when_not_matched_insert_all() \
        .execute([chunk])

LanceDB merge_insert нативно поддерживает upsert по ключу. Повторный ingest = повторное вычисление хэшей и no-op для неизменных чанков.

Кейс: приватный архив креативной студии

Типичный сценарий — креативное агентство или независимая студия накапливает архив референсов: посты в каналах коллег, PDF из музейных каталогов, аудио-интервью с подкастов, скриншоты вдохновляющих лендингов. Через год там уже несколько тысяч единиц контента, и обычный поиск по папкам не работает.

Стек: мессенджер-экспорт (для Telegram-каналов — pywinauto на Windows; для Slack — официальный export; для Discord — DiscordChatExporter) → ASR/OCR/файлы/ссылки → BGE-M3 + LanceDB → Claude Sonnet 4.6 RAG.

Архитектура — отдельный ingest-процесс, не блокирующий основной workflow. Внутри студии живёт как submodule (например, studio-assistant/ingest/).

Ручной запуск: только руками, не регистрировать как cron-job в Windows Task Scheduler или systemd. Cron создаёт race conditions с pywinauto/desktop-export-tools. Запуск — python -m archive.ingest по требованию (1–2 раза в неделю достаточно).

Запрос с rerank

def query(question: str, k: int = 20, final_k: int = 5):
    q_emb = embed([question])["dense"][0]
    candidates = db.open_table("documents").search(q_emb).limit(k).to_list()
    rerank_scores = reranker.compute_score(
        [(question, c["content"]) for c in candidates]
    )
    ranked = sorted(zip(candidates, rerank_scores), key=lambda x: -x[1])
    return [c for c, _ in ranked[:final_k]]

k=20 → rerank → final_k=5. Без rerank качество ответа просаживается на длинных архивах: dense embedding ловит «тему», но не «релевантность ответу».

Гибрид: локальная фильтрация → Sonnet

Не каждый retrieval отправляем в Sonnet. Дешёвый локальный Qwen проверяет: «это про факт, который реально есть в найденном контексте?». Если нет — отказ. Если да — финальный ответ в Sonnet.

вопрос ──► retrieval ──► top-5 chunks
              │
              ▼
         Qwen3.5: "хватает ли контекста для ответа?"
              │
              ├── нет ──► "не нашёл в архиве" (без вызова Sonnet)
              │
              └── да ──► Sonnet 4.6 финал

На архивах в сотни тысяч чанков 30–40% запросов закрываются «не нашёл» — это огромная экономия Sonnet-токенов.

Подводные камни

  • Chunk size 200 токенов — поломает контекст. BGE-M3 рассчитан на длинные чанки (512–2048 токенов). Не дробите мелко.
  • Без timestamp в metadata — RAG отвечает «по архиву» без понимания «когда это было». Время — обязательное поле.
  • OCR на низкокачественных скринах — мусор в индексе. Confidence-фильтр на OCR-выходе (отрезаем <0.5).
  • Reranker без GPU — bge-reranker-large на CPU работает 200–500 ms за пару, для k=20 это секунды. Reranker — на GPU или меньшая модель (bge-reranker-base).
  • Дубли в TG-канале — пост перепечатан 3 раза. Дедуп по близости содержимого (Jaccard на n-граммах) до ingest.
  • «Спросил у Sonnet про факт из 2024» — ответ может быть нерелевантен, даже если в архиве есть факт из 2026. Time-filter по retrieval (например, top-k по последнему году).

Recap

  • Архитектура: ingest → BGE-M3 → LanceDB → retrieve → rerank → (Qwen filter) → Sonnet.
  • LanceDB — embedded, без отдельного сервера.
  • BGE-M3 — multilingual, long context, локально; используем dense + опционально sparse.
  • Идемпотентный ingest через cache key на чанк; LanceDB merge_insert нативно.
  • Rerank после retrieve обязателен на длинных архивах.
  • Локальный фильтр перед Sonnet экономит 30–40% токенов.
