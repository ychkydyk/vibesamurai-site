A13 · Private RAG for an archive

A private archive (messenger channels, files, audio) → ASR/OCR/files/links → BGE-M3 + LanceDB → Sonnet 4.6. Idempotent re-ingest. A creative studio archive case.

What we’re building

A private RAG system over your own archive: heterogeneous sources (TG channels, files, audio, screenshots) collected into an index, retrieval runs locally, generation is hybrid (Sonnet 4.6 for the final answer, local Qwen for the routine). The archive never leaves the host — critical for business data and private correspondence.

LanceDB is embedded — no Postgres / Milvus / Qdrant required. File-on-disk; everything lives in a lance/ folder.

BGE-M3: why this embedder

  • Multilingual — Russian + English + 100+ languages in one index.
  • Long context — up to 8192 tokens per chunk; you don’t have to over-shard.
  • Local — model ~2 GB on disk, runs tolerably on CPU and fast on GPU.
  • Multi-vector — three representation types (dense, sparse, multi-vector) in one model; raises retrieval quality at the same chunk size.
from FlagEmbedding import BGEM3FlagModel

model = BGEM3FlagModel("BAAI/bge-m3", use_fp16=True)

def embed(texts: list[str]) -> dict:
    out = model.encode(
        texts,
        batch_size=12,
        max_length=8192,
        return_dense=True,
        return_sparse=True,
    )
    return {"dense": out["dense_vecs"], "sparse": out["lexical_weights"]}

For everyday RAG use only the dense vectors; turn on sparse when quality dips on short or specific queries.

Idempotent re-ingest

The main RAG pain: “I re-ran ingest and the index bloated or duplicated.” Fix: a cache key per chunk = sha256(source_id + chunk_position + content_hash).

def upsert_chunk(chunk: dict):
    key = hashlib.sha256(
        f"{chunk['source_id']}|{chunk['position']}|{chunk['content'][:512]}".encode()
    ).hexdigest()
    chunk["chunk_id"] = key
    db.merge_insert("documents") \
        .when_matched_update_all() \
        .when_not_matched_insert_all() \
        .execute([chunk])

LanceDB merge_insert natively supports upsert by key. Re-running ingest = re-hash and no-op for unchanged chunks.

Case: a creative studio’s private archive

A typical scenario — a creative agency or independent studio accumulates a reference archive: posts from peer channels, PDFs from museum catalogues, audio interviews from podcasts, screenshots of inspirational landings. A year in, there are several thousand items, and ordinary folder search no longer works.

Stack: messenger export (for Telegram channels — pywinauto on Windows; for Slack — the official export; for Discord — DiscordChatExporter) → ASR/OCR/files/links → BGE-M3 + LanceDB → Claude Sonnet 4.6 RAG.

Architecture — a separate ingest process that doesn’t block the main workflow. It lives inside the studio’s tooling as a submodule (for example, studio-assistant/ingest/).

Manual launch: hand-run only, don’t register it as a cron job in Windows Task Scheduler or systemd. Cron creates race conditions with pywinauto / desktop export tools. Run python -m archive.ingest on demand (1–2 times a week is enough).

Query with rerank

def query(question: str, k: int = 20, final_k: int = 5):
    q_emb = embed([question])["dense"][0]
    candidates = db.open_table("documents").search(q_emb).limit(k).to_list()
    rerank_scores = reranker.compute_score(
        [(question, c["content"]) for c in candidates]
    )
    ranked = sorted(zip(candidates, rerank_scores), key=lambda x: -x[1])
    return [c for c, _ in ranked[:final_k]]

k=20 → rerank → final_k=5. Without rerank, answer quality degrades on long archives: a dense embedding catches the “topic” but not the “relevance to the question.”

Hybrid: local filter → Sonnet

Don’t push every retrieval to Sonnet. A cheap local Qwen checks “is there enough context in the retrieved chunks to actually answer?” If not — refuse. If yes — final answer through Sonnet.

question ──► retrieval ──► top-5 chunks
                │
                ▼
        Qwen3.5: "is the context enough?"
                │
                ├── no  ──► "not found in archive" (no Sonnet call)
                │
                └── yes ──► Sonnet 4.6 final

On archives of hundreds of thousands of chunks 30–40 % of queries terminate at “not found” — a large savings on Sonnet tokens.

Pitfalls

  • Chunk size 200 tokens — breaks context. BGE-M3 is designed for long chunks (512–2048 tokens). Don’t over-shard.
  • No timestamp in metadata — RAG answers “from the archive” without knowing when it was. Time is a mandatory field.
  • OCR on low-quality screenshots — garbage in the index. A confidence filter on the OCR output (drop <0.5) helps.
  • Reranker without GPU — bge-reranker-large at 200–500 ms per pair on CPU adds seconds at k=20. Reranker on GPU or a smaller model (bge-reranker-base).
  • TG channel duplicates — the same post is reprinted 3 times. Dedup by content similarity (Jaccard on n-grams) before ingest.
  • “Asked Sonnet about a 2024 fact” — the answer may be irrelevant even if a 2026 fact exists in the archive. Time-filter on retrieval (e.g. top-k from the last year).

Recap

  • Architecture: ingest → BGE-M3 → LanceDB → retrieve → rerank → (Qwen filter) → Sonnet.
  • LanceDB is embedded — no separate server.
  • BGE-M3 — multilingual, long-context, local; use dense + optionally sparse.
  • Idempotent ingest via a per-chunk cache key; LanceDB merge_insert natively.
  • Rerank after retrieval is mandatory on long archives.
  • Local filter before Sonnet saves 30–40 % of tokens.
