A19 · Security in a creative studio

Secrets never in chat, watermark/IP, tunnels without .git/.env, claude-process-allowlist, security_monitor. Real incidents and the lesson each taught.

What we’re building

The applied security stack of a creative studio. Middle (M18) covered the production-security base. Advanced is about the concrete risks specific to a studio: leaks via tunnels, IP violations in generative output, secrets in logs, auto-publish of production artifacts.

Main rule: secrets stay out of context

The main rule: Claude never reads secret values and never forwards them to the network/API/context. Only the variable name + metadata (presence, mtime, hash).

Concretely:

# bad — secret leaks into chat / log / vendor API
print(f"DEBUG: API_KEY={os.environ['ANTHROPIC_API_KEY']}")

# good — metadata only
env_status = {
    "ANTHROPIC_API_KEY": {
        "present": "ANTHROPIC_API_KEY" in os.environ,
        "length": len(os.environ.get("ANTHROPIC_API_KEY", "")),
    }
}

This rule applies during debugging, agent sessions, and logs. Never “let me peek at the key.”

Announce edits before the tool call

A practical pattern: before each Edit/Write — one line ”→ file: what I’m changing.”

This is a security rule, not cosmetics: the user sees every change before it lands, and catches “not this one” before the commit.

In Claude Code it’s enabled via skill instructions or CLAUDE.md. Without the announcement the agent often edits “in the neighbourhood” — that only gets caught in the diff afterwards.

Tunnels without .git/.env

A textbook incident: an open tunnel serves .git/.env/.claude/.ssh through a network http.server launched at the repo root.

Rules:

  • before cloudflared / ngrok / http.server check the root for dotfiles;
  • serve a dedicated dist/, not the whole repo;
  • never http.server on the repo root.

Check script:

def safe_serve_root_check(root: Path):
    bad = []
    for entry in root.iterdir():
        if entry.name.startswith(".") and entry.name not in {".well-known"}:
            bad.append(str(entry))
    if bad:
        raise SystemExit(
            f"⛔ refusing to serve: dotfiles found {bad}\n"
            f"move content to dist/ and serve that."
        )

Run before every python -m http.server or cloudflared tunnel --url.

Plus: 0.0.0.0 / :: are forbidden; always --bind 127.0.0.1. The tunnel only exposes a given port on localhost, not the whole interface.

claude-process-allowlist

Practical pattern: when launching python + cloudflared — write the PID to allowed_tunnels.txt, --bind 127.0.0.1, ports from a pre-fixed claude_managed_ports.json.

# claude_managed_ports.json:
# {"reserved": [8000, 8765, 7870, 8080], "preview": [4321], "external": []}

def start_local_server(port: int, name: str):
    if port not in MANAGED_PORTS["reserved"] + MANAGED_PORTS["preview"]:
        raise SystemExit(f"port {port} not in claude_managed_ports")
    proc = subprocess.Popen([...])
    Path("allowed_tunnels.txt").open("a").write(f"{proc.pid} {name} {port}\n")
    return proc

security_monitor — a background process (PowerShell script on Windows, systemd unit on Linux) that reconciles live processes against allowed_tunnels.txt and alerts on drift. Co-located with the MCP admin panel.

Archive ingest — no cron

Practical pattern: the ingest process of a private RAG archive (see A13) runs manually only. Don’t register it as a Windows Task Scheduler / systemd timer.

Reason: cron + pywinauto kept breaking its own state (races for the Telegram UI), context was lost, the index ended up unparseable. Manual launch guarantees you know when the system writes to the archive.

IP, watermarking, faces

Generative creative is an IP/face minefield. Non-negotiable items:

  1. Using someone’s face — needs a contract. Soul ID training — explicit consent.
  2. Watermark on drafts — the client must not accidentally publish a “not approved” version. Semi-transparent watermark on previews.
  3. Output image/video licence — every vendor has its own terms. FAL/RunPod — commercial ok, Higgsfield Soul — separate tier terms. Before commercial use — check.
  4. Photoreal generated faces — regulated in RU + EU (deepfake legislation is evolving). Mark AI-generated content where applicable.

Stack security audit

Rule: before choosing a stack / new dependency check:

  • GitHub Advisories for every dependency;
  • OSV — open source vulnerability database;
  • Last release date — abandoned libraries = risk;
  • Archived status — an archived repo without maintenance.

Not “everyone uses it” — proofs. Especially for MCP servers (see A16) and ComfyUI custom nodes.

Security pre-deploy checklist

Rule: mandatory checklist before every booth deploy:

  1. .env not committed (.gitignore actual, git status clean of secrets);
  2. ports — from claude_managed_ports, no surprises;
  3. tunnels — dist/ only, no repo root;
  4. local endpoints — 127.0.0.1 only;
  5. demo token in .env, not in code;
  6. logs don’t print sensitive fields;
  7. supply chain: new MCPs / custom nodes — passed the audit.

Without the checklist any deploy is potentially broken. Roll the items into a make target / CI step so it doesn’t depend on “remembering.”

Real cases

Booth security check task

A typical pattern: a Windows Task Scheduler job every 30 minutes (via pythonw without a window) checks installation state — ports, processes, allowed_tunnels.txt. Runs independently of the booth; on drift, writes to the log and sends an alert.

MCP security_monitor

A background process reconciles state with mcp-admin/mcps.json. Any drift in ~/.claude.json triggers an alert. Used as a watchdog over the MCP panel.

Git private by default

Rule: every git repo private by default. gh repo create --private always. Public only after an explicit decision.

Pitfalls

  • “Let me paste the .env so Claude can look” — never. Names + metadata, not values.
  • 0.0.0.0 “just for a test” — gets forgotten by prod. No 0.0.0.0.
  • Watermark “we’ll add later” — usually doesn’t get added. Watermark by default on drafts.
  • gh repo create without --private — sometimes public. Habit of always --private.
  • -y on installing skills/nodes — bypasses confirmation. Don’t.

Recap

  • Secrets — never in context. Names + metadata.
  • Tunnels — dist/ only, no dotfiles, check before start.
  • 127.0.0.1 only, claude_managed_ports, allowed_tunnels.txt + security_monitor.
  • IP/faces/watermark — handled at planning, not after the fact.
  • Stack security audit before every new dependency.
  • Pre-deploy checklist is mandatory; git repos private by default.
