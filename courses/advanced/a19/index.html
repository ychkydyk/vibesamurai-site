A19 · Security в творческой студии

Secrets никогда в чате, watermark/IP, tunnels без .git/.env, claude-process-allowlist, security_monitor. Реальные инциденты и их урок.

Что строим

Прикладной security-стек креативной студии. Middle (M18) дал базу production security. Advanced — про конкретные риски, которые возникают именно у студии: утечки через tunnels, IP-нарушения в генеративке, секреты в логах, авто-publish прод-артефактов.

Главное правило: secrets не в контексте

Главное правило: Claude никогда не читает значения секретов и не передаёт их в сеть/API/контекст. Только имя переменной + метаданные (наличие, mtime, hash).

Конкретно:

# плохо — секрет уходит в чат / лог / vendor API
print(f"DEBUG: API_KEY={os.environ['ANTHROPIC_API_KEY']}")

# хорошо — только метаданные
env_status = {
    "ANTHROPIC_API_KEY": {
        "present": "ANTHROPIC_API_KEY" in os.environ,
        "length": len(os.environ.get("ANTHROPIC_API_KEY", "")),
    }
}

Это правило применяется при отладке, при agent-сессиях, в логах. Никогда не «давайте я гляну на ключ».

Анонс правок перед инструментом

Практический паттерн: перед каждым Edit/Write — одна строка «→ файл: что меняю».

Это секьюрити-правило, не косметика: пользователь видит каждое изменение до его исполнения, ловит «не то» до commit.

В Claude Code включается через скилл-инструкцию или CLAUDE.md. Без анонса агент часто правит «по соседству» — это ловится только в diff после.

Tunnels без .git/.env

Типовой инцидент: открытый туннель отдаёт .git/.env/.claude/.ssh через сетевой http.server, запущенный с корня репо.

Правило:

  • перед cloudflared/ngrok/http.server проверяем root на dotfiles;
  • отдаём отдельный dist/, не весь репо;
  • никогда не http.server на корне репозитория.

Скрипт проверки:

def safe_serve_root_check(root: Path):
    bad = []
    for entry in root.iterdir():
        if entry.name.startswith(".") and entry.name not in {".well-known"}:
            bad.append(str(entry))
    if bad:
        raise SystemExit(
            f"⛔ нельзя серверить: найдены dotfiles {bad}\n"
            f"перенесите контент в dist/ и серверьте его."
        )

Запускать до каждого python -m http.server или cloudflared tunnel --url.

Дополнительно: 0.0.0.0/:: запрещены, всегда --bind 127.0.0.1. Туннель экспонирует только заданный port на localhost, не весь интерфейс.

Claude-process-allowlist

Практический паттерн: при запуске python + cloudflared — пишем PID в allowed_tunnels.txt, --bind 127.0.0.1, порты из заранее зафиксированного claude_managed_ports.json.

# claude_managed_ports.json:
# {"reserved": [8000, 8765, 7870, 8080], "preview": [4321], "external": []}

def start_local_server(port: int, name: str):
    if port not in MANAGED_PORTS["reserved"] + MANAGED_PORTS["preview"]:
        raise SystemExit(f"порт {port} не в claude_managed_ports")
    proc = subprocess.Popen([...])
    Path("allowed_tunnels.txt").open("a").write(f"{proc.pid} {name} {port}\n")
    return proc

security_monitor — фоновый процесс (для Windows — PowerShell-скрипт, для Linux — systemd unit), который сверяет реальные процессы с allowed_tunnels.txt и алертит расхождения. Поднимать рядом с MCP-админкой.

Archive-ingest — никаких cron

Практический паттерн: ingest-процесс приватного RAG-архива (см. A13) только ручной запуск. Не регистрировать как Windows Task Scheduler / systemd timer.

Причина: cron + pywinauto регулярно ломали себе состояние (гонки за UI Telegram), терялся контекст, индекс приходил в нерасшифровываемое состояние. Ручной запуск — гарантия, что вы знаете, когда система пишет в архив.

IP, watermarking, лица

Генеративный креатив — минное поле IP/лиц. Что обязательно:

  1. Использование чужих лиц — нужен договор. Для Soul ID training — explicit consent.
  2. Watermark на черновиках — клиент не должен случайно опубликовать «непринятую» версию. Полупрозрачный watermark на превью.
  3. Лицензия выходных image/video — у каждого провайдера свои условия. FAL/RunPod — commercial ok, Higgsfield Soul — отдельные tier-условия. Перед coммерч. использованием — проверяем.
  4. Реалистичные сгенерированные лица — в РФ + ЕС регулируется (deepfake-законы развиваются). Маркируйте AI-generated content где применимо.

Stack security audit

Правило: перед выбором стека/новой зависимости проверяем:

  • GitHub Advisories для каждой dependency;
  • OSV — open source vulnerability база;
  • Дату последнего релиза — заброшенные библиотеки = риск;
  • archived-статус — заархивированный репо без maintenance.

Не «так все делают», а пруфами. Особенно для MCP-серверов (см. A16) и custom nodes ComfyUI.

Security pre-deploy чеклист

Правило: обязательный чеклист перед каждым деплоем стенда:

  1. .env не в commit (.gitignore actual, git status clean от секретов);
  2. порты — из claude_managed_ports, никаких неожиданных;
  3. tunnels — только dist/, без корня репо;
  4. локальные эндпоинты — 127.0.0.1 only;
  5. демо-токен в .env, не в коде;
  6. logs не пишут чувствительные поля;
  7. supply chain: новые MCP / custom nodes — прошли аудит.

Без чеклиста любой деплой потенциально неисправен. Раскатываем пункты в make-таргет / CI-шаг, чтобы не зависело от «помню или не помню».

Реальные кейсы

Booth security check task

Типовой паттерн: Windows-задача в Task Scheduler раз в 30 минут (через pythonw без окна) проверяет state инсталляции — порты, процессы, allowed_tunnels.txt. Работает независимо от самой инсталляции; если что-то расходится — пишет в лог и шлёт алерт.

MCP security_monitor

Фоновый процесс сверяет состояние с mcp-admin/mcps.json. Любое расхождение в ~/.claude.json алертит. Используется как сторожевой над панелью MCP.

Git private by default

Правило: все git-репо приватные по умолчанию. gh repo create --private всегда. Public — только вручную после явного намерения.

Подводные камни

  • «Вылью .env в чат, чтобы клод посмотрел» — никогда. Имена + метаданные, не значения.
  • 0.0.0.0 «временно для теста» — забывается до прода. Никаких 0.0.0.0.
  • Watermark «потом добавим» — обычно не добавляют. Watermark по умолчанию на drafts.
  • gh repo create без --private — иногда public. Привычка ставить --private.
  • -y при установке skills/nodes — bypass подтверждения. Не используйте.

Recap

  • Secrets — никогда в контексте. Имена + метаданные.
  • Tunnels — только dist/, никаких dotfiles, проверка перед стартом.
  • 127.0.0.1 only, claude_managed_ports, allowed_tunnels.txt + security_monitor.
  • IP/лица/watermark — учитываются на стадии планирования, не пост-фактум.
  • Stack security audit перед каждой новой dependency.
  • Pre-deploy checklist обязателен; git-репо private by default.
