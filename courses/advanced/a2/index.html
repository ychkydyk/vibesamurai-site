A2 · Multi-modal pipeline architecture

Контракт между модальностями: text → image → video → control. Идемпотентность, кэширование, lineage. Чтобы пайплайн не пересоздавал то, что уже сделано.

Что строим

Архитектуру multi-modal пайплайна — программу, которая последовательно вызывает генеративные сервисы и собирает финальный артефакт (СММ-пост, ролик, инсталляцию). Главное требование: пайплайн не должен пересоздавать то, что уже сгенерировано, и должен давать понятную lineage (откуда взялся каждый кадр/секунда/звук).

Контракт между шагами

Каждый шаг пайплайна имеет три обязательных поля. То же правило, что в M2 (Architecting agent boundaries) — но теперь не для LLM, а для генеративных сервисов.

PIPELINE STEP
├── INPUT             хэш-стабильный набор: текст промпта, refs, params
├── OUTPUT            артефакт + метаданные (модель, версия, seed, cost)
└── CACHE KEY         хэш INPUT → если файл с таким ключом есть — return

Без CACHE KEY каждый прогон пайплайна перегенерирует кадры заново. На партии в 200 видео это 200 × 8 секунд × $0.30 = $480, и это деньги без видимой пользы.

Lineage: откуда взялся кадр

Каждый артефакт сопровождается JSON-сайдкаром (или metadata-таблицей):

{
  "id": "frame_a83f",
  "step": "image:hero",
  "engine": "flux-2",
  "engine_version": "2.0.3",
  "prompt_hash": "sha256:7c1e...",
  "seed": 42,
  "refs": ["brand_token://wedoit/main"],
  "cost_usd": 0.04,
  "created_at": "2026-05-29T14:22:11Z",
  "input_step_ids": []
}

Что это даёт:

  • через месяц вы понимаете, почему ролик выглядит так, как выглядит;
  • клиент просит «как тот, только синее» — вы меняете один ref и кэш переиспользует всё, что не зависит от цвета;
  • бухгалтерия видит точную стоимость заказа.

Архитектура потока

flowchart LR
    BR[Brand tokens<br/>JSON]
    PR[Prompt template]
    BR --> PR
    PR --> H1{Cache hit?}
    H1 -->|yes| OUT1[hero image]
    H1 -->|no| IMG[FLUX 2]
    IMG --> OUT1
    OUT1 --> V1{Cache hit?}
    V1 -->|yes| OUT2[motion clip]
    V1 -->|no| VID[Seedance 2.0]
    VID --> OUT2
    OUT2 --> POST[Post compositor]
    POST --> FIN[final mp4]

Каждая стрелка — это шаг с собственным cache key. Любой шаг можно перезапустить отдельно, остальное не пересчитывается.

Реальный кейс: статичный креатив → motion-pin для СММ

Задача: серия из 12 постов для бренда в Instagram. Каждый пост — статичный hero (бренд-токены) + 6-секундный motion-pin.

Без пайплайна (наивно): запускаем FLUX 12 раз руками, Seedance 12 раз руками, два дня работы, half-baked консистентность.

С пайплайном:

  1. Брифы — YAML-файл posts/spring-collection.yaml с 12 записями (тема, hero-prompt, motion-hint).
  2. Шаг hero — FLUX 2 с brand-prompt-шаблоном, seed зафиксирован per-post. CACHE KEY = sha256(prompt + seed + ref_hash).
  3. Шаг motion — Seedance 2.0 с hero как start_image. CACHE KEY = sha256(hero_hash + motion_hint + camera_params).
  4. Шаг compose — ffmpeg-сборка с лого + субтитрами. CACHE KEY = sha256(motion_hash + caption + logo_version).

Поменялся copy для одного поста? Пересчитывается только шаг compose, hero и motion — из кэша. Стоимость пересборки — копейки.

Копируемый скелет orchestrator

import hashlib, json
from pathlib import Path

CACHE = Path("./cache")

def cache_key(step: str, payload: dict) -> str:
    blob = json.dumps(payload, sort_keys=True).encode()
    return f"{step}_{hashlib.sha256(blob).hexdigest()[:16]}"

def run_step(step: str, payload: dict, runner):
    key = cache_key(step, payload)
    out_path = CACHE / f"{key}.bin"
    meta_path = CACHE / f"{key}.json"
    if out_path.exists():
        return out_path, json.loads(meta_path.read_text())
    result, meta = runner(payload)
    out_path.write_bytes(result)
    meta_path.write_text(json.dumps(meta, indent=2))
    return out_path, meta

# usage:
hero_path, hero_meta = run_step(
    "image:hero",
    {"prompt": prompt, "seed": 42, "ref": "brand:wedoit"},
    runner=flux_runner,
)

Это минимально жизнеспособный orchestrator: 20 строк, idempotent, audit-friendly. Любой scale-up начинается отсюда.

Что считать неизменяемым входом

Cache key должен быть стабилен между прогонами. Это значит:

  • Промпт-шаблон — версионируется (v1.0, v1.1). Изменение шаблона = новая версия = инвалидация кэша.
  • Бренд-токены — версионируются. brand:wedoit@2026-05 — это явный ref.
  • Random seed — фиксируется (не берётся «случайным» при каждом запуске).
  • Параметры модели — temperature/cfg/steps/aspect — часть payload.

Что не должно попадать в cache key: timestamp, имя пользователя, current working directory.

Подводные камни

  • Скрытая случайность — если модель использует random seed по умолчанию, два прогона с одинаковым payload дают разные кадры. Cache key становится бесполезным. Фиксируйте seed на уровне runner-а.
  • Версии моделей — провайдер обновил движок, ваши кэшированные кадры устарели. Включайте engine_version в cache key, чтобы при апгрейде кэш честно инвалидировался.
  • Разные модальности — разный размер кэша — image 1MB × 1000 = 1GB, video 30MB × 1000 = 30GB. На больших партиях кэш переезжает на S3 / R2 / Yandex Object Storage.
  • «Cache, который никто не чистит» — настройте TTL и cron-чистку артефактов старше N дней (или старых версий движка). Иначе диск кончится в худший момент.

Recap

  • Каждый шаг пайплайна = INPUT + OUTPUT + CACHE KEY. Без CACHE KEY проект быстро становится дорогим.
  • Lineage (JSON-сайдкар) — для повторяемости, ревью клиенту, бухгалтерии.
  • Стабильный cache key требует фиксации seed, версии шаблона, версии движка.
  • Перезапуск пайплайна = пересчёт только тех шагов, чьи входы изменились. Это и есть «дёшево итерировать».
