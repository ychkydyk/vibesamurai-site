A2 · Multi-modal pipeline architecture

Contract between modalities: text → image → video → control. Idempotency, caching, lineage. So the pipeline never regenerates what's already been built.

What we’re building

The architecture of a multi-modal pipeline — the program that calls generative services in sequence and assembles a final artifact (SMM post, clip, installation). One non-negotiable: the pipeline must not regenerate what’s already been generated, and it must give you a clear lineage (where every frame/second/sound came from).

The contract between steps

Every step has three mandatory fields. Same rule as M2 (Architecting agent boundaries), now applied to generative services rather than LLMs.

PIPELINE STEP
├── INPUT             hash-stable set: prompt text, refs, params
├── OUTPUT            artifact + metadata (model, version, seed, cost)
└── CACHE KEY         hash(INPUT) → if a file with that key exists, return it

Without a CACHE KEY every run regenerates the same frames. For a batch of 200 videos that’s 200 × 8 s × $0.27 ≈ $432 in pure regeneration cost.

Lineage: where did this frame come from

Every artifact ships with a JSON sidecar (or a metadata table):

{
  "id": "frame_a83f",
  "step": "image:hero",
  "engine": "flux-2",
  "engine_version": "2.0.3",
  "prompt_hash": "sha256:7c1e...",
  "seed": 42,
  "refs": ["brand_token://wedoit/main"],
  "cost_usd": 0.03,
  "created_at": "2026-05-29T14:22:11Z",
  "input_step_ids": []
}

What you get:

  • A month later you can answer why the clip looks the way it does.
  • The client asks “same one but blue” — you change one ref, and the cache reuses everything that doesn’t depend on colour.
  • Accounting sees the exact unit cost of the order.

Flow

flowchart LR
    BR[Brand tokens<br/>JSON]
    PR[Prompt template]
    BR --> PR
    PR --> H1{Cache hit?}
    H1 -->|yes| OUT1[hero image]
    H1 -->|no| IMG[FLUX 2]
    IMG --> OUT1
    OUT1 --> V1{Cache hit?}
    V1 -->|yes| OUT2[motion clip]
    V1 -->|no| VID[Seedance 2.0]
    VID --> OUT2
    OUT2 --> POST[Post compositor]
    POST --> FIN[final mp4]

Each arrow is a step with its own cache key. Any step can be restarted in isolation; the rest is reused.

Real case: static hero → motion pin for SMM

Brief: a series of 12 Instagram posts for a brand. Each post = brand-token hero still + a 6-second motion pin.

Without a pipeline (naive): run FLUX 12 times by hand, run Seedance 12 times by hand, two days of work, half-baked consistency.

With a pipeline:

  1. Briefs — posts/spring-collection.yaml with 12 entries (theme, hero-prompt, motion-hint).
  2. Hero step — FLUX 2 with the brand prompt template, seed pinned per post. CACHE KEY = sha256(prompt + seed + ref_hash).
  3. Motion step — Seedance 2.0 with the hero as start_image. CACHE KEY = sha256(hero_hash + motion_hint + camera_params).
  4. Compose step — ffmpeg assembly with logo and captions. CACHE KEY = sha256(motion_hash + caption + logo_version).

Copy changed for one post? Only the compose step reruns; hero and motion come from cache. Re-render cost is rounding error.

Copyable orchestrator skeleton

import hashlib, json
from pathlib import Path

CACHE = Path("./cache")

def cache_key(step: str, payload: dict) -> str:
    blob = json.dumps(payload, sort_keys=True).encode()
    return f"{step}_{hashlib.sha256(blob).hexdigest()[:16]}"

def run_step(step: str, payload: dict, runner):
    key = cache_key(step, payload)
    out_path = CACHE / f"{key}.bin"
    meta_path = CACHE / f"{key}.json"
    if out_path.exists():
        return out_path, json.loads(meta_path.read_text())
    result, meta = runner(payload)
    out_path.write_bytes(result)
    meta_path.write_text(json.dumps(meta, indent=2))
    return out_path, meta

# usage:
hero_path, hero_meta = run_step(
    "image:hero",
    {"prompt": prompt, "seed": 42, "ref": "brand:wedoit"},
    runner=flux_runner,
)

Minimum viable orchestrator: 20 lines, idempotent, audit-friendly. Every scale-up starts here.

What counts as an immutable input

A cache key has to be stable between runs. That means:

  • Prompt template is versioned (v1.0, v1.1). Editing the template = new version = cache invalidated.
  • Brand tokens are versioned. brand:wedoit@2026-05 is an explicit ref.
  • Random seed is pinned (not auto-generated on every run).
  • Model parameters — temperature/cfg/steps/aspect — are part of the payload.

What must not end up in the cache key: timestamps, usernames, the current working directory.

Pitfalls

  • Hidden randomness — if the model defaults to a random seed, two runs with the same payload produce different frames. The cache key becomes meaningless. Pin the seed inside the runner.
  • Model versions — the vendor ships an updated engine, your cached frames are stale. Include engine_version in the cache key so an upgrade invalidates the cache honestly.
  • Different modalities, different cache footprints — image 1 MB × 1000 = 1 GB; video 30 MB × 1000 = 30 GB. At scale, the cache moves to S3 / R2 / Yandex Object Storage.
  • “Cache nobody cleans” — set a TTL and a cron job that prunes artifacts older than N days (or older engine versions). Otherwise disk fills up at the worst possible moment.

Recap

  • Every pipeline step = INPUT + OUTPUT + CACHE KEY. Without the cache key the project gets expensive fast.
  • Lineage (JSON sidecar) — for reproducibility, client review, and accounting.
  • A stable cache key requires a pinned seed, a versioned template, and a versioned engine.
  • A pipeline rerun = only the steps whose inputs changed get recomputed. That’s what “cheap to iterate” actually means.

Further reading

  • DVC — Data Version Control — dvc.org. Off-the-shelf caching/lineage for ML pipelines; trivial to port to generative ones.
  • MLflow Tracking — mlflow.org/docs/latest/tracking. An alternative for metadata and lineage if you’re already on an ML stack.
  • Prefect / Dagster — modern orchestration frameworks with caching as a first-class citizen: prefect.io, dagster.io.
