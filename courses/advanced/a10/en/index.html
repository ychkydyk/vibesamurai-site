A10 · Audio-reactive AV installations

TD audio (VB-Cable loopback), spectral / onset / envelope, mapping into visuals. 27 flows, Tier 1 (in-house AudioReact) vs Tier 2 (audiobridge.js).

What we’re building

An audio-reactive AV installation: audio input (system sound or microphone) → feature extraction (spectral / onset / envelope) → mapping into visual parameters → render in TD or in the browser. Latency <50 ms (above that the audience hears motion “lag” behind the music).

Getting audio into TD

WASAPI loopback isn’t available in the current TD build (memory project_td_audioreact_audio). The fix is VB-Cable:

SYSTEM AUDIO ──► VB-Cable (virtual sink) ──► TD Audio Device In (loopback)
                            │
                            └─► Windows Listen-Through ──► real speakers

VB-Cable captures system audio, TD reads it as a regular mic input, Listen-Through forwards it to the physical speakers/headphones. Without Listen-Through the audience hears nothing while the visuals react — a classic first-run mistake.

Container layout in the .toe:

INPUT  (subcontainer)   audio in CHOP + analyzer CHOPs
OUTPUT (subcontainer)   visual rendering, projector mapping

Reference project file — Clode_React.toe on the desktop.

Audio features

FeatureWhat it showsWhere to use
Volume RMSLoudnessBase “amplitude” of motion
Spectrum (FFT)Frequency distributionColour, shape, texture
OnsetSharp changes (drum, hit)Triggers, flash
BeatTempoPulse, rhythmic motif
Envelope (ADSR)Attack/sustain/releaseSmooth fading
CentroidSpectral brightness”Brightness” of the scene

Simple rule: for impact/trigger → onset; for smooth motion → envelope; for colour/palette → spectrum.

Audio analyzer

From production (memory project_td_audioreact_flows): a separate Python module audio_analyzer.py is Phase B. Pre-processes the signal before it reaches the TD flow.

import numpy as np
import librosa

def analyze(samples: np.ndarray, sr: int = 48000):
    rms = float(np.sqrt(np.mean(samples ** 2)))
    spec = np.abs(np.fft.rfft(samples)[:1024])
    onset = float(librosa.onset.onset_strength(y=samples, sr=sr).mean())
    centroid = float(librosa.feature.spectral_centroid(y=samples, sr=sr).mean())
    return {
        "rms": rms,
        "spectrum_bands": spec.tolist(),
        "onset": onset,
        "centroid": centroid,
    }

Into TD via a DAT (for metadata) or CHOP (for continuous values). Again — not HTTP; latency is critical.

Tier 1 vs Tier 2

The collection of 27 flows (memory project_td_audioreact_flows) has two tiers:

  • Tier 1 — an in-house AudioReact stack running entirely in TD. Minimum dependencies, low latency, but written from scratch.
  • Tier 2 — via audiobridge.js, ready-made bridges from web/Wekinator. Faster start, but adds ~30 ms over the bridge.

Tier 1 — for production (a festival, a client install). Tier 2 — for a prototype or an AV evening where a small delay is acceptable.

There are also 55 factory presets — a starting sketch, not the finished product.

Real case: av_jam_lab_demo AV evening

From memory project_av_jam_lab_demo: a series of AV evenings, the first demo on 2026-05-08 in the office lab (24-hour event), tdmkii + TD on a 4070, installations A/B/C/F.

Stack:

  • A — sound → spectrum → 3D particles;
  • B — onset → trigger generative video (StreamDiffusion from A9);
  • C — envelope + centroid → projection colour;
  • F — voice input → text → visual typography (via STT — see A11).

Day-of prep: load 55 factory presets, pick the closest matches to the concept, set up VB-Cable, test for 30 minutes on a live signal, freeze the setup. Without the factory presets you spend the whole day on basic CHOP plumbing.

Scene Player

From memory project_td_audioreact_flows: Scene Player is Phase C. It’s a control layer on top of the flows — switching scenes via timeline, MIDI, OSC, or hotkeys.

The architecture is simple: one parameter container holds the active scene; switching changes the routes to the output TOPs.

TIMELINE / MIDI / OSC ──► Scene Player ──► scene_id (int)
                                              │
                                              ├──► [0] flow_a active
                                              ├──► [1] flow_b active
                                              ├──► [2] flow_c active
                                              └──► [3] flow_f active

Without Scene Player the performer jumps between .toe files by hand — and the audience sees it in the projection.

Pitfalls

  • WASAPI loopback isn’t in your TD build — don’t spend hours hunting “how to enable it”; install VB-Cable.
  • Forgot Listen-Through — the audience hears nothing but sees the reaction. Catch this in a 30-second test before the public.
  • Smoothing kills onset — if you smooth the onset CHOP, the hits disappear. Smooth the envelope, not the onset.
  • Big FFT buffer — the spectrum “lags.” Keep 512–1024 samples per FFT block at 48 kHz.
  • TD main-thread hang — a known issue (memory project_td_audioreact_hang). Don’t run WS servers on the same thread as frame_exec.

Recap

  • VB-Cable for system audio in TD; Listen-Through is mandatory.
  • 6 base audio features; for triggers use onset, for motion use envelope, for palette use spectrum.
  • Tier 1 (in-house AudioReact) — production; Tier 2 (audiobridge.js) — prototype/AV evening.
  • Scene Player + Tier 1 flows — working configuration for an hour-long installation.
  • Latency budget <50 ms; FFT 512–1024 per block at 48 kHz.
