A15 · RunPod Flash + vLLM

Serverless inference on a CUDA 12.9 4090 pool. RunPod Flash GA April 2026: @Endpoint decorator, GpuType enum, FlashBoot, 500MB code limit. When to move from local to cloud.

What we’re building

Scalable inference of an open-weight model in the cloud: the same Qwen3.5-21B, but instead of a local 4070 it runs on a RunPod 4090 pool with autoscale. Used when local 5 t/s isn’t enough (a 10k-request batch or a service under 5+ rps).

RunPod Flash (GA April 2026)

Key properties this solution is built on (verified 2026-05-29 against the RunPod docs):

  • Two Endpoint patterns: @Endpoint(...) as a decorator for queue-based batch / async work, or Endpoint(...) as an instance with route methods for load-balanced low-latency HTTP.
  • Custom Docker images — supports ready workers (vLLM, TensorRT-LLM, ComfyUI, your own image).
  • GpuType enum + GpuGroup — typed hardware selection: GpuType.NVIDIA_GEFORCE_RTX_4090, GpuType.NVIDIA_A100_80GB_PCIe, or GpuGroup.ANY / GpuGroup.AMPERE_80 for flexibility.
  • FlashBoot enabled by default on new GPU/CPU endpoints; cold start 1–3 seconds instead of 30–60 on classic Serverless.
  • flash deploy vs flash dev — deploy packages the code, installs deps into the worker image, and provisions production endpoints; dev is the live-test mode.
  • 500 MB code limit — the package is thin; model weights live separately (network volume or HF cache).
  • Pay per second — pay only for real inference seconds + warm-up.

Architecture

client ──► RunPod API ──► Flash endpoint ──► vLLM(qwen3.5-21b)
              │              │
              │              ├── FlashBoot (1-3s cold)
              │              └── pre-loaded weights (network volume)
              │
              └── queue → workers → autoscale

vLLM is a production-grade inference engine: PagedAttention, continuous batching, OpenAI-compat API. Under several concurrent requests it delivers what single-shot llama.cpp can’t.

Minimum endpoint (Flash GA)

Variant 1 — queue / batch via decorator:

# handler.py
from runpod.flash import Endpoint, GpuType
from vllm import LLM, SamplingParams

llm = LLM(
    model="Qwen/Qwen3.5-21B-Instruct",
    quantization="awq",
    gpu_memory_utilization=0.92,
    max_model_len=16384,
)

@Endpoint(name="qwen-flash", gpu=GpuType.NVIDIA_GEFORCE_RTX_4090)
def handler(event):
    messages = event["input"]["messages"]
    sampling = SamplingParams(
        temperature=event["input"].get("temperature", 0.0),
        max_tokens=event["input"].get("max_tokens", 1024),
    )
    out = llm.chat(messages, sampling)
    return {"choices": [{"message": {"content": out[0].outputs[0].text}}]}

Variant 2 — load-balanced HTTP via instance + a custom image (vLLM worker):

from runpod.flash import Endpoint, GpuType

vllm = Endpoint(
    name="vllm-server",
    image="runpod/worker-vllm:stable-cuda12.1.0",
    gpu=GpuType.NVIDIA_A100_80GB_PCIe,
    env={"MODEL_NAME": "Qwen/Qwen3.5-21B-Instruct"},
)

Deploy:

flash deploy            # production
flash dev               # live-test mode

No Dockerfile (if you use the built-in pattern). For a custom image — point at a ready worker like runpod/worker-vllm.

When to move from local

Triggers:

  1. >1000 real-time requests/day — the local node falls behind.
  2. Latency-critical — cloud + FlashBoot at 1–3 s is still slower than a warm local node, but it’s built for parallelism.
  3. Need a bigger model — 70B doesn’t fit on 8 GB VRAM, period.
  4. Batches of tens of thousands of requests — weeks locally, hours on a 4090 pool.

If none of the triggers apply, stay local. The cloud bill is always above zero.

CUDA 12.9 on 4090 EU-RO-1

Practical note: CUDA 12.9 in the EU-RO-1 4090 pool ✅ has been working since 2026-05-07 — the vLLM path is ready for deploy.

Queue metrics

Before start and after restart/stop — inQueue and inProgress (see also A11):

import requests

def queue_status(endpoint_id: str, api_key: str):
    r = requests.get(
        f"https://api.runpod.ai/v2/{endpoint_id}/status",
        headers={"Authorization": api_key},
    ).json()
    return {
        "ready": r["workers"]["ready"],
        "in_queue": r["jobs"]["inQueue"],
        "in_progress": r["jobs"]["inProgress"],
    }

Show queue_status in real time in CI / dashboards. If the queue grows faster than it drains, raise max_workers.

Cost calc

Rough estimate for Qwen3.5-21B on 1 × 4090:

  • 4090 RunPod ~ $0.69/hour;
  • FlashBoot + inference ~ 200 ms per response (warm);
  • 1 hour = ~18 000 answers under steady load.

At 1 rps the real cost = $0.69 / 3600 ≈ $0.0002 per request. Cheaper than Claude Haiku on tokens, but more expensive on cold-start scenarios.

Tokenomics gate (see A17): before launching a batch — compute the cost and require [y/N] confirmation.

Cascade with local

The production pattern is a cascade:

request ──► local Qwen 4070 ──► ok ──► OUTPUT
                  │
                  └── overload or fail ──► RunPod Flash Qwen ──► OUTPUT

The local node absorbs 80 % of baseline load for free. Cloud kicks in when local overflows. Without a cloud node in the architecture, the booth stalls at peak; without a local node, money leaks on every request.

Pitfalls

  • Cold start even with FlashBoot — first request after idle 1–3 s. Warm by cron.
  • GPU pool is regional — EU-RO-1 has CUDA 12.9, US-CA-1 may lag. Check before deploy.
  • vLLM isn’t for every GGUF — vLLM works with its own format and AWQ/GPTQ. For DavidAU GGUF — Ollama path (dead) or convert.
  • Flash’s 500 MB code limit — model + venv won’t fit. Use network volume or HF cache.
  • No queue-check after restart — workers may not be ready, requests pile up.

Recap

  • RunPod Flash (GA April 2026) — serverless without your own Dockerfile, FlashBoot 1–3 s.
  • vLLM as the inference engine for concurrent load.
  • Move from local: >1000 rpd real-time, need 70B, batches of tens of thousands.
  • Cascade local → cloud is a working production architecture.
  • Queue check after restart is mandatory; cost calc before a batch.
