A15 · RunPod Flash + vLLM

Serverless inference на CUDA 12.9 4090 пуле. RunPod Flash 2026-04-30 GA: @Endpoint без Docker, FlashBoot, 500MB limit. Когда переходим с локалки на cloud.

Что строим

Масштабируемый inference open-weight модели в облаке: один и тот же Qwen3.5-21B, но не на локальном 4070, а на пуле 4090 RunPod с автоскейлом. Используется, когда 5 t/s локалки не хватает (батч на 10k запросов, или сервис с нагрузкой 5+ rps).

RunPod Flash (GA апрель 2026)

Ключевые свойства, на которых строится решение (сверка 2026-05-29 по RunPod docs):

  • Два паттерна Endpoint: @Endpoint(...) как decorator для очередей и batch-async, либо Endpoint(...) как instance с route-методами для load-balanced HTTP с низкой задержкой.
  • Custom Docker images — поддержка готовых воркеров (vLLM, TensorRT-LLM, ComfyUI, ваш собственный образ).
  • GpuType enum + GpuGroup — типизированный выбор железа: GpuType.NVIDIA_GEFORCE_RTX_4090, GpuType.NVIDIA_A100_80GB_PCIe, либо GpuGroup.ANY / GpuGroup.AMPERE_80 для гибкости.
  • FlashBoot включён по умолчанию на новых GPU/CPU эндпоинтах; cold start 1–3 секунды вместо 30–60 у классического Serverless.
  • flash deploy vs flash dev — deploy пакетит код, ставит зависимости в worker image и провижинит production endpoints; dev — режим live-test.
  • 500 MB code limit — упаковка тонкая, веса модели — отдельно (network volume или HF cache).
  • Pay per second — платим только за реальные секунды inference + warm-up.

Архитектура

client ──► RunPod API ──► Flash endpoint ──► vLLM(qwen3.5-21b)
              │              │
              │              ├── FlashBoot (1-3s cold)
              │              └── pre-loaded weights (network volume)
              │
              └── queue → workers → autoscale

vLLM — production-grade inference engine: PagedAttention, continuous batching, OpenAI-compat API. Под нагрузку нескольких параллельных запросов делает то, что llama.cpp одиночно не вытянет.

Минимальный endpoint (Flash GA)

Вариант 1 — очередь / batch через декоратор:

# handler.py
from runpod.flash import Endpoint, GpuType
from vllm import LLM, SamplingParams

llm = LLM(
    model="Qwen/Qwen3.5-21B-Instruct",
    quantization="awq",
    gpu_memory_utilization=0.92,
    max_model_len=16384,
)

@Endpoint(name="qwen-flash", gpu=GpuType.NVIDIA_GEFORCE_RTX_4090)
def handler(event):
    messages = event["input"]["messages"]
    sampling = SamplingParams(
        temperature=event["input"].get("temperature", 0.0),
        max_tokens=event["input"].get("max_tokens", 1024),
    )
    out = llm.chat(messages, sampling)
    return {"choices": [{"message": {"content": out[0].outputs[0].text}}]}

Вариант 2 — load-balanced HTTP через instance + custom image (vLLM-worker):

from runpod.flash import Endpoint, GpuType

vllm = Endpoint(
    name="vllm-server",
    image="runpod/worker-vllm:stable-cuda12.1.0",
    gpu=GpuType.NVIDIA_A100_80GB_PCIe,
    env={"MODEL_NAME": "Qwen/Qwen3.5-21B-Instruct"},
)

Деплой:

flash deploy            # production
flash dev               # live-test режим

Без своего Dockerfile (если используете встроенный паттерн). Для custom image — указываем готовый runpod/worker-vllm.

Когда переходим с локалки

Триггеры перехода:

  1. >1000 запросов/день в реальном времени — локалка не успевает.
  2. Латентность критична — облако с FlashBoot 1–3 c всё равно медленнее тёплой локалки, но рассчитано на параллелизм.
  3. Нужна большая модель — 70B не помещается на 8GB VRAM никак.
  4. Партии в десятки тысяч запросов — на локалке = недели, на пуле 4090 = часы.

Если триггеров нет — оставайтесь на локалке. Cloud-bill всегда больше нуля.

CUDA 12.9 на 4090 EU-RO-1

Практический момент: CUDA 12.9 в пуле 4090 EU-RO-1 ✅ работает с 2026-05-07 — vLLM-путь готов к деплою.

Альтернативный путь — Ollama для DavidAU GGUF — мёртв (issue #15747). Не тратьте время.

Рабочие конфиги храните в отдельной папке archive/runpod-vllm/ в git-репо студии — версионируются вместе с кодом workers.

Queue-метрики

Перед стартом и после restart/stop — inQueue и inProgress (см. также A11):

import requests

def queue_status(endpoint_id: str, api_key: str):
    r = requests.get(
        f"https://api.runpod.ai/v2/{endpoint_id}/status",
        headers={"Authorization": api_key},
    ).json()
    return {
        "ready": r["workers"]["ready"],
        "in_queue": r["jobs"]["inQueue"],
        "in_progress": r["jobs"]["inProgress"],
    }

В CI / dashboard стенда — отображайте queue_status в real-time. Если queue растёт быстрее, чем drains — поднимайте max_workers.

Cost calc

Грубая оценка для Qwen3.5-21B на 1 × 4090:

  • 4090 RunPod ~ $0.69/час;
  • FlashBoot + inference ~ 200 ms per response (warm);
  • 1 час = ~18000 ответов при стабильной нагрузке.

При нагрузке 1 rps реальная себестоимость = $0.69 / 3600 ≈ $0.0002 за запрос. Это дешевле, чем Claude Haiku по токенам, но дороже на cold start scenarios.

Tokenomics gate (см. A17): перед запуском партии — расчёт и [y/N] confirm обязательны.

Cascade с локалкой

Production-паттерн — каскад:

запрос ──► локальный Qwen 4070 ──► ok ──► OUTPUT
                     │
                     └── overload или fail ──► RunPod Flash Qwen ──► OUTPUT

Локальный узел гасит 80% базовой нагрузки бесплатно. Cloud подключается, когда локалка перегружена. Без cloud-узла в архитектуре в час пик стенд встаёт; без локального — деньги текут на каждый запрос.

Подводные камни

  • Cold start даже с FlashBoot — первый запрос после простоя 1–3 c. Прогревайте по cron.
  • GPU pool региональный — EU-RO-1 имеет CUDA 12.9, US-CA-1 может отставать. Перед deploy — проверяйте.
  • vLLM не для всех GGUF — vLLM работает с собственным форматом и AWQ/GPTQ. Для DavidAU GGUF — Ollama путь (но он мёртв) или конвертация.
  • 500MB code limit Flash — модель + venv в 500MB не лезут. Используйте network volume или HF cache.
  • Без queue-check после restart — workers могут быть не ready, запросы скапливаются.

Recap

  • RunPod Flash (GA 2026-04-30) — серверлесс без Docker, FlashBoot 1–3 c.
  • vLLM как inference engine для параллельной нагрузки.
  • Переход с локалки: >1000 rpd real-time, нужна 70B, партии в десятки тысяч.
  • Cascade local → cloud — рабочая прод-архитектура.
  • Queue-check после restart обязателен; cost calc перед партией.
