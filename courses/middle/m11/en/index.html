M11 · Evaluating agents: metrics over vibes

What 'the agent works well' actually means. Metrics, dashboards, evals. How to move past 'feels like it got worse'.

Hook

WITHOUT metrics:                    WITH metrics:

"Feels like the agent     →         Run on 50 reference cases:
 has been duller this                42/50 → 38/50
 week"                               breaks more often at step 4

zero data, all feelings              concrete signal, something to fix

Hey folks, welcome 👋

This chapter opens Part IV — Evaluation & Tuning. The most delicate part of the course. In the first three parts we built the architecture of agents: frames, tools, memory. Now — how to measure their work so we know they work well, instead of “seem to.”

This topic is dry to most. I sat on it for too long myself — “I can feel when it’s bad.” Then two strikes in a row happened: the model silently regressed, I silently lost time, and loudly cursed myself for the lack of metrics.

An hour, maybe an hour and a half. This is hygiene, without which Middle isn’t really Middle.

Concept: “works” is subjective until you quantified it

When we say “the agent works well” we usually mean something like:

  • “Understands my requests”
  • “Doesn’t lie”
  • “Does it fast”
  • “Doesn’t do extra”

These are all vibes. In a team they don’t work — everyone has their own. Over time they don’t work either — last week “felt fine,” today “feels worse,” and there’s no data.

┌─────────────────────────────────────────────────────┐
│  Without metric: "kind of works"                    │
│  With metric:    "pass rate 87% (±3%)               │
│                   on 50 reference cases"            │
└─────────────────────────────────────────────────────┘

Big difference. You can compare weeks, model builds, configurations.

Three measurement tiers

You don’t need to start with a “production eval pipeline.” Begin with what’s accessible.

Tier 1. Qualitative log (zero setup)

In CLAUDE.md / in a personal note, keep evals/log.md. After each significant session write:

## 2026-05-15
- Task: DB migration Postgres 15 → 17, 4 microservices
- Model: Opus 4.6
- Did well: correct migrations, understood dependencies
- Did poorly: spent 30 minutes on table formatting instead of logic
- Reverts?: 1 — auth-service wrong JOIN side
- Time: 2.5h (planned 2.0)

Cost — 2 minutes after a session. Payoff — in a month patterns appear: “consistently struggles on JOINs,” “after that model update got better on migrations.”

Tier 2. Pass/fail on reference cases (half an hour setup)

Collect 5–20 typical tasks you regularly verify on. For example:

  • “Refactor auth.py — extract session handling into a separate module”
  • “Add pagination to /api/posts endpoint”
  • “Find why test_user_login is flaky”

Each task has an acceptance criterion (like in B3, but stricter — programmatically checkable or by checklist).

Once every 2–4 weeks run all of them on the current configuration. Record the pass rate. That’s your benchmark.

## Reference Suite 2026-05-01
| # | Task | Pass? | Notes |
|---|---|---|---|
| 1 | auth.py refactor | ✓ | OK |
| 2 | pagination | ✓ | + edge: didn't handle page=0 |
| 3 | flaky test fix | ✗ | proposed retry instead of root-cause |
...
Pass rate: 14/20 = 70%

A month later → 76%. Progress is visible.

Tier 3. Automated eval (hours of setup, pays off at scale)

Full pipeline:

  • Reference tasks in YAML/JSON
  • A script runs the agent on each
  • Output is verified programmatically (regex, AST diff, unit tests)
  • Results go to CSV/dashboard

You need this when:

  • You have many developers on agents and need a common metric
  • You run a SaaS on agents and need a production metric
  • You tune agents for clients and need to prove progress

For one person — tier 1–2 is usually enough.

Which metrics to measure

Qualitative (binary / scalar)

  • Pass rate on reference suite — % of tasks the agent did correctly
  • Revert rate — % of accepted edits later rolled back
  • Bypass rate — how many times permissions had to be bypassed (see M3 — this is an anti-metric, must stay low)
  • Approval friction — number of approval prompts per task (see M3)

Efficiency (numeric)

  • Time-to-PR — from prompt to ready result
  • Iteration count — how many reformulations
  • Token usage per task — economics, especially on paid APIs
  • Tool call distribution — which tools the agent calls most

Trust (meta)

  • Inspectability score — % of agent steps you saw and understood (see M3 — 5 artifacts)
  • Post-mortem count — how often you had to dig into “why did the agent do this”

Measurement anti-patterns

1. “Good number” instead of “right number”

Most common miss. You picked the metric “lines of code per day” — the agent started writing more, but worse. Metric grew, reality got worse.

Cure: the metric must reflect what you actually want, not a surrogate. Pass rate on a reference suite is outcome. Lines of code is activity, not outcome.

2. One run ≠ a metric

Ran the suite, got 78 %. Cheers. Ran again — 65 %. What happened?

Nothing. The model has high variance. One run is a point, not a signal.

Cure: for serious decisions — 3+ runs, means ± standard deviation. For tracking — pin seeds for random components.

3. “Metric passed, therefore works”

The trickiest anti-pattern (see B10). You hit 94 % on the suite. You think — “great, the agent works.”

The suite is 30 cases. In reality you have 300 types of task. The suite doesn’t cover 90 % of reality.

Cure: periodically update the suite — add cases from actual screw-ups. Every time the agent seriously dropped the ball — that case goes into the suite. In a year the suite becomes representative.

4. Metric → pressure → manipulation

In the team you announce “pass rate must be >85 %.” The team starts tuning the suite to hit 85 %. Cases the agent reliably fails on — deleted.

Cure: separate tracking from pressure. Metric — for information, not for “KPI.” Same conversation as developer productivity metrics (commits/day, etc.) — they rot the moment they become the goal.

A short aside ⚡

A real story. One of the folks running an agentic pipeline initially measured “token cost per task.” Convenient metric: cheap = good.

Two months in the agent forgot how to do hard tasks — because we trained it to minimise tokens, not deliver outcomes. Did it cheap and badly.

Changed the metric to “pass rate × (1 − cost-relative-to-baseline)” — a “result and cost” compromise. Got better.

I think about this every time someone proposes “let’s just measure X.” X doesn’t exist in a vacuum. Every metric is pressure, and pressure warps the system.

Example: stand up a reference suite in an hour

Take a typical dev project (CRUD web app).

Step 1. Pick 10 typical tasks (15 minutes)

In evals/cases.md:

# Reference Suite v1

## Case 01 — refactor controller
**Input:** In `src/api/users.ts` the `createUser` method mixes validation, business logic, and DB access. Extract validation into a separate module `validators/userValidator.ts`.
**Expected:**
- File `src/validators/userValidator.ts` exists
- `createUser` imports from the new module
- Existing tests `users.test.ts` pass

## Case 02 — bug fix
**Input:** Test `cart.test.ts → calculates discount` fails. Symptom: when coupon='WELCOME10' returns 0 instead of 10 %. Find the cause and fix.
**Expected:**
- Test green
- Changes only in `cart.ts` or `discount.ts`
- No hardcoded branches for a specific coupon

## Case 03 — feature add
...

10 cases cover: refactors, bug fixes, new features, docs, tests, security.

Step 2. Manual run (30 minutes)

For each case:

  1. Clean branch
  2. Feed input to the agent
  3. Time it, observe-mode
  4. On completion — verify expected by hand
  5. Record pass/fail + a note
## Run 2026-05-15
| # | Pass | Time | Notes |
|---|---|---|---|
| 01 | ✓ | 8m | OK |
| 02 | ✗ | 12m | proposed a workaround, not root-cause |
| 03 | ✓ | 6m | OK, stray change in comments |
...

Step 3. Analysis (15 minutes)

  • Pass rate: 7/10 = 70 %
  • Where does it fail? Hard debug tasks (02), security (08)
  • Trend? Last month was 6/10 — there’s progress

A month later — repeat. That’s measurement.

Eval as a team practice

If the team has >1 developer on agents — a shared suite is mandatory. Otherwise:

  • Each tunes the agent “for themselves”
  • Configs drift apart
  • Nobody knows if the collective got better or worse

Rules for a team suite:

  1. Common suite in the repo — evals/cases.yaml in the same branch as code
  2. Owned by a maintainer — the suite has an owner, otherwise it rots
  3. Update on incident — every prod incident → new case
  4. Quarterly review — quarterly, drop what’s stale
  5. No blame on numbers — numbers = information, not KPI

Practice (60–90 minutes)

Task 1 (30 minutes) — build your reference suite

Find 5 real tasks you gave the agent in the last month — successes and failures. Write them into evals/cases.md by the template: input + expected.

That’s your first baseline. In 2 weeks — run. In a month — run again.

Task 2 (15–30 minutes) — formal log

Start evals/log.md. Record the last 5 significant sessions after the fact: task, what was good, what bad, time, reverts.

This is the minimum measurement level, and it pays back in 2–3 weeks — patterns become visible.

Task 3 (15 minutes) — pick a “not-your” metric

Think about which metric you could introduce in the team and why it would warp the system. A useful exercise before you introduce a metric — spot the anti-pattern in advance.

Recap

  • No measurement of quality — no Middle level. “Feels worse” is not a signal, it’s noise.
  • Three measurement tiers: session log (5 min/week) → reference suite (one hour setup) → automated pipeline.
  • Anti-patterns: “good number instead of right number,” “one run = metric,” “passed the suite = ok,” “metric as KPI.”

Vibe Samurai

🥷 Vibe Samurai asks: “This is bureaucracy! I work alone, why would I need a reference suite?”

Answer: The suite isn’t “bureaucracy,” it’s your external memory of quality. In 3 months you won’t remember which tasks the agent was good at in March. Without a suite you’ll guess. With it — you check the numbers and know. Same story as git: “why commit, I work alone” → because without commits you lose control fast. The suite is commits for quality. Minimum level — 5 minutes/week, and it pays off the first “feels like the agent got duller, but actually I just framed the task wrong.” Not perfect, but actually works.
