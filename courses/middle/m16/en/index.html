M16 · When multi-agent is overkill. Anti-patterns

The other side of M15: where multi-agent isn't needed, how to convince yourself or the team not to build it.

Hey folks, welcome 👋

This chapter is the flip side of M15. M15 was “how to build,” M16 is “when not to.” And — spoiler — that’s most of the time.

I’ve personally seen dozens of projects where people started with multi-agent for aesthetic reasons (“sounds cool”), and half of them later came for advice with “help me unpack this, I don’t even know what’s where anymore.” This chapter is the vaccine against that.

An hour. This is a strong chapter, and it closes Part V.

Concept: multi-agent = a distributed system

Anyone who’s worked with distributed systems in regular dev knows this line:

“Microservices are a solution that creates more problems than it solves, except in specific cases.”

With multi-agent — the same. Multi-agent is a distributed system of agents. And all the problems of distributed systems are cubed here:

┌─────────────────────────────────────────────────────┐
│  Usual distributed-systems problems:                │
│  - Network failures                                 │
│  - Eventual consistency                             │
│  - Partial failures                                 │
│  - Debugging complexity                             │
│                                                     │
│  + multi-agent adds:                                │
│  - Non-determinism of each agent                    │
│  - LLM-cost amplification                           │
│  - Coordination overhead                            │
│  - "Whose context is right?" — constantly           │
└─────────────────────────────────────────────────────┘

So: pipeline first (M14). Multi-agent — only when a pipeline demonstrably can’t cope.

Signals that multi-agent is NOT needed

A “can we skip multi-agent” checklist. If any one answer is yes — don’t build.

SignalWhat it means
One-person taskIf you’re solo and the work fits in a day — a pipeline is enough
Context fits in 200K tokensNobody is forced to split contexts
No real parallelismIf steps are sequential by meaning — pipeline
All “roles” handled well by one modelDifferent models is a common argument; if one handles all — don’t split
No autonomy requirementIf a human reviews after each major step — pipeline is enough
Tight budgetMulti-agent costs multiples more on the API
Team < 3 peopleMaintaining multi-agent is a job for a dedicated engineer
MVP stageOn MVP — you never need multi-agent, too early to over-engineer

In reality — that’s 80 % of cases. If the task lands in most of these — strictly pipeline.

The main anti-patterns

Anti-pattern 1. “Cargo cult multi-agent”

Read an article “how we automated SRE on 7 agents.” Decided — “we need it too.” Didn’t figure out why the article picked 7.

Signs:

  • You can’t briefly explain why exactly 5 (or 7) agents and not 2
  • You can’t point to a number that improves with multi-agent vs pipeline
  • The architecture “looks like” successful case studies

Cure: the “one-for-one” rule — name one concrete workflow where multi-agent is objectively faster/better than a pipeline. Can’t name one — don’t build.

Anti-pattern 2. “A role per agent”

The temptation: “we have planner, coder, tester, reviewer, doc-writer, release-manager — let each be its own agent.” No, let them not.

Signs:

  • 5+ subagents, each doing a narrow function (one-two commands)
  • 80 % of their time is waiting for each other
  • The task burns 3× more tokens than one-shot

Cure: merge roles. planner+architect = one role (same context). coder+formatter = one role (same tools). If there’s no real reason to split — don’t split.

Anti-pattern 3. “The agent must have full autonomy”

“We want the system to handle everything itself, no human.” Sounds pretty, in reality — path to problems.

Signs:

  • No human integration points in the pipeline
  • “Supervisor decides when to escalate”
  • Budgets on “time / iterations / tokens” not strictly defined

Cure: see M3 — HITL is mandatory at critical steps. Even the most advanced production agents in 2026 still require a human on:

  • merge to main
  • deploy to prod
  • changes to permissions / billing
  • decisions with irreversible consequences

Anti-pattern 4. “Multiple supervisors”

“A master supervisor runs sub-supervisors that run workers.” Hierarchy of hierarchies.

Signs:

  • Depth > 3
  • One supervisor pulls another supervisor
  • In diagnostics it’s unclear whose fault the final outcome is

Cure: flat hierarchy. One supervisor, up to 5 workers. If it truly doesn’t fit — these aren’t multi-agent, these are separate products to be built separately and linked via API/queue.

Anti-pattern 5. “Free communication — let them sort it out”

“I’ll give them shared memory, let them write to each other however.” Ends in infinite dialogue without a result.

Signs:

  • No message format
  • No rules about when the agent should stop talking
  • The log shows “agents talked about it for 47 messages, no conclusion”

Cure: a hard protocol (JSON Schema), hard limits (max N messages per discussion), a mandatory supervisor to take the final call.

Anti-pattern 6. “Build first, measure later”

Half a year of work, a multi-stage system — and nobody compared with a simple pipeline.

Signs:

  • No A/B benchmark vs single-agent / pipeline
  • “Improvement” metrics are subjective (“the team is happy”)
  • When asked “why” — generic words about “scalability”

Cure: before project start — a harness (M12) on 3 representative tasks. Run on single-agent, pipeline, multi-agent. Compare. If multi-agent isn’t significantly better — don’t build.

Anti-pattern 7. “Multi-agent for economy of scale”

“If we have 1000 tasks/day — multi-agent will pay off.” Not necessarily.

Signs:

  • The argument leans on “scale”
  • 1000 tasks/day = 1 task every 1.4 minutes
  • Multi-agent per task — 15–30 minutes of wall-clock due to coordination overhead

Cure: for mass tasks it’s often cheaper to run a parallel pipeline (N pipelines at once) than multi-agent. Simpler and cheaper.

A big B insert — a real case

Story from my own life. A client had a PR factory idea: “let’s make a supervisor that dispatches tasks from Linear to 5 worker agents, and they autonomously open PRs, pass review, merge.” On paper — pretty. In a deck — impressive.

We started building. Three months of work by a team of 4:

  • supervisor — Opus 4.6, special prompt
  • coder-worker — Sonnet 4.6, repo access
  • test-worker — Sonnet 4.6, sandbox
  • reviewer-worker — Opus 4.6, read-only + comments
  • release-worker — Haiku 4.5, only merges approved
  • shared bus — Redis + Postgres for state
  • monitoring — Grafana, dashboards per agent

Dev infra cost: 4 person-months. Production API cost (first week): $1200.

What worked: simple features (CRUD endpoint, small bug fixes) — yes, ran itself.

What didn’t:

  • Any non-trivial task (migration, security-sensitive, refactor) — deadlocked between coder and reviewer
  • $200–300/day on agent “discussions”
  • Team constantly monitoring “did it hang”
  • For every complex case — still called a human
  • After 2 months it turned out a pipeline (planner→coder→tester, no reviewer-agent, human review) did the same at a quarter of the cost

What stayed: a simplified pipeline + one subagent for tests. Multi-agent infra — ripped out.

Full menu of the lesson: we built a complex tool because it sounded cool. No harness before the project. No A/B comparison. Believing “we’re building the future.”

I’m not against multi-agent in principle. I’m against multi-agent before the pipeline is exhausted. And against multi-agent without a benchmark proving the win.

When multi-agent REALLY pays off

So I don’t sound like I’m categorically against. Cases where multi-agent is justified:

1. Long autonomous processes (24/7)

E.g. an on-call assistant — answers alerts, diagnoses, escalates. Here you really need:

  • Parallel processes (several alerts at once)
  • Different roles (triage, investigator, human communicator)
  • Long uptime

2. Adversarial workflows (generation + critique)

E.g. content review — one agent generates, another independently critiques. If they share context and model, the critic has the same blind spot as the generator.

3. Real model specialisation

E.g. text by one model (Claude), code by another (Codex), images by a third (Midjourney via API). Here multi-agent is natural — you can’t merge them into one.

4. High parallelism with independent tasks

E.g. processing 10 000 PDFs — 10 workers in parallel on dispatched tasks. Pure parallelism, no coordination among workers.

In these cases multi-agent is a natural solution. Elsewhere — overkill.

Checklist “deciding to build multi-agent”

Before starting — walk through:

  • Pipeline (M14) demonstrably doesn’t cope — there’s a benchmark
  • I can explain in one sentence why exactly multi-agent (not “scales,” concretely)
  • I have budget discipline — iterations, time, tokens
  • I have a harness (M12) to measure progress on
  • I have a communication format (JSON Schema)
  • I have HITL points at critical steps
  • I know who on the team will own the multi-agent infra
  • I’m ready to roll all of this back if the benchmark doesn’t show a win

All boxes — yes → build. Any one — no → not yet.

Finale — register C

Part V overall is about what you can avoid building. It’s counterintuitive: a course is supposed to “teach technology,” and here I’m teaching not to build. But the honest position is this: 80 % of the industry in 2026 over-invests in multi-agent. Months of teams burned, millions of dollars, and they come back to simple.

That doesn’t mean multi-agent is useless. It means it has a narrow zone of applicability, and the entry threshold is high. If you’re inside — you have a big budget, a big team, big pain. That’s rare.

The principle I apply everywhere, not just to multi-agent — YAGNI (You Aren’t Gonna Need It). If you’re not sure you need a complex thing — you probably don’t. In six months it turns out you got just fine without it — and you’ll thank yourself for not building it.

That’s an engineer’s maturity: not what they can build, but what they chose not to. The full package is knowing how to do complex and not doing it when it’s not needed. Not perfect, but actually works.

Recap

  • Multi-agent = distributed system. All its problems + LLM non-determinism.
  • 80 % of cases — a pipeline suffices. The “not needed” signals are spot-on.
  • Anti-patterns: cargo cult, fine-grained split, full autonomy, free-form communication, no budgets, no benchmark.

📚 Further reading

This chapter is mine, but the “multi-agent or not” debate in the industry is live and both sides have arguments. Read both — you’ll know who’s arguing about what.

“Don’t build multi-agents” — argument A

  • Cognition — Don’t Build Multi-Agents — the post that started the debate; key thought: context management breaks when the task is split across agents: cognition.ai/blog/dont-build-multi-agents.
  • LangChain — How and when to build multi-agent systems — a more pragmatic take from the LangGraph team: when multi-agent does make sense, when not: blog.langchain.com/how-and-when-to-build-multi-agent-systems.

“Build, but carefully” — argument B

  • Anthropic — How we built our multi-agent research system — a case where multi-agent gave +90 % over single-agent baseline, with an honest discussion of cost and pitfalls: anthropic.com/engineering/multi-agent-research-system.
  • Inside the Multi-Agent Debate (Shivanand Roy / Medium) — a compact side-by-side of the Cognition and Anthropic positions: snrspeaks.medium.com/inside-the-multi-agent-debate-why-cognition-labs-says-dont-and-anthropic-says-do-carefully-7b8a253e0b1e.

Tip: if you’ve read both sides and still want multi-agent — say the price out loud (latency, debugging, state). If a concrete number makes you uncomfortable — that’s the honest answer.
