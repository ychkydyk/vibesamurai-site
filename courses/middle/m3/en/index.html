M3 · Observability and human-in-the-loop

The autonomy ladder, inspectable artifacts, and how to keep HITL without losing velocity.

What this chapter is about

Hey folks, welcome 👋

In M1 we agreed on the paradigm. In M2 — on the contract and boundaries. Now we close Part I with the third pillar: observability and human-in-the-loop.

These are two topics usually merged because they answer the same question: “how much freedom do I give the agent, and how do I know what’s happening while it works?”

Across Part I we spend a lot of time in “I don’t trust you” mode. That’s not paranoia — that’s a sensible engineering stance towards a new tool. In Part II we’ll go further and dig into the tools themselves. But without today’s chapter — you can’t get there.

The autonomy ladder: where do you want to stand

Agent autonomy isn’t a binary (“manual vs autonomous”). It’s a ladder with discrete rungs. Each rung is a concrete set of permissions and a concrete set of observability requirements.

LEVEL 5  │ Full-auto + self-merge                  │ almost nobody works here
LEVEL 4  │ Auto-execute + auto-PR, human-merge     │ advanced teams
LEVEL 3  │ Auto-execute with approval gates         │ typical production
LEVEL 2  │ Plan-first, manual approve each step    │ early integration
LEVEL 1  │ Plan-only, does nothing                 │ learning start / R&D
LEVEL 0  │ Read-only, explains                     │ "look and tell me"

Where each rung fits

SituationLevelWhy
First session with a new agent0–1You don’t know its weaknesses
Walking an unknown repo0Reading + explanation, no actions
Light refactor, tests exist3Tests are the auto-gate
Hotfix in prod2High risk, every step by hand
Routine task (format, rename, boilerplate)4Narrow scope, no room for surprises
Full feature with an architectural decision2–3Plan is mandatory, actions stepwise
Anything with secrets / prod creds0–1No alternatives

Rule: default to the lowest rung on which the task can actually be done, and climb only with a concrete reason. Low rung = less speed, more chance to catch problems early. High rung = more speed, larger blast radius.

Also: the autonomy level isn’t fixed across an agent’s “career.” In one repo it can run at 4 for routine tasks and at 2 for architectural decisions. It’s a switch, not a property of the tool.

Inspectable artifacts: what should remain after the agent

If the agent worked — there must be enough traces so any other person (or you a week later) can reconstruct what happened without opening the chat.

Minimum set of artifacts:

1. Plan artifact

The YAML/JSON/markdown we discussed in M2. Saved in the repo: .agent/plans/2026-05-16-add-pkce.yaml or similar. Not in chat, not in a log — in the repo, so it’s in history and under version control.

2. Action log

A chronological log of everything the agent did: which tools it called, with which arguments, what the result was. Not “thought process” — exactly the actions. In modern tools this often comes out of the box:

  • Claude Code — ~/.claude/projects/<...>/ stores JSONL sessions
  • Cursor — Composer history panel
  • Codex CLI — --log-file option

For a production flow I usually require the action log to be exported into the repo as part of the PR (e.g. as a PR comment or attached file).

3. Diff artifact

The diff itself, obviously — that’s git. But notice: the diff must match the plan. If the plan has 3 steps and the diff touches 17 files, that’s the first red flag at review.

4. Evaluation report

What exactly was checked after the action and what came of it. It can be just the test output, or a separate JSON “criterion 1: passed, criterion 2: failed.” The point — the result is tied to the criterion from the plan, not to subjective “works.”

5. Bumps & retries log

How many times the agent tried and revised the plan. An important metric — if every task takes 5+ iterations, you have a problem in the task (poorly framed), or the agent (out of its depth), or the context (insufficient).

Principle: an artifact = something that survives the session closing. If it only lives while the chat is open — it’s not an artifact, it’s noise. Noise is fine, but noise can’t be the only trace of work.

HITL without killing velocity

Good human-in-the-loop isn’t “slam the brakes every step” — it’s a side bus of events: the agent works, events fly to a log / observability sink, the human steps in on signal and returns corrections to the agent. Roughly:

flowchart LR
    A[Agent<br/>action] --> E[Emit event<br/>log / span]
    E --> O[(Observability<br/>sink)]
    O --> H{Human<br/>review}
    H -->|approve| Next([Continue])
    H -->|correct| C[Correction<br/>note / diff]
    C -.-> A
    A --> Next

Events are written always (that is observability), and the human steps in by rule — not on every tool-call, but on sensitive steps, anomalies, and approaching budgets. Then HITL doesn’t drag velocity.

The most common objection from management and experienced colleagues:

“If you have to approve every step, it’s slower than doing it yourself.”

The objection is fair in the case of bad HITL and unfair in the case of good HITL. The difference comes down to three rules.

Rule 1: approval must be fast

Approving a step = 3–10 seconds, not “read 5 pages of context.” Which means:

  • The plan is structured (M2)
  • The diff is minimal (narrow steps instead of wide)
  • The success criterion is explicit (it’s clear what to check)
  • The approval interface is (y/n) level, not “open 4 tabs”

If approval takes minutes — your flow is wrong, not your HITL.

Rule 2: approval should be batched where possible

Not every action needs approval. Often what’s needed is approval of a series of related actions: “run tests + fix on results + next run” = one logical step.

Modern tools support this:

  • Claude Code — you can approve Plan + execute steps 1-3 in one action, then review
  • Codex CLI — auto-edit mode within scope
  • Cursor — Composer apply for a whole series of edits

Principle: approve logical units, not atomic tool calls.

Rule 3: approval default = “no,” but “no” requires no effort

A subtle point. Psychologically:

  • If the default button is [Y], the human presses Y without reading.
  • If the default button is [N], the human pauses and thinks.

But if every “no” requires a justification essay — the human will start “pressing Y to avoid suffering.” Balance: default — N, but Y is one click after reading the plan.

In most modern agent tools the default is N. If yours isn’t — change the setting or change the tool.

HITL anti-patterns

1. “Approval theatre”

The team formally requires approval at every step, but nobody actually reads the plan. Everyone presses Y by habit. Result: the process exists, the protection doesn’t.

How to catch it: periodically insert a deliberately bad step into the plan (e.g. deletion of a random file) and see who notices. If nobody does — you have approval theatre.

2. “Bypass culture”

Everyone knows the “magic flag” that turns off all gates (--dangerously-skip-permissions, --auto). One person started using it “because I’m in a hurry” — in two weeks the whole team works that way.

How to catch it: in team policy any autonomy flag above the base rung must be logged and visible across the team. Bypass is an incident, not a normal mode.

3. “Single point of approval”

One lead approves everything. They went on holiday — the whole process froze. Or worse — they started pressing Y in a row because of the queue.

How to catch it: approval must be role-based, not personal. Any of a set of people can approve — but always a human, never an automation.

4. “Trust escalation”

“This agent crushed the previous task — let’s give it this one too, a bigger one.” Step by step the agent gets more scope until it finally drops something important.

How to catch it: the autonomy level is a property of the task class, not of the agent’s “past wins.” A prod hotfix needs Level 2 even if the agent did 1000 good ones in a row.

Metrics worth collecting

If you’re seriously building an agentic flow in a team — collect:

MetricWhat it shows
Plans/weekVolume of agent work
Plan-to-action ratioHow many plans actually got to action (low = poor task framing)
Replans per taskAverage plan revisions (>3 = task too big)
Approval latency (median)Average wait for approve (>30 min = HITL drags)
Reverts post-mergeHow many agent PRs were rolled back (the main “harm” metric)
Bypass eventsHow often the autonomy bypass was used (must be low and visible)

You don’t have to start with all of them. At minimum — reverts post-merge and bypass events. They send the “things got worse” signal fastest.

Programmatic finale (register C)

Friends.

Part I — Foundations — is done. If you sum up everything in M1–M3 in one sentence:

An agent is a contributor working under a contract with explicit boundaries, under observation, with inspectable artifacts and a negotiated degree of autonomy.

All subsequent parts of the course — Tools & Environment, Production Patterns, Security, GH-600 prep — will unpack the technical implementation of that formula. Specific tools will change, specific APIs will go stale, specific rate-limit numbers will be rewritten.

What won’t change:

  • The contractual nature of the task
  • The plan/action split
  • Artifacts that survive the session
  • The human who keeps responsibility

The model zoo will keep growing. Tools will shuffle every quarter. Train your eye, not your fingers. Your head matters more than the AI — especially at the moment of plan approval.

Recap

  • Autonomy ladder 0–5. Default to the low rung; climb only with a concrete reason.
  • 5 artifacts after the agent: plan, action log, diff, evaluation report, bumps&retries. All in the repo, not in chat.
  • HITL doesn’t drag if 3 rules hold: approval fast, batched, default = N.
  • 4 HITL anti-patterns: approval theatre, bypass culture, single point of approval, trust escalation.
  • Minimum metrics: reverts post-merge + bypass events.

📚 Further reading

The topic is large, so keep three anchors: a telemetry spec, a ready observability platform, and Anthropic’s official position on HITL/auto-mode.

Telemetry standard for agents

  • OpenTelemetry — Semantic conventions for GenAI — opentelemetry.io/docs/specs/semconv/gen-ai/. Which spans, metrics, and attributes to put in traces for an LLM/agent system. If you’re writing a production agent — a standard worth aiming for instead of inventing your own.

Ready observability stack

  • Langfuse — LLM Observability docs — langfuse.com/docs/observability/overview. Open platform: traces, sessions, cost metrics, under OpenTelemetry. Convenient as a starting point, especially when you don’t want to build your own analytics.

HITL at Anthropic

  • Anthropic — “Claude Code auto mode: a safer way to skip permissions” — anthropic.com/engineering/claude-code-auto-mode. How Anthropic finely splits “no HITL” from “HITL by rule,” and why the second strategy is a reasonable default.

Tip: if you have a prod agent and none of these components — start with Langfuse and the base OTel convention. It’s 1–2 evenings and the blindness goes away immediately.
