M6 · Claude Code CLI mastery

Installation, configuration, hooks, skills, custom commands, subagents and .claude/settings.json patterns.

What this chapter is about

Hey folks, welcome 👋

Heads-up — this is the first chapter that steps outside GH-600. Microsoft’s certification says little about specific tools: general principles, abstract patterns. That’s fine for a certification, because it has to age well. But if you came here to actually work with an agent — you can’t avoid a deep tour of the specific tools.

So in M6, M7, M8 I give you the full menu on the three main tools of May 2026:

  • M6 — Claude Code (CLI and IDE plugin from Anthropic)
  • M7 — GPT-5.3 Codex CLI (CLI from OpenAI)
  • M8 — Cursor Composer + other IDE agents (Cline, Aider, Windsurf)

We start with Claude Code. Not because I love it more (spoiler — yes I do, I spend most of my time in it), but because architecturally it’s the most “reference” version: how hooks, skills, settings are done has become the de-facto standard everyone else aligns to.

Installation and first run

Minimum install (May 2026):

# npm channel, the recommended one
npm install -g @anthropic-ai/claude-code

# first run — authorisation
claude

Install alternatives per platform:

  • macOS native — brew install claude-code or the .dmg from anthropic.com
  • Windows native — .exe installer or MS Store. I’m on Windows myself — runs natively, no WSL required.
  • Linux — npm or .deb/.rpm for the major distros

On the first run claude will ask you to log in via browser (Pro/Max subscription) or paste an API key (pay-per-use). I’m personally on the Max plan — it pays for itself on day one if you write daily.

Base layout

After install and login you get:

~/.claude.json                     # global config (mcp servers, settings)
~/.claude/                         # user folder
├── commands/                      # global slash-commands
├── agents/                        # global subagents
├── skills/                        # global skills
├── projects/                      # session history per project
└── settings.json                  # global permissions + hooks

Per project there can be a local config:

project-root/
├── .claude/
│   ├── settings.json              # project-level overrides
│   ├── settings.local.json        # gitignored, personal
│   ├── commands/                  # project slash-commands
│   ├── agents/                    # project subagents
│   └── skills/                    # project skills
├── CLAUDE.md                      # project "memo" for the agent
└── .mcp.json                      # project MCP servers

Rule: global is for you, project is for the team (under git, via PR). settings.local.json is your personal toggle, never enters the repo.

CLAUDE.md — the main file

The most important artifact. CLAUDE.md in the project root is the persistent context Claude Code reads on every session. This is what separates “agent with memory” from “single-use agent.”

What to put in CLAUDE.md:

  • Project type (Node/Python/Rust/etc., version)
  • Commands: how to run tests, lint, build
  • Architectural conventions of the project
  • What NOT to touch (legacy code, generated files, other people’s zones)
  • Commit style, branch convention
  • Any “tribal knowledge” the team learned the hard way

A minimal CLAUDE.md for a Node project:

# Project context for Claude Code

This is a TypeScript SaaS backend on Node 22 + Fastify + Prisma + Postgres.

## Commands
- Tests: `npm test` (vitest, ~30s)
- Lint: `npm run lint` (eslint + prettier)
- Typecheck: `npm run typecheck`
- Dev: `npm run dev` (auto-reload on 3000)
- Build: `npm run build` (output to dist/)

## Conventions
- Path aliases: use `@/` (mapped to src/)
- Imports: named only, no default
- Async: top-level await is fine
- Errors: throw typed errors from `src/errors/`, never strings

## Do NOT touch
- `src/legacy/` — being deprecated, frozen
- `prisma/migrations/` — never edit existing, only add new
- `.env*` — secrets, ask the human

## Commit style
Conventional commits with type prefix: `feat:`, `fix:`, `refactor:`, `chore:`

This is the first artifact you create in a new project before serious work with the agent. Without it, every session the agent will re-open the repo and re-derive your conventions from scratch.

Pro tip: CLAUDE.md can also be generated by Claude Code itself. The /init command on a fresh project — it reads the structure and proposes a first version. You fix it by hand from there.

.claude/settings.json — permissions and hooks

File shape:

{
  "permissions": {
    "allow": [...],
    "deny": [...]
  },
  "hooks": {
    "PreToolUse": [...],
    "PostToolUse": [...],
    "Stop": [...]
  },
  "env": {...},
  "model": "claude-opus-4-7"
}

Permissions (M4 revisited, specifically here)

We covered this in M4. Key patterns:

"allow": [
  "Bash(npm run test:*)",
  "Bash(npm run lint)",
  "Bash(git status)",
  "Bash(git diff:*)",
  "Bash(git log:*)",
  "Read",
  "Edit",
  "Glob",
  "Grep"
],
"deny": [
  "Bash(rm:*)",
  "Bash(git push:*)",
  "Bash(git reset --hard:*)",
  "Read(./.env*)",
  "Read(~/.ssh/**)",
  "Read(~/.aws/**)"
]

Hooks — Claude Code’s most powerful feature

A hook is a command that runs deterministically in response to an agent event. It’s not “ask the agent to do something” — it’s a guaranteed script the harness will execute.

Events:

  • PreToolUse — before each tool call
  • PostToolUse — after each tool call
  • UserPromptSubmit — when the user submitted a prompt
  • Notification — on notifications (e.g. waiting for input)
  • Stop — when the session ends

Example: auto prettier after every edit:

"hooks": {
  "PostToolUse": [
    {
      "matcher": "Edit|Write",
      "hooks": [
        {
          "type": "command",
          "command": "npx prettier --write $CLAUDE_TOOL_INPUT_file_path"
        }
      ]
    }
  ]
}

Example: a ban on editing files inside legacy/:

"hooks": {
  "PreToolUse": [
    {
      "matcher": "Edit|Write",
      "hooks": [
        {
          "type": "command",
          "command": "node .claude/scripts/check-legacy.js"
        }
      ]
    }
  ]
}

And in check-legacy.js:

#!/usr/bin/env node
const path = process.env.CLAUDE_TOOL_INPUT_file_path;
if (path?.includes('/legacy/')) {
  console.error('Editing files in legacy/ is forbidden by hook.');
  process.exit(2);  // exit 2 = block the tool use
}

This is a physical restriction, not “we asked the model not to touch this.” Pass exit 2 — the tool call is blocked, the agent sees the error and revises.

Principle: any rule that is critical to follow belongs in a hook, not in CLAUDE.md. Instructions the model can “forget”; a hook can’t.

Slash-commands — repeatable tasks

A slash-command is a named task you can run from any session. Stored in .claude/commands/<name>.md.

Simplest example, .claude/commands/review.md:

---
description: Review uncommitted changes against project conventions
---

Read the git diff of current uncommitted changes and do a code review against the checklist:
- Conformance to CLAUDE.md conventions
- Naming
- Error handling
- Test coverage for new functions
- Security (no hardcoded secrets)

End with a verdict: "ok / needs work / blocking issues".

Use it in chat: just /review — and Claude knows what to do.

Commands can take arguments:

---
description: Generate test for a specific function
---

Generate a test file for function $1 in file $2.
Use vitest. Cover happy path + edge cases + error path.

Use: /test parseDate src/utils/date.ts

Commands come in global (~/.claude/commands/) and project (.claude/commands/) flavours. Project ones go into git, shared by the team. This is the templating layer that genuinely speeds up daily work.

Subagents — isolated executors

A subagent is a separate agent with its own system prompt the main agent can delegate to. Stored in .claude/agents/<name>.md.

Why they matter:

  • Isolated context — the subagent doesn’t see your main session, doesn’t pollute it.
  • Specialised prompt — e.g. a reviewer-agent with a different system prompt than a writer-agent.
  • Parallel work — the main agent can spawn 3 subagents in parallel (e.g. on different parts of the codebase).

Example .claude/agents/code-reviewer.md:

---
name: code-reviewer
description: Strict code reviewer. Use when reviewing diffs against project conventions.
tools: Read, Grep, Glob, Bash(git diff:*), Bash(git log:*)
---

You are a strict but constructive code reviewer.
Your job: read the diff, compare it against CLAUDE.md conventions, and produce a structured review.

Focus on:
- Convention violations
- Logical errors
- Missing tests
- Security issues

Do NOT modify any files.
Output as markdown with sections: BLOCKING, MAJOR, MINOR, NITS.

In session, the main agent can delegate via the Task tool (available by default):

Use Task tool, subagent_type=code-reviewer, prompt="Review the diff in src/api/auth.ts"

You get back only the reviewer’s final answer, none of the intermediate scuffle. The main agent’s context stays clean.

This is one of the most underrated features of Claude Code. If you have a long session with a big context — subagents for side tasks are your main defence against context dilution.

Skills — packaged capabilities

Skills are folders with instructions + optional scripts that the agent can “activate” by trigger words.

Skill structure:

~/.claude/skills/skill-name/
├── SKILL.md                # description + frontmatter with triggers
├── scripts/                # helper scripts
└── reference/              # files the skill loads on demand

Example SKILL.md:

---
name: webp-conversion
description: Convert PNG/JPG to WebP with quality optimisation. Triggers when user asks to compress/optimise images.
---

When user wants to convert images to WebP:

1. Use `cwebp` (from libwebp-tools) with quality 85 by default
2. For batch: parallelise with xargs
3. Output to same directory unless told otherwise

Script: `bash scripts/webp_batch.sh <input_dir> [quality]`

Skills differ from commands in that:

  • They activate automatically by context (the agent decides on use)
  • They can include scripts, extra files, reference documents
  • They suit reusable capabilities not tied to one project

Heads up (a safety rule): installing a third-party skill = reading SKILL.md in full before launch. SKILL.md may contain hidden directives and transitive prompt injection. I have an iron personal rule on this: npx skills add always without -y. Manual confirmation of every install — mandatory.

Useful dev practices

1. One project — one .claude/

Don’t try to make a “universal” .claude/settings.json for all projects via user-level. Every project has its own context, permissions, hooks.

2. .claude/settings.local.json for personal experiments

If you want to try a new hook without committing to the team — put it in settings.local.json (gitignored). Liked it — move to settings.json via PR.

3. CLAUDE.md grows over time

Don’t write it in full at once. Run /init, iterate on the result. Every time the agent did something wrong because of missing context — add that context to CLAUDE.md. In 2–3 weeks you’ll have the perfect file.

4. Hooks for security-critical rules

Any rule like “don’t touch .env,” “don’t push to main,” “don’t run rm in these folders” — must be a hook, never “we asked the model.”

5. Subagents for side tasks

If the main session is going through a big task and you suddenly need a “different context” (say, look at a neighbouring repo) — subagent, not the main session. Keep context clean.

Practice

20–30 minutes. Step by step:

  1. Open your current working project. If no CLAUDE.md — run /init, polish the result.
  2. Create .claude/settings.json with a minimal allow/deny (see M4) + at least one hook (e.g. prettier on edit).
  3. Make one slash-command — something you regularly ask the agent for. Save as .claude/commands/<name>.md.
  4. Try a subagent. Create .claude/agents/reviewer.md with a minimal system prompt. Delegate a task via the Task tool.
  5. Commit .claude/ to the project git (but add .claude/settings.local.json to .gitignore).

Recap

  • CLAUDE.md = persistent project context. The main artifact, grows over time.
  • .claude/settings.json = permissions + hooks at the project level. settings.local.json is personal, not in git.
  • Hooks — deterministic scripts on agent events. Security-critical rules must be hooks.
  • Slash-commands — repeatable tasks. .claude/commands/<name>.md.
  • Subagents — isolated executors with their own prompt. Keep main-session context clean.
  • Skills — packaged capabilities with auto-activation. Third-party skills — mandatory manual audit of SKILL.md.
