M6 · Claude Code CLI mastery

Установка, конфигурация, hooks, skills, custom commands, subagents и .claude/settings.json паттерны.

Черновик M6 в голосе DmitriyM (см. scripts/tg_voice_analysis/_summary.md). Авторский блок — выходит за рамки GH-600.

О чём эта глава

Друзья, всем привет! 👋

Сразу скажу — это первая глава, которая выходит за рамки GH-600. Майкрософт-сертификация про конкретные тулы говорит мало: общие принципы, абстрактные паттерны. Это нормально для сертификации, потому что она долгоживущая. Но если вы пришли учиться реально работать с агентом — без глубокого знакомства с конкретными тулами это не работает.

Поэтому в M6, M7, M8 я даю «полный фарш» по трём основным тулзам мая 2026:

  • M6 — Claude Code (CLI и IDE-плагин от Anthropic)
  • M7 — GPT-5.3 Codex CLI (CLI от OpenAI)
  • M8 — Cursor Composer + другие IDE-агенты (Cline, Aider, Windsurf)

Начнём с Claude Code. Не потому что я его больше люблю (хотя, спойлер, да — я в нём провожу больше всего времени), а потому что архитектурно он наиболее «эталонный»: то как сделаны hooks, skills, settings — стало де-факто стандартом, на который ориентируются остальные.

Установка и первый запуск

Минимальная установка (май 2026):

# npm-канал, рекомендованный
npm install -g @anthropic-ai/claude-code

# первый запуск — авторизация
claude

Альтернативы установки на разных платформах:

  • macOS native — brew install claude-code или .dmg с anthropic.com
  • Windows native — .exe инсталлер или MS Store. На Windows я сам сижу — работает нативно, не требует WSL.
  • Linux — npm или .deb/.rpm для основных дистрибутивов

После первого запуска claude попросит залогиниться через браузер (Pro/Max-подписка) или ввести API ключ (pay-per-use). Лично я сижу на Max-подписке — окупается в первый день, если вы пишете каждый день.

Базовая структура установки

После установки и логина у вас появляются:

~/.claude.json                     # глобальный конфиг (mcp servers, settings)
~/.claude/                         # папка пользователя
├── commands/                      # global slash-commands
├── agents/                        # global subagents
├── skills/                        # global skills
├── projects/                      # session history по проектам
└── settings.json                  # global permissions + hooks

В каждом проекте может быть локальный конфиг:

project-root/
├── .claude/
│   ├── settings.json              # project-level overrides
│   ├── settings.local.json        # gitignored, personal
│   ├── commands/                  # project slash-commands
│   ├── agents/                    # project subagents
│   └── skills/                    # project skills
├── CLAUDE.md                      # «памятка» проекта для агента
└── .mcp.json                      # project MCP servers

Правило: глобальное — для тебя, проектное — для команды (под git, через PR). settings.local.json — твоя личная отсебятина, не уходит в репо.

CLAUDE.md — главный файл

Самый важный артефакт. CLAUDE.md в корне проекта — это постоянный контекст, который Claude Code читает в каждой сессии. Это и есть то, что отличает «агент с памятью» от «агент-однодневка».

Что писать в CLAUDE.md:

  • Тип проекта (Node/Python/Rust/etc., версия)
  • Команды: как запустить тесты, линтер, сборку
  • Архитектурные конвенции проекта
  • Что НЕ трогать (legacy-код, генерируемые файлы, чужие зоны)
  • Стиль коммитов, конвенция веток
  • Любые “tribal knowledge”, которое команда узнала на горьком опыте

Пример минимального CLAUDE.md для Node-проекта:

# Project context for Claude Code

This is a TypeScript SaaS backend on Node 22 + Fastify + Prisma + Postgres.

## Commands
- Tests: `npm test` (vitest, ~30s)
- Lint: `npm run lint` (eslint + prettier)
- Typecheck: `npm run typecheck`
- Dev: `npm run dev` (auto-reload on 3000)
- Build: `npm run build` (output to dist/)

## Conventions
- Path aliases: use `@/` (mapped to src/)
- Imports: named only, no default
- Async: top-level await is fine
- Errors: throw typed errors from `src/errors/`, never strings

## Do NOT touch
- `src/legacy/` — being deprecated, frozen
- `prisma/migrations/` — never edit existing, only add new
- `.env*` — secrets, ask the human

## Commit style
Conventional commits with type prefix: `feat:`, `fix:`, `refactor:`, `chore:`

Это — первый артефакт, который ты создаёшь в новом проекте перед серьёзной работой с агентом. Без него агент будет каждую сессию заново открывать репо и переоткрывать ваши конвенции.

Лайфхак: CLAUDE.md тоже можно сгенерировать самим Claude Code. Команда /init в свежем проекте — он прочитает структуру и предложит первую версию. Дальше вы её правите вручную.

.claude/settings.json — permissions и hooks

Структура файла:

{
  "permissions": {
    "allow": [...],
    "deny": [...]
  },
  "hooks": {
    "PreToolUse": [...],
    "PostToolUse": [...],
    "Stop": [...]
  },
  "env": {...},
  "model": "claude-opus-4-7"
}

Permissions (повтор M4 в специфике)

Уже разбирали в M4. Ключевые паттерны:

"allow": [
  "Bash(npm run test:*)",
  "Bash(npm run lint)",
  "Bash(git status)",
  "Bash(git diff:*)",
  "Bash(git log:*)",
  "Read",
  "Edit",
  "Glob",
  "Grep"
],
"deny": [
  "Bash(rm:*)",
  "Bash(git push:*)",
  "Bash(git reset --hard:*)",
  "Read(./.env*)",
  "Read(~/.ssh/**)",
  "Read(~/.aws/**)"
]

Hooks — самая мощная штука Claude Code

Hook — это команда, которая запускается детерминированно в ответ на агентское событие. Это не «попросить агента что-то сделать», это гарантированный скрипт, который выполнит харнес.

События:

  • PreToolUse — перед каждым вызовом тула
  • PostToolUse — после каждого вызова тула
  • UserPromptSubmit — когда пользователь отправил промпт
  • Notification — на уведомления (ждёт ввод, например)
  • Stop — когда сессия завершается

Пример: автоматический prettier после каждого edit:

"hooks": {
  "PostToolUse": [
    {
      "matcher": "Edit|Write",
      "hooks": [
        {
          "type": "command",
          "command": "npx prettier --write $CLAUDE_TOOL_INPUT_file_path"
        }
      ]
    }
  ]
}

Пример: запрет на правку файлов в legacy/:

"hooks": {
  "PreToolUse": [
    {
      "matcher": "Edit|Write",
      "hooks": [
        {
          "type": "command",
          "command": "node .claude/scripts/check-legacy.js"
        }
      ]
    }
  ]
}

И в check-legacy.js:

#!/usr/bin/env node
const path = process.env.CLAUDE_TOOL_INPUT_file_path;
if (path?.includes('/legacy/')) {
  console.error('Editing files in legacy/ is forbidden by hook.');
  process.exit(2);  // exit 2 = block the tool use
}

Это физическое ограничение, не «попросили модель не трогать». Прошёл exit 2 — тул-вызов заблокирован, агент видит ошибку и пересматривает.

Принцип: любое правило, которое критично соблюсти — оно должно быть hook-ом, не инструкцией в CLAUDE.md. Инструкции модель может «забыть»; hook — нет.

Slash-commands — повторяемые задачи

Slash-command — это именованная задача, которую вы можете запустить из любой сессии. Хранится в .claude/commands/<name>.md.

Простейший пример .claude/commands/review.md:

---
description: Review uncommitted changes against project conventions
---

Прочитай git diff текущих uncommitted изменений и проведи code review по чек-листу:
- Соответствие conventions из CLAUDE.md
- Naming
- Error handling
- Test coverage для новых функций
- Безопасность (нет ли хардкоженных секретов)

В конце — итог "ok / needs work / blocking issues".

Использование в чате: просто /review — и Claude знает, что делать.

Командам можно передавать аргументы:

---
description: Generate test for a specific function
---

Сгенерируй тест-файл для функции $1 из файла $2.
Используй vitest. Покрой нормальные кейсы + edge cases + error path.

Использование: /test parseDate src/utils/date.ts

Команды бывают глобальные (~/.claude/commands/) и проектные (.claude/commands/). Проектные ходят в git, разделяются командой. Это уровень шаблонизации, который сильно ускоряет повседневную работу.

Subagents — изолированные исполнители

Subagent — это отдельный агент с собственным system prompt, к которому основной агент может делегировать задачу. Хранится в .claude/agents/<name>.md.

Зачем нужны:

  • Изолированный контекст — subagent не видит вашей основной сессии, не путается в неё.
  • Специализированный prompt — например, агент-ревьювер с другим системным промптом, чем агент-писатель.
  • Параллельная работа — основной агент может запустить 3 subagent-а параллельно (например, на разные части кодовой базы).

Пример .claude/agents/code-reviewer.md:

---
name: code-reviewer
description: Strict code reviewer. Use when reviewing diffs against project conventions.
tools: Read, Grep, Glob, Bash(git diff:*), Bash(git log:*)
---

You are a strict but constructive code reviewer.
Your job: read the diff, compare it against CLAUDE.md conventions, and produce a structured review.

Focus on:
- Convention violations
- Logical errors
- Missing tests
- Security issues

Do NOT modify any files.
Output as markdown with sections: BLOCKING, MAJOR, MINOR, NITS.

В сессии основной агент может делегировать через тул Task (доступен по умолчанию):

Use Task tool, subagent_type=code-reviewer, prompt="Review the diff in src/api/auth.ts"

И вернётся только итоговый ответ ревьювера, без всей промежуточной возни. Контекст основного агента сохраняется чистым.

Это один из самых недооценённых features Claude Code. Если у вас длинная сессия с большим контекстом — subagents для побочных задач это ваша основная защита от размытия контекста.

Skills — упакованные капабилити

Skills — это папки с инструкциями + опциональными скриптами, которые агент может «активировать» по ключевым словам.

Структура skill:

~/.claude/skills/skill-name/
├── SKILL.md                # описание + frontmatter с триггерами
├── scripts/                # вспомогательные скрипты
└── reference/              # файлы, которые skill подгружает по требованию

Пример SKILL.md:

---
name: webp-conversion
description: Convert PNG/JPG to WebP with quality optimization. Triggers when user asks to compress/optimize images.
---

When user wants to convert images to WebP:

1. Use `cwebp` (from libwebp-tools) with quality 85 by default
2. For batch: parallelize with xargs
3. Output to same directory unless told otherwise

Script: `bash scripts/webp_batch.sh <input_dir> [quality]`

Skills отличаются от commands тем, что:

  • Activate automatically по контексту (агент сам решает использовать)
  • Могут включать скрипты, доп. файлы, reference-документы
  • Подходят для переиспользуемых капабилитей, не привязанных к одному проекту

Внимание (правило безопасности): установка стороннего skill = чтение SKILL.md целиком до запуска. В SKILL.md могут быть скрытые директивы и транзитивные prompt injection. У меня по этому поводу железное правило в личной памяти: npx skills add всегда без -y. Ручное подтверждение каждой установки — обязательно.

Полезные dev-практики

1. Один проект — один .claude/

Не пытайтесь сделать «универсальный» .claude/settings.json для всех проектов через user-level. Каждый проект — свой контекст, свои permissions, свои hooks.

2. .claude/settings.local.json для своих экспериментов

Если хотите попробовать новый hook без коммита в команду — кладите его в settings.local.json (gitignored). Понравилось — переносите в settings.json через PR.

3. CLAUDE.md растёт постепенно

Не пишите его сразу полностью. Запускайте /init, дорабатывайте по факту. Каждый раз когда агент сделал что-то неправильно из-за нехватки контекста — добавьте этот контекст в CLAUDE.md. Через 2-3 недели у вас будет идеальный файл.

4. Hooks для security-критичных правил

Любое правило вроде «не трогать .env», «не push на main», «не делать rm в этих папках» — обязательно hook, никогда не «попросили модель».

5. Subagents для побочных задач

Если основная сессия идёт по большой задаче, а вам внезапно нужен «другой контекст» (например, посмотреть что в соседнем репо) — subagent, не главная сессия. Сохраняйте чистоту контекста.

Practice

20-30 минут. По шагам:

  1. Откройте свой текущий рабочий проект. Если нет CLAUDE.md — запустите /init, доработайте результат.
  2. Создайте .claude/settings.json с минимальным allow/deny (см. M4) + хотя бы одним hook (например, prettier на edit).
  3. Сделайте один slash-command — что-то, что вы регулярно просите у агента. Сохраните как .claude/commands/<name>.md.
  4. Попробуйте subagent. Создайте .claude/agents/reviewer.md с минимальным system prompt. Делегируйте ему task через Task tool.
  5. Закоммитьте .claude/ в проектный git (но добавьте .claude/settings.local.json в .gitignore).

Recap

  • CLAUDE.md = постоянный контекст проекта. Главный артефакт, растёт со временем.
  • .claude/settings.json = permissions + hooks на уровне проекта. settings.local.json — личное, не в git.
  • Hooks — детерминированные скрипты на агентские события. Безопасность-критичные правила должны быть hooks.
  • Slash-commands — повторяемые задачи. .claude/commands/<name>.md.
  • Subagents — изолированные исполнители с собственным prompt. Сохраняют контекст основной сессии.
  • Skills — упакованные капабилити с авто-активацией. Сторонние skills — обязательно ручной аудит SKILL.md.
