M12 · Регрессионные тесты для агентов

Golden examples, snapshot harness, CI-интеграция. Ловить регрессии в момент изменения, а не через две недели.

Черновик M12 в голосе DmitriyM.

Обновил модель           →        git commit + push
Через 2 недели заметил            CI: 12/20 → 9/20
что-то стало хуже                 Конкретные кейсы упали
                                  Откат до решения
"воспроизвести непросто"          "вот reproducible diff"

Друзья, всем привет! 👋

Эта глава — про harness, который ловит регрессии в момент изменения, а не через две недели. Это то, что в обычной разработке называется «регрессионные тесты», только переложенное на агентов.

Час, может полтора. Это переход с уровня 2 (M11) на уровень 3 — автоматизированный eval, который сам говорит «хуже стало».

Концепция: golden example как unit-тест для агента

В обычной разработке regression test — это:

def test_add_returns_sum():
    assert add(2, 3) == 5

Жёстко: вход 2,3 → ожидаемый выход 5. Если меняется код add и тест падает — регрессия, ловим сразу.

С агентом так нельзя. Агент не детерминирован — на одном промпте даст разные ответы. Но мы можем построить похожую структуру:

┌─────────────────────────────────────────────────────┐
│  Golden example = input + criteria, не input+output │
│  Criteria — это набор автоматических проверок:      │
│   - regex match                                     │
│   - юнит-тесты после применения diff                │
│   - синтаксические проверки (AST)                   │
│   - judge-LLM (GPT смотрит и оценивает)             │
└─────────────────────────────────────────────────────┘

Не «output должен быть точно такой», а «output должен удовлетворять этим условиям».

Структура harness

project/
├── evals/
│   ├── cases/
│   │   ├── case-01-refactor-controller/
│   │   │   ├── setup.sh           # подготавливает чистое состояние
│   │   │   ├── prompt.md          # вход для агента
│   │   │   ├── verify.sh          # проверка результата
│   │   │   └── expected.md        # что должно быть (для глаз)
│   │   ├── case-02-bug-fix/
│   │   │   └── ...
│   ├── run.sh                     # прогон всех кейсов
│   ├── runs/
│   │   ├── 2026-05-15/
│   │   │   ├── case-01.log
│   │   │   ├── case-01.pass
│   │   │   └── summary.csv
│   │   └── 2026-05-01/
│   └── trends.md                  # ручной анализ

Каждый кейс — изолированная единица: setup → prompt → verify. Прогон каждого — независимый, последовательность не имеет значения.

Пример case: refactor controller

evals/cases/case-01-refactor-controller/setup.sh:

#!/usr/bin/env bash
set -e
# Сбрасываем рабочую копию к точке "до" задачи
git checkout main -- src/api/users.ts
git checkout main -- tests/users.test.ts
# Удаляем потенциально оставшиеся файлы от предыдущих прогонов
rm -f src/validators/userValidator.ts

evals/cases/case-01-refactor-controller/prompt.md:

В файле `src/api/users.ts` метод `createUser` смешивает три ответственности:
валидацию входных данных, бизнес-логику и доступ к БД.

Задача: вынеси валидацию в отдельный модуль `src/validators/userValidator.ts`.

Требования:
- В `createUser` должен остаться импорт из нового модуля
- Существующие тесты `tests/users.test.ts` должны проходить без изменений
- Никаких новых зависимостей

evals/cases/case-01-refactor-controller/verify.sh:

#!/usr/bin/env bash
set -e

# 1. Файл существует
[ -f src/validators/userValidator.ts ] || { echo "FAIL: validator file missing"; exit 1; }

# 2. Импорт в createUser
grep -q "from.*validators/userValidator" src/api/users.ts || { echo "FAIL: no import"; exit 1; }

# 3. Тесты проходят
npm test -- tests/users.test.ts || { echo "FAIL: tests broken"; exit 1; }

# 4. Нет новых зависимостей
git diff --quiet package.json package-lock.json || { echo "FAIL: deps changed"; exit 1; }

echo "PASS"

Это полный автоматизированный кейс. Запускается за минуту, даёт бинарный ответ.

Раннер: run.sh

#!/usr/bin/env bash
set -u

RUN_ID="$(date -u +%Y-%m-%dT%H-%M-%S)"
OUT_DIR="evals/runs/$RUN_ID"
mkdir -p "$OUT_DIR"

SUMMARY="$OUT_DIR/summary.csv"
echo "case,pass,duration_s,notes" > "$SUMMARY"

for case_dir in evals/cases/*/; do
    name="$(basename "$case_dir")"
    log="$OUT_DIR/$name.log"
    start=$(date +%s)

    # Чистое состояние
    bash "$case_dir/setup.sh" > "$log" 2>&1

    # Запуск агента
    cat "$case_dir/prompt.md" | claude --print --output-format=json >> "$log" 2>&1

    # Verify
    if bash "$case_dir/verify.sh" >> "$log" 2>&1; then
        pass="1"
    else
        pass="0"
    fi

    dur=$(( $(date +%s) - start ))
    echo "$name,$pass,$dur," >> "$SUMMARY"
done

# Финальная статистика
total=$(( $(wc -l < "$SUMMARY") - 1 ))
passed=$(grep -c ",1," "$SUMMARY" || true)
echo "Run $RUN_ID: $passed/$total"

Важно: claude --print --output-format=json — это non-interactive режим. Агент получает промпт, выполняет (с CLAUDE.md из проекта), завершается. Это то, что вам нужно для CI.

CI-интеграция (GitHub Actions)

.github/workflows/agent-evals.yml:

name: agent-evals

on:
  schedule:
    - cron: "0 4 * * MON"      # каждый понедельник, чтобы ловить
                               # дрейф модели + изменения в проекте
  workflow_dispatch:           # вручную, когда надо
  pull_request:
    paths:
      - "CLAUDE.md"
      - ".claude/**"
      - "evals/cases/**"       # запуск при смене конфигов агента

jobs:
  eval:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install Claude Code
        run: npm install -g @anthropic-ai/claude-code

      - name: Run evals
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: bash evals/run.sh

      - name: Upload artifacts
        uses: actions/upload-artifact@v4
        with:
          name: eval-run
          path: evals/runs/

      - name: Fail if pass rate drops
        run: |
          # Сравнение с baseline в evals/baseline.txt
          baseline=$(cat evals/baseline.txt)
          latest=$(ls -t evals/runs/ | head -n 1)
          passed=$(grep -c ",1," "evals/runs/$latest/summary.csv" || echo 0)
          total=$(( $(wc -l < "evals/runs/$latest/summary.csv") - 1 ))
          rate=$(( passed * 100 / total ))
          echo "Pass rate: $rate% (baseline: $baseline%)"
          if [ "$rate" -lt "$baseline" ]; then
            echo "::error::Regression detected"
            exit 1
          fi

Это продакшен harness. Запускается по расписанию + при изменении конфигов агента + вручную. Если pass rate ниже baseline — CI падает, видно сразу.

Типы проверок (verify.sh)

1. Файл/строка

[ -f path/to/file ] || exit 1
grep -q "pattern" file || exit 1

Простые exists/contains. Хороши для «должен появиться модуль» / «должен быть импорт».

2. AST/синтаксис

# TypeScript — проверка, что код парсится
tsc --noEmit src/api/users.ts || exit 1

# Python
python -m py_compile src/auth.py || exit 1

Защита от «агент написал не валидный код».

3. Тесты

npm test -- tests/users.test.ts || exit 1
pytest tests/test_auth.py || exit 1

Самая ценная проверка — существующие тесты проходят.

4. Diff scope

# Изменился ли только ожидаемый набор файлов?
expected_files=$(cat case_dir/allowed_files.txt)
actual_files=$(git diff --name-only)
diff <(echo "$expected_files" | sort) <(echo "$actual_files" | sort) || exit 1

Защита от «лазит не туда» — критично для security-сценариев.

5. LLM-as-judge

Если программно проверить нельзя (стиль кода, понятность сообщения):

# verify.sh пишет prompt для judge-модели, парсит её ответ
judge_response=$(cat <<EOF | claude --print
Ты — судья. Перед тобой результат рефакторинга. Оцени по шкале 1-5:
- читаемость нового модуля
- степень изоляции ответственностей

Файл результата:
$(cat src/validators/userValidator.ts)

Ответ строго в формате: SCORE=N
EOF
)
score=$(echo "$judge_response" | grep -oP "SCORE=\K\d+")
[ "$score" -ge 4 ] || exit 1

Осторожно: judge-LLM имеет свою дисперсию. Для статистики — несколько прогонов и усреднение.

Антипаттерны harness

1. Жёсткие snapshot’ы

«Agent должен сгенерировать точно этот код». Через два дня модель сделает то же самое другими словами — упадёт, хотя семантически правильно.

Лекарство: проверяйте семантику (тесты, AST, regex по структуре), не точный текст.

2. Слишком короткие/слишком длинные кейсы

  • Короткий (10 секунд) — нет сигнала о сложных задачах
  • Длинный (30+ минут) — нельзя запускать часто, дорого

Лекарство: 1-5 минут на кейс. 20 кейсов = 20-100 минут на прогон. Один раз в неделю — приемлемо.

3. Flaky-кейсы оставлять как есть

Кейс падает раз через два — «временно игнорируем». Через месяц у вас 5 flaky кейсов, suite не верифицируем.

Лекарство: flaky-кейс — сразу в work item «починить или удалить». Лучше меньше стабильных, чем много шумных.

4. Suite от продакшена изолирован

Reference suite живёт в отдельном репо. Никто его не апдейтит, когда продакшен меняется.

Лекарство: suite в том же репо, что код, в той же ветке. Часть PR-чек-листа: «если изменил CLAUDE.md, проверь, что suite зелёный».

5. Замер только pass/fail

Бинарный — слишком грубо. Кейс «прошёл», но занял 15 минут вместо обычных 3 — это деградация, но не зафиксирована.

Лекарство: записывайте duration, token cost, number of tool calls. Регрессия по эффективности — тоже регрессия.

Тонкий момент: «дрейф модели»

У Anthropic, OpenAI и других провайдеров снапшоты модели меняются. Иногда явно (новый ID), иногда в фоне на той же API-точке. Это означает:

  • Ваш harness может внезапно показать падение pass rate, хотя вы ничего не меняли
  • Это не баг, это дрейф — модель чуть другая

Что делать:

  1. Зафиксируйте конкретный model ID в конфиге (claude-opus-4-7, не «opus-latest»)
  2. Логируйте model ID в каждом прогоне (поле в summary.csv)
  3. Когда меняется ID — отдельный baseline для новой версии

Без фикса model ID ваши тренды — шум.

Practice (90-120 минут)

Задание 1 (60 минут) — собрать minimal harness

Возьмите 3 кейса из M11 reference suite. Превратите в формат setup.sh + prompt.md + verify.sh. Прогоните руками раз — должны быть зелёные.

Если не зелёные — починить verify.sh, не задачу. Verify должен точно отражать критерий приёмки.

Задание 2 (30 минут) — раннер

Напишите evals/run.sh (по шаблону выше). Прогоните на 3 кейсах. Получите summary.csv.

Задание 3 (30 минут) — CI

Заведите минимальный .github/workflows/agent-evals.yml. Прогоните вручную через workflow_dispatch. Убедитесь, что artifacts выгружаются.

Опционально (60 минут) — расширение

Добавьте ещё 5 кейсов, заведите evals/baseline.txt, включите проверку «pass rate ≥ baseline» в CI. Теперь у вас полноценный agent regression harness.

Recap

  • Golden example = input + criteria, не input+output. Проверяете семантику, не текст.
  • Harness = setup → prompt → verify на каждый кейс. Раннер прогоняет всё, CI следит.
  • Фиксируйте model ID + duration + token cost — иначе тренды шумные. Дрейф модели — отдельная категория сигналов.

Скептик Самурай

🥷 Скептик Самурай спрашивает: «Это же толстый процесс! У меня 3 разработчика на агентах, не Google, зачем CI и harness?»

Ответ: Harness — это дешевле в постройке, чем кажется. Базовый — 3 часа работы, окупается первым же случаем «модель тихо стала хуже». Если в команде 3+ человека на агентах — без общего harness вы не знаете, чьи правки в CLAUDE.md / .claude/ улучшают или ухудшают agent. Это становится источником конфликтов: каждый «настраивает под себя», общая производительность падает, никто не знает почему. Harness — это общая правда. Не идеально, но реально работает. 3-часовая инвестиция в простой harness возвращает 3 часа в первую же неделю, когда поймаете тихую регрессию.
