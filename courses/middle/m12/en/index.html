M12 · Regression tests for agents

Golden examples, snapshot harness, CI integration. Catch regressions at the moment of change, not two weeks later.

Hook

WITHOUT a regression harness:    WITH a regression harness:

Updated the model        →       git commit + push
Two weeks later spotted          CI: 12/20 → 9/20
something got worse              specific cases failed
                                 roll back until solved
"hard to reproduce"              "here's a reproducible diff"

Hey folks, welcome 👋

This chapter is about a harness that catches regressions at the moment of change, not two weeks later. In normal development we call this regression tests, here transposed onto agents.

An hour, maybe an hour and a half. This is the step up from M11 tier 2 to tier 3 — automated eval that tells you “got worse.”

Concept: a golden example as a unit test for an agent

In normal development a regression test is:

def test_add_returns_sum():
    assert add(2, 3) == 5

Hard: input 2,3 → expected output 5. If add changes and the test fails — regression, caught immediately.

You can’t do that with an agent. An agent is non-deterministic — same prompt, different answers. But you can build a similar shape:

┌─────────────────────────────────────────────────────┐
│  Golden example = input + criteria, not input+output│
│  Criteria = a set of automatic checks:              │
│   - regex match                                     │
│   - unit tests after applying the diff              │
│   - syntax checks (AST)                             │
│   - judge-LLM (GPT looks and scores)                │
└─────────────────────────────────────────────────────┘

Not “output must be exactly this,” but “output must satisfy these conditions.”

Harness structure

project/
├── evals/
│   ├── cases/
│   │   ├── case-01-refactor-controller/
│   │   │   ├── setup.sh           # prepares a clean state
│   │   │   ├── prompt.md          # input for the agent
│   │   │   ├── verify.sh          # verifies the result
│   │   │   └── expected.md        # what should be (for eyes)
│   │   ├── case-02-bug-fix/
│   │   │   └── ...
│   ├── run.sh                     # runs all cases
│   ├── runs/
│   │   ├── 2026-05-15/
│   │   │   ├── case-01.log
│   │   │   ├── case-01.pass
│   │   │   └── summary.csv
│   │   └── 2026-05-01/
│   └── trends.md                  # manual analysis

Each case is an isolated unit: setup → prompt → verify. Each run is independent, order doesn’t matter.

Example case: refactor controller

evals/cases/case-01-refactor-controller/setup.sh:

#!/usr/bin/env bash
set -e
# Reset working copy to the "before" point
git checkout main -- src/api/users.ts
git checkout main -- tests/users.test.ts
# Remove leftovers from previous runs
rm -f src/validators/userValidator.ts

evals/cases/case-01-refactor-controller/prompt.md:

In `src/api/users.ts` the `createUser` method mixes three responsibilities:
input validation, business logic, and DB access.

Task: extract validation into a separate module `src/validators/userValidator.ts`.

Requirements:
- `createUser` should import from the new module
- Existing tests `tests/users.test.ts` must pass unchanged
- No new dependencies

evals/cases/case-01-refactor-controller/verify.sh:

#!/usr/bin/env bash
set -e

# 1. File exists
[ -f src/validators/userValidator.ts ] || { echo "FAIL: validator file missing"; exit 1; }

# 2. Import in createUser
grep -q "from.*validators/userValidator" src/api/users.ts || { echo "FAIL: no import"; exit 1; }

# 3. Tests pass
npm test -- tests/users.test.ts || { echo "FAIL: tests broken"; exit 1; }

# 4. No new dependencies
git diff --quiet package.json package-lock.json || { echo "FAIL: deps changed"; exit 1; }

echo "PASS"

That’s a complete automated case. Runs in a minute, gives a binary answer.

Runner: run.sh

#!/usr/bin/env bash
set -u

RUN_ID="$(date -u +%Y-%m-%dT%H-%M-%S)"
OUT_DIR="evals/runs/$RUN_ID"
mkdir -p "$OUT_DIR"

SUMMARY="$OUT_DIR/summary.csv"
echo "case,pass,duration_s,notes" > "$SUMMARY"

for case_dir in evals/cases/*/; do
    name="$(basename "$case_dir")"
    log="$OUT_DIR/$name.log"
    start=$(date +%s)

    # Clean state
    bash "$case_dir/setup.sh" > "$log" 2>&1

    # Run the agent
    cat "$case_dir/prompt.md" | claude --print --output-format=json >> "$log" 2>&1

    # Verify
    if bash "$case_dir/verify.sh" >> "$log" 2>&1; then
        pass="1"
    else
        pass="0"
    fi

    dur=$(( $(date +%s) - start ))
    echo "$name,$pass,$dur," >> "$SUMMARY"
done

# Final stats
total=$(( $(wc -l < "$SUMMARY") - 1 ))
passed=$(grep -c ",1," "$SUMMARY" || true)
echo "Run $RUN_ID: $passed/$total"

Important: claude --print --output-format=json is non-interactive mode. The agent gets the prompt, runs (with the project’s CLAUDE.md), exits. That’s what you need for CI.

CI integration (GitHub Actions)

.github/workflows/agent-evals.yml:

name: agent-evals

on:
  schedule:
    - cron: "0 4 * * MON"      # every Monday, to catch
                               # model drift + project changes
  workflow_dispatch:           # manual when needed
  pull_request:
    paths:
      - "CLAUDE.md"
      - ".claude/**"
      - "evals/cases/**"       # run on agent config changes

jobs:
  eval:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Install Claude Code
        run: npm install -g @anthropic-ai/claude-code

      - name: Run evals
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: bash evals/run.sh

      - name: Upload artifacts
        uses: actions/upload-artifact@v4
        with:
          name: eval-run
          path: evals/runs/

      - name: Fail if pass rate drops
        run: |
          # Compare to baseline in evals/baseline.txt
          baseline=$(cat evals/baseline.txt)
          latest=$(ls -t evals/runs/ | head -n 1)
          passed=$(grep -c ",1," "evals/runs/$latest/summary.csv" || echo 0)
          total=$(( $(wc -l < "evals/runs/$latest/summary.csv") - 1 ))
          rate=$(( passed * 100 / total ))
          echo "Pass rate: $rate% (baseline: $baseline%)"
          if [ "$rate" -lt "$baseline" ]; then
            echo "::error::Regression detected"
            exit 1
          fi

This is a production harness. Runs on schedule + on agent-config changes + manually. If the pass rate dips below baseline — CI fails, you see it immediately.

Check types (verify.sh)

1. File/string

[ -f path/to/file ] || exit 1
grep -q "pattern" file || exit 1

Simple exists/contains. Great for “a module should appear” / “an import should exist.”

2. AST/syntax

# TypeScript — check the code parses
tsc --noEmit src/api/users.ts || exit 1

# Python
python -m py_compile src/auth.py || exit 1

Guard against “the agent wrote invalid code.”

3. Tests

npm test -- tests/users.test.ts || exit 1
pytest tests/test_auth.py || exit 1

The most valuable check — existing tests still pass.

4. Diff scope

# Did only the expected files change?
expected_files=$(cat case_dir/allowed_files.txt)
actual_files=$(git diff --name-only)
diff <(echo "$expected_files" | sort) <(echo "$actual_files" | sort) || exit 1

Guard against “wandered into the wrong place” — critical for security scenarios.

5. LLM-as-judge

If you can’t check programmatically (code style, message clarity):

# verify.sh writes a prompt for the judge model and parses the response
judge_response=$(cat <<EOF | claude --print
You are the judge. Below is a refactor result. Rate 1-5 on:
- readability of the new module
- degree of separation of responsibilities

Result file:
$(cat src/validators/userValidator.ts)

Reply strictly in format: SCORE=N
EOF
)
score=$(echo "$judge_response" | grep -oP "SCORE=\K\d+")
[ "$score" -ge 4 ] || exit 1

Caution: the judge-LLM has its own variance. For statistics — multiple runs and averaging.

Harness anti-patterns

1. Hard snapshots

“The agent must generate exactly this code.” Two days later the model does the same with different words — fails, while semantically correct.

Cure: verify semantics (tests, AST, structural regex), not exact text.

2. Cases too short / too long

  • Short (10 seconds) — no signal for hard tasks
  • Long (30+ minutes) — can’t run often, expensive

Cure: 1–5 minutes per case. 20 cases = 20–100 minutes per run. Once a week — acceptable.

3. Leaving flaky cases as is

A case fails one run in two — “we’ll temporarily ignore.” A month later you have 5 flaky cases, the suite is not verifiable.

Cure: flaky case → immediate work item “fix or delete.” Better fewer stable cases than many noisy ones.

4. Suite isolated from production

The reference suite lives in a separate repo. Nobody updates it when production changes.

Cure: the suite in the same repo as the code, in the same branch. Part of the PR checklist: “if you changed CLAUDE.md, check the suite is green.”

5. Pass/fail only

Binary is too coarse. A case “passed” but took 15 minutes instead of the usual 3 — that’s degradation, not captured.

Cure: record duration, token cost, number of tool calls. Efficiency regression is regression.

A delicate point: “model drift”

At Anthropic, OpenAI, and others, model snapshots change. Sometimes explicitly (new ID), sometimes silently on the same API endpoint. Which means:

  • Your harness can suddenly show a drop in pass rate, while you changed nothing
  • That’s not a bug — it’s drift — the model is slightly different

What to do:

  1. Pin the concrete model ID in the config (claude-opus-4-7, not “opus-latest”)
  2. Log the model ID on every run (a field in summary.csv)
  3. When the ID changes — a separate baseline for the new version

Without a pinned model ID your trends are noise.

Practice (90–120 minutes)

Task 1 (60 minutes) — build a minimal harness

Take 3 cases from the M11 reference suite. Convert into setup.sh + prompt.md + verify.sh. Run by hand once — should go green.

If not green — fix verify.sh, not the task. Verify must precisely mirror the acceptance criterion.

Task 2 (30 minutes) — runner

Write evals/run.sh (template above). Run on 3 cases. Get summary.csv.

Task 3 (30 minutes) — CI

Drop in a minimal .github/workflows/agent-evals.yml. Trigger it manually via workflow_dispatch. Confirm artifacts upload.

Optional (60 minutes) — extension

Add 5 more cases, create evals/baseline.txt, turn on the “pass rate ≥ baseline” check in CI. Now you have a full-fledged agent regression harness.

Recap

  • Golden example = input + criteria, not input+output. You verify semantics, not text.
  • Harness = setup → prompt → verify per case. The runner sweeps everything, CI watches.
  • Pin model ID + duration + token cost — otherwise trends are noisy. Model drift is its own signal category.

Vibe Samurai

🥷 Vibe Samurai asks: “That’s heavy process! I have 3 developers on agents, not Google — why CI and a harness?”

Answer: A harness is cheaper to build than it sounds. The base — 3 hours of work, pays back the first “the model quietly got worse.” If the team is 3+ on agents — without a common harness you don’t know whose CLAUDE.md / .claude/ tweaks help or hurt the agent. It becomes a source of conflicts: everyone “tunes for themselves,” collective throughput drops, nobody knows why. A harness is shared truth. Not perfect, but actually works. A 3-hour investment in a simple harness returns those 3 hours in the first week you catch a silent regression.
