M7 · GPT-5.3 Codex CLI and OpenAI agents

Codex CLI, approval modes, sandbox. When Codex beats Claude — on benchmarks and personal practice.

What this chapter is about

Hey folks, welcome 👋

Last chapter we dug into Claude Code. Now the parallel tool focus — GPT-5.3 Codex CLI from OpenAI. And let me say this up front: I’m not going to do “X is better than Y” — I sit on both myself, pick by task, and most of my colleagues do the same.

Personally, content in the “Claude beats everyone / GPT beats everyone” style honestly grates on me — that’s marketing nonsense and a misunderstanding of the actual landscape. The reality: models are different, each has its own strengths profile, and a sharp middle developer in 2026 knows how to work with both.

In this chapter:

  1. Codex CLI — what it is technically
  2. Approval modes — the main architectural difference from Claude Code
  3. Sandbox — how Codex isolates execution
  4. When Codex beats Claude, and vice versa (benchmark + personal practice)
  5. A minimal pricing breakdown

Codex CLI in 5 minutes

codex is OpenAI’s CLI agent that runs the GPT-5.3 or GPT-5.3 Codex model (the specialised coding fork) under the hood. Install:

# npm channel
npm install -g @openai/codex

# or brew on mac
brew install codex

# first run
codex

Auth — via a ChatGPT account (Plus/Pro/Enterprise) or via OpenAI API key (pay-per-use).

After launch you land in the interactive TUI — terminal UI where you type a prompt, see the model’s response + suggested actions, approve or reject.

Architecturally Codex CLI is very similar to Claude Code: the same plan→act→evaluate loop, the same tool set (read, edit, bash, search, mcp), the same config + hooks concept. But there are three principal differences:

  1. Approval modes — a centralised autonomy switch
  2. Sandbox — built-in OS-level execution isolation
  3. AGENTS.md instead of CLAUDE.md (same concept, different name)

We’ll cover each.

Approval modes — the key architectural choice

In Claude Code autonomy is more distributed: permissions allow/deny, hooks, manual confirmations. In Codex it’s collected into one parameter — --approval-mode:

ModeWhat it meansClaude Code equivalent
suggestSuggests only, does nothing--permission-mode plan
auto-editEdits files itself, commands — manual approvaltypical permissions
full-autoEdits and executes itself, no approval--dangerously-skip-permissions
codex --approval-mode suggest "refactor auth module"     # plan only
codex --approval-mode auto-edit "fix tests in auth/"      # edits itself, commands approve
codex --approval-mode full-auto "run npm test, fix until green"  # autonomous

The plus of this architecture — simplicity. One value controls it all. The minus — coarser than Claude Code: no “allow this bash command, deny that one” granularity. What Claude Code solves with allow/deny patterns, Codex solves by mode + sandbox + custom rules in AGENTS.md.

Critical point: full-auto in Codex CLI defaults to a sandboxed bash, while --dangerously-skip-permissions in Claude Code is “straight on the host.” Which means full-auto in Codex is significantly safer than it sounds. More on this below.

Sandbox — built-in OS-level isolation

Codex CLI’s strongest architectural choice is the built-in sandbox. When you launch a bash command via the agent:

  • macOS — sandbox-exec with a preset profile
  • Linux — seccomp + landlock + namespaces
  • Windows — Job Objects + restricted token
  • Containerised — optional Docker/Podman

By default the sandbox:

  • Blocks internet egress (optional, behind a flag)
  • Limits writes to workdir (you can’t accidentally change a file outside the project)
  • Blocks reads from sensitive paths (~/.ssh, ~/.aws, /etc/shadow)
  • Blocks setuid binaries

That is, OS-level enforcement of what Claude Code does through hooks + permissions. There are pluses and minuses:

Plus: no manual setup needed. The agent is isolated by default.

Minus: less flexible. If your agent legitimately needs to hit the internet (e.g. pull deps), you have to allow it explicitly.

Minimal override:

# Allow egress
codex --network-access allow ...

# Extend writable paths
codex --writable-paths ~/.local/share/app ...

Config in ~/.codex/config.toml:

[sandbox]
network = "deny"            # deny | allow | allowlist
writable_extra = []         # additional writable paths
readable_extra = []         # additional readable paths
forbidden_commands = ["rm -rf /", "curl ... | sh"]

This is architecturally more correct than “hoping for hooks” in Claude Code. But less tunable for fine scenarios.

AGENTS.md — same as CLAUDE.md

AGENTS.md in the project root is the persistent context. Codex CLI reads it on session start. Structure and philosophy — identical to CLAUDE.md (see M6).

Tip: in projects where both Claude Code and Codex CLI work — keep both files with identical content, via symlink or auto-sync through a hook. Don’t duplicate content by hand — someone on the team will forget to update one.

In my own prod config I keep AGENTS.md as a symbolic link to CLAUDE.md. On Windows — via mklink /H (hardlink, so git is happy).

When Codex beats Claude (in personal practice)

No bias. I noted every case where, in real work, one tool came out better than the other. Six-month assembled picture:

Codex CLI wins on:

  1. Long autonomous cycles. When the task is “run tests, fix, repeat until green,” and you need 5–10 iterations without intervention. Codex with sandbox + full-auto runs more stably and cheaper on tokens.
  2. Large volume of same-type edits. E.g. an API migration across 200 files. Codex is faster on mass uniform edits.
  3. Strict adherence to specifications. If you have a formal spec and you need it followed strictly — Codex is slightly better at instruction following without “improvements along the way.”
  4. Complex commercial code with heavy refactoring. On some typing-heavy TypeScript codebases GPT-5.3 holds steadier.

Claude Code wins on:

  1. Multi-step research with a big context. When the task is “understand someone else’s 50k-line repo and propose a solution,” Claude Opus 4.7 sees the relationships better.
  2. Creative tasks (layout, UI, copy in code). Claude has slightly better taste.
  3. Strict adherence to hooks and permissions. The allow/deny + hooks architecture gives more behavioural control.
  4. Subagents for parallel decomposition. The subagent pattern in Claude Code is more cohesive than in Codex.
  5. Long read-heavy sessions (M6 — Skills, prompt caching). Anthropic-side prompt caching + long context is noticeably cheaper.

Using Codex in a hands-on flow

Minimal example — “fix the tests”:

# Sandbox + auto-edit (edits files, bash — approval)
codex --approval-mode auto-edit --network-access deny

> "tests/auth.test.ts has 3 failing tests. Run tests in repo:
   npm test -- tests/auth.test.ts
   Fix all 3 tests without changing source logic.
   Success criterion: npm test -- tests/auth.test.ts exit 0."

Codex will propose a plan, ask to approve the npm test run, then look at results, edit tests, ask to approve a re-run, and so on until green. This cycle in Codex usually takes 3–8 iterations.

In full-auto with sandbox the same goes through without your intervention:

codex --approval-mode full-auto --network-access deny

> "tests/auth.test.ts has 3 failing tests..."

But personally I always first try a small task in auto-edit, confirm Codex actually understood, and only then flip to full-auto.

Anti-patterns (Codex-specific)

  1. --approval-mode full-auto --network-access allow without sandbox. That’s the same as --dangerously-skip-permissions in Claude Code. Don’t do it on a dev machine.
  2. --writable-paths / “to avoid restrictions.” This kills the whole sandbox idea. If you need more rights — extend specific paths surgically.
  3. The same AGENTS.md across projects. Context must be project-specific. Common rules go into ~/.codex/AGENTS.md (user-level).
  4. Ignoring ~/.codex/config.toml. At minimum put forbidden_commands there. If you haven’t configured it — do.

Practice

20–30 minutes:

  1. Install Codex CLI (if you haven’t).
  2. Create AGENTS.md in one of your projects (symlink to CLAUDE.md if you have one).
  3. Run codex --approval-mode suggest on a simple task — see what the plan looks like. Compare with what Claude Code produces on the same task.
  4. Try --approval-mode auto-edit with sandbox on fixing a failing test (break one on purpose). Watch the behaviour.
  5. Read ~/.codex/config.toml. Add at least one forbidden_command.

Recap

  • Codex CLI is an architectural close relative of Claude Code. Same tools, same loop. Three differences: approval modes, OS-level sandbox, AGENTS.md.
  • Codex sandbox is built-in. It makes full-auto much safer than --dangerously-skip-permissions in Claude.
  • Approval modes are simpler but coarser. One parameter vs granular permissions.
  • Different tasks — different tools. Codex for long autonomous cycles and mass uniform edits. Claude for research, creative, subagents, hook control.
  • AGENTS.md = CLAUDE.md. Don’t duplicate — symlink or sync.
