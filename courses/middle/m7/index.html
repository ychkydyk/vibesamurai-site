M7 · GPT-5.3 Codex CLI и OpenAI agents

Codex CLI, approval modes, sandbox. Когда Codex выгоднее Claude — на benchmark и личной практике.

Черновик M7 в голосе DmitriyM (см. scripts/tg_voice_analysis/_summary.md). Сравнительный авторский блок.

О чём эта глава

Друзья, всем привет! 👋

В прошлой главе мы залезли в Claude Code. Сейчас параллельный тул-фокус — GPT-5.3 Codex CLI от OpenAI. И сразу скажу важное: я не буду тут устраивать «X лучше Y» — я сам сижу на обоих, выбираю по задаче, и большая часть моих коллег так же.

Лично меня выбешивает контент в стиле «Claude лучше всех / GPT лучше всех» — это инфоцыганщина, маркетинг и непонимание реальности. Реальность: модели разные, у каждой свой профиль сильных сторон, и грамотный middle-разработчик в 2026-м умеет работать с обеими.

В этой главе:

Codex CLI за 5 минут

codex — это CLI-агент от OpenAI, который под капотом использует модели GPT-5.3 или GPT-5.3 Codex (специализированный coding fork). Установка:

# npm-канал
npm install -g @openai/codex

# или brew на mac
brew install codex

# первый запуск
codex

Авторизация — через ChatGPT-аккаунт (Plus/Pro/Enterprise) или через OpenAI API key (pay-per-use).

После запуска вы попадаете в interactive TUI — терминальный UI, в котором печатаете запрос, видите ответ модели + предложенные действия, одобряете или отклоняете.

Архитектурно Codex CLI очень похож на Claude Code: тот же loop plan→act→evaluate, тот же набор тулов (read, edit, bash, search, mcp), та же концепция конфига и hooks. Но есть три принципиальных отличия:

  1. Approval modes — централизованный механизм управления автономией
  2. Sandbox — встроенная OS-level изоляция исполнения
  3. AGENTS.md вместо CLAUDE.md (та же концепция, другое имя)

Разберём каждое.

Approval modes — главное архитектурное решение

В Claude Code управление автономией более распределённое: permissions allow/deny, hooks, manual confirmations. В Codex это собрано в один параметр — --approval-mode:

ModeЧто означаетАналог в Claude Code
suggestТолько предлагает, ничего не делает--permission-mode plan
auto-editСам правит файлы, команды — ручной approvaltypical permissions
full-autoСам правит и выполняет, без approval--dangerously-skip-permissions
codex --approval-mode suggest "refactor auth module"     # план-only
codex --approval-mode auto-edit "fix tests in auth/"      # правит сам, команды — approve
codex --approval-mode full-auto "run npm test, fix until green"  # автоном

Плюс этой архитектуры — простота. Одно значение управляет всем. Минус — грубее, чем у Claude Code: нет такой гранулярности «allow вот эту bash-команду, deny вон ту». То что в Claude Code решается allow/deny паттернами, в Codex решается выбором mode + sandbox + custom rules в AGENTS.md.

Принципиальный момент: full-auto в Codex CLI дефолтно работает с sandboxed bash, тогда как --dangerously-skip-permissions в Claude Code — это «прямо на хост-машину». Это значит, что full-auto в Codex значительно безопаснее, чем кажется. Об этом подробнее ниже.

Sandbox — built-in OS-level изоляция

Самое сильное архитектурное решение Codex CLI — встроенная sandbox. Когда вы запускаете bash-команду через агента:

  • macOS — используется sandbox-exec с предустановленным профилем
  • Linux — seccomp + landlock + namespaces
  • Windows — Job Objects + restricted token
  • Containerized — опционально Docker/Podman

По умолчанию sandbox:

  • Запрещает egress в интернет (опционально, через флаг)
  • Ограничивает запись только в workdir (нельзя случайно изменить файл за пределами проекта)
  • Запрещает чтение чувствительных путей (~/.ssh, ~/.aws, /etc/shadow)
  • Запрещает выполнение setuid binaries

То есть OS-level enforcement того, что в Claude Code достигается через hooks + permissions. У этого есть плюсы и минусы:

Плюс: не требует ручной настройки. По умолчанию агент изолирован.

Минус: менее гибко. Если вашему агенту нужно legitimately ходить в интернет (например, тянуть зависимости), надо отдельно разрешать.

Минимальный override:

# Разрешить egress
codex --network-access allow ...

# Расширить writable paths
codex --writable-paths ~/.local/share/app ...

Конфиг в ~/.codex/config.toml:

[sandbox]
network = "deny"            # deny | allow | allowlist
writable_extra = []         # additional writable paths
readable_extra = []         # additional readable paths
forbidden_commands = ["rm -rf /", "curl ... | sh"]

Это архитектурно правильнее, чем «надеемся на hooks» в Claude Code. Но менее настраиваемо для тонких сценариев.

AGENTS.md — то же что CLAUDE.md

AGENTS.md в корне проекта — постоянный контекст. Codex CLI читает его на старте сессии. Структура и философия — идентичны CLAUDE.md (см. M6).

Совет: в проектах, где работают и Claude Code, и Codex CLI — держите оба файла с одинаковым содержимым, симлинк или с автоматической синхронизацией через hook. Не повторяйте контент вручную — там в любой команде кто-то забудет обновить.

В прод-конфиге у себя я держу AGENTS.md как символическую ссылку на CLAUDE.md. На Windows — через mklink /H (hardlink, чтобы работало в Git).

Когда Codex выгоднее Claude (на личной практике)

Никакого пристрастия. Я записал каждый случай, когда у меня в реальной работе один тул показал себя лучше другого. Сейчас собранная картина за полгода:

Codex CLI выигрывает на:

  1. Длинные автономные циклы. Когда задача — «прогони тесты, исправь, повтори до зелёных», и нужно 5-10 итераций без вмешательства. Codex с sandbox + full-auto работает стабильнее и дешевле на токенах.
  2. Большой объём правок одного типа. Например, миграция API на новую версию через 200 файлов. Codex быстрее в массовых однотипных правках.
  3. Strict adherence to specifications. Если у вас есть формальная spec и надо строго по ней — Codex чуть лучше следует instruction following без «улучшений по ходу».
  4. Сложный коммерческий код с heavy refactoring. На некоторых typing-heavy TypeScript кодовых базах GPT-5.3 показывает себя ровнее.

Claude Code выигрывает на:

  1. Многоступенчатый ресёрч с большим контекстом. Когда задача — «разобраться в чужом репо на 50К строк и предложить решение», Claude Opus 4.7 видит лучше связи.
  2. Творческие задачи (вёрстка, UI, текст в коде). Claude чуть «вкуснее».
  3. Точное следование hooks и permissions. Архитектура с allow/deny + hooks даёт больше контроля над поведением.
  4. Subagents для параллельной декомпозиции. Pattern subagent-ов в Claude Code сделан целостнее, чем в Codex.
  5. Длинные read-heavy сессии (M6 — Skills, prompt caching). Anthropic-side prompt caching + длинный контекст ощутимо дешевле.

Использование Codex в hands-on flow

Минимальный пример — задача «исправь тесты»:

# Sandbox + auto-edit (правит файлы, на bash — approval)
codex --approval-mode auto-edit --network-access deny

> "В tests/auth.test.ts есть 3 падающих теста. Прогон тестов в repo:
   npm test -- tests/auth.test.ts
   Исправь все 3 теста, не меняя логику источников.
   Success criterion: npm test -- tests/auth.test.ts exit 0."

Codex предложит план, попросит одобрить запуск npm test, дальше — посмотрит результаты, отредактирует тесты, попросит одобрить повторный запуск, и так до зелёных. Подобный цикл в Codex обычно занимает 3-8 итераций.

В full-auto режиме с sandbox то же самое идёт без вашего вмешательства:

codex --approval-mode full-auto --network-access deny

> "В tests/auth.test.ts есть 3 падающих теста..."

Но я лично всегда сначала проверяю на маленькой задаче в auto-edit, понимаю что Codex действительно понял, и только тогда перевожу на full-auto.

Антипаттерны (Codex-специфичные)

  1. --approval-mode full-auto --network-access allow без sandbox. Это то же самое, что --dangerously-skip-permissions в Claude Code. Не делайте на dev-машине.
  2. --writable-paths / «чтобы не упереться в ограничения». Это отключение всей идеи sandbox. Если вам надо больше прав — точечно расширяйте конкретные пути.
  3. Один и тот же AGENTS.md между проектами. Контекст должен быть проектным. Общие правила можно положить в ~/.codex/AGENTS.md (user-level).
  4. Игнорирование ~/.codex/config.toml. Туда стоит положить хотя бы базовые forbidden_commands. Если не настроили — настройте.

Practice

20-30 минут:

  1. Установите Codex CLI (если ещё не).
  2. Создайте AGENTS.md в одном из ваших проектов (симлинк на CLAUDE.md, если уже есть).
  3. Запустите codex --approval-mode suggest на простую задачу — посмотрите, как выглядит план. Сравните с тем что выдаёт Claude Code на той же задаче.
  4. Попробуйте --approval-mode auto-edit с sandbox на исправление падающего теста (можно намеренно сломать). Понаблюдайте за поведением.
  5. Прочитайте ~/.codex/config.toml. Добавьте минимум один forbidden_command.

Recap

  • Codex CLI — архитектурный близкий родственник Claude Code. Те же тулы, тот же loop. Три отличия: approval modes, OS-level sandbox, AGENTS.md.
  • Sandbox в Codex — встроенная. Это делает full-auto намного безопаснее, чем --dangerously-skip-permissions в Claude.
  • Approval modes проще, но грубее. Один параметр vs гранулярные permissions.
  • На разных задачах — разные тулы. Codex для длинных автономных циклов и массовых однотипных правок. Claude для ресёрча, творчества, subagents, hooks-контроля.
  • AGENTS.md = CLAUDE.md. Не дублируйте, используйте симлинк / синхронизацию.
