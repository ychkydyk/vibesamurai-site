M9 · Memory strategies

Short-term, long-term и external memory. Scoping, pruning, Copilot Memory model.

Черновик M9 в голосе DmitriyM (см. scripts/tg_voice_analysis/_summary.md). Открывает Part III — Memory & State.

О чём эта глава

Друзья, всем привет! 👋

В Part I и Part II мы говорили про одну сессию: вы открыли агента, дали задачу, он сделал, вы закрыли. А что происходит между сессиями? Что агент помнит, что забывает, какой контекст у него есть «по умолчанию» при следующем старте?

Это и есть тема памяти. И — спойлер — там целое семейство ошибок, которые ловят middle-разработчиков впервые, кто переходит на агентный flow всерьёз. Самая частая жалоба новых пользователей агентов:

«Я вчера ему всё это объяснил! А сегодня он опять не помнит!»

Не помнит. Потому что по умолчанию агент не помнит ничего за пределами текущей сессии. И это нормально — это архитектурное решение, а не баг. В этой главе разберём:

  1. Три типа памяти агента: short-term, long-term, external
  2. Конкретные механизмы памяти в Claude Code, Codex CLI, Cursor, Copilot
  3. Scoping — что класть в память, что не класть
  4. Pruning — как и когда чистить
  5. Copilot Memory model — отдельно, потому что это эталонная реализация для GH-600

Три типа памяти

Если рисовать слоями — снизу самое медленное и долгоживущее, сверху самое горячее:

flowchart LR
    Agent[Agent<br/>session] -->|read/write| ST[Short-term<br/>context window]
    Agent -->|read/write| Cache[(Prompt cache<br/>5 min / 1h)]
    Agent -->|read| LT[Long-term<br/>CLAUDE.md / AGENTS.md]
    Agent -->|read/write| Ext[(External memory<br/>vector DB / mem0)]
    Agent -.read.-> Ref[Reference<br/>wiki / issues / ADR]
    LT -.git.-> Repo[(repo)]

Каждый слой решает свою задачу: short-term держит «прямо сейчас», cache бьётся по идентичному префиксу, long-term живёт в репо под версионированием, external накапливает между сессиями, reference — это внешняя истина команды.

Short-term memory (рабочий контекст сессии)

Это то, что в контекстном окне модели прямо сейчас: ваши сообщения, ответы агента, прочитанные файлы, результаты тулов. Живёт только пока открыта сессия.

Размер: ограничен контекстом модели. На май 2026 — Opus 4.7 даёт 1М токенов, Sonnet 4.6 даёт 1М, Haiku 4.5 — 200K. На практике эффективная работа идёт с 50-150К токенов; дальше начинается «context dilution» — модель хуже фокусируется на конкретике.

Управление:

  • Размер сессии — короткие фокусированные > длинные «всё подряд»
  • Compaction — в современных тулзах модель сама суммаризирует старый контекст
  • Subagents (M6) — выгружают побочные задачи из основного контекста

Самый частый антипаттерн: «одна гигантская сессия на день». Через 4 часа агент уже не помнит, что было утром, путается в собственных решениях. Лучше: много фокусированных сессий, каждая под одну задачу.

Long-term memory (между сессиями, в проекте)

Это то, что переживает закрытие сессии и читается при старте следующей. Реализуется через файлы:

  • CLAUDE.md / AGENTS.md / .cursor/rules/ — главный артефакт памяти
  • .claude/memory/ (опционально, через hook или skill)
  • ADR-файлы (Architecture Decision Records) в репо
  • Project README

Это то что должно быть в git, под версионированием, для всей команды.

Сильная сторона: deterministic. Файл существует — агент его прочитает. Файл удалили — агент забыл.

Слабая сторона: не автоматическое накопление. Если вы хотите чтобы агент «запомнил» решение из сегодняшней сессии — вы (или агент по вашему запросу) должны его записать в CLAUDE.md. Само не появится.

Лайфхак: можно в конце сессии попросить агента сформулировать «что мы решили / что нужно зафиксировать в CLAUDE.md» и предложить diff. Принимаете — закоммитили. Отклоняете — не зафиксировалось.

External memory (между сессиями и между тулами)

Это память более широкого scope — между сессиями, между тулами, между членами команды. Реализуется через:

  • Системы заметок / wiki (Notion, Confluence, Obsidian)
  • Issue tracker — Jira, Linear, GitHub issues
  • Personal memory systems (например, ~/.claude/memory/ с user-level индексом — это паттерн, который Anthropic ввели в стандартных «auto-memory» сетапах)
  • MCP-серверы для памяти — например, mem0, memori и подобные

Сильная сторона: универсальность. Один источник — много потребителей.

Слабая сторона: затраты на поддержание. Если вы кладёте всё подряд — через месяц это болото, в котором ничего не найти.

Конкретные механизмы по тулзам

Claude Code: auto-memory

В Claude Code (от 2025-Q4) есть встроенная auto-memory: папка ~/.claude/memory/ с индексным файлом MEMORY.md и отдельными memory-файлами.

Структура:

~/.claude/memory/
├── MEMORY.md                       # индекс, всегда в контексте
├── user_role.md                    # auto: запись о роли пользователя
├── feedback_no_userbot.md          # auto: feedback-запись
├── project_x.md                    # auto: проектный контекст
└── reference_y.md                  # auto: ссылка на внешний ресурс

Агент сам решает, когда записать в память (по триггерам типа «пользователь дал указание не делать X», «пользователь подтвердил подход Y»). При следующем старте Claude Code читает MEMORY.md и пробрасывает релевантные записи в контекст.

Типы memory:

  • user — о пользователе (роль, опыт, специализация)
  • feedback — что пользователь корректировал или подтверждал
  • project — состояние проектов
  • reference — ссылки на внешние системы

Это личная, не проектная память. Хранится у вас локально, не уходит в git.

Cursor: chat history

Cursor хранит историю чатов в локальном файле, и в новой сессии может «вспомнить» предыдущие разговоры. Менее формальный механизм, чем у Claude — скорее «история», чем «структурированная память».

.cursor/rules/ — это long-term project memory (см. M8).

Codex CLI: history

Codex CLI хранит историю сессий в ~/.codex/history/. Структурно проще, формально не отличается от Claude project memory.

AGENTS.md — long-term project memory (см. M7).

GitHub Copilot: Copilot Memory

Это отдельный, более enterprise-ориентированный механизм, который у нас в GH-600 покрывается отдельным subdomain. Разберём его особо.

Copilot Memory model (для GH-600)

GitHub Copilot (особенно в варианте Copilot Workspace / Copilot Coding Agent) использует Copilot Memory — структурированную систему памяти, у которой есть конкретная модель:

Scopes

ScopeКто видитЧто хранится
UserТолько один пользовательЛичные предпочтения, паттерны работы
RepositoryВсе коллабораторы репоПроектные конвенции, do-not-touch
OrganizationВся организацияКорпоративные стандарты
EnterpriseВсе в энтерпрайзеCompliance-правила

Lifecycle

  • Capture — момент записи. Может быть автоматическим (триггер на «пользователь сказал не делать X») или ручным (commit-style «remember this»).
  • Storage — где хранится. Repository scope — в самом репозитории (как .github/copilot-memory.yml), user scope — у GitHub-аккаунта, и так далее.
  • Retrieval — когда читается. По умолчанию — всегда при старте сессии. Опционально — semantic retrieval по контексту.
  • Decay / pruning — устаревание. Memory может «угаснуть» по времени или по явному delete.
  • Conflict resolution — что если user-scope конфликтует с repository-scope. Default: более узкий scope побеждает (user перебивает org, project перебивает user).

Принципиальные паттерны

  • Не дублируйте memory между scopes. Если правило «не редактируй legacy/» — это repository-scope. Не нужно дублировать в user-scope.
  • User scope — personal-preferences only. Не кладите туда проектное.
  • Org-scope — compliance and policy. Не кладите туда «как мы предпочитаем именовать переменные».
  • Conflict — это сигнал, что архитектура памяти течёт. Пересмотрите scoping.

Scoping: что класть в память, что не класть

Класть в long-term/external:

✅ Архитектурные решения и почему они были приняты ✅ Конвенции, которые не очевидны из кода ✅ Команды для запуска (тесты, линтер, сборка) ✅ Список путей, которые не трогать ✅ Tribal knowledge, появившееся из инцидентов ✅ Личные паттерны работы (для user-scope) ✅ Решённые архитектурные вопросы (ADR)

НЕ класть в long-term/external:

❌ То, что можно прочитать из кода (структура папок, имена функций) ❌ Что меняется быстро (текущие задачи, ad-hoc решения) ❌ Секреты (любого вида — pwds, tokens, ключи) ❌ Подробности фейлов «как мы вчера баг чинили» (это для commit message) ❌ Обещания, которые могут не выполниться (lien-style) ❌ Эмоциональные комментарии («этот код написал стажёр, ну тупой»)

Это аналог git — что-то стоит коммитить, что-то не стоит. С памятью та же логика.

Pruning: как чистить

Память без чистки превращается в свалку. Через 6 месяцев у вас 500 записей, 200 из них устарели, 50 противоречат друг другу, агент не знает что слушать.

Минимум pruning routine

Раз в месяц (или в спринт-ритме команды):

  1. Прочитайте MEMORY.md / индекс целиком. Если за минуту не пробежать — слишком большой.
  2. По каждой записи спросите:
    • Это ещё актуально? (если нет — удалить)
    • Это противоречит чему-то другому? (если да — разобрать конфликт)
    • Это можно сгруппировать с похожими? (если да — объединить)
  3. Удалите устаревшее. Не «пометьте как deprecated», а удалите. Если оно ещё нужно — оно в коде или в commit history.
  4. Перепишите громоздкие записи в одну короткую формулировку с «why» и «how to apply».

Сигналы что pruning нужен срочно

  • Memory-индекс не помещается в один экран — pruning нужен
  • Агент применяет устаревшее правило — pruning нужен
  • В команде «никто не понимает что в memory» — pruning + redocumentation
  • Bypass-события «потому что memory мешает» — pruning нужен ещё вчера

Аналогия: пробег у машины не бесконечный. Память без чистки — это та же машина, у которой не меняли масло. Работает, но всё хуже.

Антипаттерны памяти

1. «Memory как лог»

Каждый чих агента записывается в память. Через месяц — 1000 записей, 95% мусор. Memory — это структурированный отбор, не лог.

2. «Memory как замена документации»

Команда пишет всё в Copilot Memory, README пустеет. Когда приходит новый человек — он не может прочитать README, он не имеет доступа к memory. Memory дополняет, не заменяет.

3. «Stale memory как закон»

В memory записали правило 6 месяцев назад. С тех пор стек изменился — а правило в memory. Агент ему слепо следует. Memory должна проверяться, не догматизироваться.

4. «Memory одна на всех»

Project-scope правила в user-scope. Или org-policy в project-scope. Каждое правило — в свой scope.

5. «Memory без owner-а»

В команде memory есть, но никто не следит. Через год — болото. Память должна иметь owner-а, как код имеет maintainer-а.

Practice

30 минут:

  1. Откройте свой MEMORY.md или эквивалент. Если у вас не настроена память — настройте минимальную (Claude Code auto-memory или ручной MEMORY.md в ~/.claude/).
  2. Прочитайте индекс целиком. Если занимает > 1 экрана — pruning candidate.
  3. Удалите 3-5 самых очевидных stale записей. «Это уже не актуально».
  4. Сгруппируйте 2-3 похожих записи в одну. Если памяти много, она дробится — собирайте.
  5. Откройте проектный CLAUDE.md / AGENTS.md одного из ваших проектов. Сравните: то что в нём есть — это сейчас актуально? Что-то стоит удалить? Что-то добавить?

Recap

  • Три типа памяти: short-term (контекст сессии), long-term (project files), external (wiki/notes/MCP).
  • Auto-memory в Claude Code: ~/.claude/memory/ с типизированными записями (user/feedback/project/reference).
  • Copilot Memory model: scopes (user/repo/org/ent), lifecycle (capture→storage→retrieval→pruning), conflict-resolution «узкий scope побеждает».
  • Scoping: в память — конвенции, решения, tribal knowledge. НЕ в память — секреты, lien-обещания, быстро меняющееся.
  • Pruning раз в месяц обязателен. Без чистки — болото за 6 месяцев.
  • 5 антипаттернов: memory-как-лог, memory-вместо-документации, stale-as-law, без scope-разделения, без owner-а.

Тема памяти — где особенно полезно посмотреть, как её делают «у больших». Три якоря: prompt cache от Anthropic, дедушка memory-агентов (Letta/MemGPT) и популярный memory-layer для агентов (mem0).

Кеш и контекст в LLM

  • Anthropic — Prompt caching — platform.claude.com/docs/en/build-with-claude/prompt-caching. Как кешируется идентичный префикс (system prompt + tools + CLAUDE.md), при чём тут компакция. Прямо в нашу тему «горячих слоёв».

Stateful agents с памятью

  • Letta (formerly MemGPT) — docs — docs.letta.com/concepts/memgpt/. Архитектурный референс: как агент работает с разделением core memory / archival memory, как переживает рестарты.
  • mem0 — docs — docs.mem0.ai/introduction. Memory-layer, который вешается сверху обычного агента; episodic/semantic/procedural memory как отдельные сущности.

Совет: Letta — для тех, кому интересна архитектура «изнутри», mem0 — для тех, кому нужна готовая прослойка к существующему агенту. Не ставьте оба сразу.
