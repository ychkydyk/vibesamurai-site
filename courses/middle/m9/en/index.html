M9 · Memory strategies

Short-term, long-term, and external memory. Scoping, pruning, the Copilot Memory model.

What this chapter is about

Hey folks, welcome 👋

In Part I and Part II we talked about one session: you opened the agent, gave it a task, it ran, you closed it. What happens between sessions? What does the agent remember, what does it forget, what context does it have “by default” at the next start?

That’s the memory topic. And — spoiler — there’s a whole family of mistakes that catches middle developers stepping into agentic flow for the first time. The most common complaint from new agent users:

“I explained all of this to it yesterday! Today it doesn’t remember!”

It doesn’t. Because by default the agent remembers nothing outside the current session. And that’s fine — it’s an architectural decision, not a bug. In this chapter:

  1. Three memory types: short-term, long-term, external
  2. Concrete memory mechanisms in Claude Code, Codex CLI, Cursor, Copilot
  3. Scoping — what to keep in memory, what not to
  4. Pruning — how and when to clean
  5. Copilot Memory model — separately, because it’s the reference implementation for GH-600

Three memory types

In layers — bottom slowest and longest-lived, top hottest:

flowchart LR
    Agent[Agent<br/>session] -->|read/write| ST[Short-term<br/>context window]
    Agent -->|read/write| Cache[(Prompt cache<br/>5 min / 1h)]
    Agent -->|read| LT[Long-term<br/>CLAUDE.md / AGENTS.md]
    Agent -->|read/write| Ext[(External memory<br/>vector DB / mem0)]
    Agent -.read.-> Ref[Reference<br/>wiki / issues / ADR]
    LT -.git.-> Repo[(repo)]

Each layer solves its own problem: short-term holds “right now,” cache hits identical prefixes, long-term lives in the repo under version control, external accumulates across sessions, reference is the team’s external source of truth.

Short-term memory (the session’s working context)

This is what’s in the model’s context window right now: your messages, the agent’s responses, files read, tool results. Lives only while the session is open.

Size: bounded by the model’s context. As of May 2026 — Opus 4.7 gives 1M tokens, Sonnet 4.6 gives 1M, Haiku 4.5 — 200K. In practice effective work happens around 50–150K tokens; beyond that “context dilution” starts — the model focuses less well on specifics.

Management:

  • Session size — short focused > long “kitchen sink”
  • Compaction — modern tools auto-summarise old context
  • Subagents (M6) — offload side tasks from the main context

The most common anti-pattern: “one giant session per day”. After 4 hours the agent doesn’t remember the morning, gets tangled in its own decisions. Better: many focused sessions, one per task.

Long-term memory (between sessions, in the project)

This is what survives session close and is read on next start. Implemented through files:

  • CLAUDE.md / AGENTS.md / .cursor/rules/ — the main memory artifact
  • .claude/memory/ (optional, via hook or skill)
  • ADR files (Architecture Decision Records) in the repo
  • Project README

This belongs in git, under version control, for the whole team.

Strength: deterministic. The file exists — the agent reads it. Delete the file — the agent forgets.

Weakness: no automatic accumulation. If you want the agent to “remember” a decision from today’s session — you (or the agent on your request) have to write it into CLAUDE.md. It won’t appear by itself.

Pro tip: at the end of a session ask the agent to formulate “what did we decide / what should be captured in CLAUDE.md” and offer a diff. Accept it — committed. Reject — not captured.

External memory (between sessions and across tools)

Memory of broader scope — across sessions, tools, team members. Implemented through:

  • Notes / wiki systems (Notion, Confluence, Obsidian)
  • Issue tracker — Jira, Linear, GitHub issues
  • Personal memory systems (e.g. ~/.claude/memory/ with a user-level index — a pattern Anthropic introduced in their standard “auto-memory” setups)
  • MCP servers for memory — mem0, memori, and similar

Strength: universality. One source, many consumers.

Weakness: maintenance overhead. If you stuff everything in — in a month it’s a swamp where nothing’s findable.

Mechanisms per tool

Claude Code: auto-memory

In Claude Code (from 2025-Q4) there’s a built-in auto-memory: the folder ~/.claude/memory/ with an index file MEMORY.md and individual memory files.

Structure:

~/.claude/memory/
├── MEMORY.md                       # index, always in context
├── user_role.md                    # auto: user role record
├── feedback_no_userbot.md          # auto: feedback record
├── project_x.md                    # auto: project context
└── reference_y.md                  # auto: external reference

The agent decides when to write to memory (on triggers like “user gave a directive not to do X,” “user confirmed approach Y”). On next start Claude Code reads MEMORY.md and pulls relevant entries into context.

Memory types:

  • user — about the user (role, experience, specialisation)
  • feedback — what the user corrected or confirmed
  • project — state of projects
  • reference — links to external systems

This is personal, not project memory. Stored locally on your machine, doesn’t go into git.

Cursor: chat history

Cursor stores chat history in a local file and can “remember” previous conversations in a new session. Less formal than Claude’s mechanism — more “history” than “structured memory.”

.cursor/rules/ is long-term project memory (see M8).

Codex CLI: history

Codex CLI stores session history in ~/.codex/history/. Structurally simpler; formally no different from Claude project memory.

AGENTS.md is long-term project memory (see M7).

GitHub Copilot: Copilot Memory

A separate, more enterprise-oriented mechanism, which in our GH-600 is covered as its own subdomain. Worth treating separately.

Copilot Memory model (for GH-600)

GitHub Copilot (especially in the Copilot Workspace / Copilot Coding Agent variant) uses Copilot Memory — a structured memory system with a concrete model:

Scopes

ScopeWho sees itWhat’s stored
UserOnly one userPersonal preferences, work patterns
RepositoryAll repo collaboratorsProject conventions, do-not-touch
OrganizationThe whole organisationCorporate standards
EnterpriseEveryone in the enterpriseCompliance rules

Lifecycle

  • Capture — moment of writing. Can be automatic (trigger on “user said don’t do X”) or manual (commit-style “remember this”).
  • Storage — where it lives. Repository scope — in the repo itself (e.g. .github/copilot-memory.yml), user scope — on the GitHub account, and so on.
  • Retrieval — when it’s read. By default — always on session start. Optionally — semantic retrieval by context.
  • Decay / pruning — staleness. Memory can “fade” over time or via explicit delete.
  • Conflict resolution — what if user-scope conflicts with repository-scope. Default: narrower scope wins (user beats org, project beats user).

Principal patterns

  • Don’t duplicate memory across scopes. If a rule “don’t edit legacy/” — that’s repository-scope. No need to duplicate at user-scope.
  • User scope — personal preferences only. Don’t put project material there.
  • Org scope — compliance and policy. Don’t put “how we prefer to name variables” there.
  • Conflict is a signal that your memory architecture is leaking. Revisit scoping.

Scoping: what goes in memory, what doesn’t

Keep in long-term/external:

✅ Architectural decisions and why they were made ✅ Conventions that aren’t obvious from code ✅ Run commands (tests, lint, build) ✅ Paths-not-to-touch list ✅ Tribal knowledge from incidents ✅ Personal work patterns (for user scope) ✅ Resolved architectural questions (ADR)

Don’t keep in long-term/external:

❌ What can be read from the code (folder structure, function names) ❌ What changes fast (current tasks, ad-hoc decisions) ❌ Secrets (any kind — passwords, tokens, keys) ❌ Failure details “how we fixed yesterday’s bug” (that’s for commit messages) ❌ Promises that may not be kept (lien-style) ❌ Emotional comments (“an intern wrote this code, the dummy”)

It’s like git — some things belong in a commit, some don’t. With memory the logic is the same.

Pruning: how to clean

Memory without cleaning turns into a dump. In 6 months you’ve got 500 entries, 200 stale, 50 contradicting each other, the agent doesn’t know what to follow.

Minimum pruning routine

Once a month (or in the team’s sprint rhythm):

  1. Read the full MEMORY.md / index. If it doesn’t skim in a minute — it’s too big.
  2. For each entry ask:
    • Still relevant? (if no — delete)
    • Conflicts with something else? (if yes — resolve)
    • Can it be merged with similar ones? (if yes — combine)
  3. Delete stale. Don’t “mark deprecated” — delete. If it’s still needed, it lives in code or commit history.
  4. Rewrite bulky entries into one short statement with “why” and “how to apply.”

Signals you need pruning now

  • Memory index doesn’t fit on one screen — pruning needed
  • The agent applies an outdated rule — pruning needed
  • “Nobody on the team understands what’s in memory” — pruning + redocumentation
  • Bypass events “because memory gets in the way” — pruning was needed yesterday

Analogy: a car’s mileage isn’t infinite. Memory without cleaning is the same car with no oil changes. It runs, but worse.

Memory anti-patterns

1. “Memory as a log”

Every agent sneeze recorded into memory. In a month — 1000 entries, 95 % garbage. Memory is a structured selection, not a log.

2. “Memory replaces documentation”

The team writes everything into Copilot Memory; the README empties. New joiner — can’t read the README, doesn’t have access to memory. Memory complements; it doesn’t replace.

3. “Stale memory as law”

A rule was written 6 months ago. The stack has changed; the rule is still in memory. The agent blindly follows it. Memory must be reviewed, not dogmatised.

4. “One memory for everyone”

Project-scope rules in user scope. Or org policy in project scope. Each rule in its scope.

5. “Memory without an owner”

The team has memory but nobody curates it. A year later — a swamp. Memory must have an owner, just like code has a maintainer.

Practice

30 minutes:

  1. Open your MEMORY.md or equivalent. If memory isn’t configured — set up the minimum (Claude Code auto-memory or a manual MEMORY.md in ~/.claude/).
  2. Read the index in full. If it takes > 1 screen — pruning candidate.
  3. Delete 3–5 of the most obviously stale entries. “Not relevant anymore.”
  4. Group 2–3 similar entries into one. If memory is large, it fragments — consolidate.
  5. Open the project CLAUDE.md / AGENTS.md in one of your projects. Compare: is what’s in there current? Anything to drop? Anything to add?

Recap

  • Three memory types: short-term (session context), long-term (project files), external (wiki/notes/MCP).
  • Auto-memory in Claude Code: ~/.claude/memory/ with typed entries (user/feedback/project/reference).
  • Copilot Memory model: scopes (user/repo/org/ent), lifecycle (capture→storage→retrieval→pruning), conflict-resolution “narrower scope wins.”
  • Scoping: in memory — conventions, decisions, tribal knowledge. NOT in memory — secrets, lien promises, fast-changing data.
  • Pruning monthly is mandatory. Without it — a swamp in 6 months.
  • 5 anti-patterns: memory-as-log, memory-replaces-docs, stale-as-law, no scope separation, no owner.

📚 Further reading

Memory is where it’s especially useful to see how “big shops” do it. Three anchors: Anthropic’s prompt cache, the granddaddy of memory agents (Letta/MemGPT), and a popular memory layer for agents (mem0).

Cache and context in LLMs

  • Anthropic — Prompt caching — platform.claude.com/docs/en/build-with-claude/prompt-caching. How an identical prefix (system prompt + tools + CLAUDE.md) is cached and how it relates to compaction. Right in our “hot layer” topic.

Stateful agents with memory

  • Letta (formerly MemGPT) — docs — docs.letta.com/concepts/memgpt/. Architectural reference: how an agent splits core memory / archival memory and survives restarts.
  • mem0 — docs — docs.mem0.ai/introduction. A memory layer that hooks onto a regular agent; episodic/semantic/procedural memory as separate entities.

Tip: Letta — for those interested in architecture “from the inside”; mem0 — for those who need a ready layer for an existing agent. Don’t install both at once.
