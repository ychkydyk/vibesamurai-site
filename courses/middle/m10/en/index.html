M10 · State, drift, continuity

Resumable agents, durable artifacts. How to catch and correct drift. Cross-tool / cross-env state sharing.

What this chapter is about

Hey folks, welcome 👋

In M9 — about memory: what the agent remembers and how we manage it. Now about state — a broader concept. Memory is part of state. But beyond memory there’s:

  • What the agent is doing right now (current task, current plan step)
  • What it has on hand (open files, loaded context, active tools)
  • Where it stopped (if the session was interrupted — where to resume)
  • What it is in general (version, config, permission set)

Managing all of this is state management in agentic systems. And this is exactly where middle developers catch the meanest bugs: “worked yesterday — broken today,” “resumed from the same spot — got garbage,” “two agents made conflicting changes.” All state.

In this chapter:

  1. Resumable agents — how to continue an interrupted session without losses
  2. Durable artifacts — what must exist for continuation to be possible
  3. Drift — when the agent’s state diverges from reality, how to catch it
  4. Cross-tool state sharing — when Claude Code, Cursor, and Codex work on the same project

Resumable agents: what an “interrupted session” means

An agent session can break for many reasons:

  • You closed the terminal
  • Token limit hit / provider rate-limited
  • Network blip
  • Tool crash (happens)
  • You urgently switched to something else
  • Daily limit on the subscription

What happens to state at that moment? Depends on the tool. Modern sensible agents do autosave session state:

  • Claude Code — every session is saved in full to ~/.claude/projects/<repo>/<session-id>.jsonl.
  • Codex CLI — ~/.codex/history/<session-id>/
  • Cursor Composer — in the IDE state per project.

Which means — you can resume. Usually claude --resume / codex --resume.

What resume restores

  • Session context — all messages, files read, tool results
  • The active plan, if structured
  • Unfinished tool calls

What resume does NOT restore

  • File changes the agent made — those are already on disk (or not, if rolled back)
  • Running processes (dev servers, watchers, side effects)
  • MCP server state (if stateful, which is rare)
  • Local changes outside the session log (anything you did manually between interrupt and resume)

So resume is “bring back the conversation,” not “bring back the world.” Important to understand.

Durable artifacts: what must remain even without resume

If you want the agent’s work to survive any failure — don’t rely only on session state. Save durable artifacts in the repo.

Minimum durable set

  1. Plan artifact — YAML/markdown in .agent/plans/ (M2).
  2. Progress journal — list of completed steps with marks. .agent/journals/2026-05-16-task-x.md.
  3. Commits after every risk: high step. Not one big commit at the end, but small atomic ones — so each can be rolled back independently.
  4. CI runs. If steps relate to tests/builds — CI runs as artifact.

If all this is in place — even without resume you can read .agent/plans/x.yaml + .agent/journals/x.md and figure out where you are, what’s done, what’s left.

Analogy: imagine a new person joins the team, no “old” people available. From your artifacts they must be able to continue. If yes — the system is durable. If not — state lives only in heads (and in agent session state), and that’s fragile.

Drift: when the world diverged from the picture

Drift is when the agent’s picture of state doesn’t match reality. As moves pile up the model drifts from the original context, and at some point it’s cheaper to checkpoint and reset than to keep reasoning forward:

flowchart LR
    Fresh[T0: fresh<br/>clean context] --> T1[T+N: light drift<br/>facts still fresh]
    T1 --> T2[T+M: serious drift<br/>cached stale data]
    T2 --> Check{Checkpoint?}
    Check -->|durable artifact| Reset[Compact / reset<br/>new session]
    Reset -.-> Fresh
    Check -->|no| Bad([silent divergence<br/>quiet bugs])

Key idea: drift always accumulates — the question is whether you notice before it harms. Durable artifacts (plan + journal + atomic commits) are the points where you can safely reset and continue.

The most common kinds:

1. File drift

The agent thinks src/auth.ts has a verifyToken function because it read the file at session start. But you (or another agent) changed the file since. The agent works on stale context.

How to catch: before every serious edit the agent should re-read the file, not work from cache. In Claude Code this is built in (Read before Edit). In others — verify.

How to avoid: don’t work in parallel on the same file from different sessions. If you must — git status / git diff before each serious step.

2. Schema drift

The agent cached that table users has columns X, Y, Z. Actually it’s X, Y, Z, W now — a migration ran. The agent writes code on the old schema.

How to catch: periodically tell the agent “re-read the current DB schema.” Don’t rely on what’s memorised.

3. API drift

The agent cached that endpoint /api/users returns email. Actually it’s contact_email now. All the agent’s calls break.

How to catch: for critical integrations — versioned schemas and conformance checks.

4. Config drift

The agent worked on config A. You changed it to B. The agent keeps going on A (didn’t reload).

How to catch: config is a stateful resource, re-read on substantial transitions.

5. Process drift

The most painful. You used to do X before Y. The team agreed: now it’s X1 → Y. The agent follows old memory and does X → Y.

How to catch: see M9 — pruning. Memory must be updated, otherwise it itself becomes a drift source.

Detecting drift: automatically and by hand

Automatically

  • Pre-flight checks. At the start of a serious step the agent runs a script that verifies basic invariants (“are tests green before my edits? Does the DB schema match expectations? Is the dependency version still X?”).
  • Hooks (M6) — a PreToolUse hook can verify a file’s mtime hasn’t changed since the last read.
  • CI — every PR runs tests; if they fail “not because of you,” that’s drift.

By hand

  • Regular “situational briefings” at the start of a session. Before giving a task — ask the agent “read the current state of X, Y, Z and summarise.” If the summary differs from your expectations — drift.
  • Diff session memory vs reality — sometimes it’s worth asking directly “what do you currently think about Z?” and comparing with git/db/config.

Correcting drift

When drift is detected:

  1. Stop. Don’t keep working in that context.
  2. Reset closed/cached state — close the session, start a fresh one.
  3. Re-read everything current — files, configs, schemas relevant to the task.
  4. Update memory. If drift came from a stale long-term entry — update it before returning to the task, not after.
  5. Only then — keep working.

This is disruptive to speed — every drift incident throws you 10–30 minutes off. So prevention (durable artifacts + regular reconciliation) is cheaper than correction.

Cross-tool / cross-env state sharing

The modern reality: several different agents work on the same repo in different environments. One dev on Claude Code on macOS, another on Cursor on Windows, the CI pipeline running Codex CLI in Docker. And each “remembers” the project its own way.

What must be shared (via git)

✅ CLAUDE.md + symlinked AGENTS.md + .cursor/rules/ ✅ .claude/settings.json (project-level) — for those on Claude Code ✅ .codex/config.toml (project, if applicable) ✅ .mcp.json — list of connected MCP servers ✅ .agent/plans/ and .agent/journals/ (durable artifacts) ✅ Hook scripts in .claude/scripts/ or .cursor/hooks/ ✅ ADR files in docs/adr/

What MUST NOT enter git

❌ .claude/settings.local.json — personal toggles ❌ ~/.claude/memory/ — user-scope memory, personal ❌ Session history (~/.claude/projects/, ~/.codex/history/) ❌ Local secrets / .env

The problem: cross-tool semantic gap

Cursor may not understand .claude/settings.json, and Claude Code doesn’t read .cursor/rules/ by default. What to do?

Minimum approach: one AGENTS.md (or CLAUDE.md) at root as single source of truth. Symlinks to the required names. .cursor/rules/ may contain one line apply_from: ../../CLAUDE.md (supported in newer Cursor versions).

Extended approach: a .agent/sync.sh script that regenerates tool-specific configs from a common source. Runs in a pre-commit hook.

Anti-pattern: “every developer has their own agent”

The worst of them all. Every team member configured the agent “for themselves,” all rules in user-scope memory, in the project repo — nothing.

What will happen:

  • A new person joins — nothing works “as intended,” they don’t know the rules.
  • One person on holiday — their “knowledge” goes with them.
  • Code review of agent PRs is unpredictable: one agent writes one way, another differently, unclear if it’s “right.”

Fix: project memory — in the repo, project settings — in the repo, project hooks — in the repo. User-scope is for personal preferences only that don’t affect the project (your commit style, say).

We close Part III — Memory & State.

A short take on all ten modules of Parts I–III:

An agent = a contributor working under a contract, observed, within explicit boundaries, with memory under version control and state that survives any failure.

This is an architectural formula, not “how to configure a tool.” The specific Claude Code 2026 settings will age. Specific file names will change. The models themselves will be different.

The formula won’t. Contract, observation, boundaries, memory, state — these are disciplinary elements at the heart of any decent system with agents, human or machine.

In Part IV we tackle how to evaluate whether the agent did well or badly (success criteria and evaluation). Same topic, one level deeper.

Train your eye, not your fingers. Your head matters more than the AI — especially when judging “does it work or not.”

Recap

  • Resume restores the conversation, not the world. Rely on durable artifacts, not just session state.
  • Durable artifacts: plans, journals, atomic commits after high-risk steps, CI runs.
  • 5 drift kinds: file, schema, API, config, process. Each has its own catch.
  • Detection: pre-flight checks + hooks + regular “briefings.”
  • Correction: stop → reset → reread → update memory → continue.
  • Cross-tool sharing: in git — conventions, configs, MCP, hooks, ADR. Not in git — personal toggles, secrets, session history.
  • Anti-pattern: “every dev has their own agent” — project rules must live in the repo.

📚 Further reading

Drift and continuity are essentially a conversation about context-window management. So a short list — Anthropic’s cache/compaction and stateful agents as reference.

Context and compaction

  • Anthropic — Prompt caching — platform.claude.com/docs/en/build-with-claude/prompt-caching. What exactly is cached, how compaction reuses the same prefix. Helps explain why “a small edit to CLAUDE.md” invalidates half the cache.
  • OpenTelemetry — Semantic conventions for GenAI — opentelemetry.io/docs/specs/semconv/gen-ai/. Which metrics and attributes to put in traces so drift is visible in observability rather than “surfacing” through a PR incident.

Stateful agents and durable state

  • Letta — MemGPT concept — docs.letta.com/concepts/memgpt/. The reference for splitting “hot” context from “cold” archival store so resume is “remembered the world,” not “read the conversation.”

Tip: if drift isn’t a one-off but systemic — don’t start with another hook; start by making every long task write a journal into .agent/journals/. Half the drifts vanish.
