M10 · State, drift, continuity

Resumable agents, durable artifacts. Как ловить и исправлять drift. Cross-tool / cross-env state sharing.

Черновик M10 в голосе DmitriyM (см. scripts/tg_voice_analysis/_summary.md). Закрывает Part III — Memory & State.

О чём эта глава

Друзья, всем привет! 👋

В M9 мы про память — что агент помнит и как мы этим управляем. Сейчас про состояние — это более широкое понятие. Память — это часть состояния. Но кроме памяти есть:

  • Что агент сейчас делает (текущая задача, текущий шаг плана)
  • Что у него на руках (открытые файлы, прогруженный контекст, активные тулы)
  • Где он остановился (если сессия прервалась — где продолжить)
  • Какой он в принципе (версия, конфиг, набор разрешений)

Управление этим всем — это и есть state management в агентских системах. И именно тут middle-разработчики ловят самые подлые баги: «работало вчера — сегодня не работает», «продолжил с того же места — получил мусор», «два агента сделали взаимоисключающие изменения». Это всё про state.

В этой главе:

  1. Resumable agents — как продолжать прерванную сессию без потерь
  2. Durable artifacts — что должно остаться, чтобы продолжение было возможно
  3. Drift — когда состояние агента расходится с реальностью, как это ловить
  4. Cross-tool state sharing — когда Claude Code, Cursor и Codex работают над одним проектом

Resumable agents: что значит «прерванная сессия»

Сессия агента может прерваться по куче причин:

  • Вы закрыли терминал
  • Истёк лимит токенов / провайдер дал rate-limit
  • Сетевой сбой
  • Краш тула (бывает)
  • Нужно срочно переключиться на другое
  • Дневной лимит на подписке

Что происходит с состоянием в этот момент? Зависит от тула. Современные нормальные агенты делают autosave session state:

  • Claude Code — каждая сессия сохраняется в ~/.claude/projects/<repo>/<session-id>.jsonl целиком.
  • Codex CLI — ~/.codex/history/<session-id>/
  • Cursor Composer — в IDE-state по проекту.

Это значит — можно resume. Команда обычно claude --resume / codex --resume.

Что resume восстанавливает

  • Контекст сессии — все сообщения, прочитанные файлы, результаты тулов
  • Активный план, если он был structured
  • Незавершённые тул-вызовы

Что resume НЕ восстанавливает

  • Изменения в файлах, которые сделал агент — они уже в файловой системе (или нет, если откатили)
  • Запущенные процессы (dev servers, watchers, side effects)
  • MCP-серверные state-ы (если они были stateful, что бывает редко)
  • Локальные изменения, не вошедшие в сессионный лог (то что вы вручную сделали между прерыванием и resume)

То есть resume — это «вернуть диалог», не «вернуть мир». Это важно понимать.

Durable artifacts: что должно остаться даже без resume

Если вы хотите, чтобы работа агента переживала любые сбои — не полагайтесь только на session state. Сохраняйте durable artifacts в репо.

Минимальный набор durable

  1. Plan-артефакт — YAML/markdown, в .agent/plans/ (M2).
  2. Progress journal — список выполненных шагов с отметками. .agent/journals/2026-05-16-task-x.md.
  3. Commits после каждого risk: high шага. Не одним большим коммитом в конце, а маленькими атомарными — чтобы каждый можно было отдельно откатить.
  4. CI runs. Если шаги связаны с тестами/сборкой — CI run-ы как artifact.

Если вся эта инфраструктура есть — даже без resume вы можете прочитать .agent/plans/x.yaml + .agent/journals/x.md и понять, на каком вы шаге, что сделано, что осталось.

Аналогия: представьте что в команду пришёл новый человек, никто из «старых» недоступен. По вашим артефактам он должен суметь продолжить. Если может — система durable. Если нет — состояние держится только в головах живых людей (и в session state агента), и это fragile.

Drift: когда мир разошёлся с представлением

Drift — это когда представление агента о состоянии не совпадает с реальностью. По мере накопления ходов модель уходит от исходного контекста, и в какой-то момент проще не «дорассуждать», а сделать checkpoint и reset:

flowchart LR
    Fresh[T0: fresh<br/>контекст чистый] --> T1[T+N: lite drift<br/>факты ещё свежие]
    T1 --> T2[T+M: serious drift<br/>есть устаревший кеш]
    T2 --> Check{Checkpoint?}
    Check -->|durable artifact| Reset[Compact / reset<br/>новая сессия]
    Reset -.-> Fresh
    Check -->|нет| Bad([silent divergence<br/>тихие баги])

Ключевая идея: drift накапливается всегда — вопрос только в том, заметите ли вы его раньше, чем он успеет навредить. Durable artifacts (plan + journal + atomic commits) — это и есть точки, где можно безопасно сбросить и продолжить.

Самые частые виды:

1. File drift

Агент думает, что файл src/auth.ts содержит функцию verifyToken, потому что прочитал его в начале сессии. Но вы (или другой агент) с тех пор поменяли файл. Агент работает по устаревшему контексту.

Как ловить: перед каждой серьёзной правкой агент должен перечитать файл, не работать по кешу. В Claude Code это поведение встроено (Read перед Edit). В других — проверяйте.

Как избегать: не работайте параллельно с одним файлом из разных сессий. Если приходится — git status / git diff перед каждым серьёзным шагом.

2. Schema drift

Агент закешировал что в БД таблица users имеет колонки X, Y, Z. Реально там уже X, Y, Z, W — добавили миграцию. Агент пишет код по старой схеме.

Как ловить: периодически давайте агенту команду «перечитай актуальную схему БД». Не полагайтесь на запомненное.

3. API drift

Агент закешировал что endpoint /api/users возвращает поле email. Реально оно теперь contact_email. Все запросы агента ломаются.

Как ловить: для критичных интеграций — версионирование схем и проверка соответствия.

4. Config drift

Агент работал по конфигу A. Вы изменили на B. Агент продолжает по A (не перечитал).

Как ловить: конфиг — это stateful resource, перечитывайте при существенных переходах.

5. Process drift

Самый болезненный. Раньше вы делали X перед Y. Команда договорилась: теперь надо делать X1 → Y. Агент следует старой памяти, делает X → Y.

Как ловить: см. M9 — pruning. Память должна обновляться, иначе она и есть источник drift.

Detecting drift: автоматически и руками

Автоматически

  • Pre-flight checks. В начале серьёзного шага агент вызывает скрипт, который верифицирует базовые инварианты («тесты до моих правок зелёные? Схема БД соответствует ожидаемой? Версия зависимости такая?»).
  • Hooks (M6) — PreToolUse hook может проверить что mtime файла не изменилось с момента последнего read.
  • CI — каждый PR прогоняет тесты; если они падают на «не моих» причинах, это drift.

Руками

  • Регулярные «ситуационные брифинги» в начале сессии. Прежде чем дать задачу — спросите агента «прочитай актуальное состояние X, Y, Z и сделай резюме». Если резюме отличается от ваших ожиданий — drift.
  • Diff session memory vs reality — иногда полезно явно спросить «что ты сейчас думаешь о состоянии Z?» и сравнить с git/db/конфигом.

Correcting drift

Когда drift обнаружен:

  1. Стоп. Не продолжайте работу с этим контекстом.
  2. Сбросьте closed/cached state — закройте сессию, начните новую с чистого листа.
  3. Перечитайте всё актуальное — файлы, конфиги, схемы, что относится к задаче.
  4. Update memory. Если drift возник из устаревшей записи в long-term memory — обновите её до возврата к задаче, не после.
  5. Только тогда — продолжайте работу.

Это разрушительно для скорости — каждый drift-incident отбрасывает на 10-30 минут. Поэтому предотвращение (durable artifacts + регулярная сверка) дешевле, чем коррекция.

Cross-tool / cross-env state sharing

Современная реальность: над одним репо работают несколько разных агентов в разных средах. Один разработчик в Claude Code на macOS, второй в Cursor на Windows, CI-pipeline запускает Codex CLI в Docker. И все они «помнят» проект по-своему.

Что обязательно должно быть общим (через git)

✅ CLAUDE.md + симлинк AGENTS.md + .cursor/rules/ ✅ .claude/settings.json (project-level) — для тех кто на Claude Code ✅ .codex/config.toml (project, если применимо) ✅ .mcp.json — список подключённых MCP-серверов ✅ .agent/plans/ и .agent/journals/ (durable artifacts) ✅ Hooks-скрипты в .claude/scripts/ или .cursor/hooks/ ✅ ADR-файлы в docs/adr/

Что НЕ должно ходить в git

❌ .claude/settings.local.json — личная отсебятина ❌ ~/.claude/memory/ — user-scope memory, личное ❌ Session history (~/.claude/projects/, ~/.codex/history/) ❌ Локальные секреты / .env

Проблема: cross-tool semantic gap

Cursor может не понимать .claude/settings.json, а Claude Code — не читать .cursor/rules/ по умолчанию. Что делать?

Минимальный подход: один AGENTS.md (или CLAUDE.md) в корне как single source of truth. Симлинки на нужные имена. .cursor/rules/ может содержать одну строчку apply_from: ../../CLAUDE.md (в новых версиях Cursor поддерживается).

Расширенный подход: скрипт .agent/sync.sh, который пере-генерирует tool-specific конфиги из общего источника. Запускается в pre-commit hook.

Antipattern: «у каждого разработчика свой агент»

Самый тяжёлый из всех. Каждый член команды настроил агента «под себя», все правила в user-scope memory, в проектном репо — ничего.

Что произойдёт:

  • Новый человек приходит — ничего не работает «как надо», правила не знает.
  • Один человек уходит в отпуск — его «знание» уходит с ним.
  • Code review агентских PR непредсказуемо: один агент пишет так, другой иначе, неясно «правильно» это или нет.

Решение: проектная память — в репо, проектные настройки — в репо, проектные hooks — в репо. User-scope — только для личных предпочтений, которые не касаются проекта (например, ваш стиль коммитов).

Если коротко по всем десяти модулям Part I-III:

Агент = контрибьютор, работающий по контракту, под наблюдением, с явными границами, с памятью под версионированием и со state, который переживает любые сбои.

Это архитектурная формула, не «как настроить тул». Конкретные настройки Claude Code 2026 года устареют. Конкретные имена файлов поменяются. Сами модели будут другими.

Формула — нет. Контракт, наблюдение, границы, память, state — это дисциплинарные элементы, которые лежат в основе любой нормальной системы с агентами, человеческими или machine.

В Part IV мы будем заниматься тем, как оценивать что агент сделал хорошо или плохо (success criteria и evaluation). Это та же тема, ровно один уровень глубже.

Тренируйте насмотренность, а не пальцы. Ваша голова важнее чем ИИ — особенно при оценке «работает или нет».

Recap

  • Resume восстанавливает диалог, не мир. Полагайтесь на durable artifacts, не только session state.
  • Durable artifacts: plans, journals, atomic commits после high-risk шагов, CI runs.
  • 5 видов drift: file, schema, API, config, process. Каждый ловится по-своему.
  • Detection: pre-flight checks + hooks + регулярные «брифинги».
  • Correction: стоп → reset → reread → update memory → продолжать.
  • Cross-tool sharing: в git — конвенции, конфиги, MCP, hooks, ADR. Не в git — личная отсебятина, secrets, session history.
  • Антипаттерн: «у каждого свой агент» — проектные правила должны быть в репо.

Drift и continuity — это, по сути, разговор про управление окном контекста. Поэтому короткий список — про кеш/компакцию у Anthropic и про stateful-агентов как референс.

Контекст и компакция

  • Anthropic — Prompt caching — platform.claude.com/docs/en/build-with-claude/prompt-caching. Что именно кешируется, как компакция переиспользует тот же префикс. Поможет понять, почему «маленькая правка CLAUDE.md» инвалидирует половину кеша.
  • OpenTelemetry — Semantic conventions for GenAI — opentelemetry.io/docs/specs/semconv/gen-ai/. Какие метрики и атрибуты класть в трейсы, чтобы drift был виден в наблюдалке, а не «всплыл» через PR-инцидент.

Stateful agents и durable state

  • Letta — концепция MemGPT — docs.letta.com/concepts/memgpt/. Эталон того, как разделить «горячий» контекст и «холодный» archival store, чтобы resume был не «прочитал диалог», а «вспомнил мир».

Совет: если drift у вас не разовый, а системный — начните не с очередного хука, а с того, чтобы все длинные задачи писали journal в .agent/journals/. Половина drift’ов отпадает.
