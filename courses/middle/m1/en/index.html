M1 · What is agentic AI in the SDLC

Plan → Act → Evaluate, the contributor model, and why an agent is a colleague, not a reference manual.

Why this chapter, if you’re already shipping prod

Hey folks, welcome 👋

If you’re a middle developer with two years in production repos, I’ve got bad news: everything you know about IDE assistants and autocomplete from the last six months is out of date. Not because the tools got worse — because the paradigm shifted.

The AI tool used to be a reference manual with a hand — it suggested a line, completed a function, explained an API. It fundamentally couldn’t open a PR, run tests, refactor three files in a row, notice that the build broke, and fix it.

Now it can. And not one or two experimental tools — the whole zoo: Claude Code, Cursor, Codex CLI, Devin, GitHub Copilot Workspace, Aider, Continue. In 2026 each of them isn’t an “assistant,” it’s a contributor. They work through the same interface you do: files, branches, PRs, reviews.

This chapter is about how to stop thinking in terms of “help in code” and start thinking in terms of “how do I build a team of humans and agents and keep control over the repo.”

The three phases of an agent: plan → act → evaluate

In any sensibly structured agentic pipeline there are three mandatory phases that sit on top of the usual SDLC — the agent assists at every step, but the cycle stays recognisable:

flowchart LR
    Task([Task from you]) --> Plan[PLAN<br/>agent describes<br/>what and in what order]
    Plan -->|you approved| Act[ACT<br/>agent changes<br/>code / tests / PR]
    Act --> Eval[EVALUATE<br/>tests, linter,<br/>success criterion]
    Eval -->|ok| Ship([Merge / ship])
    Eval -->|did not converge| Replan([Revise / replan])
    Replan -.-> Plan
  1. Plan. The agent states what exactly it’s about to do, in what order, under what assumptions, before it changes anything. It’s an explicit artifact — a markdown plan, JSON, structured output.
  2. Act. The agent executes the plan in a clearly bounded scope: a branch, a sandbox, a set of allowed tools. Each action is a separate step with inputs/outputs.
  3. Evaluate. The agent or an external system compares the result against the success criterion. If it doesn’t converge — back to plan for revision.

The critical point: the plan is not a by-product, it’s the contract. If you skipped the plan, you skipped the moment when you could still say “stop, let’s do it differently.” Once act has started — fixes are an order of magnitude more expensive.

If something just clicked in your head — “this is waterfall with feedback” — yes, it is. The difference is that the cycle now takes minutes, not weeks. And that’s exactly why the discipline of plan/act/evaluate matters more than ever: without it the cycle degenerates into chaos at 1000 commits per hour.

A contributor, not a reference manual

The core mental model of this chapter:

An agent is a new team member on probation.

Almost everything we’ll do from here on derives from this model:

What you do with a new juniorWhat you do with an agent
You don’t hand them prod credentials on day oneYou don’t pass --dangerously-skip-permissions
You do code reviewYou read the plan and the diff before merge
You explain context and conventionsYou write CLAUDE.md / .cursorrules / AGENTS.md
You give a small task with a clear DoDYou give a clear success criterion
You ask “why did you do it that way”You request justification in the plan
You teach them to admit they don’t knowYou structure tools so it can say “I don’t know”

This isn’t an analogy for the sake of it. It’s a working management model: reuse your people-management skills, and most of the agent-work patterns become obvious immediately.

GitHub as system of record and control plane

In a world where your team has both humans and agents — you need a single source of truth. In 2026 that point is GitHub (or GitLab, or Forgejo — for the purposes of this course the difference is small).

Why GitHub:

  • Every change is visible. The PR is the artifact of the plan and execution.
  • Every change is reversible. Revert works the same for humans and agents.
  • Every change is auditable. Who did what, when, against which ticket — all in commit log, PR review, issue tracker.
  • Every change is delegatable. An agent can open a PR; a human can merge it. Or the other way — a human opens a plan, an agent finishes the work.

We’ll repeat this thought across the course until it hurts: GitHub isn’t code hosting, it’s the control plane of your hybrid human/agent team. If your agent doesn’t work through PRs, you’ve lost half the benefits.

In M3 we’ll talk about the degree-of-autonomy ladder. At the bottom — the agent only proposes a plan, everything is merged by a human. At the top — the agent opens PRs by itself, runs tests, replies to review comments. Where exactly you want to sit on that ladder is an architectural decision; it depends on risk and on the maturity of your observation system.

The anti-patterns I see most often

The four kinds of mistakes that middle developers make in the first couple of months of agentic flow honestly drive me up the wall. Let’s get them out of the way so you don’t repeat them.

1. “Give the agent prod creds, it’s smart”

No. It’s not smart — it’s capable. Those aren’t the same thing. A capable employee without context will crash prod faster than an incapable one, because the capable one has the hands. No prod creds in an agent session. Ever. More on this in M4 (least-privilege scoping).

2. “I told it in one line, it’ll figure out the rest”

The agent isn’t a telepath. If the task doesn’t state a success criterion, the agent will decide for itself what counts as “done.” And it almost always won’t be what you meant. A good request is a spec, not a command. We unpack this in M7 (prompt engineering).

3. “Why read the plan, I trust the model”

Trust isn’t the absence of checking. It’s the willingness to accept the consequences of checking. If you don’t read the plan, you don’t trust — you just closed your eyes. Those are different things.

4. “It’s a PR from the agent, it’s easier to accept than to dig in”

The most dangerous attitude. An agent PR has to go through the same review as a human PR — and in the first months of adoption, often a stricter one. In M11 (review patterns) we’ll assemble a checklist for reviewing agent code.

Your first deliberate agentic cycle

Take a minimal, safe, illustrative example. Open Claude Code in any of your repos and ask it to build a plan with no right to change anything:

claude --permission-mode plan \
  "Read the repo. Suggest three places where cognitive complexity could be reduced. \
   For each suggestion — file, reason, and rough cost in lines. \
   Do not modify any files."

What matters here:

  • --permission-mode plan is an architectural boundary, not a cosmetic one. The agent physically cannot modify a file.
  • The request contains a success criterion (“three places, with file, reason, cost”).
  • The request explicitly bounds the scope of actions (“do not modify”).

Read the plan. Estimate how well it matches your own picture of the repo. If it matches at 80 %+ — the model sees your code adequately, you can go further. If it matches at 30 % — the model lacks the right context, the problem is not the model, the problem is the context you gave it. Maybe a CLAUDE.md is needed, or extra explainer files.

This exercise is the foundation of the whole course. Reading the plan before allowing action is the skill that separates a middle developer in the agentic world from a middle developer in a world without agents.

A programmatic note (register C)

Friends.

The zoo of models and tools will grow every month. What’s Claude Code today might still be Claude Code in a year, or be something completely different, or be embedded in the IDE. You can’t memorise specific buttons and flags — they go out of date faster than you can read about them.

What won’t go stale:

  • Plan → Act → Evaluate as a discipline.
  • The contributor model as a mental frame.
  • A GitHub-like control plane as the source of truth.
  • Taste, discipline, understanding the client as the foundation.

Train your eye, not your fingers. Your head matters more than the AI.

Recap

  • The paradigm shifted: the AI tool became a contributor, not a reference manual.
  • Plan → Act → Evaluate — three mandatory phases. The plan is a contract, not a by-product.
  • Agent = new employee on probation. Manage it like a junior.
  • GitHub is the control plane. Every agentic action must be visible, reversible, auditable.
  • Four anti-patterns: prod creds, unclear request, unread plan, lenient review.
  • Baseline exercise: --permission-mode plan + explicit success criterion + reading the plan before actions.
