M2 · Architecting agent boundaries

Inputs, outputs, success criteria. How to separate plan from action so the agent doesn't break the important things.

What this chapter is about

Hey folks, welcome 👋

In M1 we agreed that an agent is a contributor, not a reference manual, and that the core of its work is the plan → act → evaluate cycle. That was the mental frame.

Now we drop one level down — to engineering. When you launch an agent not “to play around” but for a real task in a real repo, four questions show up immediately:

  1. What exactly am I feeding it as input?
  2. What exactly do I want as output?
  3. How should the agent state its plan so I can verify it before action?
  4. Where and how do I install the stop-cord if things go wrong?

The answers to these four questions are architecting boundaries. If you haven’t thought them through — it doesn’t matter how “smart” your agent is, the cycle will fall apart.

In this chapter we walk through each of the four questions with concrete patterns, code, and examples of how it all fits into a production flow.

The contract: inputs, outputs, success criteria

Any agent task is a contract. Not a request, not “help me,” not “let’s try” — a contract with three mandatory fields.

TASK CONTRACT
├── INPUTS              — what the agent has at the start
├── OUTPUTS             — what must exist when done
└── SUCCESS CRITERION   — how we know the output is valid

Inputs

This is the full set of what the agent sees: the repo, specific files, environment variables, API access, specific tools. Principle — least context, not most context: give the minimum sufficient for the task. Not “here’s the whole repo, figure it out,” but “here are these three files, this ticket, this spec.”

Why minimum:

  • Extra context = noise = more hallucinations
  • Extra context = more tokens = more money
  • Extra context = blurred responsibility (“it dug in the wrong place”)

Outputs

A concrete artifact. Not “an answer,” not “a solution,” but a physical thing: a file, a patch, a PR, a test case, a JSON. If you can’t sketch the structure of the output — the task isn’t ready for delegation.

Bad: “help with authorization.” Good: “output — a patch on auth/oauth.py + a new test in tests/test_oauth.py, both committed to branch feat/oauth-pkce.”

Success criterion

The most often-skipped field. It’s a verifiable condition by which you decide “accept” or “rollback.” Not “works,” not “looks ok,” but a concrete test:

  • All unit tests in tests/ green
  • Linter ruff check quiet
  • Function verify_pkce_challenge with the stated signature exists in auth/oauth.py
  • Test coverage on the new function ≥ 80 %

When the criterion is explicit, the agent gains the ability to say “didn’t work.” Without a criterion, the agent defaults to “done,” even if the result is garbage. Not because it lies, but because it has no measure.

Plan ↔ Reasoning ↔ Action: separate layers, not one cloth

Inside a single agent session three qualitatively different things happen:

LayerWhat it isWho watches what
ReasoningInternal model deliberation, “why I think this way”Mostly for debugging
PlanA structured list of steps with verifiable itemsThe main review artifact
ActionA concrete tool call (write_file, run_bash, etc.)The change trigger

When beginners work with an agent without separating these layers — everything is mashed into one stream of consciousness: “I’ll think — I’ll do — I’ll check — I’ll think again.” Result: you don’t know what happened and can’t stop the agent at “don’t do that.”

When you split the layers — you get gates where stops are possible. Think of them as concentric trust rings: from what’s inside (the model’s reasoning) out to what’s outside (files, network, production) — each ring has an explicit check.

flowchart LR
    R[Reasoning<br/>internal] --> G1{Gate 1<br/>need a plan?}
    G1 -->|yes| P[Plan<br/>structured]
    P --> G2{Gate 2<br/>plan ok?}
    G2 -->|approve| A[Action<br/>file / cmd / net]
    A --> G3{Gate 3<br/>result<br/>matches?}
    G3 -->|ok| Next([Next iteration])
    G3 -.-> P

At every gate a simple rule applies: the further from the centre, the more expensive the mistake. Fixing reasoning is cheap, an action on a file is more serious, an action against the network/prod is the most expensive. Architecturally you place control points from the inner ring outward, not the other way round.

  • Gate 1 (after reasoning): is a plan needed or is the task trivial?
  • Gate 2 (after plan): is the plan good, can we act?
  • Gate 3 (after action): does the result match the criterion?

In current agentic tools these gates are implemented differently:

  • Claude Code — --permission-mode plan (physical separation of plan from action), --ask-before-each-tool-use (gate before each action)
  • Cursor Composer — separate “Plan” and “Edit” modes; review changes before apply
  • Codex CLI — --approval-mode suggest vs auto-edit vs full-auto
  • GitHub Copilot Workspace — separate Spec → Plan → Implementation steps with the ability to edit in between

Critically: a tool where these layers aren’t separated isn’t an agentic tool, it’s autocomplete on steroids. Don’t use such tools for serious work.

Structured plan output

The plan must be structural, not free text. Otherwise you can’t validate it and you’ll “read into” every plan, losing 10 minutes per chapter.

Minimum plan structure (which you can require from any current agent):

goal: "Add PKCE to the OAuth flow"
context_inputs:
  - auth/oauth.py
  - docs/spec-auth-v2.md
steps:
  - id: 1
    action: "edit"
    target: "auth/oauth.py"
    intent: "Add a verify_pkce_challenge function"
    risk: "low"
    rollback: "git restore auth/oauth.py"
  - id: 2
    action: "create"
    target: "tests/test_pkce.py"
    intent: "Test for verify_pkce_challenge"
    risk: "low"
  - id: 3
    action: "run"
    target: "pytest tests/test_pkce.py"
    intent: "Verification"
    risk: "none"
success_criterion:
  - "pytest -q exit code 0"
  - "ruff check exit code 0"
  - "coverage on auth/oauth.py >= 80%"
risks_and_mitigations:
  - risk: "Refresh-token flow may break if the signature changes"
    mitigation: "Before the change — run existing tests, confirm they pass"

Every step has:

  • action — a verb (edit, create, delete, run, search)
  • target — concrete file / command / artifact
  • intent — why this step (this field is for you, not for the agent)
  • risk — assessment (low / medium / high)
  • rollback — how to roll back (if risk ≥ medium — mandatory)

When the plan is in YAML/JSON/markdown with clear structure — you spend 30 seconds checking a 10-step plan. When the plan is a paragraph of free text — you spend 5 minutes and still miss things.

Asking for such a plan is easy: add a phrase to the prompt: “Output the plan as structured YAML with fields: goal, context_inputs, steps (id/action/target/intent/risk/rollback), success_criterion, risks_and_mitigations. Do not execute anything until the plan is approved.”

Validation gates: where to install the stop-cord

Between phases there must be gates — check points. Minimum set:

Gate 1 — After plan, before action

What we check:

  • Every step in the plan matches the original task (the agent didn’t “expand scope”)
  • No risk: high steps without an explicit rollback
  • Every target file is real (the agent didn’t invent a path)
  • Success criterion is actually verifiable (not “works well”)

Who checks: you (or a senior on the team if the process is multi-party).

Gate 2 — After every risk: high action

What we check:

  • The step result matches the intent
  • Tests are still green (if they were green before)
  • No unrelated files were touched

Who checks: you or CI.

Gate 3 — Before merge

What we check:

  • All success criteria from the plan are satisfied
  • The diff is reviewed (an agent PR is still a PR)
  • No “bonus” changes outside the plan

Who checks: a human, mandatory.

Gate 4 — After deploy

What we check: observability shows the expected behaviour.

Who checks: at this level — monitoring. Details in M3.

Anti-pattern: “contract after the fact”

The most common anti-pattern of middle developers when they first build a serious pipeline: they state the success criterion after the agent has already done everything.

The logic goes: “well, let’s see how it turns out, then I’ll decide if it’s good.” This doesn’t work. Once you have a result, you subconsciously bend the criterion to fit it, because redoing the work is unappealing.

The rule: success criterion is fixed before action and doesn’t change during. If you realised the criterion is bad — roll everything back, reformulate, start over. Don’t patch on the fly.

That’s exactly the discipline without which Plan → Act → Evaluate turns into Plan → Improvise → Rationalize.

Practice: build the contract for your own task

Take a real task from your backlog that you’d like to delegate to an agent. Not a training task — one you were going to do this week anyway.

Fill the contract by template:

inputs:
  files: []
  docs: []
  envs: []
  tools_allowed: []

outputs:
  artifacts: []   # PR, files, test cases, JSON report — something measurable
  branch: ""

success_criterion:
  - ""
  - ""

forbidden:
  - "do not touch files outside inputs.files"
  - "do not run commands outside tools_allowed"

risks:
  - risk: ""
    mitigation: ""

If even one field doesn’t fill — the task is not ready to delegate. Go finish formulating it, don’t try to “explain it somehow.” That’s the most expensive mistake at this level.

Pro tip: you can ask the agent to help fill the contract before doing anything. So the first session is task formalisation, not execution. Sounds odd, saves hours.

Recap

  • Contract = inputs + outputs + success criterion. All three fields are mandatory, the criterion is fixed before action.
  • Three layers — reasoning, plan, action — must be separated, otherwise there are no points to stop at.
  • Plan structured (YAML/JSON), not free text. Fields: action, target, intent, risk, rollback.
  • Validation gates sit between phases. Minimum 4: after plan, after high-risk action, before merge, after deploy.
  • Anti-pattern “contract after the fact” — a success criterion set after the result always gets bent to fit it.
