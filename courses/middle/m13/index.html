M13 · Tuning: что менять, когда метрики не растут

Лестница улучшений — от промпта до архитектуры. В каком порядке настраивать и с чего точно НЕ начинать.

Черновик M13 в голосе DmitriyM. Закрывает Часть IV — Evaluation & Tuning.

Hook

ПЛОХО:                            ХОРОШО:

Pass rate 65% →           →       Pass rate 65% →
"Попробую сменить модель"         1. Промпт (5 минут)
                                  2. Контекст (15 минут)
                                  3. Разбиение задачи (час)
                                  4. Тулы / hooks
                                  5. Модель (последним)
                                  
рулетка                           систематично

Друзья, всем привет! 👋

Эта глава закрывает Часть IV. У вас уже есть метрики (M11) и harness (M12). Pass rate где-то между «хочется» и «реальностью». Эта глава — порядок действий, чтобы поднять этот gap.

Ловушка номер один здесь: люди начинают с того, что они слышали — «попробуем GPT-5», «давайте reasoning-режим», «может temperature крутануть». Это не оптимальный порядок. Большинство выигрышей дешевле и раньше в лестнице.

Час. Структурно, без воды.

Концепция: лестница улучшений (от дешёвого к дорогому)

┌─────────────────────────────────────────────────────┐
│  1. Промпт          5 минут       (дешевле всего)   │
│  2. Контекст        15-30 минут                     │
│  3. Разбиение        1-3 часа                       │
│  4. Тулы и hooks    несколько часов                 │
│  5. Память и state  дни                             │
│  6. Модель          часы (тестов) + $$              │
│  7. Архитектура     недели                          │
└─────────────────────────────────────────────────────┘

Правило: сначала пройдите 1-3, потом трогайте 4+. Если на 1-3 не разрулили — это сигнал, что задача либо плохо сформулирована, либо не в зоне агента.

Шаг 1. Промпт (5 минут)

Если pass rate 50-70% — в половине случаев виноват запрос, не модель. Возьмите 3 упавших кейса. Прочитайте промпт глазами. Спросите себя:

  • Есть ли ЧТО + ГДЕ + КАК ПОЙМЁМ ГОТОВЫМ (B3 формула)?
  • Указано ли, что НЕ трогать?
  • Просили ли план перед кодом (если задача нетривиальная)?
  • Указан ли формат вывода там, где это важно?

Простой эксперимент: переформулируйте промпт тщательно. Прогоните только этот кейс — ещё 3 раза. Если pass — проблема была в промпте.

Что меняется на этом уровне

СимптомПромпт-фикс
Агент лазит не тудаДобавить «Не изменяй файлы вне <dir>»
Делает «бонусные» правкиДобавить «Только запрошенные изменения, остальное не трогай»
Промахивается мимо требованийУсилить критерии «как поймём готовым», добавить юнит-тесты
Тратит контекст на болтовню«Без преамбулы, отвечай по делу»

Это самый дешёвый уровень. Большую часть «всё хуже стало» лечит он.

Шаг 2. Контекст (15-30 минут)

Если промпт ок, а агент всё равно мажет — у него не хватает контекста. Что значит «не хватает»:

  • В CLAUDE.md нет описания конвенций проекта
  • В CLAUDE.md есть слишком много конвенций, и важные тонут в шуме
  • Агент не видит связанных файлов (только users.ts, не видит users.test.ts)
  • Агент видит не то, что нужно (читает node_modules/)

Конкретные манёвры

1. Чистка CLAUDE.md — выкиньте всё, что не помогает в 90% задач. Идеальный CLAUDE.md — 2-3 страницы, не 20.

2. Hint-промпт в кейсе — если задача про auth, явно укажите:

Релевантные файлы (для контекста):
- src/auth/login.ts
- src/auth/types.ts
- tests/auth/login.test.ts

Так агент читает их сразу, не угадывает.

3. Negative context — что не читать:

Не читай node_modules/, .next/, .git/. Они не релевантны.

4. Workflow-doc — в CLAUDE.md добавьте «как у нас делается X»:

## Adding a new endpoint
1. Add route handler in src/api/<resource>.ts
2. Add types in src/types/<resource>.ts
3. Add test in tests/api/<resource>.test.ts
4. Run `npm test` to verify

Это процесс, не код. Агент использует его как рецепт.

Шаг 3. Разбиение задачи (1-3 часа)

Если задача большая (рефакторинг 5 файлов, миграция, фича на пол-дня) и стабильно падает — она слишком крупная для одного шота.

ПЛОХО (one-shot):
"Перепиши authentication
 на JWT, обнови все 12 endpoints,
 миграция БД"

ХОРОШО (этапами):
1. "Сначала покажи план — какие файлы,
    что в них менять, в каком порядке."
2. "Сделай миграцию БД и тесты на неё."  → проверить → коммит
3. "Сделай JWT util-модуль + тесты."     → проверить → коммит
4. "Перенеси endpoints на JWT (батчами по 3)." → ...

Каждый шаг — отдельный мини-кейс с своими критериями приёмки.

Когда разбиение реально помогает

  • Задача > 30 минут agent-time
  • Несколько независимых подзадач (можно отделить)
  • Высокий risk (security, миграция БД) — хотите проверять между шагами

Когда разбиение не поможет

  • Задача связная (нельзя отделить часть без поломки)
  • Каждый шаг сам по себе требует контекста всех остальных

Здесь нужно другое — см. шаг 5 (память) или 7 (архитектура).

Шаг 4. Тулы и hooks (несколько часов)

Если задача правильная, контекст норм, разбиение есть — но pass rate всё ещё низкий — смотрите на инструменты.

Манёвры

1. Дать тулы, которых не хватает — например, MCP для специфичной БД, или скрипт для линтинга. Агент чаще ошибается там, где нет удобного тула.

2. Hooks для гигиены (см. M4):

  • PreToolUse — запрет на опасные операции
  • PostToolUse — авто-форматирование, авто-тест после edit
  • Stop — финальная валидация перед «готово»

Например, hook, который после каждого edit прогоняет tsc --noEmit и возвращает ошибки агенту — заставляет фиксить сразу, а не оставлять «потом».

3. Subagents для четко очерченных подзадач — например, test-runner subagent, который только запускает тесты и репортит фейлы. Освобождает контекст основного агента.

Шаг 5. Память и state (дни)

Если задача многосессионная — агент забывает свои собственные решения между запусками. Это state drift из M10.

Манёвры:

  • Журналы решений в .agent/journals/
  • Plan-файл, который агент обновляет
  • Externalised state (M10)

Не «модель тупит», а state plumbing нет. Это инфраструктурная работа на дни.

Шаг 6. Модель (часы + $$)

Только сейчас — переключение модели. Не раньше.

Когда модель действительно поможет

  • На 1-5 уже отжали, pass rate упёрся в потолок
  • Задача в принципе сложная (системный дизайн, нетривиальный math, длинный контекст)
  • На разных моделях разная сильная сторона (Sonnet — фронтенд, Opus — архитектура, Codex — рефактор)

A/B протокол

1. Зафиксируйте текущий model_id (например, claude-sonnet-4-6)
2. Прогон harness 3 раза → среднее pass rate, std
3. Сменить на kandidate (claude-opus-4-7)
4. Прогон 3 раза → среднее, std
5. Сравните. Если delta > 2σ — значимо.

Без 3 прогонов разница может быть просто шумом. Я несколько раз радовался «новая модель лучше», прогонял ещё раз — становилось хуже. Дисперсия.

Когда модель не поможет

  • На уровнях 1-3 ещё есть, что отжимать
  • Задача в зоне слабости обеих моделей (например, медицинские диагнозы — никакая модель сейчас не Reliable)
  • Метрика не та (вы мерите токены, а не результат)

Шаг 7. Архитектура (недели)

Самый дорогой уровень. Сюда — только когда остальное реально исчерпано.

Признаки:

  • Pass rate упирается на 60-70% на любой модели и любом промпте
  • Задача в принципе не помещается в одного агента
  • Нужны специализированные агенты под разные роли

Это multi-agent территория, тема Части V (M14-M16). Декомпозиция задачи на роли (planner / executor / reviewer / etc.), оркестрация, межагентная коммуникация.

Короткая вставка ⚡

Самый частый разговор в чате курса:

«Я попробовал GPT-5, не помогло. Может, попробовать Sonnet? Или Gemini?»

— А промпт переписал? — Нет. — А контекст почистил? — Нет. — А задачу разбил? — Нет.

Это типовое. Люди сразу прыгают на «попробуем другую модель», потому что это звучит проще, чем «переформулировать запрос». На деле — это самое дорогое и самое медленное изменение. И в 70% случаев не помогает, потому что проблема раньше по цепочке.

Поэтому я в Middle-треке так настойчиво — сначала промпт, потом контекст, потом разбиение. Это не догма, это арифметика.

Пример: tuning pipeline на одной задаче

Стартовая ситуация

Кейс: «refactor controller» (см. M12). Текущий pass rate 60%. Хочу 85%.

Шаг 1 — промпт (15 минут)

Прочитал свой prompt.md. Заметил:

  • Не указано, какой стиль валидатора (class-based vs functional)
  • Не сказано, сохранять ли существующие комментарии в createUser

Добавил два предложения. Прогон 3 раза → 70%. Помогло, но мало.

Шаг 2 — контекст (20 минут)

Заметил, что агент не открывает src/validators/ (где уже есть валидаторы для других ресурсов с конвенциями).

Добавил в prompt.md:

Релевантные файлы:
- src/validators/productValidator.ts (эталон стиля)

Прогон 3 раза → 80%. Хорошее улучшение.

Шаг 3 — разбиение (10 минут)

Внутренне разделил инструкцию на 2 шага:

  1. Создать userValidator.ts с полным содержимым.
  2. Обновить createUser для использования валидатора.

Прогон 3 раза → 88%. Цель достигнута.

Не потребовалось менять модель. Не потребовалось писать hooks. Полный фарш на трёх первых уровнях.

Антипаттерны tuning

1. «Один большой experiment»

«Я поменял промпт, добавил hooks и сменил модель — стало лучше». Что именно помогло? Неизвестно. Cargo cult.

Лекарство: меняйте по одному. Замеряйте после каждого. Скучно, но знаете, что работает.

2. Tuning на random sample

«Прогнал 1 кейс, стало хуже — откатил». Один кейс — это шум.

Лекарство: минимум 3 прогонов, минимум 10 кейсов в suite.

3. Promo bias

«Anthropic выпустил новую модель, у них в маркетинге +30%». В их бенчмарках. На их задачах. На ваших — может быть +5%, может -5%, может ровно так же.

Лекарство: доверяйте своему harness. Маркетинг — это сигнал попробовать, не факт.

4. «Все ручки крутить одновременно»

«У меня прокси, hooks, custom MCP, jailbreak prompt и temperature 0.3» — а pass rate 55%. Что из этого работает?

Лекарство: минимализм в конфиге. Если базовая конфигурация даёт 60%, и хочется 85% — добивайте на 1-3. Не громоздите.

5. Optimisation против harness

Постепенно ваш harness становится «игрой»: оптимизируете под него. Pass rate растёт, реальность не улучшается.

Лекарство: периодически обновляйте harness кейсами из реальных факапов. Защита от overfit.

Финал — регистр C

Если посмотреть глобально на всю Часть IV — она про переход от «искусства» к «ремеслу». Без метрик работа с агентом — это искусство, основанное на ощущениях. С метриками — это ремесло, в котором есть процесс, поведение, тренды.

Это не отменяет интуицию. Интуиция + данные — это сильно. Интуиция без данных — это галлюцинация, которой вы доверяете. Данные без интуиции — это робот, который не видит смысла.

Tuning — это не магия. Это дисциплина: знаешь, где смотреть; знаешь, в каком порядке трогать; знаешь, как измерить, помогло или нет. Через полгода такой практики вы становитесь профессионалом — человеком, у которого есть процесс, а не «по настроению».

Это и есть Middle-уровень в работе с агентами. Дальше — Multi-agent, где мы строим системы из агентов, а не просто настраиваем одного.

Recap

  • Лестница улучшений: промпт → контекст → разбиение → тулы/hooks → память → модель → архитектура. В этом порядке.
  • Сначала дёшево, потом дорого. 70% выигрышей лежит на первых трёх ступенях.
  • A/B всё, что меняете. Одна переменная за раз, минимум 3 прогона, harness как референс.

Скептик Самурай

🥷 Скептик Самурай спрашивает: «Это всё интересно, но я чувствую, что новая модель лучше. Зачем мне твой harness — я и так вижу.»

Ответ: «Чувствую» — это первая ступень в дискуссии, которую серьёзные разработчики ведут с самим собой каждую неделю. «Я чувствую, что лучше» — нормальный импульс. «У меня есть числа, и я знаю, что лучше» — это позиция, которую сложно оспорить (и в команде, и через две недели, когда захочется откатиться). Я бывал на обеих сторонах: на стороне «я чувствую» — терял время, на стороне «у меня есть числа» — экономил. Цена harness — три часа. Цена ошибки без harness — недели на не той модели. Считается. Не идеально, но реально работает.

Дальше

В Части V (M14-M16) мы переходим к multi-agent — системам, где несколько агентов работают вместе. Когда это нужно, когда избыточно, как организовать коммуникацию, как избежать хаоса.

В M14 — Один агент, много шагов: пайплайны и оркестрация разберём, как из одного агента сделать пайплайн — последовательность специализированных вызовов, каждый со своей ролью. Это первая ступень перед multi-agent.

Если зашло — поделитесь главой с тем, кто «настраивает агента, но не знает, что трогать». Если у вас есть свой tuning-кейс — расскажите в комменты, разберём порядок. Ваша обратная связь реально помогает шлифовать материал.
