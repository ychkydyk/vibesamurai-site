M13 · Tuning: what to change when metrics don't move

The improvement ladder — from prompt to architecture. The order to tune in, and where definitely NOT to start.

Hook

BAD:                              GOOD:

Pass rate 65% →           →       Pass rate 65% →
"I'll try a different model"      1. Prompt (5 min)
                                  2. Context (15 min)
                                  3. Task decomposition (1h)
                                  4. Tools / hooks
                                  5. Model (last)

a coin toss                       systematic

Hey folks, welcome 👋

This chapter closes Part IV. You already have metrics (M11) and a harness (M12). Pass rate sits between “what we’d like” and “reality.” This chapter is the order of operations to lift that gap.

Trap number one here: people start with what they heard about — “let’s try GPT-5,” “let’s flip reasoning mode,” “maybe twiddle temperature.” That’s not optimal. Most wins are cheaper and earlier on the ladder.

An hour. Structured, no fluff.

Concept: the improvement ladder (cheap → expensive)

┌─────────────────────────────────────────────────────┐
│  1. Prompt          5 minutes        (cheapest)     │
│  2. Context         15–30 minutes                   │
│  3. Decomposition    1–3 hours                      │
│  4. Tools and hooks  several hours                  │
│  5. Memory and state days                           │
│  6. Model            hours (tests) + $$             │
│  7. Architecture     weeks                          │
└─────────────────────────────────────────────────────┘

Rule: first run through 1–3, then touch 4+. If 1–3 didn’t fix it — that’s a signal that the task is either poorly framed or outside the agent’s zone.

Step 1. Prompt (5 minutes)

If pass rate is 50–70 % — in half the cases the culprit is the request, not the model. Take 3 failing cases. Read the prompt by eye. Ask yourself:

  • Is there WHAT + WHERE + HOW WE KNOW IT’S DONE (the B3 formula)?
  • Is what NOT to touch stated?
  • Did you request a plan before code (for non-trivial tasks)?
  • Is the output format specified where it matters?

Simple experiment: rewrite the prompt carefully. Run only that case — 3 more times. If it passes — the problem was the prompt.

What changes at this level

SymptomPrompt fix
Agent edits the wrong placeAdd “Do not modify files outside <dir>”
Makes “bonus” editsAdd “Only the requested changes, leave the rest alone”
Misses requirementsSharpen “how we know it’s done,” add unit tests
Wastes context on chat”No preamble, answer to the point”

This is the cheapest level. Most “everything got worse” is fixed here.

Step 2. Context (15–30 minutes)

If the prompt is fine and the agent still misses — it’s short on context. What “short” means:

  • CLAUDE.md doesn’t describe the project’s conventions
  • CLAUDE.md has too many conventions, and important ones drown in noise
  • The agent doesn’t see related files (only users.ts, not users.test.ts)
  • The agent sees the wrong thing (reads node_modules/)

Concrete moves

1. Clean CLAUDE.md — drop anything that doesn’t help in 90 % of tasks. The ideal CLAUDE.md is 2–3 pages, not 20.

2. Hint prompt in the case — if the task is about auth, name relevant files explicitly:

Relevant files (for context):
- src/auth/login.ts
- src/auth/types.ts
- tests/auth/login.test.ts

The agent reads them upfront instead of guessing.

3. Negative context — what not to read:

Don't read node_modules/, .next/, .git/. Not relevant.

4. Workflow doc — in CLAUDE.md add “how we do X”:

## Adding a new endpoint
1. Add route handler in src/api/<resource>.ts
2. Add types in src/types/<resource>.ts
3. Add test in tests/api/<resource>.test.ts
4. Run `npm test` to verify

That’s process, not code. The agent uses it as a recipe.

Step 3. Task decomposition (1–3 hours)

If the task is big (refactor 5 files, migration, a half-day feature) and consistently fails — it’s too big for one shot.

BAD (one-shot):
"Rewrite authentication to JWT,
 update all 12 endpoints,
 DB migration"

GOOD (in stages):
1. "First show me the plan — which files,
    what to change, in what order."
2. "Do the DB migration and tests for it."  → verify → commit
3. "Build the JWT util module + tests."     → verify → commit
4. "Move endpoints to JWT (batches of 3)."  → ...

Each step is a separate mini-case with its own acceptance criteria.

When decomposition really helps

  • Task > 30 minutes of agent time
  • Several independent subtasks (separable)
  • High risk (security, DB migration) — you want to verify between steps

When decomposition won’t help

  • The task is tightly coupled (you can’t separate a piece without breaking the rest)
  • Each step on its own needs the context of all the others

Here you need something else — see step 5 (memory) or 7 (architecture).

Step 4. Tools and hooks (several hours)

If the task is right, context is fine, decomposition is in place — but pass rate is still low — look at the tooling.

Moves

1. Provide missing tools — e.g. an MCP for a specific DB, or a linting script. The agent misses more often where a convenient tool is missing.

2. Hooks for hygiene (see M4):

  • PreToolUse — block dangerous operations
  • PostToolUse — auto-format, auto-test after edit
  • Stop — final validation before “done”

For example, a hook that after every edit runs tsc --noEmit and returns errors to the agent — forces fixing immediately, not “later.”

3. Subagents for clearly bounded subtasks — e.g. a test-runner subagent that only runs tests and reports failures. Frees the main agent’s context.

Step 5. Memory and state (days)

If the task is multi-session — the agent forgets its own decisions between runs. That’s state drift from M10.

Moves:

  • Decision journals in .agent/journals/
  • A plan file the agent updates
  • Externalised state (M10)

Not “the model is dumb” — state plumbing is missing. Infrastructure work taking days.

Step 6. Model (hours + $$)

Only now — model switching. Not earlier.

When the model actually helps

  • 1–5 are wrung out, pass rate hits a ceiling
  • The task is fundamentally hard (system design, non-trivial math, long context)
  • Different models have different strengths (Sonnet — frontend, Opus — architecture, Codex — refactor)

A/B protocol

1. Pin the current model_id (e.g. claude-sonnet-4-6)
2. Run the harness 3 times → mean pass rate, std
3. Switch to a candidate (claude-opus-4-7)
4. Run 3 times → mean, std
5. Compare. If delta > 2σ — significant.

Without 3 runs the difference may just be noise. I’ve cheered “new model is better” several times, run again, got worse. Variance.

When the model won’t help

  • 1–3 still have juice to squeeze
  • The task is in a weakness zone of both models (e.g. medical diagnoses — no model is currently reliable)
  • The wrong metric (you measure tokens, not outcome)

Step 7. Architecture (weeks)

The most expensive level. Get here only when everything else is genuinely exhausted.

Signs:

  • Pass rate stalls at 60–70 % on any model and any prompt
  • The task fundamentally doesn’t fit one agent
  • You need specialised agents for different roles

That’s multi-agent territory, Part V (M14–M16). Decomposing the task into roles (planner / executor / reviewer / etc.), orchestration, inter-agent communication.

A short aside ⚡

The most common course-chat conversation:

“I tried GPT-5, didn’t help. Maybe try Sonnet? Or Gemini?”

— Did you rewrite the prompt? — No. — Did you clean context? — No. — Did you decompose the task? — No.

This is typical. People immediately jump to “let me try a different model” because it sounds simpler than “reframe the request.” In fact it’s the most expensive and slowest change. And in 70 % of cases it doesn’t help, because the problem is earlier in the chain.

That’s why in the Middle track I’m so insistent — prompt first, then context, then decomposition. It’s not dogma, it’s arithmetic.

Example: tuning pipeline on one task

Starting point

Case: “refactor controller” (see M12). Current pass rate 60 %. I want 85 %.

Step 1 — prompt (15 minutes)

Re-read my prompt.md. Noticed:

  • Didn’t say which style of validator (class-based vs functional)
  • Didn’t say whether to keep existing comments in createUser

Added two sentences. 3 runs → 70 %. Helped, modestly.

Step 2 — context (20 minutes)

Noticed the agent didn’t open src/validators/ (where other validators with conventions already live).

Added to prompt.md:

Relevant files:
- src/validators/productValidator.ts (style reference)

3 runs → 80 %. Solid improvement.

Step 3 — decomposition (10 minutes)

Internally split the instruction into 2 steps:

  1. Create userValidator.ts with full contents.
  2. Update createUser to use the validator.

3 runs → 88 %. Goal hit.

Didn’t need to swap models. Didn’t need hooks. Full package on the first three levels.

Tuning anti-patterns

1. “One big experiment”

“I changed the prompt, added hooks, and switched the model — got better.” What exactly helped? Unknown. Cargo cult.

Cure: change one thing at a time. Measure after each. Boring, but you know what worked.

2. Tuning on a random sample

“Ran 1 case, got worse — rolled back.” One case is noise.

Cure: minimum 3 runs, minimum 10 cases in the suite.

3. Promo bias

“Anthropic shipped a new model, +30 % in their marketing.” In their benchmarks. On their tasks. On yours it might be +5 %, −5 %, or exactly the same.

Cure: trust your harness. Marketing is a signal to try, not a fact.

4. “Twist every knob at once”

“I have a proxy, hooks, a custom MCP, a jailbreak prompt, and temperature 0.3” — and pass rate 55 %. What of these actually works?

Cure: minimalism in config. If the baseline gives 60 % and you want 85 % — squeeze 1–3. Don’t pile on.

5. Optimising against the harness

Over time your harness becomes a “game”: you optimise against it. Pass rate grows, reality doesn’t improve.

Cure: periodically update the harness with cases from actual screw-ups. Overfit defence.

Finale — register C

Step back across Part IV — it’s about the shift from “art” to “craft”. Without metrics, agent work is art based on feelings. With metrics it’s a craft with a process, behaviour, trends.

This doesn’t kill intuition. Intuition + data = strong. Intuition without data = a hallucination you trust. Data without intuition = a robot that doesn’t see meaning.

Tuning isn’t magic. It’s discipline: you know where to look; you know what order to touch in; you know how to measure whether it helped. Six months of this practice — you’re a professional, someone with a process, not “by mood.”

That’s Middle level in agent work. Next — Multi-agent, where we build systems of agents, not just tune one.

Recap

  • Improvement ladder: prompt → context → decomposition → tools/hooks → memory → model → architecture. In that order.
  • Cheap first, expensive last. 70 % of wins sit on the first three rungs.
  • A/B everything you change. One variable at a time, minimum 3 runs, harness as reference.

Vibe Samurai

🥷 Vibe Samurai asks: “All interesting, but I feel the new model is better. Why do I need your harness — I can already tell.”

Answer: “I feel” is the first rung in the conversation serious developers have with themselves every week. “I feel it’s better” is a healthy impulse. “I have numbers and I know it’s better” is a position hard to dispute (with the team, and two weeks later when you want to roll back). I’ve been on both sides: on “I feel” — lost time, on “I have numbers” — saved time. Cost of the harness — three hours. Cost of a mistake without it — weeks on the wrong model. Math holds. Not perfect, but actually works.

📚 Further reading

The “improvement ladder” is my own scheme, but every rung sits on well-written third-party material. If a specific rung is stuck — open the matching source.

Prompt and context — first rungs

  • Anthropic — Prompting best practices — the official guide worth re-reading quarterly: platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices.
  • Effective context engineering for AI agents — Anthropic Engineering post, exactly about “what to put in context and in what shape”: anthropic.com/engineering/effective-context-engineering-for-ai-agents.
  • prompt-eng-interactive-tutorial — Anthropic’s interactive trainer, useful if you haven’t written prompts “by hand” in a while: github.com/anthropics/prompt-eng-interactive-tutorial.

Tools, speed, and architecture — upper rungs

  • Building agents with the Claude Agent SDK — what to change when the prompt is wrung out and you need to dig into tool/loop architecture: anthropic.com/engineering/building-agents-with-the-claude-agent-sdk.
  • SWE-bench leaderboards — real reference for which model/architecture lives where; useful before “changing the model”: swebench.com.

Tip: before reading any prompt-engineering guide, freeze your current prompt into a dated file. A month from now it’s your most honest teacher — you’ll see what changed, what helped, what didn’t.
