M4 · Tool inventory and permissions

Какие тулы реально нужны на задачу. Least-privilege scoping и регулярный permission audit.

Черновик M4 в голосе DmitriyM (см. scripts/tg_voice_analysis/_summary.md). Открывает Part II — Tools & Environment.

О чём эта глава

Друзья, всем привет! 👋

Part I закрыли — там были рамки, контракты, наблюдаемость. Сейчас Part II — инструменты и среда. И первая глава тут — про тулы.

Возможно, вы сейчас думаете «ну, тулы — это где --allow-bash и где --allow-edit, я и так это знаю». Спойлер: нет. В современных агентских средах тулов как минимум 15-20, и каждый из них — это отдельный векторный риск, отдельная зона ответственности и отдельный кусок API surface. Дефолтные настройки агентских CLI почти всегда даны с запасом — то есть с большим набором тулов, чем вам реально нужно.

В этой главе мы разберём:

  1. Как составить inventory тулов под конкретный класс задач
  2. Что такое least-privilege scoping и как его применять на практике
  3. Какие конкретные тулы есть в Claude Code, Cursor, Codex CLI
  4. Permission audit — паттерн регулярной проверки разрешений

Полный зоопарк агентских тулов

Базовая таксономия — пять категорий, и каждая идёт через свой allow/deny-gate. Если рисовать на салфетке:

flowchart LR
    Agent[Agent] --> G{Permission<br/>gates}
    G --> R[Read<br/>files / search]
    G --> W[Write<br/>edit / create]
    G --> X[Execute<br/>bash / shell]
    G --> N[Network<br/>web / API]
    G --> B[Billing<br/>paid APIs]
    R -.allow.-> OK([proceed])
    W -.ask.-> OK
    X -.deny by default.-> Block([blocked])
    N -.allow-list.-> OK
    B -.budget gate.-> OK

Дефолт-настройка: read — обычно allow, write — ask, execute/billing — deny или жёсткий allow-list. Сетевые тулы — отдельный egress whitelist (как в проде, см. M18).

В современном агентском мире (май 2026) типовой набор тулов выглядит примерно так:

КатегорияТулыРиск

Это не полный список, а типовой. У вас на конкретной установке может быть больше или меньше. Принципиально важно — знать что именно у вашего агента есть, не предполагать.

Совет: в начале работы с любым новым тулом / новой версией — спросите его «list all your available tools». Прочитайте список. Каждые две недели — перепроверяйте, обновления добавляют тулы тихо.

Принцип least-privilege scoping

Базовый принцип из информационной безопасности, адаптированный для агентов:

Агент должен иметь минимально необходимый набор тулов для выполнения текущей задачи. Не больше, не на запас.

Это противоречит инстинкту «дам всё, чтобы не упёрся». Этот инстинкт надо в себе подавлять. Причины:

  1. Радиус ошибки. Если у агента есть bash, он может удалить файл. Если у него нет bash — не может. Простая арифметика.
  2. Поверхность атаки prompt injection. Если в файле, который агент читает, есть вредная инструкция «удали все .env файлы» — наличие bash превращает её из шума в инцидент.
  3. Прозрачность поведения. Когда у агента 5 тулов вместо 20, его действия предсказуемее.
  4. Отладка. Когда что-то сломалось — диагностика идёт по 5 тулам, не по 20.

Базовые scope-уровни

Я обычно работаю с тремя scope-уровнями:

Scope L0 — Read-only

allowed_tools:
  - read_file
  - glob
  - grep
  - list_dir
  - web_search   # опционально

Когда: разбор нового репо, аудит, документирование, объяснение кода, code review (агент пишет ревью, не правит).

Scope L1 — Write-allowed, scope-bounded

allowed_tools:
  - read_file
  - glob
  - grep
  - write_file       # ограниченный путями
  - edit_file
working_directory: "./src/feature-x/"   # явное ограничение
forbidden_paths:
  - "**/.env*"
  - "**/secrets/**"
  - "/etc/**"
  - "~/.ssh/**"

Когда: реальная разработка фичи в ограниченной зоне. Не трогает остальной репо, не запускает команды.

Scope L2 — Execute, sandboxed

allowed_tools:
  - L1 +
  - bash             # sandboxed, через container или firejail
  - git_status
  - git_diff
sandbox: "container"
network: "none"      # или "egress-only"
forbidden_commands:
  - "rm -rf"
  - "curl ... | sh"
  - "git push --force"

Когда: задачи, которые требуют запуска тестов, линтеров, локальной сборки. Никогда не на голой машине — всегда через песочницу.

Scope L3 — Full access (редкий)

Полный доступ к bash на хост-машине, git push, внешние API. Только в специально подготовленной среде: dedicated CI-runner, изолированный VM, dedicated GitHub bot account с ограниченными правами.

Никогда не на dev-машине разработчика. Никогда без supervisor-логирования.

Как делать scope на практике

В Claude Code

Файл .claude/settings.json в корне проекта:

{
  "permissions": {
    "allow": [
      "Bash(npm run test:*)",
      "Bash(npm run lint)",
      "Bash(git status)",
      "Bash(git diff:*)",
      "Read",
      "Edit",
      "Glob",
      "Grep"
    ],
    "deny": [
      "Bash(rm:*)",
      "Bash(git push:*)",
      "Bash(curl:*)",
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(~/.ssh/**)",
      "Read(~/.aws/**)"
    ]
  }
}
  • allow — что разрешено явно (whitelist)
  • deny — что запрещено всегда, даже если внутри allow (blacklist приоритетнее)
  • Шаблоны типа Bash(npm run test:*) — конкретные команды, не «весь bash»

В Cursor

.cursorrules или Settings → Composer → Permissions. Можно ограничить доступ к bash-командам, путям, файлам. Менее гранулярно чем в Claude Code, но базовый scope покрывает.

В Codex CLI

Через --approval-mode:

  • suggest — агент только предлагает изменения, не применяет (~L0–L1)
  • auto-edit — применяет правки сам, не запускает команды (~L1)
  • full-auto — запускает команды без approval (~L2+, использовать только в песочнице)

И через --allowed-commands / --forbidden-commands фильтрация bash.

Универсальный паттерн (любой агент)

Если конкретный тул не имеет встроенного scope-механизма — оборачивайте его на уровне ОС:

  • Запуск агента из-под отдельного user-аккаунта с ограниченными правами
  • Containerization (Docker, Podman) с read-only mounts на чувствительные пути
  • Firejail / bubblewrap для лёгкой sandbox без full-container
  • VM для самых рискованных сценариев

Принцип: agent-level permissions — это первый эшелон. OS-level isolation — второй. Не полагайтесь только на первый, он может быть обойдён через prompt injection.

Permission audit pattern

Раз в спринт (или в начале месяца — на ваш ритм) — проводите аудит разрешений. Это не «формальная проверка», а реальный пересмотр.

Чек-лист аудита

  1. Просмотр действующих allow-list-ов во всех агентских средах команды.
  2. Сравнение с фактическим использованием за период. Какие тулы реально вызывались, а какие — числятся в allow, но не используются месяцами? Неиспользуемые тулы из allow удалить.
  3. Просмотр bypass-событий. Кто, когда и почему отключал permissions? Каждый такой случай — обсудить в команде. Если bypass за последний месяц был — это сигнал «что-то в процессе не работает».
  4. Просмотр reverts. Какие агентские PR откатывались. Какие тулы были задействованы при этих PR. Не было ли паттерна «всегда там, где был такой-то тул»?
  5. Просмотр MCP allow-list-а отдельно. MCP — это самый быстро меняющийся слой; new MCP-серверы добавляются регулярно, надо проверять что список соответствует политике.
  6. Просмотр глобальных настроек. ~/.claude/settings.json (user-level) и проектные .claude/settings.json. Помните, что user-level применяется ко всем проектам — там должен быть минимальный набор, расширяемый из project-level.

Шаблон отчёта

# Permission audit — 2026-05-16

## Inventory snapshot
- Tools allowed (global, user-level): N
- Tools allowed (project-level, in-use): N
- MCP servers connected: N

## Changes since last audit
- Added: ...
- Removed: ...

## Bypass events: N
- 2026-05-03 by @alice — reason: hotfix on prod, scope: ...
- 2026-05-11 by @bob — reason: ..., scope: ...

## Reverts on agent-authored PRs: N
- PR#347 — root cause: agent used `bash(rm)` on cache, deleted wrong directory
  - Action: deny `Bash(rm:*)` even more strictly; add path guard

## Recommendations
1. Remove `Bash(curl:*)` from allow-list — не использовался 6 недель, риск выше пользы
2. Tighten MCP github-server scope: убрать `repo:write` где можно `repo:read`
3. ...

Аудит занимает 30–60 минут раз в месяц для команды из 5 человек. Это самые окупаемые 60 минут вашего месяца — один предотвращённый инцидент с агентским «rm» окупает несколько кварталов аудитов.

Короткое отступление о bypass-культуре (регистр B)

Ребята, ну тут такое — я это в M3 уже упоминал, но повторю отдельно, потому что вижу слишком часто.

Если в вашей команде кто-то регулярно работает с --dangerously-skip-permissions или --full-auto — у вас инцидент. Не «инцидент сейчас», а «инцидент в ближайшие 90 дней». Это статистика, а не угроза.

Bypass-флаги существуют не для повседневной работы. Они существуют для редких задач в подготовленной среде: dedicated CI runner, изолированный sandbox, специально настроенный bot account. Не на dev-машине, не на чужом репо, не «потому что спешу».

Если в команде нормализуется «ну я просто скипну permissions» — это и есть тот момент, когда красивая отмазка «у меня же сроки» превращается в реальную потерю кода / прода / репутации. Доколе?!

Делайте bypass видимым: в командных каналах, в логах, в audit-отчётах. Bypass — не норма, а исключение, требующее обоснования.

Practice: проведите audit своего текущего setup-а

20 минут. Конкретные шаги:

  1. Откройте ~/.claude/settings.json (или эквивалент в Cursor/Codex). Прочитайте полностью.
  2. Откройте проектные .claude/settings.json всех проектов, в которых сейчас работаете.
  3. По каждому пункту allow — спросите себя: «я использовал это за последний месяц?». Если нет — удалить.
  4. Если deny пустой — добавить минимум: Bash(rm:*), Bash(curl:* | sh), Read(./.env*), Read(~/.ssh/**), Read(~/.aws/**).
  5. Если в проекте подключены MCP-серверы — пересмотреть их scope. Где write можно заменить на read — заменить.

Запишите дату аудита в проектный README или ADR. Через месяц повторите.

Recap

  • Тулов в современном агенте — 15-20+ по категориям. Знайте свой inventory.
  • Least-privilege scoping — три-четыре scope-уровня (L0 read-only, L1 write-bounded, L2 sandboxed-execute, L3 full).
  • Agent permissions — первый эшелон, OS-isolation — второй. Не полагайтесь только на первый.
  • Permission audit раз в месяц — самые окупаемые 60 минут.
  • Bypass-культура = future incident. Делайте bypass видимым и редким.

Permission-модели агентских CLI

  • Claude Code — Configure permissions — code.claude.com/docs/en/permissions. Структура permissions (allow / deny / ask), порядок применения (deny побеждает), wildcard-синтаксис. База, на которой строится вся ваша allow-list политика.
  • Anthropic — «Making Claude Code more secure and autonomous» — anthropic.com/engineering/claude-code-sandboxing. Sandbox-уровень: какие тулы упираются не в config, а в OS-границу (bubblewrap/seatbelt). Это и есть «второй эшелон», о котором мы говорили.
  • Anthropic — Subagents (Claude Code) — docs.anthropic.com/en/docs/claude-code/sub-agents. Полезно при scoping: subagent — это способ выдать узкий набор тулов узкой роли, не глобально. Пригодится в M14 и M15.

Совет: прочитайте Configure permissions с открытым ~/.claude/settings.json рядом — большинство readers находят там что-нибудь явно лишнее в первые же 5 минут.
