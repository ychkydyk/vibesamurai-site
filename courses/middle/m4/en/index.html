M4 · Tool inventory and permissions

Which tools you actually need for a task. Least-privilege scoping and regular permission audits.

What this chapter is about

Hey folks, welcome 👋

Part I is closed — that was frames, contracts, observability. Now Part II — tools and environment. And the first chapter here is about tools.

You’re probably thinking “well, tools — that’s where --allow-bash is and where --allow-edit is, I know this.” Spoiler: no. In current agentic environments there are at least 15–20 tools, and each is a separate risk vector, a separate responsibility zone, and a separate slice of API surface. Default agent CLI settings almost always come with a surplus — that is, with a larger tool set than you actually need.

In this chapter we’ll cover:

  1. How to assemble a tool inventory for a specific class of tasks
  2. What least-privilege scoping is and how to apply it in practice
  3. The concrete tools available in Claude Code, Cursor, Codex CLI
  4. Permission audit — the pattern of regular permission review

The full zoo of agent tools

Base taxonomy — five categories, each gated by its own allow/deny. On a napkin:

flowchart LR
    Agent[Agent] --> G{Permission<br/>gates}
    G --> R[Read<br/>files / search]
    G --> W[Write<br/>edit / create]
    G --> X[Execute<br/>bash / shell]
    G --> N[Network<br/>web / API]
    G --> B[Billing<br/>paid APIs]
    R -.allow.-> OK([proceed])
    W -.ask.-> OK
    X -.deny by default.-> Block([blocked])
    N -.allow-list.-> OK
    B -.budget gate.-> OK

Default config: read — usually allow, write — ask, execute/billing — deny or a strict allow-list. Network tools — separate egress whitelist (as in prod, see M18).

In the current agentic world (May 2026) the typical tool set looks roughly like this:

CategoryToolsRisk

This isn’t an exhaustive list, it’s typical. Your specific install may have more or less. The critical point — know what your agent actually has, don’t assume.

Tip: on starting with any new tool / new version — ask it “list all your available tools.” Read the list. Every two weeks — recheck; updates add tools quietly.

The least-privilege scoping principle

Base principle from infosec, adapted for agents:

The agent must have the minimum set of tools required for the current task. Not more, not “just in case.”

This contradicts the instinct of “I’ll give it everything so it doesn’t get stuck.” Suppress that instinct. Reasons:

  1. Blast radius. If the agent has bash, it can delete a file. If it doesn’t have bash — it can’t. Simple arithmetic.
  2. Prompt injection attack surface. If a file the agent reads contains a malicious instruction “delete all .env files” — the presence of bash turns it from noise into an incident.
  3. Behaviour transparency. With 5 tools instead of 20, the agent’s actions are more predictable.
  4. Debugging. When something breaks — diagnosis runs through 5 tools, not 20.

Base scope levels

I usually work with three scope levels:

Scope L0 — Read-only

allowed_tools:
  - read_file
  - glob
  - grep
  - list_dir
  - web_search   # optional

When: walking a new repo, audits, documentation, code explanation, code review (the agent writes review, doesn’t edit).

Scope L1 — Write-allowed, scope-bounded

allowed_tools:
  - read_file
  - glob
  - grep
  - write_file       # limited by paths
  - edit_file
working_directory: "./src/feature-x/"   # explicit limit
forbidden_paths:
  - "**/.env*"
  - "**/secrets/**"
  - "/etc/**"
  - "~/.ssh/**"

When: actual feature development inside a bounded zone. Doesn’t touch the rest of the repo, doesn’t run commands.

Scope L2 — Execute, sandboxed

allowed_tools:
  - L1 +
  - bash             # sandboxed, via container or firejail
  - git_status
  - git_diff
sandbox: "container"
network: "none"      # or "egress-only"
forbidden_commands:
  - "rm -rf"
  - "curl ... | sh"
  - "git push --force"

When: tasks that require running tests, linters, local builds. Never on a bare machine — always through a sandbox.

Scope L3 — Full access (rare)

Full bash access on the host machine, git push, external APIs. Only in a specifically prepared environment: dedicated CI runner, isolated VM, dedicated GitHub bot account with limited rights.

Never on a dev machine. Never without supervisor logging.

How to do scope in practice

In Claude Code

.claude/settings.json in the project root:

{
  "permissions": {
    "allow": [
      "Bash(npm run test:*)",
      "Bash(npm run lint)",
      "Bash(git status)",
      "Bash(git diff:*)",
      "Read",
      "Edit",
      "Glob",
      "Grep"
    ],
    "deny": [
      "Bash(rm:*)",
      "Bash(git push:*)",
      "Bash(curl:*)",
      "Read(./.env)",
      "Read(./.env.*)",
      "Read(~/.ssh/**)",
      "Read(~/.aws/**)"
    ]
  }
}
  • allow — what’s explicitly allowed (whitelist)
  • deny — what’s always forbidden, even if inside allow (blacklist wins)
  • Patterns like Bash(npm run test:*) — specific commands, not “all of bash”

In Cursor

.cursorrules or Settings → Composer → Permissions. You can limit access to bash commands, paths, files. Less granular than Claude Code, but the base scope is covered.

In Codex CLI

Via --approval-mode:

  • suggest — agent only suggests changes, doesn’t apply (~L0–L1)
  • auto-edit — applies edits itself, doesn’t run commands (~L1)
  • full-auto — runs commands without approval (~L2+, only in a sandbox)

And via --allowed-commands / --forbidden-commands bash filtering.

Universal pattern (any agent)

If a specific tool doesn’t have a built-in scope mechanism — wrap it at the OS level:

  • Run the agent under a separate user account with limited rights
  • Containerization (Docker, Podman) with read-only mounts on sensitive paths
  • Firejail / bubblewrap for a light sandbox without a full container
  • VM for the riskiest scenarios

Principle: agent-level permissions are the first line. OS-level isolation is the second. Don’t rely on the first alone — it can be bypassed via prompt injection.

Permission audit pattern

Once a sprint (or at the start of the month — pick your rhythm) — run a permission audit. Not a “formal check” — a real review.

Audit checklist

  1. Review of effective allow-lists across every agent environment in the team.
  2. Compare with actual usage over the period. Which tools were actually called, and which are in allow but unused for months? Drop the unused ones.
  3. Review bypass events. Who, when, and why turned permissions off? Discuss every such case in the team. A bypass in the last month is a signal “something in the process isn’t working.”
  4. Review reverts. Which agent PRs got rolled back. Which tools were involved. Is there a “always when this tool is used” pattern?
  5. Review the MCP allow-list separately. MCP is the fastest-changing layer; new MCP servers get added regularly, the list must match policy.
  6. Review global settings. ~/.claude/settings.json (user-level) and project .claude/settings.json. Remember: user-level applies to every project — keep it minimal, extend at the project level.

Report template

# Permission audit — 2026-05-16

## Inventory snapshot
- Tools allowed (global, user-level): N
- Tools allowed (project-level, in-use): N
- MCP servers connected: N

## Changes since last audit
- Added: ...
- Removed: ...

## Bypass events: N
- 2026-05-03 by @alice — reason: hotfix on prod, scope: ...
- 2026-05-11 by @bob — reason: ..., scope: ...

## Reverts on agent-authored PRs: N
- PR#347 — root cause: agent used `bash(rm)` on cache, deleted wrong directory
  - Action: deny `Bash(rm:*)` more strictly; add path guard

## Recommendations
1. Remove `Bash(curl:*)` from allow-list — unused for 6 weeks, risk > benefit
2. Tighten MCP github-server scope: drop `repo:write` where `repo:read` works
3. ...

The audit takes 30–60 minutes once a month for a team of 5. These are the highest-return 60 minutes of your month — one prevented agent-rm incident pays back several quarters of audits.

A short aside on bypass culture (register B)

Folks, look — I covered this briefly in M3, but I’ll repeat it separately because I see it too often.

If anyone on your team regularly works with --dangerously-skip-permissions or --full-auto — you have an incident. Not “an incident now,” but “an incident in the next 90 days.” That’s statistics, not a threat.

Bypass flags don’t exist for everyday work. They exist for rare tasks in a prepared environment: dedicated CI runner, isolated sandbox, specially configured bot account. Not on a dev machine, not on someone else’s repo, not “because I’m in a hurry.”

When a team normalises “I’ll just skip permissions” — that’s the moment the pretty excuse “but I have deadlines” turns into actual loss of code / prod / reputation. How long are we going to keep doing this?

Make bypass visible: in team channels, in logs, in audit reports. Bypass isn’t the norm — it’s an exception that requires justification.

Practice: audit your current setup

20 minutes. Concrete steps:

  1. Open ~/.claude/settings.json (or the equivalent in Cursor/Codex). Read it through.
  2. Open project .claude/settings.json in every project you’re currently working in.
  3. For each item in allow — ask yourself: “have I used this in the last month?” If not — drop.
  4. If deny is empty — add the minimum: Bash(rm:*), Bash(curl:* | sh), Read(./.env*), Read(~/.ssh/**), Read(~/.aws/**).
  5. If MCP servers are connected — revisit their scopes. Where write can be read — replace.

Note the audit date in the project README or an ADR. Repeat a month later.

Recap

  • A current agent has 15-20+ tools by category. Know your inventory.
  • Least-privilege scoping — three-four scope levels (L0 read-only, L1 write-bounded, L2 sandboxed-execute, L3 full).
  • Agent permissions are the first line, OS isolation is the second. Don’t rely on the first alone.
  • Permission audit once a month — the highest-return 60 minutes.
  • Bypass culture = future incident. Make bypass visible and rare.

Permission models of agent CLIs

  • Claude Code — Configure permissions — code.claude.com/docs/en/permissions. The permissions structure (allow / deny / ask), evaluation order (deny wins), wildcard syntax. The base your whole allow-list policy sits on.
  • Anthropic — “Making Claude Code more secure and autonomous” — anthropic.com/engineering/claude-code-sandboxing. Sandbox level: which tools hit not config but an OS boundary (bubblewrap/seatbelt). That’s the “second line” we talked about.
  • Anthropic — Subagents (Claude Code) — docs.anthropic.com/en/docs/claude-code/sub-agents. Useful for scoping: a subagent is a way to give a narrow tool set to a narrow role, not globally. Comes back in M14 and M15.
