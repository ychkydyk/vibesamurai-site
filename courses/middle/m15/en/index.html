M15 · Multi-agent: roles, communication, supervisor

Several independent agents talking to each other. When a supervisor, when a flat network, how to avoid chaos.

Hook

PIPELINE (M14):              MULTI-AGENT (M15):

planner → impl → tests       supervisor
                                 ↓
one agent                   ┌────┴────┬─────────┐
several roles             coder    reviewer   ops
                          ↑↓        ↑↓        ↑↓
                          ════════ event bus ════════

                          different processes, shared bus

Hey folks, welcome 👋

This chapter is about real multi-agent. Not “one agent in several hats” (that was M14), but several independent processes talking to each other.

Honestly: in reality you rarely need this. Most tasks close with one agent or a pipeline. Multi-agent pays off in specific scenarios — long autonomous cycles, parallel work, specialisation with a strong conflict of interests between roles. If you’re not sure you need it — you probably don’t. M16 will be entirely about anti-patterns; but we cover this chapter so you understand what’s available.

An hour and a half.

Concept: when “several roles” becomes “several agents”

In M14 we had one Claude playing different roles through subagents. That works while:

  • Roles rotate (not parallel)
  • Context between roles transfers through files
  • One source of truth (one CLAUDE.md)

Signals that a pipeline is already too small:

SignSignal
Roles must work in parallelyes
Roles use different base models (Opus for architect, Haiku for tests)yes
Roles have different contexts (different repos, different docs)yes
Roles must disagree with each other (review, adversarial)yes
The cycle is long (hours-days) with autonomous decisionsyes
One agent’s context can’t hold all rolesyes

If 2–3 of those are “yes” — time for multi-agent.

Multi-agent architectures

Most common in practice — supervisor + workers. Picture:

flowchart TB
    User([User task]) --> Sup[Supervisor<br/>planning / dispatch]
    Sup --> Coder[Coder<br/>src/*]
    Sup --> Tester[Tester<br/>tests/*]
    Sup --> Reviewer[Reviewer<br/>read-only + review.md]
    Coder --> Sup
    Tester --> Sup
    Reviewer --> Sup
    Sup --> Result([Synthesized result])

The supervisor is the only role that decides “done / redo.” Workers return results to it, not to each other. That alone removes half the “infinite debate” risks discussed below.

A. Supervisor + workers (hierarchy)

        ┌────────────────┐
        │   Supervisor    │
        │  (orchestrator) │
        └────────┬────────┘
                 │ task dispatch
       ┌─────────┼─────────┐
       ↓         ↓         ↓
   ┌────────┐ ┌────────┐ ┌────────┐
   │ Coder  │ │ Tester │ │Reviewer│
   └────────┘ └────────┘ └────────┘

Supervisor:

  • Breaks a big task into subtasks
  • Dispatches to workers
  • Gathers results
  • Makes the final “done/redo” call

Workers — narrowly specialised, minimum rights.

Fits: structured tasks with obvious decomposition (feature = code + tests + review).

B. Peer-to-peer (flat network)

   ┌────────┐ ←─────→ ┌────────┐
   │Agent A │         │Agent B │
   └────────┘ ←─────→ └────────┘
       ↕                 ↕
   ┌────────┐ ←─────→ ┌────────┐
   │Agent C │         │Agent D │
   └────────┘         └────────┘

All agents are equal, talk directly. Decisions are made locally or by voting.

Fits: creative / research tasks (brainstorm, debate). Rarely needed in development.

Danger: easy to descend into infinite dialogue with no result. You need hard termination rules (“after 5 rounds — supervisor decides”).

C. Event-driven (through a bus)

   Agent A ──→ ┌─────────────────┐ ──→ Agent C
              │   Event Bus     │
   Agent B ──→ │  (queue/pub-sub) │ ──→ Agent D
              └─────────────────┘

Agents don’t know about each other. Each listens to event types, reacts, publishes its own.

Fits: long processes, partly human workflows (CI/CD-like).

Tech: Redis pub/sub, RabbitMQ, GitHub webhooks, or just a filesystem queue (tasks/inbox/ → tasks/done/).

Communication: message format

The main rule — a shared format, without which Agent B won’t understand Agent A.

Minimum packet

{
  "from": "coder-agent-01",
  "to": "tester-agent-01",
  "task_id": "feature-y-2026-05-15",
  "type": "request_test",
  "payload": {
    "files": ["src/api/users.ts"],
    "context": "Implemented GET /api/users/me. See PR #142 for diff."
  },
  "timestamp": "2026-05-15T14:32:00Z"
}

That format is the protocol. Every agent knows it (via CLAUDE.md or system prompt). Without a format — you don’t have multi-agent, you have chaos.

Typing levels

Weak (free-form inside a JSON block): “do it how you feel, just JSON.” Medium (schema + examples): “here’s a JSON Schema, validate.” Strong (typed bus): protocol buffers, FlatBuffers, etc.

For most cases — medium. JSON Schema + parser at every agent’s input.

Supervisor: how it works

The supervisor is a separate prompt (or agent) with a different role from the workers. Its jobs:

  1. Decomposition: split the task into sub-tasks
  2. Dispatch: which worker takes which sub-task
  3. Synthesis: gather results, check consistency
  4. Decision: finish, iterate, escalate to a human

supervisor.md:

You are a supervisor agent. You do NOT write code.
You orchestrate workers (coder, tester, reviewer).

For each incoming task:
1. Decompose into 1-5 worker tasks.
2. Dispatch via .agent/queue/<worker>/<task-id>.json
3. Poll .agent/queue/<worker>/done/ for results.
4. On all-done: synthesize result, return.
5. On worker error: retry once, then escalate.

Hard rules:
- Max 5 iterations per task.
- Total budget: 30 minutes wall-clock.
- On budget exhaustion: stop, return what we have, flag for human.

It’s a fine setup. The supervisor is the hardest role in multi-agent because it has to decide about other agents. Often the first place where something goes wrong.

A short aside ⚡

A chat story — folks assembled a supervisor + 4 workers for an auto-PR factory. Launched on one task. Six hours later:

  • 47 commits in the feature branch
  • supervisor “cancelled” its own decisions three times
  • coder and reviewer fell into an infinite argument about one variable’s naming
  • $80 on the API overnight

What was done wrong: no hard limits (iteration count, $ budget, time). The supervisor ran on “the agent knows best.” It doesn’t.

Multi-agent without budgets is a deadlock machine. Always set an upper bound: iterations, time, tokens. Exceeded — stop, escalate to a human.

Which architecture to pick when

Supervisor + workers — the most common

Fits almost everywhere the task decomposes obviously:

  • Feature development: planner + coder + tester + doc-writer
  • Codebase analysis: split by subsystems, sub-agents on each
  • PR prep: code-review-tests-description-merge

Limits: max 5 workers (more — hard to debug), max 3 levels of depth.

Peer-to-peer — rare

Only when you specifically need dialogue dynamics (brainstorm, debate, generation+critique loops).

Mandatory: a hard termination rule (N rounds, or a supervisor-judge on top).

Event-driven — for long processes

When steps can be hours-days apart. CI/CD, customer tickets, on-call rotation with a first-line agent.

Don’t use when the task fits one hour of wall-clock — overhead won’t pay back.

Example: supervisor + 3 workers (minimal implementation)

.agent/queue/ layout:

.agent/queue/
├── inbox/                    # supervisor reads from here
│   └── task-001.json
├── coder/
│   ├── todo/
│   ├── doing/
│   └── done/
├── tester/
│   ├── todo/
│   ├── doing/
│   └── done/
└── reviewer/
    ├── todo/
    ├── doing/
    └── done/

runner.sh:

#!/usr/bin/env bash
set -e

# One supervisor cycle
while [ -f .agent/queue/inbox/*.json ]; do
    TASK=$(ls .agent/queue/inbox/*.json | head -1)
    TASK_ID=$(basename "$TASK" .json)

    # 1. Supervisor decomposes
    claude --print --system-prompt-file supervisor.md \
        "Decompose task in $TASK. Output subtasks to .agent/queue/{worker}/todo/$TASK_ID-{n}.json"

    mv "$TASK" .agent/queue/processing/

    # 2. Workers process in parallel
    parallel -j 3 ::: \
        "claude --print --system-prompt-file coder.md --json-mode" \
        "claude --print --system-prompt-file tester.md --json-mode" \
        "claude --print --system-prompt-file reviewer.md --json-mode"

    # 3. Supervisor synthesizes
    claude --print --system-prompt-file supervisor.md \
        "Tasks done. Synthesize result for $TASK_ID."
done

This is a simplification. In reality you also need timeouts per worker, retry policy, error handling, deduplication, persistent state. But the skeleton is this.

Multi-agent anti-patterns

1. “Too fine-grained split”

Supervisor + 8 narrow workers. Each does one function. Coordination overhead exceeds the gain from specialisation.

Cure: 2–4 workers. More — only if you actually need different models or rights.

2. Workers without hard rights

The “reviewer” has full write to the repo. Why? It only reads and writes a comment.

Cure: minimum rights per role. Reviewer — read-only + write only in .agent/reviews/.

3. Free-form communication

Workers write “how they feel.” The supervisor parses guessing. Chaos.

Cure: JSON Schema per message type. Parser at input. Invalid — drop and re-ask.

4. No limits

“Let the supervisor decide when to stop.” It won’t. Run to a budget, then stop.

Cure: hard upper bounds on iterations, time, tokens. Any breach — escalate to a human, not an autonomous decision.

5. Every agent on one model

You can. But expensive and overkill for simple roles.

Cure: Supervisor / planner — Opus. Coder — Sonnet or Opus. Tester / formatter — Haiku or Sonnet (faster, cheaper). Reviewer — Opus (for quality).

6. Multi-agent for a task that doesn’t need multi-agent

The most common. “Sounds cool, let’s build it.” Hype-driven design.

Cure: see M16, the whole chapter is about this.

Practice (90–120 minutes)

Task 1 (60 minutes) — assemble a minimal supervisor + 2 workers

Not on a live project — on a sandbox. Script + JSON queue + two subagents. Run on a simple task (e.g. “refactor function X, add a test”).

Record:

  • How long to verify the result?
  • What broke first (if anything)?
  • Was it worth it vs just a pipeline?

Task 2 (30 minutes) — message format

Describe a JSON Schema for one message type in your pipeline. Decide which fields are mandatory, which optional. This exercise is critical before attempting real multi-agent.

Optional (60+ minutes) — parallel pipeline

If you have a real task with parallel branches (e.g. migrating two independent modules), run 2 workers concurrently via parallel. Compare wall-clock vs sequential.

Recap

  • Multi-agent pays off on parallelism, different models/contexts, adversarial setups. In most cases — a pipeline is enough.
  • Architectures: supervisor+workers (most common), peer-to-peer (rare), event-driven (long processes).
  • Hard communication format + hard limits — without them multi-agent turns into a deadlock machine.

Vibe Samurai

🥷 Vibe Samurai asks: “All cool, but why me? I have a pet project, I work alone. I’ll never touch multi-agent in my life.”

Answer: And you’ll be right. Multi-agent is architectural territory, paying off on large projects with teams and production. I put this chapter in precisely so you understand that multi-agent exists and how it works — in case you land on a project where it’s actually needed (or seems needed — half the cases). But building multi-agent for yourself — overhead. A pipeline (M14) — yes, I recommend to everyone. Multi-agent — only when the pain from its absence costs more than the pain of its presence. Not perfect, but actually works.

📚 Further reading

Multi-agent is a topic with much hype and little good writing. The minimum worth aligning on:

Academic reference

  • AutoGen — paper (Microsoft, arXiv:2308.08155) — arxiv.org/abs/2308.08155. Original paper on conversable agents and multi-agent conversation programming. Still the best mental model: “dialogue between roles” as a first-class concept.

Multi-agent frameworks

  • CrewAI — docs — docs.crewai.com. Role-based crews + flows. A good way to feel what supervisor+workers looks like without your own skeleton.
  • LangGraph — Overview — docs.langchain.com/oss/python/langgraph/overview. The same DAG as in M14, but conveniently shows how state is split across several agent nodes.

Anthropic-native

  • Anthropic — Subagents — docs.anthropic.com/en/docs/claude-code/sub-agents. Limited multi-agent inside one session. Useful if you want roles without an inter-session bus.

Tip: read the AutoGen paper before installing any framework. Half the design questions are solved there, and without it you’ll be surprised by CrewAI/LangGraph’s “weird” behaviour.
