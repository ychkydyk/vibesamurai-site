M15 · Multi-agent: роли, коммуникация, supervisor

Несколько независимых агентов, общающихся между собой. Когда supervisor, когда плоская сеть, как избежать хаоса.

Черновик M15 в голосе DmitriyM.

Hook

PIPELINE (M14):              MULTI-AGENT (M15):

planner → impl → tests       supervisor
                                 ↓
один агент                  ┌────┴────┬─────────┐
несколько ролей           coder    reviewer   ops
                          ↑↓        ↑↓        ↑↓
                          ════════ event bus ════════
                          
                          разные процессы, общая шина

Друзья, всем привет! 👋

Эта глава — про настоящий multi-agent. Не «один агент в нескольких шляпах» (это был M14), а несколько независимых процессов, общающихся между собой.

Сразу честно: в реальности это редко нужно. Большинство задач закрывается одним агентом или pipeline. Multi-agent окупается на конкретных сценариях — длительных автономных циклах, параллельной работе, специализации с сильным конфликтом интересов между ролями. Если не уверены, что нужно — скорее всего не нужно. M16 будет полностью посвящена антипаттернам — но эту главу проходим для понимания, что вообще доступно.

Час-полтора.

Концепция: когда «несколько ролей» становится «несколько агентов»

В M14 у нас был один Claude, играющий разные роли через subagents. Это работает, пока:

  • Роли сменяют друг друга (не параллельно)
  • Контекст между ролями переносится через файлы
  • Один источник истины (один CLAUDE.md)

Сигналы, что pipeline уже мал:

ПризнакСигнал
Роли должны работать параллельнода
У ролей разные базовые модели (Opus для архитекта, Haiku для тестов)да
У ролей разные контексты (разные репо, разные доки)да
Роли должны спорить между собой (review, adversarial)да
Цикл — долгий (часы-дни) с автономными решениямида
Контекст одного агента не вмещает все ролида

Если 2-3 из этого — да — пора multi-agent.

Архитектуры multi-agent

Самая частая на практике — supervisor + workers. Чтобы было перед глазами:

flowchart TB
    User([User task]) --> Sup[Supervisor<br/>планирование / dispatch]
    Sup --> Coder[Coder<br/>src/*]
    Sup --> Tester[Tester<br/>tests/*]
    Sup --> Reviewer[Reviewer<br/>read-only + review.md]
    Coder --> Sup
    Tester --> Sup
    Reviewer --> Sup
    Sup --> Result([Synthesized result])

Supervisor — единственная роль, которая принимает решение «готово / переделать». Workers возвращают результаты ему, не друг другу. Это убирает половину рисков «бесконечного спора», о которых ниже.

A. Supervisor + workers (иерархия)

        ┌────────────────┐
        │   Supervisor    │
        │  (orchestrator) │
        └────────┬────────┘
                 │ task dispatch
       ┌─────────┼─────────┐
       ↓         ↓         ↓
   ┌────────┐ ┌────────┐ ┌────────┐
   │ Coder  │ │ Tester │ │Reviewer│
   └────────┘ └────────┘ └────────┘

Supervisor:

  • Разбивает большую задачу на подзадачи
  • Раздаёт workers
  • Собирает результаты
  • Решает финальное «готово/переделать»

Workers — узко специализированы, минимум прав.

Когда подходит: структурированные задачи с очевидной декомпозицией (фича = код + тесты + ревью).

B. Peer-to-peer (плоская сеть)

   ┌────────┐ ←─────→ ┌────────┐
   │Agent A │         │Agent B │
   └────────┘ ←─────→ └────────┘
       ↕                 ↕
   ┌────────┐ ←─────→ ┌────────┐
   │Agent C │         │Agent D │
   └────────┘         └────────┘

Все агенты равны, общаются напрямую. Решения принимаются локально или голосованием.

Когда подходит: креативные/исследовательские задачи (brainstorm, debate). Редко нужно в разработке.

Опасность: легко уйти в бесконечный диалог без результата. Нужны жёсткие правила завершения («после 5 раундов — supervisor решает»).

C. Event-driven (через шину)

   Agent A ──→ ┌─────────────────┐ ──→ Agent C
              │   Event Bus     │
   Agent B ──→ │  (queue/pub-sub) │ ──→ Agent D
              └─────────────────┘

Агенты не знают друг о друге. Каждый слушает типы событий, реагирует, публикует свои.

Когда подходит: длинные процессы, частично-человеческие workflow (CI/CD-like).

Технологии: Redis pub/sub, RabbitMQ, GitHub webhooks, или просто файловая очередь (tasks/inbox/ → tasks/done/).

Коммуникация: формат сообщений

Главное правило — общий формат, без которого Agent B не поймёт Agent A.

Минимальный пакет

{
  "from": "coder-agent-01",
  "to": "tester-agent-01",
  "task_id": "feature-y-2026-05-15",
  "type": "request_test",
  "payload": {
    "files": ["src/api/users.ts"],
    "context": "Implemented GET /api/users/me. See PR #142 for diff."
  },
  "timestamp": "2026-05-15T14:32:00Z"
}

Этот формат — протокол. Все агенты его знают (через CLAUDE.md или системный prompt). Без формата — у вас не multi-agent, а хаос.

Уровни типизации

Слабый (free-form в JSON-блок): «делай как чувствуешь, главное JSON». Средний (схема + примеры): «вот JSON Schema, валидируй». Сильный (типизированный bus): protocol buffers, FlatBuffers и проч.

Для большинства случаев — средний уровень. JSON Schema + парсер на входе каждого агента.

Supervisor: как работает

Supervisor — отдельный prompt (или agent), с отличной от workers ролью. Его задачи:

  1. Декомпозиция: разбить задачу на под-таски
  2. Dispatch: какой worker какую под-таску
  3. Synthesis: собрать результаты, проверить consistency
  4. Decision: завершить, итерировать, эскалировать к человеку

supervisor.md:

You are a supervisor agent. You do NOT write code.
You orchestrate workers (coder, tester, reviewer).

For each incoming task:
1. Decompose into 1-5 worker tasks.
2. Dispatch via .agent/queue/<worker>/<task-id>.json
3. Poll .agent/queue/<worker>/done/ for results.
4. On all-done: synthesize result, return.
5. On worker error: retry once, then escalate.

Hard rules:
- Max 5 iterations per task.
- Total budget: 30 minutes wall-clock.
- On budget exhaustion: stop, return what we have, flag for human.

Это тонкая настройка. Supervisor — самая сложная роль в multi-agent, потому что ему надо принимать решения о других агентах. Часто это первое место, где что-то идёт не так.

Короткая вставка ⚡

В чате ребят встречается история — собрали супервайзер + 4 worker’а для авто-PR-фабрики. Запустили на одной задаче. Через 6 часов:

  • 47 коммитов в feature-ветке
  • supervisor три раза «отменил» свои же решения
  • coder и reviewer ушли в бесконечный спор о наименовании одной переменной
  • $80 на API за ночь

Что было сделано не так: не было жёстких лимитов (число итераций, бюджет $, время). Supervisor работал на «агент знает лучше». Не знает.

Multi-agent без бюджетов — это deadlock-машина. Всегда верхний предел: итераций, времени, токенов. Превышен — stop, эскалация к человеку.

Когда выбирать какую архитектуру

Supervisor + workers — самый частый

Подходит почти везде, где задача очевидно декомпозируется:

  • Разработка фичи: planner + coder + tester + doc-writer
  • Анализ кодовой базы: разделение по подсистемам, sub-агенты на каждую
  • Подготовка PR: коды-ревью-тесты-описание-merge

Лимиты: макс. 5 workers (больше — сложно отлаживать), макс. 3 уровня глубины.

Peer-to-peer — редко

Только когда специально нужна диалоговая динамика (brainstorm, debate, generation+critique loops).

Обязательно: жёсткое правило завершения (N раундов или supervisor-judge сверху).

Event-driven — для длинных процессов

Когда между шагами могут быть часы-дни. CI/CD, заявки клиентов, on-call rotation с агентом-первой-линии.

Не используйте, когда задача укладывается в один час wall-clock — overhead не окупится.

Пример: supervisor + 3 workers (минимальная реализация)

.agent/queue/ структура:

.agent/queue/
├── inbox/                    # supervisor читает отсюда
│   └── task-001.json
├── coder/
│   ├── todo/
│   ├── doing/
│   └── done/
├── tester/
│   ├── todo/
│   ├── doing/
│   └── done/
└── reviewer/
    ├── todo/
    ├── doing/
    └── done/

runner.sh:

#!/usr/bin/env bash
set -e

# Один цикл supervisor
while [ -f .agent/queue/inbox/*.json ]; do
    TASK=$(ls .agent/queue/inbox/*.json | head -1)
    TASK_ID=$(basename "$TASK" .json)
    
    # 1. Supervisor decomposes
    claude --print --system-prompt-file supervisor.md \
        "Decompose task in $TASK. Output subtasks to .agent/queue/{worker}/todo/$TASK_ID-{n}.json"
    
    mv "$TASK" .agent/queue/processing/
    
    # 2. Workers process in parallel
    parallel -j 3 ::: \
        "claude --print --system-prompt-file coder.md --json-mode" \
        "claude --print --system-prompt-file tester.md --json-mode" \
        "claude --print --system-prompt-file reviewer.md --json-mode"
    
    # 3. Supervisor synthesizes
    claude --print --system-prompt-file supervisor.md \
        "Tasks done. Synthesize result for $TASK_ID."
done

Это упрощение. В реальности нужны: timeout per worker, retry policy, error handling, deduplication, persistent state. Но скелет именно такой.

Антипаттерны multi-agent

1. «Слишком тонкое разбиение»

Supervisor + 8 узкоспециализированных workers. Каждый делает одну функцию. Overhead на координацию больше, чем выигрыш от специализации.

Лекарство: 2-4 worker’а. Больше — только если реально нужны разные модели или права.

2. Workers без жёстких прав

«Reviewer» имеет полный write-доступ к репо. Зачем? Он только читает и пишет коммент.

Лекарство: минимум прав на роль. Reviewer — read-only + write только в .agent/reviews/.

3. Свободная коммуникация без формата

Workers пишут «как чувствуют». Supervisor парсит наугад. Хаос.

Лекарство: JSON Schema на каждый тип сообщения. Парсер на входе. Невалидное — отбросить и переспросить.

4. Без лимитов

«Пусть супервайзер сам решит, когда хватит». Не решит. Гоняйте до бюджета, дальше stop.

Лекарство: жёсткие верхние границы по итерациям, времени, токенам. Любое превышение — эскалация к человеку, не autonomous decision.

5. Все агенты на одной модели

Можно. Но дорого и избыточно для простых ролей.

Лекарство: Supervisor / planner — Opus. Coder — Sonnet или Opus. Tester / formatter — Haiku или Sonnet (быстрее, дешевле). Reviewer — Opus (для качества).

6. Multi-agent для задачи, которая не нужна multi-agent

Самый частый. «Звучит круто, давайте сделаем». Овеководство, hype-driven design.

Лекарство: см. M16, целиком про это.

Practice (90-120 минут)

Задание 1 (60 минут) — собрать минимальный supervisor + 2 workers

Не на боевом проекте, а на песочнице. Скрипт + JSON-queue + два subagent. Прогон на одной простой задаче (например, «refactor function X, добавь тест»).

Запишите:

  • Сколько времени отвалидировать результат?
  • Что сломалось первым (если сломалось)?
  • Стоило ли оно того vs просто pipeline?

Задание 2 (30 минут) — формат сообщений

Опишите JSON Schema для одного типа сообщения в вашем pipeline. Загляните, какие поля обязательно нужны, какие — опциональны. Это упражнение критично, до того как пробовать настоящий multi-agent.

Опционально (60+ минут) — параллельный pipeline

Если в практической работе есть задача с параллельными ветвями (например, миграция двух независимых модулей), запустите 2 workers одновременно через parallel. Сравните wall-clock vs sequential.

Recap

  • Multi-agent окупается на параллелизме, разных моделях/контекстах, adversarial сценариях. В большинстве случаев — pipeline хватает.
  • Архитектуры: supervisor+workers (самая частая), peer-to-peer (редко), event-driven (для долгих процессов).
  • Жёсткий формат коммуникации + жёсткие лимиты — без них multi-agent превращается в deadlock-машину.

Вайб Самурай

🥷 Вайб Самурай спрашивает: «Это всё круто, но мне-то зачем? У меня pet-project, я один. Я к multi-agent за всю жизнь не подойду.»

Ответ: И правильно сделаете. Multi-agent — это архитектурная территория, окупающаяся на больших проектах с командами и продакшеном. Я в эту главу включил именно, чтобы вы понимали, что multi-agent существует и как устроен — на случай, если попадёте в проект, где это реально нужно (или кажется, что нужно — половина случаев). Но строить multi-agent для себя — overhead. Pipeline (M14) — да, всем рекомендую. Multi-agent — только когда боль от его отсутствия начнёт дороже боли от его наличия. Не идеально, но реально работает.

📚 Дальше: посмотреть и почитать

Multi-agent — тема, где много hype и мало хорошей литературы. Минимум, на котором стоит ориентироваться:

Академический референс

  • AutoGen — paper (Microsoft, arXiv:2308.08155) — arxiv.org/abs/2308.08155. Original paper про conversable agents и multi-agent conversation programming. До сих пор лучшая ментальная модель: «диалог между ролями» как первый класс системы.

Фреймворки multi-agent

  • CrewAI — docs — docs.crewai.com. Role-based crews + flows. Хороший способ почувствовать, как выглядит supervisor+workers без своего скелета.
  • LangGraph — Overview — docs.langchain.com/oss/python/langgraph/overview. Тот же DAG, что и в M14, но удобно показывает, как state делится между несколькими нодами-агентами.

Anthropic-родное

  • Anthropic — Subagents — docs.anthropic.com/en/docs/claude-code/sub-agents. Лимитированный multi-agent внутри одной сессии. Полезно, если хочется ролей без межсессионной шины.

Совет: прочитайте AutoGen-paper до того как ставить любой фреймворк. Половина дизайн-вопросов решается там, и без этого вы будете удивляться «странному» поведению CrewAI/LangGraph.
