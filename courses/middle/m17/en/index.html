M17 · Production: deploying agents to live environments

From local dev to an agent system in prod. Persistence, observability, secret management — what changes.

Hook

LOCAL DEV:                       PRODUCTION:

claude in terminal      →        container on a server
↓                                ↓
reads your files                 reads other people's tasks
you see every step               logs to a centralised bus
mistake → roll back              mistake → incident → on-call
$5/day on your key               $1000/day on the company key

Hey folks, welcome 👋

This chapter opens Part VI — Production. The final part of the Middle track. We already know how to design agents (Part I), what tools are available (Part II), how to manage memory (Part III), how to measure (Part IV), and when not to do multi-agent (Part V). Now — the final step: how all of this ships to production.

An hour, dense. After the chapter you’ll have an architectural map of what’s needed to deploy an agent into a live regime.

Concept: production isn’t “the same thing, but on a server”

The most common slip: “it works in my Claude Code — let’s move it to a server.” Not so.

┌─────────────────────────────────────────────────────┐
│  In local you have:                                 │
│  - You, the human (HITL by default)                 │
│  - A free terminal with confirmations               │
│  - Access to your own files                         │
│  - Low volume (1-10 tasks/day)                      │
│                                                     │
│  In prod you have none of that:                     │
│  - Many user requests                               │
│  - Can't confirm every action                       │
│  - Access to company data, not yours                │
│  - 1000+ tasks/day, possibly in parallel            │
└─────────────────────────────────────────────────────┘

So the architecture in prod is different. Not “same + server” — a different architecture with different rules. The deploy flow looks roughly like:

flowchart LR
    Local[Local dev<br/>claude in terminal] --> Staging[Staging<br/>headless / Docker]
    Staging --> Eval{Eval harness<br/>pass?}
    Eval -->|fail| Local
    Eval -->|ok| Canary[Canary<br/>1-5% of traffic]
    Canary --> Health{Metrics ok?}
    Health -->|fail| Rollback([Rollback<br/>kill switch])
    Health -->|ok| Prod[Prod<br/>100% of traffic]
    Rollback -.-> Staging

Each transition is its own gate with its own signals: on staging — the eval harness (M11), on canary — error rate / cost / latency, on prod — alerts and the kill switch.

Layers of a production agent

┌─────────────────────────────────────────────────────┐
│  6. Observability       (logs, metrics, alerts)     │
│  5. Cost & rate limiting (budgets, throttling)      │
│  4. Persistence         (state, history, audit)     │
│  3. Security            (secrets, sandbox, ACL)     │
│  2. Orchestration       (queue, retry, error)       │
│  1. Agent runtime       (LLM calls, tool calls)     │
└─────────────────────────────────────────────────────┘

Locally you usually have only layer 1 (and partly 2). In prod all six are mandatory.

Layer 1. Agent runtime in prod

Locally you run claude interactively. In prod you need:

Headless mode

Claude Code, Codex CLI, and most modern agents support a non-interactive mode:

claude --print --json-mode \
       --system-prompt-file system.md \
       --max-iterations 10 \
       --timeout 600 \
       < task.json > result.json

No interactivity — task in, result out.

Container-based

Deploy in Docker / Podman / Kubernetes. Advantages:

  • Isolation — each task in its own container, no cross-talk
  • Versioning — pin agent version together with its context
  • Resource limits — memory, CPU, time

Dockerfile:

FROM node:22-slim

# Install Claude Code
RUN npm install -g @anthropic-ai/claude-code

# Agent context
COPY CLAUDE.md /workspace/
COPY .claude/ /workspace/.claude/

# Entrypoint
WORKDIR /workspace
ENTRYPOINT ["claude", "--print", "--json-mode"]

In reality — more complex (dependencies, multi-stage build, security scan), but the skeleton is this.

Managed Agents

Alternative to running your own infra — Managed Agents from providers (Anthropic, …). Anthropic runs a per-session container, the agent works on mounted files, an SSE event stream returns to the client.

Plus: no infra of your own, firewall/sandbox from the provider, ready event stream. Minus: vendor lock-in, more expensive per unit, compute limits.

For B, Anthropic Managed Agents is a working option for teams that don’t want a DevOps team for agent infra.

Layer 2. Orchestration

In prod tasks come from a queue, not from a terminal.

User request → API gateway → Queue → Agent runner → Result store → Notification
                              ↑              ↓
                              └──── retry ───┘

Minimum stack:

  • Queue — Redis, RabbitMQ, AWS SQS, or just Postgres with advisory locks
  • Runner — spawns an agent container per task
  • Result store — where the outcome lands (S3, DB)
  • Notification — webhook / Slack / e-mail to the client

Orchestration rules:

  1. Idempotency — a task can arrive twice (retry); the result must be the same
  2. Timeout per task — hard upper bound (15–30 minutes)
  3. Retry policy — exponential backoff, max 3 attempts
  4. Dead letter queue — tasks that died after retries, separately for inspection

Layer 3. Security

A whole chapter (M18); here the map at “what exists” level:

  • Secrets — never in code/config, only through a secret manager (AWS Secrets Manager, HashiCorp Vault, K8s Secrets)
  • Sandbox — container with minimum rights; separate filesystem; whitelisted network
  • ACL — which tasks can be given to which agent; multi-tenant isolation
  • Audit log — every tool call, every file change — into a centralised bus

Layer 4. Persistence

Locally state lives in .agent/journals/, files (M10). In prod — a DB.

What to store

  • Sessions — each agent “session”: id, user, start/end, status
  • Messages — every user/assistant/tool message
  • Tool calls — what was called, with what args, what returned, duration
  • Artifacts — files created/changed by the agent (or S3 links)
  • Costs — tokens, $ per session

Why

  • Audit — who did what when, for compliance
  • Replay — reproduce an incident step by step
  • Analytics — which tasks are expensive, where the agent misses more often
  • HITL handoff — if the agent is stuck, a human sees context and continues

sessions table (simplified):

CREATE TABLE agent_sessions (
  id            UUID PRIMARY KEY,
  user_id       UUID NOT NULL,
  agent_version VARCHAR(50) NOT NULL,
  model_id      VARCHAR(50) NOT NULL,
  status        VARCHAR(20) NOT NULL,  -- pending/running/done/failed
  started_at    TIMESTAMP NOT NULL,
  finished_at   TIMESTAMP,
  input_tokens  INT,
  output_tokens INT,
  cost_usd      DECIMAL(10, 4),
  result_url    TEXT
);

CREATE INDEX idx_sessions_user_started
  ON agent_sessions (user_id, started_at DESC);

Layer 5. Cost & rate limiting

The most underrated layer. Without it your agent can quietly generate $5000/day on the API.

Budgets

At three levels:

  1. Per-task — task no more than N tokens / $X
  2. Per-user/tenant — user no more than N tasks/day / $X/month
  3. Per-system — global daily / monthly cap

Breach → stop, not “degrade gracefully.” Better refuse than bankrupt.

Throttling

LLM-provider requests are rate-limited (per minute / TPM). On excess — queue, not retry forever.

Cost guardrails

Hooks (see M4) that before an expensive tool call (large prompt, long run) show the estimate and require approval above a threshold.

For example:

# pre-tool hook
if estimated_cost(tool_call) > 0.50:  # 50 cents
    require_approval(...)

This saves you from runaway loops where the agent runs itself 100 times.

Layer 6. Observability

“In prod you don’t see what the agent does” — the main difference from local. So:

Logs

  • Structured logs (JSON) per step
  • Trace ID linking all steps of one session
  • Centralised aggregation (ELK, Loki, Datadog)

Metrics

  • Sessions/min (RPS)
  • Pass rate (if you have automated validation)
  • p50/p95/p99 latency
  • Cost per session
  • Error rate

Tracing

OpenTelemetry-style traces — every tool call as a span, the call hierarchy visible. Especially valuable for multi-agent (though hopefully you don’t have multi-agent — see M16).

Alerts

Triggers:

  • Error rate > 5 % → page on-call
  • Cost rate > 2× baseline → page on-call
  • P99 latency > 30 min → warning
  • No sessions in last 10 minutes (if flow was expected) → warning

Example: minimum prod stack

For a small project (one agent, up to 100 tasks/day):

1. K8s deployment:
   - Replicas 2 (HA)
   - Resource limits: 1 CPU, 2 GB RAM
   - Init container: pulls latest CLAUDE.md from git

2. PostgreSQL:
   - tables: sessions, messages, tool_calls, costs
   - daily backup → S3

3. Redis:
   - task queue
   - rate-limiting counters

4. Monitoring:
   - Loki for logs
   - Prometheus + Grafana for metrics
   - PagerDuty/OpsGenie for alerts

5. Secrets:
   - ANTHROPIC_API_KEY in K8s Secret
   - injected as env var

6. Cost guardrails:
   - per-task: 100K tokens / $1
   - per-tenant: 1000 tasks / day
   - global: $100 / day initially

That’s already complete. Beyond — scaling (multi-region, blue-green deploys, etc.).

Pre-prod checklist

Before deploying your first agent to live traffic:

  • Headless mode works on 95 %+ of tasks locally
  • All secrets — in a secret manager, not in the repo
  • Sandbox — container has no access outside the workdir
  • Budgets at three levels (task, user, system)
  • Idempotency keys on every task from the API
  • Timeout per task (max 30 minutes wall-clock)
  • Logs structured + centralised
  • Alerts on error rate, cost, latency
  • Runbook for typical incidents
  • Kill switch — ability to disable the agent in seconds (env var / feature flag)
  • Harness has run on the staging environment
  • On-call rotation defined

Without every box — no deploy. That’s the minimum.

Production anti-patterns

1. “Ship and watch later”

Without observability — blind. The first incident will be in prod, not staging.

Cure: observability before launch.

2. “One container, many tasks”

One pod handles 100 tasks in parallel. Context leaks, state crosses.

Cure: one container = one session. More compute, but isolated.

3. Hard-coded API key

The key in an env file in the repo. A week later — on GitHub.

Cure: secret manager, always, no exceptions (see M18).

4. “Retry until success”

Task failed → retry → failed → retry. Forever.

Cure: max 3 retries, exponential backoff, dead letter queue.

5. “No kill switch”

When something goes wrong — no way to quickly disable the agent, short of “delete the deployment.”

Cure: feature flag (LaunchDarkly, ConfigCat, or just an env var). Flip to false → the agent stops taking new tasks.

Practice (60–90 minutes)

Task 1 (30 minutes) — headless mode locally

Take one of your agent workflows. Move to claude --print --json-mode. Run 5 tasks. Measure:

  • Which tasks broke without interactivity?
  • What else is needed (input schema, exit codes, output schema)?

Task 2 (30 minutes) — minimal Dockerfile

Build a Dockerfile for a headless agent. Run locally in Docker. Verify a task end-to-end inside the container.

Task 3 (30 minutes) — observability skeleton

Sketch the minimum set of logs and metrics for your future prod agent. What’s critical to see in prod? What’s nice-to-have?

Recap

  • Production = different architecture. Not “same + server.”
  • Six layers: runtime → orchestration → security → persistence → cost → observability. All mandatory.
  • Pre-prod checklist. Without every box — no deploy.

Vibe Samurai

🥷 Vibe Samurai asks: “I have a pet project with 10 users. Do I really need all this?”

Answer: For 10 users — not all of it. Minimum: headless mode + secrets in env-var + an API budget ($5/day) + a basic log. That’s enough to not go broke and to see what’s happening. The full prod stack starts paying off when: you charge money for the service, or you have >100 users, or it’s a regulated domain (finance, medicine). Otherwise — minimum viable observability. Rule: match infra to what scale pays for. Not perfect, but actually works.

📚 Further reading

Deploying agents is the area where “general principles” go only so far; the fastest way up is to read the docs of the actual platform. Three anchors: managed Bedrock, sandbox infra for code, and observability for a prod agent.

Managed runtime for agents

  • AWS — Amazon Bedrock AgentCore — docs.aws.amazon.com/bedrock-agentcore/latest/devguide/what-is-bedrock-agentcore.html. Serverless host for agents: session isolation, VPC, PrivateLink. A good reference for “what does prod infra for agents from a big vendor look like.”
  • AWS — Deploy an Amazon Bedrock agent — docs.aws.amazon.com/bedrock/latest/userguide/agents-deploy.html. Concretely on alias/version/InvokeAgent flow — the “don’t quietly switch agent version in prod” piece.

Sandbox infra for agent code

  • E2B — Docs — e2b.dev/docs. Cloud sandboxes for executing agent-generated code. Start in ~80 ms, cut your infrastructure yak-shaving.

Observability in prod

  • Langfuse — Observability overview — langfuse.com/docs/observability/overview. A ready platform for traces / sessions / cost; up in an evening, sees the whole flow at once.

Tip: before the first prod deploy pick exactly one of these three. “All at once” is a guaranteed way to drown in infra for a month.
