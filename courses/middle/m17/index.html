Концепция: продакшн — это не «то же самое, но на сервере»

Самый частый промах: «у меня в Claude Code работает — давайте перенесём на сервер». Не так.

┌─────────────────────────────────────────────────────┐
│  В локалке у вас есть:                              │
│  - Вы как человек (HITL by default)                 │
│  - Свободный terminal с подтверждениями              │
│  - Доступ к своим файлам                            │
│  - Низкий volume (1-10 задач/день)                  │
│                                                     │
│  В проде нет ничего из этого:                       │
│  - Заявок от пользователей много                    │
│  - Подтверждать каждое действие нельзя              │
│  - Доступ к данным компании, не вашим               │
│  - 1000+ задач/день, может, параллельно             │
└─────────────────────────────────────────────────────┘

Поэтому архитектура в проде отличается. Не «то же + сервер», а другая архитектура с другими правилами.

Слои продакшн-агента

│  6. Observability       (логи, метрики, алёрты)     │
│  5. Cost & rate limiting (бюджеты, throttling)      │
│  4. Persistence         (state, history, audit)     │

│  2. Orchestration       (queue, retry, error)       │

└─────────────────────────────────────────────────────┘

Локально у вас обычно только слой 1 (и частично 2). В проде все шесть обязательны.

Слой 1. Agent runtime в проде

В локалке вы запускаете claude интерактивно. В проде нужно:

Headless mode

Claude Code, Codex CLI и большинство современных агентов поддерживают non-interactive режим:

claude --print --json-mode \
       --system-prompt-file system.md \
       --max-iterations 10 \
       --timeout 600 \
       < task.json > result.json

Никакой интерактивности — задача на вход, результат на выход.

Container-based

Деплоим в Docker / Podman / Kubernetes. Преимущества:

  • Изоляция — каждая задача в своём контейнере, не пересекается с другими
  • Versioning — фиксируем версию агента вместе с его контекстом
  • Resource limits — память, CPU, время

Dockerfile:

FROM node:22-slim

# Установка Claude Code
RUN npm install -g @anthropic-ai/claude-code

# Контекст агента
COPY CLAUDE.md /workspace/
COPY .claude/ /workspace/.claude/

# Запуск
WORKDIR /workspace
ENTRYPOINT ["claude", "--print", "--json-mode"]

В реальности — более сложно (зависимости, многоступенчатый билд, security scan), но скелет такой.

Managed Agents

Альтернатива своей инфре — Managed Agents от провайдеров (Anthropic, …). Anthropic держит per-session контейнер, агент работает с mounted файлами, SSE event stream возвращается клиенту.

Плюсы: не своя инфра, файрвол/sandbox от провайдера, готовый event stream. Минусы: vendor lock-in, дороже на единицу, ограничения по compute.

В B Anthropic Managed Agents — рабочий способ для команд, которые не хотят держать DevOps-команду под agent infra.

Слой 2. Orchestration

В проде задачи приходят из очереди, не из терминала.

User request → API gateway → Queue → Agent runner → Result store → Notification
                              ↑              ↓
                              └──── retry ───┘

Минимальный стек:

  • Queue — Redis, RabbitMQ, AWS SQS, или просто Postgres с advisory locks
  • Runner — поднимает контейнер с агентом на задачу
  • Result store — куда складывается итог (S3, БД)
  • Notification — webhook / Slack / e-mail клиенту

Правила orchestration:

  1. Idempotency — задача может прийти дважды (retry), результат должен быть одинаковым
  2. Timeout per task — жёсткий верхний предел (15-30 минут)
  3. Retry policy — exponential backoff, max 3 попытки
  4. Dead letter queue — задачи, упавшие после retry, отдельно для разбора

Слой 3. Security

Тема следующей главы (M18) целиком, здесь — карта на уровне «что есть»:

  • Secrets — никогда в коде/конфиге, только через secret manager (AWS Secrets Manager, HashiCorp Vault, K8s Secrets)
  • Sandbox — контейнер с минимальными правами; отдельная файловая система; whitelisted network
  • ACL — какие задачи можно дать какому агенту; multi-tenant изоляция
  • Audit log — каждый tool call, каждое изменение файла — в централизованную шину

Слой 4. Persistence

В локалке state — в .agent/journals/, файлы (M10). В проде — БД.

Что хранить

  • Sessions — каждая «сессия» агента: id, user, start/end time, status
  • Messages — все сообщения user/assistant/tool
  • Tool calls — что вызвал, с какими параметрами, что вернул, длительность
  • Artifacts — файлы, созданные/изменённые агентом (или ссылки в S3)
  • Costs — токены, $ на каждую сессию

Зачем

  • Audit — кто что когда, для compliance
  • Replay — воспроизвести инцидент шаг за шагом
  • Analytics — какие задачи дороже, где агент чаще ошибается
  • HITL handoff — если agent застрял, человек видит контекст и продолжает

sessions table (упрощённо):

CREATE TABLE agent_sessions (
  id            UUID PRIMARY KEY,
  user_id       UUID NOT NULL,
  agent_version VARCHAR(50) NOT NULL,
  model_id      VARCHAR(50) NOT NULL,
  status        VARCHAR(20) NOT NULL,  -- pending/running/done/failed
  started_at    TIMESTAMP NOT NULL,
  finished_at   TIMESTAMP,
  input_tokens  INT,
  output_tokens INT,
  cost_usd      DECIMAL(10, 4),
  result_url    TEXT
);

CREATE INDEX idx_sessions_user_started 
  ON agent_sessions (user_id, started_at DESC);

Слой 5. Cost & rate limiting

Самый недооценённый слой. Без него ваш агент тихо может сгенерировать $5000/день на API.

Бюджеты

На трёх уровнях:

  1. Per-task — задача не больше N токенов / $X
  2. Per-user/tenant — пользователь не больше N задач в день / $X в месяц
  3. Per-system — глобальный потолок на день / месяц

Превышение → stop, не «degrade gracefully». Лучше отказ, чем разорение.

Throttling

Запросы к LLM-провайдеру ограничены (rate limit per minute / TPM). При превышении — очередь, не retry forever.

Cost guardrails

Hooks (см. M4), которые до дорогого tool call (большой prompt, длинный run) показывают оценку и требуют подтверждения выше определённого порога.

Например:

# pre-tool hook
if estimated_cost(tool_call) > 0.50:  # 50 центов
    require_approval(...)

Это спасает от runaway loops, где агент сам себя прогоняет 100 раз.

Слой 6. Observability

«В проде вы не видите, что делает агент» — главная разница с локалкой. Поэтому:

Логи

  • Structured logs (JSON) на каждый шаг
  • Trace ID связывающий все шаги одной сессии
  • Централизованная агрегация (ELK, Loki, Datadog)

Метрики

  • Sessions/min (RPS)
  • Pass rate (если у вас есть автомат-валидация)
  • p50/p95/p99 latency
  • Cost per session
  • Error rate

Tracing

OpenTelemetry-style traces — каждый tool call как span, видна иерархия вызовов. Особенно ценно для multi-agent (хотя multi-agent у вас, надеюсь, нет — см. M16).

Алёрты

Триггеры:

  • Error rate > 5% → page on-call
  • Cost rate > 2× от baseline → page on-call
  • P99 latency > 30 минут → warning
  • No sessions in last 10 minutes (если ожидался поток) → warning

Пример: минимальный prod-стек

Для маленького проекта (один агент, до 100 задач/день):

1. K8s deployment:
   - Replica 2 (для HA)
   - Resource limits: 1 CPU, 2 GB RAM
   - Init container: pulls latest CLAUDE.md from git

2. PostgreSQL:
   - tables: sessions, messages, tool_calls, costs
   - daily backup → S3

3. Redis:
   - task queue
   - rate limiting counters

4. Monitoring:
   - Loki for logs
   - Prometheus + Grafana for metrics
   - PagerDuty/OpsGenie for alerts

5. Secrets:
   - ANTHROPIC_API_KEY in K8s Secret
   - injected as env var

6. Cost guardrails:
   - per-task: 100K tokens / $1
   - per-tenant: 1000 tasks / day
   - global: $100 / day initially

Это уже complete. Дальше — масштабирование (multi-region, blue-green deploys, etc.).

Pre-prod чек-лист

Перед deploy первого агента в боевой трафик:

  • Headless mode работает на 95%+ задач локально
  • Все секреты — в secret manager, не в репо
  • Sandbox — контейнер не имеет доступа за пределы рабочей директории
  • Бюджеты на трёх уровнях (task, user, system)
  • Idempotency keys на каждой задаче из API
  • Timeout на задаче (макс 30 минут wall-clock)
  • Logs structured + централизованы
  • Алёрты на error rate, cost, latency
  • Runbook на типичные инциденты
  • Kill switch — возможность выключить агента за секунду (env var / feature flag)
  • Harness прогнан на staging-окружении
  • On-call rotation определён

Без всех галок — не deploy. Это минимум.

Антипаттерны продакшн-агентов

1. «Запустим — потом будем смотреть»

Без observability — слепо. Первый инцидент будет в проде, не на staging.

Лекарство: observability до запуска.

2. «Один контейнер, много задач»

Один pod обрабатывает 100 задач параллельно. Контекст утекает, state пересекается.

Лекарство: один контейнер = одна сессия. Дороже по compute, но изолированно.

3. Hard-coded API key

Ключ в env-файле в репо. Через неделю — на GitHub.

Лекарство: secret manager, всегда, без исключений (см. M18).

4. «Retry until success»

Задача упала → retry → упала → retry. Бесконечно.

Лекарство: max 3 retry, exponential backoff, dead letter queue.

5. «Без kill switch»

Когда что-то идёт не так — нет способа быстро отключить агента, кроме «удалить deployment».

Лекарство: feature flag (LaunchDarkly, ConfigCat, или просто env var). Включил false → агент не берёт новые задачи.

Practice (60-90 минут)

Задание 1 (30 минут) — headless mode локально

Возьмите один из ваших агентских workflows. Переведите в claude --print --json-mode. Прогоните 5 задач. Замерьте:

  • Какие задачи сломались без интерактивности?
  • Что нужно дополнительно (input schema, exit codes, output schema)?

Задание 2 (30 минут) — minimal Dockerfile

Соберите Dockerfile для headless агента. Запустите в локальном Docker. Проверьте, что задача от-до проходит в контейнере.

Задание 3 (30 минут) — observability skeleton

Распишите минимальный набор логов и метрик для своего будущего prod-агента. Что критично видеть в проде? Что — nice-to-have?

Recap

  • Продакшн = другая архитектура. Не «то же + сервер».
  • Шесть слоёв: runtime → orchestration → security → persistence → cost → observability. Все обязательны.
  • Pre-prod чек-лист. Без всех галок — не deploy.

Скептик Самурай

🥷 Скептик Самурай спрашивает: «У меня pet-проект на 10 пользователей. Мне всё это точно надо?»

Ответ: На 10 пользователей — не всё. Минимум: headless mode + secrets в env-var + бюджет на API ($5/день) + базовый лог. Это хватает, чтобы не разориться и видеть, что происходит. Полный prod-стек начинает окупаться, когда: вы за деньги оказываете услугу, или у вас >100 пользователей, или ответственный домен (финансы, медицина). Иначе — minimum viable observability. Правило: ставите столько инфры, сколько окупается масштабом. Не идеально, но реально работает.
