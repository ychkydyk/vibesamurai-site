IDE-agent vs CLI-agent: разница не только в интерфейсе

Очевидное отличие: один в редакторе, другой в терминале. Но за этим стоит три неочевидных архитектурных различия.

1. Контекст

CLI-agent видит проект через тулы — read_file, glob, grep. Он каждый раз «выясняет», что в проекте.

IDE-agent видит проект через API редактора — открытые табы, текущая позиция курсора, выделенный фрагмент, симвoль под курсором, недавние правки. Это даёт ему контекст вашего внимания, который CLI получить не может.

То есть когда вы кликнули мышкой на функцию parseDate и сказали «оптимизируй её» — IDE-agent сразу знает, о чём речь. CLI-agent потребует «оптимизируй функцию parseDate в файле src/utils/date.ts».

2. Apply-flow

CLI-agent правит файлы напрямую через тул Edit / Write. После правки — это уже изменённый файл, вы видите его в git diff.

IDE-agent обычно сначала показывает превью правок прямо в редакторе (inline diff), и вы принимаете или отклоняете каждое изменение. Это даёт более мелкую гранулярность контроля.

3. Embedding в dev-flow

CLI-agent — это отдельный процесс. Вы переключаетесь между терминалом и редактором.

IDE-agent — это часть редактора. Не надо переключаться, не надо копировать пути файлов, не надо устанавливать «контекст руками».

Это существенно для UX, но не для архитектуры — на самом деле под капотом у IDE-agent тот же loop, те же тулы, та же модель.

Cursor Composer — что это

Cursor — это fork VS Code с встроенным AI-агентом. Существует уже несколько лет, активно развивается. В 2026 году ключевая фича — Composer, агентский режим (раньше был отдельным от обычного chat, сейчас интегрирован).

Что Composer умеет:

  • Читать и редактировать множество файлов в одной задаче (не один файл, как раньше)
  • Запускать bash-команды (тесты, линтеры, сборка) — с approval
  • Использовать MCP-серверы (как Claude Code и Codex)
  • Применять inline-diff превью с приёмом каждого изменения
  • Работать в режиме plan-first или agent-mode (автономно)

Под капотом — модель на выбор: Claude Sonnet 4.6 / Opus 4.7, GPT-5.3, Gemini 3.1 Pro, либо «cursor-small» (собственная быстрая модель Cursor для лёгких задач).

.cursorrules и .cursor/rules/

Аналог CLAUDE.md / AGENTS.md. Хранится в корне проекта:

project/
├── .cursorrules                  # старый формат, single file
└── .cursor/
    └── rules/                     # новый формат, директория с rule-файлами
        ├── conventions.md
        ├── do-not-touch.md
        └── commit-style.md

Содержимое идентично по смыслу CLAUDE.md. Cursor читает эти файлы при старте сессии.

Лайфхак: .cursor/rules/ можно структурировать так, чтобы файлы автоматически применялись только в нужном контексте. Например, auth-conventions.md с meta-тегом apply_when: "path matches src/auth/**". Это сильно экономит контекст.

Composer как multi-file editor

Главная сила Composer — много файлов в одной задаче. Пример типового запроса:

«Добавь rate limiting на endpoint /api/login. Учти что у нас уже есть Redis в src/lib/redis.ts. Добавь middleware, конфиг в .env.example, и unit test.»

Composer соберёт план, покажет, какие файлы будут изменены, выдаст inline-diff на каждый, и вы можете принять конкретные diff-ы, отклонив другие. То есть микро-гранулярность.

Это удобнее, чем в Claude Code, где вы видите изменения уже после применения. Но менее автоматизируемо: для long-running автономных задач Composer не так хорош.

Cline (бывший Claude Dev) — VS Code extension

Cline — extension для оригинального VS Code (не fork). Open-source, активно развивается. Работает с моделями Anthropic, OpenAI, Gemini, Ollama (локальные модели).

Сильные стороны:

  • Open-source — можете смотреть код, контрибьютить, форкать
  • Подключение локальных моделей через Ollama — для тех кому критичен on-prem
  • Multi-file edits с approval каждого — похоже на Composer
  • MCP support

Слабые стороны:

  • Менее «product-grade» UX по сравнению с Cursor
  • Меньше готовых интеграций
  • Документация местами догоняет development

Использую периодически, когда нужен open-source и/или локальные модели. Для повседневной работы — Cursor удобнее.

Aider — терминальный pair-programmer

Aider — формально CLI, но философски ближе к IDE-агентам, потому что продолжает редактировать тот файл, на котором вы сосредоточены. Работает через git: каждая правка — отдельный commit.

Сильные стороны:

  • Маленький, простой, понятный. Open-source, активная разработка.
  • Git-first. Каждая агентская правка — commit, легко откатить.
  • Работает с любой моделью. Anthropic, OpenAI, Gemini, locals.
  • Хорошо для focused pair-programming одного файла / небольшой фичи.

Слабые стороны:

  • Хуже для длинных автономных задач (нет полноценного agent loop)
  • Нет sandbox для исполнения
  • Не подходит для агрессивного refactoring через многие файлы

Использую когда хочется «парного программирования» — он сидит сбоку и помогает по конкретному файлу.

Windsurf — Codeium IDE-агент

Windsurf — IDE от Codeium (тот же, что делает Codeium-плагин для VSCode). Тоже fork VS Code, как Cursor.

Сильные стороны:

  • «Cascade» — мощный multi-step режим, иногда лучше Composer на длинных задачах
  • Сильная free-тарифка для одиночных разработчиков
  • Хорошая «обратная связь» — показывает, что именно агент сейчас делает на каждом шаге

Слабые стороны:

  • Меньше community, чем у Cursor
  • Меньше готовых правил/best-practices
  • UI местами тяжеловатый

Я попробовал — нормально, но в повседневный workflow не вошёл. На вкус и цвет.

Когда IDE-agent выгоднее CLI-agent

IDE-agent выигрывает на:

  1. Активная работа с одним проектом — вы сидите в IDE целыми днями, агент рядом, не надо переключаться.
  2. Точечные правки конкретного файла — выделил фрагмент, попросил, посмотрел inline diff, принял или отклонил.
  3. Onboarding новой кодовой базы — IDE показывает контекст естественнее.
  4. UI / verstka / небольшие исправления — высокая частота мелких задач.

CLI-agent выигрывает на:

  1. Длинные автономные циклы — full-auto с sandbox, без вашего внимания.
  2. Cross-repo задачи — когда работаем сразу с несколькими репозиториями.
  3. CI / automation — агент в pipeline, без интерактива.
  4. Hooks-driven workflows — детерминированные правила (M6).
  5. Subagents и параллельная декомпозиция — отдельные изолированные процессы (M6).

Когда использовать вместе

Лично у меня типичный setup на средний рабочий день:

  • Cursor открыт — основная среда работы, daily product work, мелкие правки.
  • Claude Code в терминале — для длинных задач, ресёрча, subagents-декомпозиции.
  • Codex CLI рядом — для одной-двух специфичных задач (массовые миграции, autonomous test-fixing).

Они не конкурируют, они дополняют. Один проект, один CLAUDE.md (через симлинк AGENTS.md и .cursor/rules/), три тула — это нормальный рабочий setup.

Practice

15-20 минут:

  1. Установите один из IDE-агентов, который ещё не пробовали. Если уже сидите в Cursor — попробуйте Windsurf, или наоборот.
  2. Сделайте маленькую задачу в новом тулу — что-то, что вы обычно делаете в своём основном. Сравните UX.
  3. Запишите для себя три точки, в которых новый тул выиграл или проиграл вашему обычному. Это и есть «насмотренность» в действии.
  4. Если у вас уже есть CLAUDE.md / AGENTS.md / .cursor/rules/ — попробуйте через симлинк свести их в один источник истины.

Закрытие Part II

Друзья. Закрываем Part II — Tools & Environment.

Если суммировать всё, что мы прошли в M4-M8 одной формулой:

Тулы — это набор сменных рабочих инструментов с разными сильными сторонами. Грамотный middle-разработчик 2026 года знает несколько, выбирает по задаче, не привязан эмоционально ни к одному.

Зоопарк моделей и тулзов будет расти. Каждый квартал будет «новый король». Не цепляйтесь за бренды — цепляйтесь за паттерны (plan→act→evaluate, structured plan, sandbox, hooks, MCP). Они переживают конкретные продукты.

Тренируйте насмотренность, а не пальцы. Ваша голова важнее чем ИИ — особенно в момент выбора тула под задачу.

Recap

  • IDE-agent vs CLI-agent — три отличия: контекст внимания, apply-flow с inline diff, embedding в редактор.
  • Cursor Composer — флагман IDE-агентов. .cursor/rules/ как аналог CLAUDE.md.
  • Cline, Aider, Windsurf — три альтернативы со своими сильными сторонами.
  • IDE для daily product work, CLI для автономных циклов и cross-repo задач.
  • Лучший setup — комбинация, не выбор одного. Симлинк CLAUDE.md/AGENTS.md/cursorrules — единый source of truth.
