M8 · Cursor Composer + IDE agents

Composer-1 patterns, .cursorrules, when IDE agent vs CLI agent, a comparison table of four IDE agents.

What this chapter is about

Hey folks, welcome 👋

We close Part II with the fourth camp of agent tools — IDE agents. Up to now we’ve covered CLI agents (Claude Code, Codex CLI) that live in the terminal and operate on the project “from the outside.” IDE agents live inside the editor, and that’s a completely different philosophy.

In this chapter:

  1. How an IDE agent fundamentally differs from a CLI agent
  2. Cursor Composer — the IDE-agent flagship, in detail
  3. Cline / Aider / Windsurf — the main alternatives, briefly
  4. When an IDE agent is preferable, when a CLI one
  5. A comparison table of four IDE agents

IDE agent vs CLI agent: the difference isn’t only the interface

The obvious split: one in the editor, one in the terminal. But behind it there are three non-obvious architectural differences.

1. Context

CLI agent sees the project through tools — read_file, glob, grep. It “discovers” the project each time.

IDE agent sees the project through the editor API — open tabs, current cursor position, selected text, symbol under the cursor, recent edits. That gives it the context of your attention which a CLI can’t get.

So when you click a function parseDate and say “optimise this” — the IDE agent immediately knows what you mean. A CLI agent will require “optimise the parseDate function in src/utils/date.ts.”

2. Apply flow

CLI agent edits files directly via the Edit / Write tool. After the edit — it’s an already-changed file, visible in git diff.

IDE agent usually first shows a preview of edits right in the editor (inline diff), and you accept or reject each change. This gives finer-grained control.

3. Embedding in the dev flow

CLI agent is a separate process. You switch between terminal and editor.

IDE agent is part of the editor. No switching, no copying file paths, no “setting context by hand.”

Big deal for UX, not so much architecturally — under the hood the IDE agent runs the same loop, the same tools, the same model.

Cursor Composer — what it is

Cursor is a VS Code fork with a built-in AI agent. Been around several years, actively developed. In 2026 the key feature is Composer, the agent mode (used to be separate from the regular chat, now integrated).

What Composer does:

  • Read and edit multiple files in one task (not one file at a time)
  • Run bash commands (tests, linters, build) — with approval
  • Use MCP servers (like Claude Code and Codex)
  • Apply inline diff previews with per-change acceptance
  • Run in plan-first or agent-mode (autonomous)

Under the hood — model of choice: Claude Sonnet 4.6 / Opus 4.7, GPT-5.3, Gemini 3.1 Pro, or “cursor-small” (Cursor’s own fast model for light tasks).

.cursorrules and .cursor/rules/

The analog of CLAUDE.md / AGENTS.md. Lives in the project root:

project/
├── .cursorrules                  # legacy format, single file
└── .cursor/
    └── rules/                     # new format, directory of rule files
        ├── conventions.md
        ├── do-not-touch.md
        └── commit-style.md

Content is identical in meaning to CLAUDE.md. Cursor reads these on session start.

Pro tip: .cursor/rules/ can be structured so files auto-apply only in the relevant context. E.g. auth-conventions.md with meta apply_when: "path matches src/auth/**". Saves a lot of context.

Composer as a multi-file editor

The main strength of Composer — many files in one task. Typical request:

“Add rate limiting to the /api/login endpoint. We already have Redis in src/lib/redis.ts. Add middleware, config in .env.example, and a unit test.”

Composer assembles a plan, shows which files will be changed, gives an inline diff for each, and you can accept specific diffs while rejecting others. Micro-granularity.

That’s more convenient than Claude Code, where you see changes after they’re applied. But less automatable: for long-running autonomous tasks Composer isn’t as strong.

Cline (formerly Claude Dev) — VS Code extension

Cline is an extension for original VS Code (not a fork). Open-source, active development. Works with Anthropic, OpenAI, Gemini, Ollama (local models).

Strengths:

  • Open-source — read the code, contribute, fork
  • Connecting local models via Ollama — for those who need on-prem
  • Multi-file edits with per-change approval — Composer-like
  • MCP support

Weaknesses:

  • Less “product-grade” UX vs Cursor
  • Fewer ready integrations
  • Docs sometimes catch up to development

I use it occasionally when open-source and/or local models matter. For daily work — Cursor is more convenient.

Aider — terminal pair-programmer

Aider is formally a CLI, but philosophically closer to IDE agents because it keeps editing the file you’re focused on. Works via git: every change is a separate commit.

Strengths:

  • Small, simple, clear. Open-source, active development.
  • Git-first. Every agent change is a commit, easy to roll back.
  • Works with any model. Anthropic, OpenAI, Gemini, local.
  • Good for focused pair-programming of one file / a small feature.

Weaknesses:

  • Worse for long autonomous tasks (no full agent loop)
  • No sandbox for execution
  • Not suited for aggressive refactoring across many files

I use it when I want “pair programming” — sits beside me and helps on a specific file.

Windsurf — Codeium’s IDE agent

Windsurf is the IDE from Codeium (same shop as the Codeium VSCode plugin). Also a VS Code fork, like Cursor.

Strengths:

  • “Cascade” — a powerful multi-step mode, sometimes better than Composer on long tasks
  • Strong free tier for solo developers
  • Good “feedback loop” — shows exactly what the agent is doing at each step

Weaknesses:

  • Smaller community than Cursor
  • Fewer ready rules / best-practices
  • UI a bit heavy in places

I tried it — fine, but it didn’t make it into my daily workflow. To each their own.

When an IDE agent beats a CLI agent

IDE agent wins on:

  1. Active work in one project — you sit in the IDE all day, the agent is right there, no switching.
  2. Point edits to a specific file — selected a fragment, asked, looked at the inline diff, accepted or rejected.
  3. Onboarding a new codebase — the IDE shows context more naturally.
  4. UI / markup / small fixes — high frequency of small tasks.

CLI agent wins on:

  1. Long autonomous cycles — full-auto with sandbox, without your attention.
  2. Cross-repo tasks — when working across several repositories at once.
  3. CI / automation — agent in a pipeline, no interactive.
  4. Hooks-driven workflows — deterministic rules (M6).
  5. Subagents and parallel decomposition — separate isolated processes (M6).

When to use them together

Personally my typical setup on an average working day:

  • Cursor open — main environment, daily product work, small fixes.
  • Claude Code in the terminal — for long tasks, research, subagent decomposition.
  • Codex CLI nearby — for one or two specific tasks (mass migrations, autonomous test-fixing).

They don’t compete, they complement. One project, one CLAUDE.md (via symlink to AGENTS.md and .cursor/rules/), three tools — that’s a normal working setup.

Practice

15–20 minutes:

  1. Install an IDE agent you haven’t tried. If you’re already on Cursor — try Windsurf, or vice versa.
  2. Do a small task in the new tool — something you usually do in your main one. Compare the UX.
  3. Write down three points where the new tool won or lost vs your usual. That’s “developing your eye” in action.
  4. If you already have CLAUDE.md / AGENTS.md / .cursor/rules/ — try collapsing them to one source of truth via symlink.

Closing Part II

Friends. We close Part II — Tools & Environment.

If you sum up M4–M8 in one formula:

Tools are a set of interchangeable working instruments with different strengths. A sharp middle developer in 2026 knows several, picks by task, isn’t emotionally attached to any.

The model and tool zoo will keep growing. There will be a “new king” every quarter. Don’t anchor on brands — anchor on patterns (plan→act→evaluate, structured plan, sandbox, hooks, MCP). Those outlive specific products.

Train your eye, not your fingers. Your head matters more than the AI — especially at the moment of picking the tool for the task.

Recap

  • IDE agent vs CLI agent — three differences: attention context, apply flow with inline diff, embedding in the editor.
  • Cursor Composer — IDE-agent flagship. .cursor/rules/ as the CLAUDE.md analog.
  • Cline, Aider, Windsurf — three alternatives with their own strengths.
  • IDE for daily product work, CLI for autonomous cycles and cross-repo tasks.
  • Best setup is a combination, not a single pick. Symlink CLAUDE.md/AGENTS.md/cursorrules — one source of truth.
