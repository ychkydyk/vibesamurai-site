M19 · Incidents & post-mortems

Когда агент сломался в проде. Incident response, blameless post-mortems, системные уроки из инцидентов.

Черновик M19 в голосе DmitriyM.

Друзья, всем привет! 👋

Эта глава — про то, что делать, когда agent в проде уже сломался. Это обязательный навык для тех, кто запускает agents в продакшен. Inцидентов не избежать — можно только сделать дешевле.

Час. Глава достаточно практичная, без избытка терминологии.

Концепция: incident — это информация, не позор

В классической инженерии есть зрелое отношение к инцидентам:

┌─────────────────────────────────────────────────────┐
│  Незрелая команда:                                  │
│  - Инцидент = чья-то ошибка                         │
│  - Постмортем = поиск виноватого                    │
│  - Action items = увольнение/выговор                │
│  - Результат: люди скрывают инциденты               │
│                                                     │
│  Зрелая команда:                                    │
│  - Инцидент = сигнал о системе                      │
│  - Постмортем = blameless analysis                  │
│  - Action items = починка СИСТЕМЫ, не человека      │
│  - Результат: каждый инцидент = улучшение           │
└─────────────────────────────────────────────────────┘

С агентами — усугубляется. Поскольку агент не детерминирован, инциденты часто не воспроизводятся на 100%. Это не повод искать «случайность» — это повод смотреть системно.

Anatomy of an agent incident

Типовой incident lifecycle для agent system:

T0      Trigger (что-то пошло не так)
T+1m    Detection (алёрт / жалоба)
T+2m    Acknowledge (on-call принимает)
T+5m    Mitigation (kill switch / failover)
T+15m   Hotfix (если возможен)
T+30m   Resolution (сервис восстановлен)
T+1h    Stabilization (наблюдение)
T+24h   Post-mortem draft
T+72h   Post-mortem review
T+1w    Action items в работе

Каждая стадия имеет свои правила.

Стадия 1. Detection

Хорошее обнаружение — это когда алёрт сработал до жалобы пользователя.

Метрики, на которые обязательно должны быть алёрты:

  • Error rate > 5% за 5 минут → P1
  • Cost rate > 2× от baseline за 10 минут → P1
  • Latency p95 > 30 минут → P2
  • Saturation (очередь растёт) → P2
  • Specific tool failures (например, deploy_tool fails 3 times) → P1

Без алёртов — узнаёте о проблеме из жалоб, через час-два. Дорого.

Стадия 2. Acknowledge

On-call видит алёрт. 5 минут на acknowledge — иначе escalation. Это дисциплина.

В чате отдельный канал инцидента:

#incident-2026-05-16-agent-stall

[03:43] @oncall acknowledged
[03:44] @oncall investigating — cost 5x normal, no error spike
[03:45] @oncall pulling logs from last 30 min
...

Все действия — в канал, в timeline. Это потом — материал для post-mortem.

Стадия 3. Mitigation

Первое правило: остановите кровотечение до того, как искать корень.

Mitigations, по убыванию мягкости:

  1. Throttle — снизить rate (50% задач только)
  2. Pause new — feature flag, новые задачи не берутся, текущие доделываются
  3. Kill switch — все агенты остановлены, очередь стоит
  4. Rollback — откатить deploy к последнему известному good

Выбор зависит от природы инцидента:

  • Costs runaway → kill switch (немедленно)
  • Latency spike → throttle (мягко)
  • Регрессия после deploy → rollback

Не пытайтесь сразу «починить root cause». Сначала stop the bleeding, потом разбирайтесь.

Стадия 4. Diagnosis

После mitigation — диагностика. Тут пригодится всё, что вы построили в M17 (logs, traces, audit log).

Шаги:

  1. Timeline — что произошло, когда (из логов)
  2. What changed? — deploy, config, model version, traffic pattern
  3. Hypothesis — что могло вызвать
  4. Verification — данными подтвердить/опровергнуть гипотезу

Специфика agent-инцидентов

В отличие от обычных incident’ов:

  • Non-determinism — одна и та же задача может сломаться не каждый раз
  • Model drift — провайдер мог тихо обновить snapshot
  • Prompt injection — атака могла прийти через user input
  • Cost runaway — обычно causally связан с loops, не с одной дорогой задачей

Поэтому больше копать в:

  • Сессионных traces
  • Логах tool calls
  • Сравнении с baseline harness’а (M12)

Стадия 5. Hotfix

Если видимо, что чинит — hotfix. Маленькое изменение, только то, что нужно остановить дальнейшие повторения.

Не делайте hotfix больше нужного. Не рефакторите по ходу. Не добавляйте «давно хотел». Hotfix = минимально достаточный fix.

Стадия 6. Resolution & stabilization

Мониторинг возвращается к baseline. Очередь обрабатывается. Алёрты — clear.

Stabilization period — 1-2 часа активного наблюдения. Может повториться, или другая симптоматика вылезти.

Стадия 7. Post-mortem

Через 24-48 часов — пишем post-mortem. Не на горячую голову, не через неделю (когда уже все забыли).

Структура

# Post-mortem: Agent stall 2026-05-16

## Summary
One-paragraph: что случилось, кого затронуло, как долго.

## Impact
- Affected users: 230
- Failed sessions: 47
- Excess cost: $342
- Duration: 03:42 → 04:10 = 28 min

## Timeline (UTC)
- 03:42 — алёрт cost-rate > 2× baseline
- 03:43 — @oncall acknowledged
- 03:45 — выявлен loop в supervisor agent
- ... (полный timeline из канала инцидента)

## Root cause
Supervisor agent попал в loop из-за того, что worker возвращал invalid JSON,
а supervisor retry-policy не имела лимита. Worker'у в обновлённом prompt'е
была добавлена инструкция, ломающая существующую JSON schema.

## Contributing factors
- Промпт worker'а изменён за 12 часов до инцидента без regression-eval
- Retry-policy supervisor'а не имела жёсткого лимита (только exponential backoff)
- Cost alert pороj был слишком высокий (3× — обновили на 2×)

## What went well
- Алёрт сработал за 1 минуту до cost runaway
- Kill switch активирован за 3 минуты
- Mitigation эффективна, costs стабилизировались

## What went wrong
- Не было pre-deploy harness прогона для prompt-изменения
- Retry-policy не имела hard cap
- Post-mortem-канал создан только через 5 минут

## Action items
| # | Action | Owner | Due |
|---|---|---|---|
| 1 | Добавить hard cap (max 5 retries) в supervisor | @alice | 2026-05-20 |
| 2 | Mandatory harness run на любое изменение prompts | @bob | 2026-05-25 |
| 3 | Lower cost alert threshold to 1.5× | @carol | 2026-05-17 |
| 4 | Schema validation на worker output | @alice | 2026-05-22 |

## Lessons learned (system level)
- Agent prompts — это **код**. Должны проходить через тот же gate (eval), что и обычные изменения.
- Cost alerts — самая ранняя сигнал runaway. Держим низкий порог.
- Retry policies — всегда hard cap. Exponential backoff — не замена.

Правила хорошего post-mortem

1. Blameless. Не «Алиса сломала», а «процесс позволил Алисе сломать». Цель — система, не человек.

2. Конкретный. Не «улучшить тестирование», а «add regression suite for prompt changes by 2026-05-25».

3. С owner и сроком. Action item без owner = не сделается.

4. Доступный. Все в команде читают. Подняли уровень коллективного понимания.

5. С follow-up. Через месяц — review, что из action items сделано. Без follow-up post-mortems становятся театром.

Короткая вставка ⚡

Самая частая ошибка в post-mortem с агентами — списать на «нестабильность LLM». «Ну вот модель такая, иногда так делает, что поделаешь».

Это отговорка. Да, модель non-deterministic. Но система вокруг — детерминированная. Бюджеты, retry policies, validation, kill switch — это ваше, не «модель такая».

Если post-mortem заканчивается «LLM are stochastic, oh well» — этот post-mortem не выполнил свою работу. Должны быть системные действия.

Категории инцидентов и типовые ответы

1. Cost runaway

Симптом: cost rate спайк.

Причины:

  • Loop (агент сам себя дёргает)
  • Retry без cap
  • Один пользователь шлёт огромные prompts

Ответ:

  • Per-tenant rate limit
  • Hard cap на retries
  • Cost alert на 1.5× baseline

2. Pass rate drop

Симптом: harness показал падение pass rate после deploy.

Причины:

  • Изменение prompt
  • Обновление модели от провайдера
  • Изменение зависимости

Ответ:

  • Pre-deploy harness gate
  • Зафиксировать model ID
  • Lock dependency versions

3. Stuck session

Симптом: сессии не завершаются.

Причины:

  • Tool не возвращает (внешний API hang)
  • Agent ждёт human input
  • Deadlock между подзадачами

Ответ:

  • Timeout per tool call
  • Timeout per session
  • Heartbeat монитор

4. Data exfiltration

Симптом: подозрительные egress, или жалоба о утечке.

Причины:

  • Prompt injection
  • Compromised MCP
  • Bypassed sandbox

Ответ:

  • Немедленно kill switch
  • Rotate ALL secrets
  • Audit log review
  • Forensics, изоляция inicident

5. Cross-tenant leak

Симптом: tenant A видит данные tenant B.

Причины:

  • Shared cache без isolation
  • Wrong tenant_id в session
  • Race condition в session management

Ответ:

  • Немедленно stop affected service
  • Identify affected tenants
  • Notify клиентов (юридическое требование во многих юрисдикциях)
  • Forensics

Тренировка: chaos / gamedays

Чтобы команда не училась на боевых инцидентах — проводите gamedays. Раз в квартал:

  1. Кто-то из команды симулирует инцидент (kill процесс, испорти конфиг, simulate spike)
  2. On-call проходит через response как на боевой
  3. Замеряете: time-to-detection, time-to-mitigation
  4. Post-mortem — на учебном инциденте

Это дёшево и окупается на первом реальном инциденте.

Practice (60-90 минут)

Задание 1 (30 минут) — runbook

Напишите runbook для одного типового инцидента в вашем agent system. По шаблону:

# Runbook: Cost runaway

## How to detect
- Alert "cost-rate > 2x baseline"
- Manual: check Grafana dashboard "Agent Costs"

## Immediate actions
1. Acknowledge alert
2. Check kill switch toggle (#incident-channel)
3. If cost still climbing: activate kill switch

## Diagnosis
1. Open Grafana → Costs by Session
2. Find top-cost sessions
3. Open session logs in Loki
4. Look for loops in tool calls

## Resolution
- If loop: restart specific worker, file ticket
- If single large session: kill session, investigate

## Communication
- Update #incident channel every 5 min
- Notify affected tenants if > 30 min

Задание 2 (30 минут) — гипотетический post-mortem

Возьмите реальный инцидент из вашего опыта (хоть локального — «у меня агент проиграл $5 на API за час»). Напишите post-mortem по шаблону выше. Найдите 2-3 системных action item.

Задание 3 (15-30 минут) — gameday

Договоритесь с коллегой (или сами с собой через timer): один симулирует «алёрт» в случайный момент в течение дня, второй проходит через response. Замерьте время.

Recap

  • Incidents — это информация о системе, не позор человека. Blameless всегда.
  • Stage: detect → ack → mitigate → diagnose → hotfix → resolve → post-mortem. Каждая — отдельные правила.
  • Post-mortem с action items, owner, сроком. Без follow-up — театр.

Вайб Самурай

🥷 Вайб Самурай спрашивает: «У меня маленький сервис, мне-то зачем эти процедуры?»

Ответ: Объём процедур масштабируется. Маленький сервис: минимум — runbook на 2-3 типовых инцидента, kill switch, alert на cost runaway. Post-mortem даже в маленькой команде — окупается первым же инцидентом, который повторился бы, не запиши вы lesson. Большой сервис — формальный incident management, dedicated on-call, регулярные gamedays. Не на каждом уровне нужно всё, но что-то на каждом — обязательно. Минимум, который никогда не пропускают: kill switch + cost alert + runbook на runaway. Не идеально, но реально работает.
