M19 · Incidents & post-mortems

When the agent broke in prod. Incident response, blameless post-mortems, systemic lessons from incidents.

Hook

DURING THE INCIDENT:              AFTER:

03:42 PagerDuty       →           post-mortem
"agents stalled                   - what happened (timeline)
 cost 5x normal"                  - why (RCA, 5 whys)
03:43 acknowledge                 - what to fix (action items)
03:45 kill switch ON              - blameless
03:46 mitigation                  - systemic changes
03:50 hotfix
04:10 monitoring restored

Hey folks, welcome 👋

This chapter is about what to do when the agent in prod already broke. A mandatory skill for anyone shipping agents to production. Incidents are unavoidable — they can only be made cheaper.

An hour. Quite practical, no excess terminology.

Concept: an incident is information, not shame

In classic engineering there’s a mature attitude toward incidents:

┌─────────────────────────────────────────────────────┐
│  Immature team:                                     │
│  - Incident = someone's mistake                     │
│  - Post-mortem = blame hunt                         │
│  - Action items = firing / reprimand                │
│  - Result: people hide incidents                    │
│                                                     │
│  Mature team:                                       │
│  - Incident = a signal about the system             │
│  - Post-mortem = blameless analysis                 │
│  - Action items = fix the SYSTEM, not the human     │
│  - Result: every incident = improvement             │
└─────────────────────────────────────────────────────┘

With agents — it gets worse. Since the agent is non-deterministic, incidents often don’t reproduce 100 %. That’s not a reason to chalk it up to “randomness” — it’s a reason to look systemically.

Anatomy of an agent incident

A typical incident lifecycle for an agent system:

T0      Trigger (something went wrong)
T+1m    Detection (alert / complaint)
T+2m    Acknowledge (on-call takes it)
T+5m    Mitigation (kill switch / failover)
T+15m   Hotfix (if possible)
T+30m   Resolution (service restored)
T+1h    Stabilization (observation)
T+24h   Post-mortem draft
T+72h   Post-mortem review
T+1w    Action items in motion

Each stage has its own rules.

Stage 1. Detection

Good detection — when the alert fired before a user complaint.

Metrics that must have alerts:

  • Error rate > 5 % over 5 minutes → P1
  • Cost rate > 2× baseline over 10 minutes → P1
  • Latency p95 > 30 minutes → P2
  • Saturation (queue growing) → P2
  • Specific tool failures (e.g. deploy_tool fails 3 times) → P1

Without alerts — you hear about issues through complaints, an hour or two later. Expensive.

Stage 2. Acknowledge

On-call sees the alert. 5 minutes to acknowledge — else escalation. Discipline.

In chat — a dedicated incident channel:

#incident-2026-05-16-agent-stall

[03:43] @oncall acknowledged
[03:44] @oncall investigating — cost 5x normal, no error spike
[03:45] @oncall pulling logs from last 30 min
...

All actions in the channel, in timeline. Later — material for the post-mortem.

Stage 3. Mitigation

First rule: stop the bleeding before chasing the root.

Mitigations, from soft to hard:

  1. Throttle — lower the rate (50 % of tasks only)
  2. Pause new — feature flag, new tasks not accepted, current ones finish
  3. Kill switch — all agents stopped, queue paused
  4. Rollback — revert deploy to last known good

Choice depends on the nature of the incident:

  • Costs runaway → kill switch (immediately)
  • Latency spike → throttle (soft)
  • Regression after deploy → rollback

Don’t try to immediately “fix the root cause.” First stop the bleeding, then dig.

Stage 4. Diagnosis

After mitigation — diagnosis. Now you use everything you built in M17 (logs, traces, audit log).

Steps:

  1. Timeline — what happened, when (from logs)
  2. What changed? — deploy, config, model version, traffic pattern
  3. Hypothesis — what could have caused it
  4. Verification — confirm/refute with data

Agent-incident specifics

Unlike usual incidents:

  • Non-determinism — the same task may fail not every time
  • Model drift — the provider may have silently rolled a new snapshot
  • Prompt injection — the attack may have come via user input
  • Cost runaway — usually caused by loops, not by one expensive task

So dig more into:

  • Session traces
  • Tool-call logs
  • Comparison against the baseline harness (M12)

Stage 5. Hotfix

If visibly something fixes it — hotfix. Small change, only what’s needed to stop further recurrences.

Don’t make the hotfix bigger than needed. Don’t refactor on the side. Don’t add “long wanted.” Hotfix = minimally sufficient fix.

Stage 6. Resolution & stabilisation

Monitoring returns to baseline. Queue drains. Alerts clear.

Stabilisation period — 1–2 hours of active watch. It may recur, or another symptom may surface.

Stage 7. Post-mortem

Within 24–48 hours — write the post-mortem. Not when you’re hot, not a week later (when everyone’s forgotten).

Structure

# Post-mortem: Agent stall 2026-05-16

## Summary
One paragraph: what happened, who was affected, how long.

## Impact
- Affected users: 230
- Failed sessions: 47
- Excess cost: $342
- Duration: 03:42 → 04:10 = 28 min

## Timeline (UTC)
- 03:42 — alert cost-rate > 2× baseline
- 03:43 — @oncall acknowledged
- 03:45 — identified loop in supervisor agent
- ... (full timeline from the incident channel)

## Root cause
Supervisor agent fell into a loop because the worker returned invalid JSON,
and the supervisor's retry policy had no limit. The worker's updated prompt
added an instruction that broke the existing JSON schema.

## Contributing factors
- Worker prompt was changed 12 hours before the incident without a regression eval
- Supervisor retry policy had no hard cap (only exponential backoff)
- Cost alert threshold was too high (3× — updated to 2×)

## What went well
- Alert fired one minute before cost runaway
- Kill switch activated in 3 minutes
- Mitigation effective, costs stabilised

## What went wrong
- No pre-deploy harness run for the prompt change
- Retry policy without a hard cap
- Post-mortem channel created only 5 minutes in

## Action items
| # | Action | Owner | Due |
|---|---|---|---|
| 1 | Add hard cap (max 5 retries) in supervisor | @alice | 2026-05-20 |
| 2 | Mandatory harness run on any prompt change | @bob | 2026-05-25 |
| 3 | Lower cost alert threshold to 1.5× | @carol | 2026-05-17 |
| 4 | Schema validation on worker output | @alice | 2026-05-22 |

## Lessons learned (system level)
- Agent prompts are **code**. They must go through the same gate (eval) as ordinary changes.
- Cost alerts are the earliest runaway signal. Keep the threshold low.
- Retry policies always need a hard cap. Exponential backoff isn't a substitute.

Rules for a good post-mortem

1. Blameless. Not “Alice broke it,” but “the process let Alice break it.” Target — the system, not the person.

2. Concrete. Not “improve testing” — “add a regression suite for prompt changes by 2026-05-25.”

3. With owner and due date. Action item with no owner = won’t be done.

4. Accessible. Everyone on the team reads. Raises the collective understanding.

5. With follow-up. A month later — review which action items got done. Without follow-up post-mortems become theatre.

A short aside ⚡

The most common post-mortem mistake with agents — blaming “LLM instability.” “Well, the model’s like that, sometimes it does that, what can you do.”

That’s an excuse. Yes, the model is non-deterministic. But the system around it is deterministic. Budgets, retry policies, validation, kill switch — these are yours, not “the model’s fault.”

If a post-mortem ends with “LLMs are stochastic, oh well” — that post-mortem didn’t do its job. There must be systemic actions.

Incident categories and typical responses

1. Cost runaway

Symptom: cost-rate spike.

Causes:

  • Loop (agent re-triggers itself)
  • Retry with no cap
  • One user sending huge prompts

Response:

  • Per-tenant rate limit
  • Hard cap on retries
  • Cost alert at 1.5× baseline

2. Pass rate drop

Symptom: the harness shows a pass-rate drop after deploy.

Causes:

  • Prompt change
  • Provider model update
  • Dependency change

Response:

  • Pre-deploy harness gate
  • Pin model ID
  • Lock dependency versions

3. Stuck session

Symptom: sessions don’t finish.

Causes:

  • A tool doesn’t return (external API hang)
  • Agent waiting for human input
  • Deadlock between subtasks

Response:

  • Timeout per tool call
  • Timeout per session
  • Heartbeat monitor

4. Data exfiltration

Symptom: suspicious egress, or a leak complaint.

Causes:

  • Prompt injection
  • Compromised MCP
  • Bypassed sandbox

Response:

  • Immediately kill switch
  • Rotate ALL secrets
  • Audit log review
  • Forensics, incident isolation

5. Cross-tenant leak

Symptom: tenant A sees tenant B’s data.

Causes:

  • Shared cache without isolation
  • Wrong tenant_id in the session
  • Race condition in session management

Response:

  • Immediately stop affected service
  • Identify affected tenants
  • Notify clients (legal requirement in many jurisdictions)
  • Forensics

Training: chaos / gamedays

So the team doesn’t learn on live incidents — run gamedays. Quarterly:

  1. Someone on the team simulates an incident (kill a process, corrupt a config, simulate a spike)
  2. On-call walks through the response as on a live one
  3. Measure: time-to-detection, time-to-mitigation
  4. Post-mortem — on the training incident

It’s cheap and pays back on the first real incident.

Practice (60–90 minutes)

Task 1 (30 minutes) — runbook

Write a runbook for one typical incident in your agent system. Template:

# Runbook: Cost runaway

## How to detect
- Alert "cost-rate > 2× baseline"
- Manual: check Grafana dashboard "Agent Costs"

## Immediate actions
1. Acknowledge alert
2. Check kill switch toggle (#incident-channel)
3. If cost still climbing: activate kill switch

## Diagnosis
1. Open Grafana → Costs by Session
2. Find top-cost sessions
3. Open session logs in Loki
4. Look for loops in tool calls

## Resolution
- If loop: restart specific worker, file ticket
- If single large session: kill session, investigate

## Communication
- Update #incident channel every 5 min
- Notify affected tenants if > 30 min

Task 2 (30 minutes) — a hypothetical post-mortem

Take a real incident from your practice (even a local one — “the agent burned $5 on the API in an hour”). Write a post-mortem to the template above. Find 2–3 systemic action items.

Task 3 (15–30 minutes) — gameday

Agree with a colleague (or with yourself via a timer): one simulates an “alert” at a random moment during the day, the other walks through the response. Measure the time.

Recap

  • Incidents are information about the system, not shame on a person. Blameless always.
  • Stages: detect → ack → mitigate → diagnose → hotfix → resolve → post-mortem. Each has its own rules.
  • Post-mortem with action items, owner, due date. Without follow-up — theatre.

Vibe Samurai

🥷 Vibe Samurai asks: “I have a small service, why do I need these procedures?”

Answer: The volume of procedure scales. Small service: minimum — runbook for 2–3 typical incidents, kill switch, alert on cost runaway. Post-mortems even in a small team — pay back at the first incident that would have recurred had you not written down the lesson. Big service — formal incident management, dedicated on-call, regular gamedays. Not everything at every level, but something at every level is mandatory. The minimum you never skip: kill switch + cost alert + runbook on runaway. Not perfect, but actually works.
