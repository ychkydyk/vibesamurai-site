M14 · One agent, many steps: pipelines and orchestration

Decomposing a big task into a sequence of calls. When subagents, when an external orchestrator.

Hook

ONE-SHOT:                         PIPELINE:

big task                          plan → execute → review
   ↓                                ↓        ↓        ↓
[Claude does everything           specialised specialised specialised
 in one giant call]                subagent   subagent   subagent
                                    ↓        ↓        ↓
   ↓                              clean outputs, narrow scope,
result: ok-ish,                   easy to debug each stage
hard to debug

Hey folks, welcome 👋

This chapter opens Part V — Multi-Agent. Before we build systems of several independent agents (that’s M15), you need to master a simpler tool: a pipeline inside one agent.

Most of what people call “multi-agent” in reality is a pipeline. Several roles, one orchestrator, a clear flow. That’s much simpler and cheaper to build than real multi-agent. And in 80 % of cases — it’s enough.

An hour. After this chapter you can split a 30-minute task into 4 × 7-minute steps and debug each independently.

Concept: a pipeline = “one developer in several hats”

Picture this task: “add a feature with tests and docs.”

ONE-SHOT (one big prompt):
"Build feature X, add tests, update the docs"

→ the agent tries everything at once
→ context is stuffed, focus diluted
→ often one of the three lands worse
PIPELINE (3 steps, same agent):
Step 1. Planner       : "Give me an implementation plan, no code"
Step 2. Implementer   : "Write code for this plan"
Step 3. Test engineer : "Write tests for this code"
Step 4. Doc writer    : "Update docs/README for this feature"

→ each step is focused
→ output of one = input of the next
→ you can verify between steps
→ you can roll back only the problem step

It’s the same agent, same model. The difference is in the workflow structure. A bit more “architectural,” the pipeline is a DAG with parallel branches after a shared plan:

flowchart LR
    Plan[Planner<br/>spec / plan.md] --> Impl[Implementer<br/>src/]
    Impl --> Tests[Test engineer<br/>tests/]
    Impl --> Lint[Lint / typecheck]
    Impl --> Docs[Doc writer<br/>docs/]
    Tests --> Review[Reviewer<br/>PR description]
    Lint --> Review
    Docs --> Review
    Review --> Ship([Ship / merge])

Some steps are independent (tests / lint / docs) and can run in parallel — that’s the “orchestration” pipelines exist for.

When you need a pipeline

Signals it’s time to split into a pipeline:

SymptomSignal
Task > 20–30 minutes of agent timeyes
Several different kinds of work (plan + code + tests + docs)yes
One kind is consistently worse than the othersyes
Context stuffed midwayyes
You want to verify between stagesyes

If none apply — stay on one-shot. Don’t multiply entities.

Three ways to do a pipeline

Way 1. Sequential prompts (manual)

The simplest. You kick off steps in order:

Session 1: "Give me an implementation plan for feature Y, no code"
            → read, approve, save to .agent/plans/feature-y.md

Session 2: "Implement code per .agent/plans/feature-y.md"
            → review the diff, commit

Session 3: "Write tests for this code"
            → tests/, run

Session 4: "Update README + docs/api.md for this feature"

Plus: full control, can stop anywhere. Minus: manual switching, extra clicks.

Way 2. Subagents (Claude Code feature)

In Claude Code you can define specialised subagents via .claude/agents/<name>.md. Each subagent is a separate Claude with its own prompt and own permissions.

.claude/agents/planner.md:

---
name: planner
description: Plans implementation steps. Does NOT write code.
tools: [Read, Glob, Grep]
---

You are a planning agent. Given a feature request, output:
1. A numbered list of implementation steps
2. Files to be modified
3. Risks and dependencies

Do NOT write code. Do NOT call Edit or Write tools.
Save your plan to .agent/plans/<feature-slug>.md before returning.

.claude/agents/test-engineer.md:

---
name: test-engineer
description: Writes tests for existing code. Does NOT modify implementation.
tools: [Read, Edit, Write, Bash]
---

You are a test engineer. Read the implementation file and:
1. Identify test cases (happy path + edge cases + error paths)
2. Write tests in the project's existing test framework
3. Run them and report pass/fail

Do NOT modify any file outside of tests/.

The main agent invokes the subagent:

“Use the planner subagent to design the plan for feature X. Then implement against the plan yourself. Then hand off to test-engineer.”

Plus: subagents have narrow rights (security), focused on the role. Minus: setup; documentation needs maintenance.

Way 3. External orchestrator (script)

pipeline.sh:

#!/usr/bin/env bash
set -e

FEATURE="$1"
SLUG="$(echo "$FEATURE" | tr ' ' '-' | tr '[:upper:]' '[:lower:]')"

# Step 1. Plan
claude --print "Plan implementation for feature: $FEATURE. \
Output to .agent/plans/$SLUG.md, then exit." \
  > .agent/runs/$SLUG-plan.log

# Step 2. Implementation
claude --print "Implement feature according to .agent/plans/$SLUG.md. \
Do not modify tests/." \
  > .agent/runs/$SLUG-impl.log

# Step 3. Tests
claude --print "Write tests for the implementation. \
Read .agent/plans/$SLUG.md for context." \
  > .agent/runs/$SLUG-tests.log

# Step 4. Docs
claude --print "Update README and docs/ based on the new feature. \
Read .agent/plans/$SLUG.md for context." \
  > .agent/runs/$SLUG-docs.log

# Step 5. Final check
npm test && echo "Pipeline OK"

Plus: automation, repeatability, CI-friendly. Minus: brittle (one step falls — everything collapses), doesn’t capture interactive nuance.

Passing state between steps

The key question: how does step 2 know what step 1 did?

Way A. File artifact (most reliable)

Step 1 writes plan.md. Step 2 reads plan.md. Step 3 reads plan.md + sees what step 2 committed.

These are the durable artifacts from M10. The file is the common language between steps.

Way B. Through-prompt (via context)

PLAN=$(cat .agent/plans/$SLUG.md)
claude --print "Plan: $PLAN. Now implement."

Works, but spends input context every time. For long plans — the artifact is better.

Way C. Through git

Step 1 commits, step 2 reads git log and knows what was done.

This works when the result of a step is code, not a plan.

Example: pipeline for “add an endpoint with tests”

.agent/plans/feature-pipeline.md (after step 1)

# Feature: Add /api/users/me endpoint

## Plan
1. Add route handler in src/api/users.ts:
   - GET /api/users/me
   - Reads session from req.cookies.sessionId
   - Returns user object (without password_hash)
2. Add types in src/types/user.ts
3. Add unit test in tests/api/users.test.ts
4. Update docs/api.md

## Files
- Modify: src/api/users.ts, src/types/user.ts, docs/api.md
- Create: tests/api/users.test.ts (if not exists)

## Risks
- Need to handle missing session (return 401)
- Don't leak password_hash field in the serializer

Step 2 (implementation)

The agent reads the plan and writes code to it. It doesn’t “invent its own plan” — it follows plan.md. Result: commit “implement /api/users/me.”

Step 3 (tests)

The agent reads plan + new code. Writes tests, runs them, iterates to green.

Step 4 (docs)

The agent reads plan + code + tests. Updates docs/api.md. Final commit.

Each step — 5–10 minutes. Each commit — a rollback point. Each problem area — isolated.

Pipeline anti-patterns

1. Too many steps

“Plan → file tree → types → handler → tests → docs → changelog → release notes.” Half the steps add no value.

Cure: merge until each step has obvious value.

2. Steps without artifacts

Step 1 “thought it through,” wrote nothing. Step 2 doesn’t know what was decided.

Cure: every step writes its result to a file. No file — step not done.

3. Misaligned permissions

A “planner” subagent has full rights. Why? It doesn’t write code.

Cure: minimum rights per role. Planner — read-only. Test-engineer — write only in tests/. Implementer — write in src/, not in tests/.

4. Silent failure midway

Step 3 failed, the script kept going. Step 4 got a broken input.

Cure: set -e in the script, check npm test or similar between steps. Hard fail-fast.

5. Pipeline where one-shot is enough

“Let me build a beautiful pipeline for adding one line to the README.” Overkill.

Cure: pipeline — for big tasks (see signals above). A simple commit doesn’t need infrastructure.

Subagent vs external orchestrator: which to pick

CriterionSubagentsExternal script
Interactive workgoodbad
CI/CDbadgood
Setup complexitymediummedium
Permission isolationstrongmedium
Team visibilityvia .claude/agents/ in repovia CI logs
Flexibilitymediummax

Rule: for everyday work — subagents. For CI / data refresh / mass tasks — an external script.

Practice (60–90 minutes)

Task 1 (30 minutes) — sequential prompts manually

Take a task you usually run one-shot and split into 3–4 prompts. Run them one by one, between steps — git status, git diff, commit.

Compare: is it easier to understand what’s happening at each stage?

Task 2 (30 minutes) — set up 2 subagents

In the project:

  • .claude/agents/planner.md (read-only, outputs a plan)
  • .claude/agents/test-engineer.md (only tests/)

In one session: “Use planner to design the plan for X. Then implement. Then hand to test-engineer.”

Optional (60 minutes) — pipeline in CI

Write evals/pipeline-test.sh for one of your tasks. Wire it into CI. That’s the full automated pipeline + regression on top.

Recap

  • Pipeline = decomposition of a big task into roles. One agent, different hats, a shared artifact.
  • Three ways: manually, via subagents, via a script. Pick by scenario.
  • File artifact = common language between steps. Without it — blind steps.

Vibe Samurai

🥷 Vibe Samurai asks: “That’s overhead! I can do it all in one prompt. Why do I need 4 subagents?”

Answer: For simple tasks — yes, you don’t. A pipeline starts to pay off when the task is >30 minutes of agent time, or when errors at one stage keep you from diagnosing what happened. Without a pipeline — “it kind of all happened, but one piece is weird”; with a pipeline — “aha, step 3 had test failures, steps 1–2 are fine.” It’s not for show, it’s for diagnosis. And — don’t multiply entities preemptively: move to a pipeline when one-shot actually stops coping, not “just in case.” Not perfect, but actually works.

📚 Further reading

Pipelines is the topic where “there are more words than code.” So keep three sources real software is built on: LangGraph (DAG execution), DSPy (declarative pipelines), and Anthropic’s pattern unpacking.

Pipeline and agent patterns

  • Anthropic — “Building effective agents” — anthropic.com/research/building-effective-agents. Prompt chaining, routing, parallelization, orchestrator-workers — exactly the patterns we worked today. The shortest, most honest read on the topic.

Orchestration frameworks

  • LangGraph — Overview — docs.langchain.com/oss/python/langgraph/overview. DAG orchestrator from LangChain with durable execution and HITL out of the box. If you build a CI/CD pipeline around an agent — the default candidate.
  • DSPy (Stanford) — dspy.ai. Declarative signatures and program compilation from modules; a fundamentally different angle on the same “many steps, each focused” problem.

Claude Code in place

  • Anthropic — Subagents — docs.anthropic.com/en/docs/claude-code/sub-agents. A way to assemble a pipeline inside Claude Code, without an external orchestrator and docker-compose.

Tip: don’t jump straight to LangGraph/DSPy if your task fits inside .claude/agents/. A pipeline through subagents is the cheapest start; an external framework adds more complexity than win.
