M14 · Один агент, много шагов: пайплайны и оркестрация

Декомпозиция большой задачи в последовательность вызовов. Когда subagents, когда внешний оркестратор.

Черновик M14 в голосе DmitriyM. Открывает Часть V — Multi-Agent.

Hook

ONE-SHOT:                         PIPELINE:

big task                          plan → execute → review
   ↓                                ↓        ↓        ↓
[Claude does everything           specialised specialised specialised
 in one giant call]                subagent   subagent   subagent

Друзья, всем привет! 👋

Эта глава открывает Часть V — Multi-Agent. Прежде чем строить системы из нескольких независимых агентов (это M15), нужно освоить более простой инструмент: пайплайн внутри одного агента.

Большая часть «multi-agent» в реальности — это пайплайн. Несколько ролей, один оркестратор, понятный поток. Это сильно проще и дешевле в разработке, чем настоящий multi-agent. И в 80% случаев — этого достаточно.

Час. После главы вы сможете разбить задачу из 30 минут на 4 шага по 7 минут и разобраться с каждым шагом отдельно.

Концепция: пайплайн = «один разработчик в нескольких шляпах»

Представьте, что у вас задача: «добавить фичу с тестами и доками».

ONE-SHOT (one big prompt):
"Сделай фичу X, добавь тесты и обнови docs"

→ агент пытается всё одновременно
→ контекст забит, фокус размыт
→ часто что-то одно делает хуже
PIPELINE (3 шага, тот же агент):
Шаг 1. Планировщик    : "Дай план реализации, без кода"
Шаг 2. Имплементер    : "По этому плану напиши код"
Шаг 3. Тест-инженер   : "Напиши тесты на этот код"
Шаг 4. Документатор   : "Обнови docs/README по этой фиче"

→ каждый шаг focused
→ выход одного = вход другого
→ можно проверять между шагами
→ можно откатить только проблемный шаг

Это тот же агент, та же модель. Разница — в структуре workflow.

Когда пайплайн нужен

Сигналы, что пора разбивать на пайплайн:

СимптомСигнал
Задача > 20-30 минут agent-timeда
Несколько разных видов работы (план + код + тесты + доки)да
Один из видов стабильно хуже остальныхда
Контекст забивается на серединеда
Хочется проверять между этапамида

Если ни один не срабатывает — оставайтесь на one-shot. Не плодите сущности.

Три способа делать пайплайн

Способ 1. Sequential prompts (вручную)

Самый простой. Вы сами запускаете шаги по очереди:

Сессия 1: "Дай план реализации фичи Y, без кода"
            → читаете, утверждаете, сохраняете в .agent/plans/feature-y.md

Сессия 2: "По плану .agent/plans/feature-y.md напиши код"
            → проверяете diff, коммитите

Сессия 3: "К этому коду напиши тесты"
            → tests/, прогон

Сессия 4: "Обнови README + docs/api.md под эту фичу"

Плюсы: полный контроль, любая остановка возможна. Минусы: ручное переключение, лишний клик.

Способ 2. Subagents (Claude Code feature)

В Claude Code можно завести специализированные subagents через .claude/agents/<name>.md. Каждый subagent — это отдельный Claude с своим prompt и своими правами.

.claude/agents/planner.md:

---
name: planner
description: Plans implementation steps. Does NOT write code.
tools: [Read, Glob, Grep]
---

You are a planning agent. Given a feature request, output:
1. A numbered list of implementation steps
2. Files to be modified
3. Risks and dependencies

Do NOT write code. Do NOT call Edit or Write tools.
Save your plan to .agent/plans/<feature-slug>.md before returning.

.claude/agents/test-engineer.md:

---
name: test-engineer
description: Writes tests for existing code. Does NOT modify implementation.
tools: [Read, Edit, Write, Bash]
---

You are a test engineer. Read the implementation file and:
1. Identify test cases (happy path + edge cases + error paths)
2. Write tests in the project's existing test framework
3. Run them and report pass/fail

Do NOT modify any file outside of tests/.

Главный агент вызывает subagent:

«Используй planner subagent для разработки плана фичи X. Затем сам реализуй по плану. Потом передай test-engineer subagent.»

Плюсы: субагенты имеют узкие права (security), сфокусированы на роли. Минусы: настройка, документация требует поддержки.

Способ 3. Внешний оркестратор (скрипт)

pipeline.sh:

#!/usr/bin/env bash
set -e

FEATURE="$1"
SLUG="$(echo "$FEATURE" | tr ' ' '-' | tr '[:upper:]' '[:lower:]')"

# Шаг 1. План
claude --print "Plan implementation for feature: $FEATURE. \
Output to .agent/plans/$SLUG.md, then exit." \
  > .agent/runs/$SLUG-plan.log

# Шаг 2. Имплементация
claude --print "Implement feature according to .agent/plans/$SLUG.md. \
Do not modify tests/." \
  > .agent/runs/$SLUG-impl.log

# Шаг 3. Тесты
claude --print "Write tests for the implementation. \
Read .agent/plans/$SLUG.md for context." \
  > .agent/runs/$SLUG-tests.log

# Шаг 4. Доки
claude --print "Update README and docs/ based on the new feature. \
Read .agent/plans/$SLUG.md for context." \
  > .agent/runs/$SLUG-docs.log

# Шаг 5. Финальная проверка
npm test && echo "Pipeline OK"

Плюсы: автоматизация, повторяемость, CI-совместимо. Минусы: хрупкий (упал один шаг → всё разваливается), не подхватывает interactive нюансы.

Передача состояния между шагами

Ключевой момент: как второй шаг знает, что сделал первый?

Способ A. Файл-артефакт (самый надёжный)

Шаг 1 пишет plan.md. Шаг 2 читает plan.md. Шаг 3 читает plan.md + смотрит что закоммитил шаг 2.

Это и есть durable artifacts из M10. Файл — общий язык между шагами.

Способ B. Through prompt (по контексту)

PLAN=$(cat .agent/plans/$SLUG.md)
claude --print "Plan: $PLAN. Now implement."

Работает, но тратит входной контекст каждый раз. Для длинных планов — артефакт лучше.

Способ C. Через git

Шаг 1 коммитит, шаг 2 видит git log и понимает, что было сделано.

Это работает, когда результат шага — код, не план.

Пример: pipeline для «добавь endpoint с тестами»

.agent/plans/feature-pipeline.md (после шага 1)

# Feature: Add /api/users/me endpoint

## Plan
1. Add route handler in src/api/users.ts:
   - GET /api/users/me
   - Reads session from req.cookies.sessionId
   - Returns user object (without password_hash)
2. Add types in src/types/user.ts
3. Add unit test in tests/api/users.test.ts
4. Update docs/api.md

## Files
- Modify: src/api/users.ts, src/types/user.ts, docs/api.md
- Create: tests/api/users.test.ts (if not exists)

## Risks
- Need to handle missing session (return 401)
- Don't leak password_hash field in serializer

Шаг 2 (имплементация)

Агент читает plan, пишет код по нему. Не «придумывает свой план» — следует plan.md. Результат: коммит «implement /api/users/me».

Шаг 3 (тесты)

Агент читает plan + новый код. Пишет тесты, запускает, прогоняет до зелёного.

Шаг 4 (доки)

Агент читает plan + код + тесты. Обновляет docs/api.md. Финальный коммит.

Каждый шаг — 5-10 минут. Каждый коммит — точка отката. Каждое проблемное место — изолировано.

Антипаттерны pipeline

1. Слишком много шагов

«Plan → file-tree → types → handler → tests → docs → changelog → release-notes». Половина шагов не несут ценности.

Лекарство: объединяйте, пока каждый шаг не имеет очевидной ценности.

2. Шаги без артефактов

Шаг 1 «продумал», ничего не записал. Шаг 2 не знает, что было решено.

Лекарство: каждый шаг записывает результат в файл. Без файла — шаг не выполнен.

3. Несинхронизированные права

Subagent «planner» имеет полные права. Зачем? Он не пишет код.

Лекарство: минимум прав на каждую роль. Planner — read-only. Test-engineer — write только в tests/. Implementer — write в src/, не в tests/.

4. Глухая ошибка где-то на середине

Шаг 3 упал, скрипт продолжил. Шаг 4 получил сломанный вход.

Лекарство: set -e в скрипте, проверка npm test или подобного между шагами. Жёсткий fail fast.

5. Pipeline там, где one-shot хватает

«Сделаю-ка я красивый pipeline для добавления одной строки в README». Оверкилл.

Лекарство: pipeline — для больших задач (см. сигналы выше). Простой коммит не нуждается в инфраструктуре.

Subagent vs внешний оркестратор: что выбирать

КритерийSubagentsВнешний скрипт
Interactive работахорошоплохо
CI/CDплохохорошо
Сложность настройкисредняясредняя
Изоляция правсильнаясредняя
Видимость в командечерез .claude/agents/ в репочерез CI-логи
Гибкостьсредняямаксимум

Правило: для ежедневной работы — subagents. Для CI/обновления данных/массовых задач — внешний скрипт.

Practice (60-90 минут)

Задание 1 (30 минут) — sequential prompts вручную

Возьмите задачу, которую обычно делаете one-shot, и разбейте на 3-4 prompt’а. Делайте по одному, между шагами — git status, git diff, коммит.

Сравните: легче ли понимать, что происходит, на каждом этапе?

Задание 2 (30 минут) — настроить 2 subagents

Создайте в проекте:

  • .claude/agents/planner.md (read-only, выдаёт план)
  • .claude/agents/test-engineer.md (только tests/)

В одной сессии: «Используй planner для разработки плана X. Затем сам реализуй. Затем test-engineer».

Опционально (60 минут) — pipeline в CI

Напишите evals/pipeline-test.sh для одной из ваших задач. Поставьте в CI. Это полный автоматизированный pipeline + регресс на нём.

Recap

  • Пайплайн = декомпозиция большой задачи в роли. Один агент, разные шляпы, общий артефакт.
  • Три способа: руками, через subagents, через скрипт. Выбирайте по сценарию.
  • Файл-артефакт = общий язык между шагами. Без него — слепые шаги.

Скептик Самурай

🥷 Скептик Самурай спрашивает: «Это же overhead! Я могу одним промптом всё сделать. Зачем мне 4 субагента?»

Ответ: Для простых задач — да, не нужно. Pipeline начинает окупаться, когда задача >30 минут agent-time, или когда ошибки на одном этапе мешают вам диагностировать, что произошло. Без pipeline — «вроде всё сделалось, но один кусок странный»; с pipeline — «ага, на шаге 3 у нас тесты упали, шаги 1-2 ок». Это не для красоты, это для диагностики. И — не плодите сущности заранее: переходите на pipeline, когда один-шот реально перестал справляться, не «на всякий случай». Не идеально, но реально работает.
